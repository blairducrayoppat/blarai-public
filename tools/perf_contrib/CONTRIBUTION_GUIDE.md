## What is measured

Each record in the dataset covers one scenario (e.g., `chat`, `router`, `vlm`).
Fields present in every record:

- `name` — scenario identifier
- `timestamp` — ISO-8601 when the run was captured
- `model` + `precision` — unambiguous model identity
- `methodology` — enough detail to reproduce (prompt set, run count, config,
  warm/cold conditions, speculative decoding on/off)
- `environment.cpu` / `environment.gpu` — hardware identity
- `environment.openvino_version` / `environment.openvino_genai_version` —
  exact framework version strings (from `ov.__version__` at runtime)
- `environment.not_measured` — explicit list of what this record does NOT cover
  (GPU driver version, co-resident memory cost, subjective quality)
- `measurements` — the numbers (latency, throughput, load time, etc.)

## What is NOT measured (explicit)

Every record's `environment.not_measured` names these:

- GPU driver version — read from dxdiag / Device Manager; not introspectable
  from Python at runtime.
- Co-resident memory cost when other models are loaded alongside.
- Subjective quality (voice naturalness, visual answer quality).

## Dataset files

After running `python -m tools.perf_contrib aggregate`, the pipeline writes:

| File | Purpose |
|------|---------|
| `blarai_perf_dataset.jsonl` | Machine-readable; one JSON record per line; use for programmatic import |
| `blarai_perf_dataset.csv` | Flat CSV; one row per record; environment + measurements flattened with `env_` / `meas_` prefixes |

Both files are privacy-scrubbed: local Windows paths are replaced with
`<local-path>` and free-text reply previews are removed.

## How to submit

### Option A: HuggingFace dataset

1. Check whether a community OpenVINO benchmarking dataset already exists
   (search `openvino benchmark` on hf.co/datasets).
2. If contributing to an existing dataset, follow its contribution guide.
3. If creating a new dataset:
   - `huggingface-cli login`
   - `huggingface-cli repo create blarai-openvino-perf --type dataset`
   - Upload `blarai_perf_dataset.jsonl` and `blarai_perf_dataset.csv`.
   - Write a dataset card referencing this hardware + the OpenVINO version.

### Option B: OpenVINO issue / PR

For benchmarking issues in the OpenVINO GitHub repo (e.g., model-specific
performance issues, speculative decoding behaviour on Arc):

1. Locate the relevant open issue (check your existing contribution portfolio,
   e.g., issue #34651 and related).
2. Post the relevant record(s) as a comment — paste the JSON inline or attach
   the JSONL file.
3. Include the hardware table above and the OpenVINO version string.

### Engagement-first norm

Per the project's upstream-contribution norm: coordinate with maintainers via
the issue thread BEFORE submitting a PR or dataset. Do not scatter attention
across same-subsystem PRs while one stalls.

## Regenerating the dataset

```powershell
# From the repo root:
python -m tools.perf_contrib validate          # check all harness_*.json
python -m tools.perf_contrib aggregate         # produce JSONL + CSV in docs/performance/

# Or point at a specific directory:
python -m tools.perf_contrib aggregate --perf-dir docs/performance --out-dir /tmp/contrib
```

The pipeline rejects any record missing `not_measured`, a real `methodology`,
or a `measurements` dict — partial records are never silently included.

## Privacy

- No record contains session content, user queries, or file paths.
- `reply_preview` fields are stripped by the scrubber before output.
- Windows absolute paths (`C:\Users\...`) are replaced with `<local-path>`.
- Submission is a manual act. This pipeline prepares the data only.
