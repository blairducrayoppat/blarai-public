# BlarAI — Lessons Learned

*The distilled list, moved out of `BUILD_JOURNAL.md` on 2026-07-03 so it can be
curated as well as appended. Each numbered lesson was paid for by a real,
documented incident; the journal entries in `BUILD_JOURNAL.md` are the evidence.
This file is the compounding portfolio surface.*

## Rules of this file (the curation discipline)

1. **Never renumber or delete a ratified lesson.** Lesson numbers are cited
   throughout `BUILD_JOURNAL.md` and the ADRs; they are permanent identifiers.
   Consolidation happens by cross-reference and by the canonical tier below,
   never by renumbering.
2. **Search before you mint.** Before adding a new lesson, search this file for
   the class. If the incident is a recurrence of an existing lesson, do NOT
   mint a new number — append a dated recurrence tally to the existing lesson:
   `*(recurred: YYYY-MM-DD — <entry title>)*`.
3. **The third-instance rule.** A lesson that reaches its third instance
   (original + two recurrences) is no longer a lesson — it is a missing
   control. The change that records the third tally MUST also ship a
   structural enforcement (a gate test, a CI check, a hook, a required
   checklist line) and record it on the lesson:
   `*(control: <what was built, file/test name>)*`. A lesson that keeps
   recurring with no control is a documented failure of this discipline.
4. **Judgment here; mechanics in `FIELD_NOTES.md`.** This list is for
   transferable judgment — things an engineer could apply at a different
   company. Environment-specific gotchas (a Windows API trap, a regex-engine
   dialect, a driver quirk) go to `FIELD_NOTES.md` instead. When an incident
   yields both, the lesson carries the judgment and links the field note.
5. **Quarterly consolidation pass.** Once a quarter (next due: 2026-10-01),
   one session re-reads additions since the last pass, refreshes the canonical
   tier below, adds missing recurrence tallies, and verifies every
   third-instance lesson has its control. This is scheduled work, not
   willpower.
6. **Numbering at fold-in only.** Journal fragments propose lessons described,
   never pre-numbered (see `docs/journal_fragments/README.md`); the integrator
   assigns the next number here serially.

## The Canonical Tier — twenty classes that own the list

*The page to read first. Each canonical class names the constituent numbered
lessons that earned it. Curated 2026-07-03 from lessons 1–195; refreshed at
each quarterly pass.*

**C1. Process is not progress.** Activity — documents, sessions, audits of
audits — is not product. Build the smallest thing that genuinely runs, then
grow. Of ~259 autonomous fleet sessions, 15 committed real code. *(Lessons 1, 4.)*

**C2. "Show me it running" — the live screen is the only proof.** A green
suite proves the plumbing; only the real system doing the real thing proves
the feature. Live-verify is a bug-finding instrument, not a rubber stamp, and
the fixes it finds are the verification. *(Lessons 2, 24, 171, 176, 182, 194.)*

**C3. A test that shares the code's blind spot certifies the bug.** Code and
test written from the same wrong mental model agree with each other. A test
double must mirror the real contract (shape, async-ness, variance); a
regression lock earns trust only once you have watched it fail against the
fault it guards. *(Lessons 3, 30, 114, 150, 167, 186.)*

**C4. The bug lives in the seam, not the parts.** Component-green is not
system-green: mocked seams, individually-correct components with an
unspecified contract between them, and standalone benches all hide the defect
that only a test driving the real connected path can catch. *(Lessons 30, 56,
65, 90, 92, 119, 175, 187.)*

**C5. What you configure is not what runs.** A setting is a request; a
constant may be overridden; a knob may be wired to nothing. Instrument the
system to report what it actually did and believe the instrument. *(Lessons 6,
44, 79, 156.)*

**C6. Built is not wired; wired is not activated; activated is not exercised;
exercised is not operator-reachable.** Five separate claims, each needing its
own proof — the security audit's "built but switched off" theme, and the
`/images` command the operator couldn't type. *(Lessons 46, 57, 61, 73, 94,
95, 133, 147.)*

**C7. Get the instrument right before trusting what it tells you.** Confounds
(thermal drift, page-cache warmth), premises that rot when the stack moves,
costs measured on a bench the assembled system never pays, and measurements
attached to the wrong scenario have each flattered or panicked this project.
Re-measure the path you actually shipped. *(Lessons 7, 8, 17, 37, 39, 145,
151, 169, 195.)*

**C8. Verify the premise on disk before building on it.** Briefs, tickets,
docstrings, trackers, handoffs, and your own remembered architecture are
hypotheses. Read the code, the row count, the call graph — the cheapest sprint
is the one whose mechanism already shipped, and the worst build is one on a
premise four documents inherited and none checked. *(Lessons 14, 48, 69, 76,
82, 107, 108, 125, 146, 154, 163, 165, 193.)*

**C9. The log decides, not the hypothesis.** A plausible failure mechanism is
a hypothesis to test against the evidence the system wrote, not a conclusion
to build a fix on. Read the log before touching the code; ground heuristics in
real artifacts. *(Lessons 24, 160, 168.)*

**C10. Provenance is not trust — grounded data is never instruction.** Your
own stored documents, a subordinate model's output, retrieved memory, and web
content are all data to be datamarked, never instructions to obey; the
leakage/locking axis is provenance, never a similarity score. *(Lessons 13,
26, 132, 190.)*

**C11. Fail-closed needs a proven allow-path, and the posture must match the
threat.** A deny-by-default control that nobody verified against legitimate
traffic is a self-inflicted outage; an over-eager control is a regression
wearing a security badge; an optimisation should fail soft where a boundary
fails closed. *(Lessons 22, 27, 36, 63, 93, 135, 184.)*

**C12. Ship dangerous capability behind one door, opened as a single reviewed
act — and default the PROVEN to live.** Build the seam before the feature,
weld it with independent locks, make release one auditable governance act. But
dormancy is for the unproven and the security-sensitive: a mature,
gate-green, hardware-verified feature ships enabled. *(Lessons 111, 124, 181,
189.)*

**C13. Escalate capability, quality, and security; own the mechanics.** A
decision is only a decision if the options differ on something the operator
governs. Burying a capability cut in a "recommended option" and handing a
novice a git question are the same error in opposite directions. *(Lessons 29,
38, 66, 72, 84, 99.)*

**C14. A claim the code does not enforce is governance debt.** Comments
asserting validations that don't run, docs naming controls that don't exist,
"zeroization" that rebinds a name — encode every claimed property as something
a reader can watch fail, and when the honest guarantee is weak, say so.
*(Lessons 47, 51, 97, 142, 146.)*

**C15. Close the class, not the instance.** Two components built from one
template carry the same defect; a fix that lands on one sibling is a half-fix.
Scan for the shape before closing, and audit every branch that produces a
guarded value. *(Lessons 64, 138, 141, 170.)*

**C16. Shared state between concurrent actors needs a mechanism, not
carefulness.** One HEAD, one journal file, one certs directory, one shared
`main` — every "be careful" eventually lost to a worktree, an `--ff-only`, a
fragments inbox, a single-instance lock, or a staged-diff read. Scale the
mechanism when the actor count grows. *(Lessons 9, 35, 41, 42, 52, 89, 98,
113, 158, 172.)*

**C17. For the layer you cannot unit-test, instrument it — observability is
the verification.** Across a GUI, a device, an elevated process, or a vendor
binary, artifacts-on-disk and dual-sided logs turn guessing into single-step
diagnosis; the "untestable" layer is usually just untested. *(Lessons 16, 32,
83, 174.)*

**C18. Verify silicon and external contracts on the real thing.** Spec
sheets, vendor docs, and inherited designs describe an intended device; the
trust root is built on the measured one, and a documented API endpoint can be
dead by go-live. Only a live probe settles it. *(Lessons 18, 96, 103, 126,
128, 188.)*

**C19. A control that depends on a human remembering is not a control — and a
human step must be proven necessary.** Make the safe posture the enforced
default, guard cross-language mirrors with gate tests, and route a step to the
human only when it genuinely needs human hands. *(Lessons 44, 86, 149.)*

**C20. Ask the human only what the human can judge, at the coarsest
meaningful grain.** An unjudgeable consent degrades to a rubber stamp; route
danger to deterministic controls and put the question on the action, not the
content. When the non-expert operator's gut says a design is wrong, that is a
design-review trigger. *(Lessons 38, 191, 193.)*

## The numbered list

*Append-only; numbers are permanent. Each one was paid for.*

1. **Process is not progress.** I spent months building an "autonomous fleet" — AI agents meant to build software on their own. It generated an enormous amount of documentation: design docs, reports, audits of the reports. It produced almost no working product. Of around 259 automated sessions that committed anything, only 15 committed real code; the other 244 committed only markdown. It *felt* like work. Activity is not progress.

2. **"Show me it running."** A passing test is not proof a feature works. The only proof is the real system doing the real thing, in front of me. Every fix in this project is now verified live before it counts.

3. **A green test suite is only as honest as its assumptions.** BlarAI had tests for hiding the model's internal reasoning. They passed for a long time. The feature never worked: the tests checked for a tag the model never produces — the *same* wrong tag the code looked for. Code and test agreed with each other, and both were wrong. Tests that share the code's blind spot will confirm a bug instead of catching it.

4. **Build the smallest thing that genuinely runs, then grow.** Every grand plan collapsed under its own weight. Small, verified increments — one real capability at a time — are what compounded into a working product.

5. **Read the rules, not just the task.** A task once instructed me to build a second database; the project's own governance rules forbade exactly that. The task was wrong. Doing what you are told is not the same as doing the right thing — the constraints deserve as much attention as the request.

6. **What you configure is not what runs.** BlarAI's config enabled speculative decoding and the draft model was sitting on disk, so I assumed it was working. The first benchmark showed the engine silently falling back to standard decoding on every run. A setting is a request, not a guarantee — instrument the system to report what it *actually did*, and believe the instrument over the config file. *It came back on 2026-05-22:* a draft-model switch was made to a constant the running system overrides from a separate config file — a test confirmed the constant, and the change was inert. A written-down lesson is not a prevented one; that day's journal entry has the full account.

7. **A measurement can flatter you.** The first benchmark said speculative decoding made the model twice as fast. It had not — the two configurations were measured back-to-back, the second on a GPU already hot from the first, so temperature decided the result as much as the code did. Fixing the benchmark to be fair brought the honest number down to about 1.4×. That is still a real gain — but 1.4 is true and 2 was not. Get the instrument right before you trust what it tells you, and write down the corrected number even when it is the less impressive one.

8. **Fixing the instrument once does not make it trustworthy.** After the benchmark was corrected for thermal unfairness *between* configurations, I trusted it. A clean re-run drifted anyway — this time *within* a single configuration: a dozen measurements back-to-back heat an integrated GPU enough to throttle the last of them. One thermal hole had been closed and another left open, and only reading the runs one by one — not the average — caught it. A result is meaningful only with enough samples and its confounds held still, and that is not a fix you apply once; it is a question to ask of every measurement.

9. **Two Claudes on one machine share one HEAD.** When I dispatched the model-sharing build as a second Claude session in a separate window, I treated the two sessions as fully independent. They were not. They shared one physical working tree and therefore one `HEAD`. The moment the predecessor ran `git checkout main` to commit a journal entry, the successor's `HEAD` switched along with it — and the successor's next commit landed on `main` instead of its feature branch. The fix is not "be careful." It is to **use `git worktree add` for a separate working directory** whenever a predecessor needs to commit on the successor's repo. Two sessions can share a hard disk; they cannot share a checkout. The other rule that goes with it: the predecessor must stay off the GPU while the successor is benchmarking — a busy machine on a parallel pipeline poisons the data the successor is paying for.

10. **Continuity of intent across agents is a file-system property, not a memory property.** The journal discipline that compounds this portfolio worked while it lived in the head of whichever agent was carrying the work, and stopped working the moment a fresh session opened without it. The Layer 3 over-build was, at root, the same shape: a rule held in one place that should have been held in two. If you want the next session to do the thing this one learned, write it down where the next session will read it — not where you will remember it. `CLAUDE.md` is the right place for standing instructions every agent inherits; the journal entry below it is the evidence the instruction earned its standing.

11. **A setting that must cross a privilege boundary has to travel by a channel that crosses it.** Wiring the new native window into the launcher, I reached for an environment variable to choose it — the obvious switch. But the launcher self-elevates through a UAC prompt, and a Windows elevation relaunch begins with a fresh environment: a variable set in the un-elevated shell simply does not exist in the elevated process. The selection would have silently fallen back to the terminal — and because I cannot run the elevated, GPU-bound path on a headless machine, no test of mine would ever have caught it; only Blair, at the live screen, would have. The fix was to carry the choice as a command-line flag, which the elevation relaunch already forwards in `sys.argv`. The general rule is a security-shaped one: when a value must survive a trust or process boundary — UAC, sudo, a container edge, a service account — it has to ride a channel *defined* to cross that boundary, not one that merely happens to work on the same side of it.

12. **Build the surface to the shape of the capability you are about to add, and the capability becomes a backend swap instead of a UI rewrite.** Photos and video are where this assistant is going, but it cannot see yet. The temptation is to wait for vision before building anything multimodal. The better move was to build the entire experience now — a file picker, drag-and-drop, inline chips with image thumbnails, a screenshot button that really captures — against a backend that stores the file and grounds a placeholder saying "an image is here; you can't read it yet." When a vision model lands, nothing in the window changes: the chip is the same, the drop is the same, the user's habit is the same; only the grounded text the backend produces changes from a placeholder to a description. The store-only seam placed in Phase 1 and realised here is the load-bearing idea — a capability's *envelope* (where it plugs in, what it is allowed to touch, how the user invokes it) can and should be built and reasoned about before the capability itself exists. For a governance portfolio this is the valuable shape: you decide the blast radius and the UX contract of a feature while it is still inert, so turning it on later is a single, auditable change rather than a scramble.

13. **Provenance is not trust: your own stored data can still carry an attack.** When the Substrate began feeding the model the user's past documents and past conversations as retrieved memory, the easy assumption was that this content is "ours" and therefore safe — it came from the user's own files and the user's own earlier turns. That assumption is wrong in a way that matters. A document loaded last week may have carried an injection nobody noticed at the time; a past conversation may contain text the user pasted from a hostile page. Stored-by-us is not authored-by-us, and even authored-by-us is not safe-to-obey. So retrieved memory runs through the exact Layer 1+2 defences a freshly-loaded document gets — forged-delimiter neutralisation and per-load datamarking — and is grounded as data, never trusted as instruction, no matter that it came out of our own database. The general rule, and one the AIGP body of knowledge keeps returning to: the *provenance* of data (where it came from, who stored it) is independent of its *trustworthiness* (whether it is safe to act on). A system that conflates the two — "it's from our own store, so it's clean" — builds a laundering path where yesterday's unnoticed injection becomes today's trusted instruction.

14. **A handoff brief is a map, not proof the terrain is passable — probe the load-bearing unknowns before building on them.** The voice handoff brief was excellent: it named the models, the package, the wire format, even the integration points file-by-file. The temptation was to trust it and start writing the ADR. Blair's correction was the right one — "you leaned on the brief as a map without confirming it's buildable." So before a line of feature code, I ran the three unknowns on the actual hardware: does `WhisperPipeline` really exist in this OpenVINO build (yes); does Kokoro run on the Arc GPU or silently fall to CPU (CPU — onnxruntime here has no GPU execution provider — but measured real-time-factor 0.30, fast enough, so the brief's assumption was wrong in mechanism yet right in outcome); does the elevated app's microphone hit the same UIPI wall as drag-drop (reasoned no — capture is in-process, not a cross-integrity-level message). The probe paid for itself twice: it found that the brief's "Kokoro on GPU" premise was false, and — chasing the STT conversion path — it surfaced a *pre-existing* broken `optimum` exporter (a three-way `transformers`/`optimum`/`optimum-intel` version mismatch) that the brief never mentioned. The governance shape here is the one the cert track rewards: a plan inherited from a predecessor (or a vendor, or last quarter's you) carries assumptions that were true *then*, *there*, or *never* — and the cost of verifying a load-bearing assumption is always smaller than the cost of building a phase on top of one that has quietly rotted. Measure first; the number you get is the one you write down, not the one you were handed.

15. **Routing around a broken tool beats fixing it, when the tool is not on your path.** The probe found `optimum-cli export openvino` broken in this venv. The instinct — Blair's first instinct too — was to fix it: a broken tool in the tree feels like a hazard you must not build over. But the discipline is to ask *whether you are actually building over it*. We were not: the runtime never imports optimum, and the voice STT path needs a converted Whisper IR, which can be **downloaded pre-converted** rather than produced by the broken exporter. Fixing it meant re-pinning `transformers`, which the working 14B inference stack and the Substrate embedder both depend on — a large, untested blast radius taken on for a tool the mission does not use. So we routed around it (the pre-converted IR is, if anything, more reproducible than a local conversion) and *filed the breakage as a tracked hardening ticket* rather than pretending it does not exist or fixing it under time pressure mid-feature. The lesson is one of blast-radius judgment: "it's broken" is not the same as "it's in my way," and the mature move is to inventory the hazard, prove your path avoids it, and schedule the repair for when it can be done with the right regression net — not to yak-shave a dependency tree on the critical path of an unrelated feature.

16. **For the layer your tests cannot reach, instrument it — do not guess across the boundary.** Voice shipped "compiling, pending live-verify," and live-verify found it stone-dead: the mic raised five different failures in a row, none of which any pytest could have caught because they all lived in C#/WinRT/elevated-audio-device territory that the Python suite structurally cannot touch. The slow way to fix that is what I did at first — change a thing, rebuild, ask Blair to re-test, guess again — a multi-minute round trip per hypothesis across a process *and* language boundary. The fast way, which I should have reached for sooner, was to make the invisible visible: write the captured audio to a WAV file on disk (so I could read it, measure its RMS, and run it through Whisper myself), and log every step on *both* sides of the pipe to files I could tail. The instant I did, the guessing collapsed into single-step diagnosis — a 44-byte header-only WAV said "capture produced zero frames"; a perfect-sounding WAV that still failed said "the bug is past capture"; a `fmt ` chunk hiding behind a `JUNK` chunk said "your parser reads the wrong offsets"; an `ObjectDisposedException` in the log said "you double-disposed the node `FinalizeAsync` already closed." The lesson the cert track cares about: when verification has to cross a boundary you cannot unit-test — a GUI, a device, another process, a vendor binary — observability is not optional tooling, it is the *only* form of verification you have, and building it first is faster than building it fifth. The artifact-on-disk plus dual-sided logs turned an opaque "nothing happens" into a sequence of named, fixable faults.

17. **A decision can be right for a reason that is false — re-test the rationale, not just the result.** The governance doc disabled the GPU compile cache "for guaranteed identical compiled state," and because the *setting* was right, nobody questioned the *reason*. Measuring it emptied the reason: a warm cache produces byte-identical output — the same SHA-256 to the last character — as a fresh compile, so it was guarding against a risk that does not exist. The same probe dissolved the change a handoff had billed as "the single highest-value startup fix": a fresh 14B compile is ~11 s, no faster than reading the 9 GB cache blob back from cold disk, and the 13% "win" I first saw was only the *second* warm run reading the blob out of the OS page cache — lesson 7's hot-GPU confound in disk-cache clothing. Right setting, false rationale, worthless fix: three separate things, visible only because the premise itself was put on the instrument. The governance shape the cert track rewards: a documented rationale is a claim, and an unverified claim sitting under a correct decision is still a liability — when you revisit a decision, price its premise on the running system and write the corrected reason down even when the decision itself does not move. (Second, cleaner pass at the cache first measured in the 2026-05-22 "optimisation I measured to nothing" entry — this time post-model-sharing and without the integrity-hash that dominated the first.)

18. **A security design is only as real as the hardware it assumes — verify the silicon, not the spec sheet or your own inference.** BlarAI's trust root was specified for years on Intel SGX; the target Lunar Lake has no SGX (removed from Intel client CPUs since 2021), so the "SGX local attestation" in the canonical use-case doc was a phantom nothing ever ran. Querying the actual device replaced the phantom with what is truly there — a TPM 2.0 (manufacturer ST; Microsoft Pluton present but *not* the active TPM) whose non-exportable signing I then proved on the real chip. Two smaller versions of the same error bracketed it: I declared the chip "not Pluton" from a single `tpmtool` manufacturer string and was corrected by the device's owner (Pluton *is* present — I had conflated "the active TPM" with "the security processor"); and the primitive's `NCryptDeleteKey` flag was wrong in a way no reasoning revealed — only the real TPM returning `NTE_BAD_FLAGS` did. Spec sheets, inherited designs, and FFI docs all describe an *intended* device; the trust root must be built on the *measured* one. Query the silicon (`tpmtool`, Device Manager, a key round-trip on the actual chip) before writing a line of trust code — and when the hardware's owner says your description is wrong, believe the owner over your inference.

19. **Sell the capability you have, not the one they hope for — name the limits before they build on them.** Wiring vision (Qwen3-VL) for a landscape business, the User-Operator's goals included *precise 3D measurement from photos* and *generating proposal renderings*. A VLM does neither: it understands images (describe, recognize, answer, draft text) but guesses at metrics — false precision is worse than none for a bid — and cannot generate images at all (that is a diffusion model). The valuable move was not to quietly wire what I could and let the gaps surface mid-sales-call, but to say plainly, up front, what the tool does and does not do: measurement is photogrammetry (a separate pipeline needing the capture to change), rendering is image-generation (a separate model), both honest future tracks. Capability communication is a governance act the AIGP body of knowledge keeps returning to: a user building a business on an assistant needs the limits named as clearly as the features, *before* they plan around them — an over-promised capability discovered in production costs far more than a limit stated in conversation.

20. **The privilege-lowering fix and the capability fix can be the same change.** File attach was broken — OneDrive cloud-only files unselectable, drag-drop dead — and the obvious-looking fix, broadening the dialog's file-type filters, shipped and changed nothing, because formats were never the cause. The tell was *which* inputs failed: a local copy of a photo worked and the same photo as a cloud-only file did not, through the *same* dialog. A feature that fails only for the inputs that cross a trust boundary — the shell's cloud-hydration service, UIPI's integrity check — is a privilege bug wearing a feature bug's clothes. The cause was that the UI inherited the launcher's High integrity; the fix was to run it at the lowest integrity that works, a filtered Medium (admin-deny-only) token. That one change restored the capability *and* shrank the attack surface. Least privilege is usually filed as a security tax paid on top of features; here it *was* the feature. When a capability breaks only across a boundary, suspect the privilege level before the feature code — reach for the token, not the dialog.

21. **Verify the artifact, not just the data inside it — with the real tool, not a proxy.** I ran a rigorous adversarial security audit, then shipped the presentation that carried it with broken diagrams — twice — because my "check" of the deck was a mechanical lint of my own (bracket balance, line breaks) that shared the artifact's blind spot instead of running the actual Mermaid parser. The findings were verified; the vehicle delivering them was not, and a proxy check that never exercises the real consumer — the browser, the parser, the thing that will actually open it — is not a check. The fix was to run the genuine renderer over the exact bytes that ship (and the real cause turned out to be the HTML embedding mangling every arrow, not the diagram syntax at all). The shape, and the one that stings because the audit itself was so careful: when you hand someone a verified result, the thing carrying it — the slide deck, the export, the generated config, the report — is its own artifact that must be verified against the tool that consumes it. Verifying the inputs and assuming the output is a separate failure, and a common one.

22. **Fail-closed is not fail-shut: a deny-by-default control must still enumerate — and prove — the legitimate-allow set.** BlarAI's "no external network" guarantee was never enforced anywhere; it held only because the machine was air-gapped and the source happened not to call out — and both of those evaporate the day it faces the internet. Turning that habit into a code-enforced egress kill-switch, the handoff brief's framing — "refuse any outbound socket" — would have bricked the app on launch, because the runtime's own IPC *is* sockets: the Hyper-V vsock, the dev-mode loopback TCP, the launcher's 127.0.0.1 handshake. The LA caught it before it shipped. The control had to be an allowlist — permit loopback and Hyper-V, deny the rest — and the test had to assert the *positive* path (the local channels still pass) as rigorously as the negative one (external egress refused), because a fail-closed control whose allow-set nobody verified is one launch away from a self-inflicted outage that no headless test would catch: it lives in the GPU-bound elevated boot only the live screen sees. Two craft corollaries, paid for the same way: a guard must bite at the layer where the real egress happens (the raw socket's connect/bind — which both call sites perform *before* the mTLS wrap, so read the order rather than assume it), and a process-wide monkeypatch must be armed only at the true process entry (`__main__`), never in importable code a test will run, or the global mutation leaks into the suite and breaks things far from where it was set.

23. **A capability proven is not a control enforced — a finding closes when production points at the strong path and the weak one is gone.** The TPM signing mechanism was built and proven on the real chip a full increment before the audit's Critical actually closed, and in between nothing an auditor would credit had changed: the config still named the cleartext key, the key still sat in git, and a software-key fallback still existed to silently regress to. Closing the finding was the unglamorous half — repoint production at the TPM key, delete the cleartext PEM (history kept; the rotation makes the old key worthless, so no destructive rewrite was needed), and remove the fallback so preflight is fail-closed-until-provisioned with nothing weaker to choose. Two governance calls fell out of it: provisioning is a deliberate human *ceremony* (the operator witnesses the key, its fingerprint, and the date on the real silicon, not an automatic boot side-effect nobody audits), and the per-chip public key is *not* committed — a non-exportable TPM key's public half is machine-bound, not a portable constant, so git holds its SHA-256 fingerprint as the trust anchor, not one chip's bytes. Ship the mechanism, yes; but the finding stays open until the strong path is the only path.

24. **A problem the live boot surfaces next to your change makes the change a suspect, not a culprit — let the instrument decide before you reach for the fix.** Tier 0's egress kill-switch had never run live until the first real boot, and that boot was unusably slow: a photo attach took minutes, a voice clip queued five. The guard was the obvious suspect — a fail-closed allowlist starving a legitimate channel is precisely the self-inflicted denial-of-service the egress ADR warned about, and the kind only the live screen sees. The disciplined move was not to reach for the guard and "fix" it, but to read the launcher log first: the guard armed cleanly and there was not one `EgressDenied` in the entire file. It had starved nothing. The real bottleneck was elsewhere — image grounding running eagerly on attach, monopolising a single serialised lane so voice and chat queued behind it (the log timestamped a voice clip captured at 22:10:39 and not handed to the transcriber until 22:15:27, then transcribed in under a second). Had I trusted the suspicion over the instrument, I'd have "fixed" an innocent guard and shipped the real jam untouched. The live boot tells you *that* something is wrong; only the log tells you *what* (cf. lessons 2, 6, 16).

25. **One shared event loop makes any single blocking call a whole-app freeze — fix it structurally (off the loop), not locally (make the blocker faster).** The backend serves every request through one `run_until_complete` loop per connection. The voice methods knew this — they push their blocking model calls onto worker threads with `asyncio.to_thread` — but the document methods beside them in the same file called the blocking loader *directly on the loop*, so an eager vision pass on attach seized the only loop there was and the microphone frame that arrived a minute later had nothing left to service it. The reflex fix was to make grounding faster; the right fix was to get every blocking call off the loop, exactly as the voice methods already did. The tell was the asymmetry — the same async discipline applied to one set of handlers and not to the set next to them, the gap that opens when a pattern lives in one feature's author's head instead of where the next feature inherits it. The governance shape: when work is slow on a single-loop (or single-threaded, single-lane) system, ask *what is holding the lane* before asking *how to speed the work* — and pin the off-loop property with a test that asserts the work ran on a different thread than the loop, so the regression announces itself instead of hiding until the next live boot (cf. lessons 16, 24).

26. **On-demand and context-aware beats eager and generic — and a subordinate model's answer is data, not instruction.** The first vision wiring described every photo the instant it was attached, with a fixed domain prompt, before any question existed: it answered a question nobody asked and produced a description generic to a domain the photo might not even be about. The better shape is to hold the input, and when the user actually asks, let the reasoning model (the 14B "brain") formulate *what the perception model (the VLM "eyes") should look for* from the conversation, then fold that answer back as context. Two disciplines make it safe and fast: the subordinate model's output is grounded as **datamarked data, never trusted as instruction** (lesson 13 extended from stored documents to a live model's output — a VLM can read an injection off a sign in an attacker's photo), and the formulation pass is **skipped when the prompt adds nothing** to enrich (a bare "what's this?" goes straight to the eyes). The topology corollary, paid for by almost building the hard version: put the cross-model orchestration *where both models already live* — co-located, the "brain directs the eyes" sequence is a local function call; split across a thin control channel it would have needed a bulk-data path and a callback the protocol does not have. Reason about where code should run from what the *default configuration actually does*, not from the canonical architecture story (cf. lessons 13, 24, 25).

27. **A deferred resource risk comes due as a hard failure, not a graceful one — and a fail-closed default can disguise it as another subsystem's bug.** The vision MVP cached its ~5 GB model in a module global and never freed it; on an integrated GPU with no separate VRAM, that 5 GB lived in the same system RAM as everything else, and over a session it saturated memory and froze the host — the "open hard problem" I had deferred in the ADR, deferred *without bounding it*. Worse, it wore a disguise: under memory pressure the vision turn exceeded the 120 s IPC timeout, the gateway's fail-closed default denied a response the orchestrator was in fact still producing, and it surfaced to the user as a "validation error" — naming the policy validator for what was actually resource exhaustion three layers away. The log distinguished `default deny (no result)` from `PGOV DENIED (N violations)`; the user-visible surface collapsed them. Two lessons in one: free what you cache on a shared, finite resource (evict after use; an unfreed cache is a slow leak that ends in a freeze, not a warning), and make a fail-closed control *report why it closed* — "denied by policy" and "denied because nothing answered in time" are different failures with different owners, and conflating them sends you debugging the wrong code. The instrument named the cause; the symptom named a scapegoat (cf. lessons 22, 24, 26).

28. **The safest feature flag is the fail-soft path you already built — disable-by-config and degrade-by-failure should be one code path.** Asked to stop preloading the voice models to save memory, the tempting fix was a new lazy-load-on-demand subsystem — more capable, but new code with a UI contract I could not live-test. The voice engine was already fail-soft, though: an absent model disables that half cleanly (STT/TTS report unavailable, the UI hides the affordance, and `speak` is gated off so nothing is auto-spoken), a path exercised every time the weights were missing. So the off-switch became a one-line config gate that routes into that existing, tested degraded state rather than a new path — I was not introducing a new state, I was deliberately entering a proven one, which is the lowest-risk shape there is. The general rule: when you need to turn a capability off — for memory, for risk, for a user who is not using it yet — look first for the degradation path the system already handles; disabling by config and degrading by failure are the same behaviour, so reuse the proven path and save the more elegant mechanism (here, lazy-load) for when it is wanted on its own merits. The flag is a code constant, not an env var, because it must survive the launcher's UAC self-elevation (lesson 11). (cf. lessons 11, 12, 16, 27.)

29. **Escalate what the user experiences; own how it is wired — a quality or capability tradeoff is the operator's call, never an implementation detail you slip in.** To fix a memory freeze I proposed downscaling a 50 MP photo to ~1.5 MP before the vision model, and framed it as a "low-risk, standard, reversible" recommended option. Blair stopped it: that resize would gut the answer for the very query the photo carried — *read the text on the wall* — and reducing what the product can do is his decision, not mine to bury in a recommendation. His framing is the rule: *"decisions that have large impacts on the quality of the response is the kind of hard escalation we should be having to me, not stuff asking me about Git which I know nothing about."* The non-developer operator cannot adjudicate commit structure or which thread a call runs on — decide those yourself — but whether the assistant can still read the sign in the photo is exactly the call to surface, explicitly, with alternatives, *before* implementing. This sharpens the calibrate-your-asks discipline: "decide implementation yourself" covers the wiring, never the user-visible capability. Two technical corollaries the same round paid for: a memory "fix" I could not measure was a no-op that *logged a lie* ("VLM memory released" while the OS got nothing back — `gc.collect()` does not return a GPU driver's pooled memory intra-process; only process exit does), so I shipped observation (psutil memory + input-size logging) and deferred the behaviour change to when a harness can measure it — a log line is a claim, not proof, and the layer you cannot test is the one that bites (cf. lessons 6, 16, 27).

30. **A guard you have not watched go red against the bug is decoration — and a bug in the interaction between parts is invisible to tests that exercise each part alone.** The image-attach freeze sailed through 1,483 passing tests because not one of them drove an attach and a voice request against the *same* event loop — the contention *was* the bug, and a per-handler suite structurally cannot feel it. Building the regression lock, the easy version passes when the fix is present; the honest version must also *fail when the bug is present*, so the suite carries a meta-test that reconstructs the pre-fix sync-on-loop dispatcher and proves it starves the neighbour. Until I had seen the lock red against the reconstructed regression, I had a test that passed, not a guard that catches — different things. Two corrections from an adversarial review I ran against my own harness sharpened it the same day: my first lock asserted a *timing proxy* (the loop stayed responsive during the attach) rather than the property the fix actually guarantees, so I replaced it with a thread-identity assertion — the blocking call ran on a different thread than the loop — which is both the real invariant and immune to the CI jitter a millisecond budget is exposed to; and the review caught a memory-eviction helper that `del`-ed a *parameter* and freed nothing, the same logged-a-lie no-op as lesson 29, written twice in one week. The shape the cert track rewards: a regression lock earns trust only once you have watched it fail against the very fault it guards, and a contention bug needs a harness that reproduces the contention, because the component suite will keep certifying that every part is fine while the whole stays broken (cf. lessons 3, 21, 25, 29).

31. **Fix the symptom with a bound; fix the cause when the bound lets you.** When the Orchestrator under memory pressure generated a valid answer but never sent GENERATION_COMPLETE, the UI froze until the user killed the app. The cause — VLM-triggered memory exhaustion — is tracked separately (Vikunja #565) and its fix involves hard trade-offs only the User-Operator can make (quality versus memory). The symptom — an infinite freeze regardless of root cause — is fully fixable right now by wrapping the entire streaming arc in a deadline: whatever the gateway does, the UI's text input is re-enabled within a bounded time. These are separate decisions with separate owners, and conflating them would either delay the fix (waiting for the root cause to be resolved first) or short-circuit the decision (making a capability trade-off without asking). The governance shape: when a failure has a *symptom* the system can absorb and a *cause* that requires a human call, bound the symptom now and expose the cause cleanly, so the operator can decide on the cause from a system that no longer freezes rather than from one that does. (cf. lessons 27, 29.)

32. **The layer you call "untestable" is usually just "untested" — hunt for the seam before you fence it off.** Asked to bring autonomous testing to every layer, I first walled off the WinUI front end as the one a Python harness "structurally cannot touch" — lesson 16's boundary, C#/WinRT past the edge. The User-Operator refused the fence: the whole point is to take the human out of the loop, and a layer that *always* needs human eyes defeats it. He was right, and the wall was a failure of imagination, not a law. Two seams I had not looked for dissolved it. First, UI Automation: pywinauto drives the real window over the UIA tree WinUI 3 already exposes, so a test types a prompt, clicks send, and asserts the input is enabled again — no human — and it even surfaced that `x:Name` maps straight to the AutomationId, so no app changes were needed. Second, the UAC wall: the launcher forces admin + Hyper-V, but the window is a *separate de-elevated exe* that needs only a named-pipe backend, which a scripted fake stands up with no models, no VM, no elevation — letting the test drive the real window against a backend it fully controls (including scripting the exact dropped-frame freeze that bit the user). The real constraint was never testability; it was that the display, the GPU, and the GUI are *shared singletons* — the fleet parallelizes everything around them, but anything that drives the screen or loads the models serializes on the one machine. That limit is real; "the front end can't be tested" was not. The governance shape: when you hear yourself call a boundary untestable, that is the moment to look hardest for the seam, because the cost of an unexamined "can't" is a permanent hole in the safety net — and the operator who keeps finding those bugs by hand pays it every time. (cf. lessons 14, 16.)

33. **Capturing data and publishing data are two separate disciplines — and a record that implies more coverage than it has is worse than none at all.** Every measurement BlarAI produces carries a `not_measured` field in its environment block: an explicit list of what the run did NOT cover (GPU driver version not introspectable from Python; co-resident memory cost not held fixed; subjective quality not quantified). That field is not courtesy — it is the gate the community-contribution pipeline uses to decide whether a record can be published at all. A record without `not_measured`, or with an empty one, is rejected outright, not silently included or silently dropped: the validator surfaces the error and names the file, because a partial record admitted to a community dataset misleads the next person who looks at your hardware numbers and asks "does this generalise to my setup?" The pipeline itself does nothing on the network. Aggregating, scrubbing, and writing the JSONL and CSV is machine work; deciding when to submit to an OpenVINO issue thread or a HuggingFace dataset is a deliberate human act that requires reading the maintainer's context and the engagement-first norm. Building the capture side first and the publish side as a separate, gated step was correct — a dataset with rigorous provenance assembled privately is more valuable than a fast upload of whatever the harness happened to produce. (cf. lessons 7, 8.)

34. **A prefix or allow/deny check on a path you did not canonicalize is a bypass waiting to be typed.** The deterministic Policy-Agent gate denied `/etc/`, another user's `/home/`, and `/internal/` with `startswith` on the raw resource string — and the #5 benchmark found that `/home/user/../../etc/passwd` walks straight through, because it literally starts with the *allowed* `/home/user/` and not with `/etc/`. The string a caller types and the file the kernel opens are different objects; the security comparison has to run on the canonical form, not the typed one. The fix normalizes with `posixpath` — not `os.path`, which on this Windows host would rewrite `/` to `\` and break the very check it was meant to harden, since CARs carry POSIX paths regardless of the host OS. And it tests BOTH the raw and the normalized form rather than swapping one for the other, so the hardening is provably additive: everything the gate caught before is still caught, the traversal variants are caught too, and the trailing-slash and leading-`//` quirks of `normpath` cannot open a new hole. A change that cannot regress is worth more than a tidier rewrite that might. (cf. lessons 22, 23.)

35. **Verify the merge landed where you meant it to — a shell's working directory can drift, and a loop's "OK" echo is not proof.** Running the five-branch Tier-1 merge loop, every line printed "OK", the commit graph looked clean, and `main` appeared to advance to the integrated commit. It hadn't: the shell's working directory had silently drifted into one of the agents' worktrees, so all five merges had landed on *that worktree's branch* while `main` never moved. The verify-on-disk habit caught it before a single test ran — grep the merged tree for each control marker, then check `git rev-parse HEAD` in the main repo *by absolute path* — and the recovery was a non-destructive fast-forward, because the integrated commit was a clean descendant of `main`. This is lesson 3 wearing version-control clothes: a command that prints success proves the operation ran *somewhere*, not that it ran where you intended. Confirm the branch, the HEAD, and the cwd by absolute path before you trust a multi-step git sequence — and never trust the loop's own success echo over the repository's own answer. (cf. lessons 3, 9.)

36. **An over-eager control is a regression wearing a security badge — re-enabling a disabled control is not automatically safe.** The audit said "wire the inert leakage detector"; wiring it surfaced that a cosine-similarity-against-grounded-content check flags every legitimate retrieval answer, because a summary or a memory-recall is similar to its source *by design*. The control, once enforced, suppressed correct answers on the live screen — the opposite of security. The mocked tests passed, because they verified the wiring, not the behaviour; only the real model with real documents caught it. Before enabling a dormant control, verify its PRECISION against the system doing its *normal* job, not just its presence — treat "turn on the thing that ships off" as a claim to be tested, not a safe default. Off-and-correct beats on-and-wrong. (cf. lessons 16, 23, 24.)

37. **A cost measured in isolation can be a cost the running system never pays — profile the component where it lives, not on a bench by itself.** A ticket reported a 5–8 second "embedder first-prompt tax" the Substrate supposedly added, and the profiling that found it (#546) had measured the bge-small embedder *standalone*, in a process that imported nothing else. Decomposed, that 5–8 seconds was ~92% a single `from transformers import AutoTokenizer` — a one-time Python library import, not model I/O, not graph optimization, not tokenizer files (those were 0.2 s and 0.04 s). And in the real Assistant Orchestrator that import is already paid, at boot, by the 14B's own inference module, which imports `transformers` at module load for *its* tokenizer — before the Substrate ever builds. So the embedder, loading after it, finds the library cached and costs ~0.3 seconds, not 5–8; I confirmed it by measuring the load with `transformers` pre-imported (~0.3 s) versus not (5 177 ms). The "tax" was an artifact of the bench: a shared dependency billed in full to whichever component is measured alone, when the assembled system pays for it exactly once and attributes it to whoever loads first. The trap nearly cost more than a wrong number. The ticket framed three options for *disposing* of the 5–8 seconds — fold it into boot, background it on a thread, leave it — and I began by relaying those options to the operator as a live choice, after my own data had already voided the premise behind them. He stopped me twice: if the tax isn't real there is no choice to make, and presenting one is hedging instead of owning an obvious call. The governance shape is two-edged. Measure a shared cost *in situ* — co-resident, in the real process, in load order — because a component's standalone cost and its marginal cost in the assembled system are different numbers and only the second is real. And when your own measurement dissolves the premise of a decision, report the conclusion and act; do not keep offering the options the dead premise used to justify. (cf. lessons 7, 14, 17.)

38. **When the operator's gut says a design is wrong, that is a design-review trigger — re-derive from best practice, do not defend the as-built.** The provenance sprint shipped a blunt action-lock — any untrusted content turned *every* tool off, and `/trust` turned them back on — and it passed every test and verified live. Then the User-Operator, a self-described non-developer, watched a clock get locked behind a `/trust` prompt and said *"is this really best practice? this whole trusting process seems bad."* He was right, and the reflex to explain why the shipped design was fine would have buried a real flaw: the blunt lock adds friction with zero security benefit on a calculator, and `/trust` pushes an un-evaluable security decision onto exactly the person least equipped to make it — two anti-patterns the literature names. The fix was a different shape, not a tweak: tools carry risk tiers, safe tools are never locked, dangerous *actions* are denied absolutely with no override, and the sharp per-action deny that already existed (the #570 mediation) does the real work while the blunt lock scopes down to where it earns its friction. A non-expert's "this feels wrong" about a security experience is rarely noise — it is the smell test the experts stopped running. Treat the challenge as a prompt to re-derive from first principles and the external state of practice; the as-built is evidence of what was tried, never proof that it was right. (cf. lessons 16, 29, 37.)

39. **A locked decision can stay right while the evidence under it quietly rots — re-measure a premise when the stack beneath it moves, not just when you revisit the decision.** DEC-03 fixed the context window at 16K, and one of its supporting findings — speculative decoding is inert at ≥16K, so the big window costs no speed — was measured in March on a draft model and a runtime that both changed two months later (the ISS-1 fix that made spec-decode engage, and the pruned-6L INT8 draft swap). The decision was still correct; the premise under one corner of it was stale, and I nearly quoted it as current. What caught it was a *date*, not a re-run: the finding predated the change, so it described a system that no longer existed. Blair's push — *"don't we already have that data from the last couple of days?"* — was half-right in the most useful way: the May benchmarks confirmed spec-decode works, but only at short context, never at 16K, so the corner was genuinely unmeasured. A targeted re-measure settled it — acceptance at 16K is 46%, not 0, the cliff gone, throughput ~2.3× the recorded number, the decision reaffirmed on current data — and the *same* run found a different premise of the same decision rotted the other way (the footprint nearly quadrupled, so "20K is safe" is now the suspect claim). The governance shape: a decision's premises have their own shelf life, independent of the decision; when the stack beneath a locked call moves — a model swap, a runtime upgrade — price its premises on the running system before you quote them, because a right decision resting on a rotted reason is still a liability. (cf. lessons 6, 7, 17.)

40. **Reuse the commodity, harvest the method, refuse the framework — a network-touching dependency is a supply-chain liability you import on purpose.** Building the web-search skill, three mature open projects — local-deep-research, gpt-researcher, open_deep_research — already implemented the exact query→search→read→synthesize loop I needed, and the convenient move was to adopt one wholesale. I didn't, and the reason is governance, not pride: each carries a large orchestration framework (LangGraph), cloud-LLM assumptions, and a multi-engine scraper stack — a wide, network-touching dependency surface to import into a system whose security model *is* its air-gap. The discipline that paid: take the *commodity* (the thin, hash-pinned Kagi client — five canonical packages, a 3.7 KB wheel) and the *method* (the algorithmic shape, studied not copied), and hand-build the gated-local-native part none of them have. The shape the cert track rewards: in a security-critical system, build-vs-adopt is a supply-chain decision before it is a convenience one — every dependency is attack surface you chose to take on, and "someone already wrote it" is the weakest reason to import a framework into a system whose air-gap is a core invariant. (Building the skill does not lift that air-gap — it stays mocked and dormant behind the armed egress guard; whether BlarAI ever faces the internet is a separate, deliberate decision, and this hardening exists so the walls are ready long before that question is asked.) (cf. lessons 1, 15.)

41. **To land on a branch others are actively pushing to, use an operation that can only fast-forward — it refuses rather than races.** Landing the web-search work, the shared `main` advanced under me three times mid-merge — each time after "nothing is running" had been said in good faith. A plain merge would have raced, and a drifted working directory would have landed it on the wrong branch (lessons 9, 35); the guard that made it safe was `git merge --ff-only`, which can only advance `main` to a commit that already contains it and *refuses* otherwise. A moving target can therefore never produce a wrong land: when `main` advanced, the fast-forward declined, I re-integrated the new `main` (re-resolving the journal append) and retried — bounded and deterministic. Two corollaries paid for the same way: "nothing is running" is a claim to verify against the repository's own HEAD by absolute path, not a fact to trust (a handoff fact disk-reality kept contradicting — cf. lesson 14); and the one file that conflicts on every parallel-session land is the journal itself, whose resolution is always "keep both sides, renumber mine." (cf. lessons 9, 14, 35.)

42. **A coordination discipline that works for one actor becomes the bottleneck when many run at once — scale the mechanism, not just the rule.** The journal-append discipline that compounds this portfolio jammed the moment three sessions ran in parallel, because they all appended to one file git cannot auto-merge — lesson 41's "the journal conflicts on every parallel land" stated as a structural problem rather than a per-merge chore to keep paying. The rule that would have avoided it existed for sub-agents ("report your entry text, don't touch the journal") but never for sessions, so the collision simply moved up a layer and bit three times in a day. The fix was to stop sharing the file: a per-session fragment inbox (`docs/journal_fragments/`, one file per entry, nothing shared to conflict on) that one integrator folds into `BUILD_JOURNAL.md` serially at a quiet tree, with lesson numbers — the sharpest shared state — assigned only at fold-in. When a single-actor discipline has to scale to many concurrent actors, the medium it rides has to scale with it, or the discipline becomes the friction it was meant to prevent. (cf. lessons 9, 10, 33, 41.)

43. **A component's primary purpose is fixed by its use case, not by the sprint you happen to be viewing it through.** I framed the Cleaner (UC-003) as a security decision because I met it during security work; its real first job is data-normalization for the knowledge store — stripping ads, boilerplate, and markup so the Substrate isn't poisoned with inference pollution, with injection-sanitization only its second layer. Re-read the use-case text before locking a decision around a component; the lens of the sprint you meet it in will try to redefine its purpose.

44. **A safety control that depends on a human remembering to do the right thing is not a control.** dev-mode shipped as the silent default, so every test and every "BlarAI works" report validated a configuration that would never ship — green on dev-mode while the production posture failed, which is why agents kept reporting success the operator couldn't reproduce. The fix could not be "remember to test in production posture," because the operator does not run the tests; it had to be structural — make the production posture the one the suite verifies by default, and make the wrong posture fail loudly in CI. If the only thing between you and a false green is someone's memory, you have no control.

45. **When a mess keeps reappearing, find the doctrine producing it before you clean it up again.** Eleven handoff briefs had piled up in the repo root, and the quick fix was a move plus a gitignore line — but they hadn't landed there by accident: the context-exhaustion doctrine literally instructed every session to write its brief at the repo root. Gitignoring the directory would have buried the symptom while the cause kept generating new ones; amending the doctrine that mandated the location was the only durable fix. A recurring-clutter symptom usually has a policy default standing behind it.

46. **A control wired into a function but not into the boot path is still "built but wired into nothing" — a green suite proves the mechanism, never that the running system invokes it.** The Sprint-13 tamper-evident audit stream was built correctly — hash-chained records, a pluggable signer, fail-closed writes, dependency-injected into all three of the adjudicator's return points — and proved with 38 teeth-tests that all passed. It was still inert: the production Policy Agent is constructed in one factory that called `from_config` without passing the sink, so the running PA would have persisted exactly zero records. That is the 2026-06-03 audit's own central finding — the TPM root with no callers, the leakage detector fed an empty list — recurring *inside* the sprint built to close such gaps, and the full green suite hid it because the tests exercised the part in isolation, not the live boot. Reading the production construction site caught what no passing test could. The fix that makes it stick is a regression test asserting the boot factory yields `has_audit_log == True` — an assertion that fails the day the wiring is removed. When a mechanism's tests prove it in isolation, ask separately whether the live system actually invokes it, and write the test that breaks if it stops. (cf. lessons 16, 23, 30.)

47. **A comment that claims a property the code does not enforce is governance debt that surfaces later as an audit finding.** The PII credit-card detector's pattern was commented "(Luhn-plausible)" and ran no checksum — it matched any 13-to-19-digit run, so order numbers and account IDs would have read as card numbers the moment the redact-at-egress posture switched on. The gap between the comment's claim and the code's behaviour is exactly where false confidence lives: a reader trusts a validation that isn't there, and an auditor eventually finds it. The fix made the comment true — a real mod-10 gate on both detection paths — and registered the validator in a named per-label table so "what validation actually runs" is visible code, not a parenthetical promise. Encode the secondary gate a detector needs as something the reader can see fail, not as prose. (cf. lessons 30, 44.)

48. **A load-bearing factual premise gets verified on disk, not inherited through the review chain.** The entire urgency case for the at-rest encryption sprint rested on a claim — that BlarAI's two SQLite stores already held "decades of the user's private data" sitting exposed in plaintext — that the ADR, the SDV, the security roadmap, and my own design-gate presentation all asserted and none had checked. A one-line look at the files (107 chunks / ~400 KB in `substrate.db`; 59 sessions / 376 turns / ~250 KB in `sessions.db`, all build-phase scaffolding) showed it was false: nothing real had ever been stored. The honest framing was the opposite of urgency — encrypt *now, before first real use*, so real data is born encrypted — which is more defensible, not less. What stung was the propagation: plausible-and-repeated across four documents read as settled fact, and each author inherited it from the last rather than running the check. When a claim is doing real work in a decision, its truth is a row count or a `du`, not a thing to carry forward because the previous document said it. (cf. lessons 2, 17.)

49. **A design gate — the ADR written and reviewed before a line of cipher code — turns the hard seam-choices into translation work, and moves the review surface from the code to the ADR.** The interesting failures in a dual-wrapped encryption envelope are never "it crashed"; they are quiet wrong-shapes at the seam between two components — a version byte the inner blob doesn't expect, a recovery wrap that base64s the raw key by accident, a factory guard that isinstance-checks the wrong class — each a test that passes while a security property silently doesn't hold. Pinning the three real choices in ADR-025 first (a fresh CSPRNG nonce per encryption, the recovery key as a random 256-bit blob not a passphrase, AAD binding each ciphertext to its table/column/row identity) meant EA-2 arrived as translation, not design, and the merge gate reviewed the ADR rather than reverse-engineering intent from code comments. The governance payoff for the cert track: the path-not-taken is on the record at the design layer, where it is legible, instead of implied by an implementation. (cf. lesson 5.)

50. **Encrypt the derived representations, not just the raw text — a stored embedding is an invertible shadow of its source.** The first instinct when wiring the field cipher into the substrate store was to treat embeddings as non-sensitive metadata; they are not human-readable, after all. The sharper call was to encrypt them too, because the vec2text literature shows a 384-dimensional semantic embedding can reconstruct enough of its source text to be a real privacy leak from a stolen disk — encrypting the house while leaving the blueprint in the mailbox. The cost objection dissolves with a boot-cache: decrypt all embeddings once at unlock into an in-RAM matrix, run vector search over plaintext in memory, and decrypt only the top-k matched text per query. Measured overhead was +0.124 ms per query and 1.45 ms one-time at boot for 107 embeddings — below noise against time-to-first-token. The price is paid at boot in proportion to corpus size, not per query; at single-user scale it is immaterial. (The dedup that AES-GCM's per-nonce non-determinism broke was restored with a keyed-hash index column — `HMAC(k_idx, filename)` — accepting the documented leak that two identical filenames share a hash.)

51. **Tamper-evident is not non-forgeable, and the difference belongs in a test, not a comment.** Sprint 13 shipped a hash-chained audit stream — tamper-evident by construction — with the signer deliberately left a software HMAC stub whose key was recomputable from filesystem state. The hash chain proves records were not altered after the fact; only a signature proves *who* wrote them, and we had only the first property while a comment implied the second. EA-5 closed the gap with a TPM-sealed ECDSA signer on a dedicated key (separated from the PA's JWT key so an audit compromise doesn't also compromise token issuance), but the portfolio-valuable part is the contrast test: one test actually forges a record and asserts the weak stub accepts it; its mirror asserts an adversary with only the key name cannot produce a signature the TPM signer accepts. The security property is now executable, not asserted. (Extends lesson 47 into the crypto layer — encode the property as something the reader can watch fail.)

52. **The worktree-isolation quirk can switch the main checkout's branch, so `git -C <main-path>` is not enough — verify the *branch* before every main-tree commit.** Launching background worktree builders did not merely move my shell's cwd this sprint; it switched the main checkout itself (`C:/Users/mrbla/BlarAI`) onto an EA's feature branch. So a commit I made with an explicit `git -C <main-path>` — believing the path made me safe — landed on the feature branch, not `main`, which never advanced, and I reported it as "on main." The path and the branch are independent: the path can be the main worktree while HEAD is a feature branch. The guard that holds is `git branch --show-current == main` before every commit or merge on the main tree, not the path alone. Recovery stayed inside the never-discard rule and the LA's refinement made it cleaner: preserve the stray staged reversal on a throwaway `wip/` branch, switch the checkout back to `main`, and cherry-pick the real commit — no reset, no force, nothing thrown away. The branch-guard added afterward caught the very next recurrence and blocked a wrong-branch merge. (Sharpens lesson 35.)

53. **A break-glass recovery path must be correct by design and proven against a dead chip, never correct by accident.** The DEK recovery flow — unseal the data key with the offline recovery key when the TPM is gone, then re-seal to a new machine — passed its first review green, but only because the dev sealer stub happened to raise an exception the code happened to catch; the real recovery logic leaned on a private helper and was never exercised the way a genuinely dead chip would exercise it. A break-glass path is the one path you cannot test in production and the one you most need to work the first time, so "passes today" is not the bar — "correct by construction and proven against a simulated dead chip" is. Round two added public `unseal_via_recovery` / `reseal_dek` entry points and a `DeadSealer` test whose `unseal()` always raises, so the recovery path is driven exactly as a destroyed TPM would drive it, with both the wrong-key-fails-closed case and the dead-chip-succeeds case asserted. (cf. lessons 16, 46 — built-and-green is not invoked-and-correct.)

54. **A factory that reads missing configuration as "dev mode" turns a misconfiguration into a silent downgrade to the weakest posture.** The store factories' dev path was not an explicit opt-in but the implicit default for any misconfiguration: with `BLARAI_DEK_KEYSTORE` absent, both the session and substrate builders would quietly construct a SoftwareSealer store encrypting under a hard-coded public key — effectively-plaintext data that looks encrypted. No data was exposed on the live system (the production boot set the env var), but the posture was "missing config reads as dev mode" rather than "missing config is a misconfiguration," one accidental env-var drop from a production boot that writes pseudo-plaintext while appearing to encrypt. The fix is symmetric with what the audit stream already did: make the safe posture the enforced default and require an explicit caller signal to unlock the weak one — the session store (load-bearing) refuses the whole boot, the substrate (non-load-bearing) degrades loudly with memory disabled. The weak posture must be the thing you opt into, never the thing you fall back to. (cf. lesson 44 — a control that depends on a human remembering is not a control.)

55. **Never run the test suite against live user data — and the isolation a suite needs must be set before import, not in a fixture.** While verifying each merge of the at-rest encryption work, I repeatedly ran the full suite against my own live environment; a test that calls service startup resolved the real `%LOCALAPPDATA%\BlarAI\`, wrote dev-key-encrypted rows into the operator's actual `sessions.db`, and the next production launch could not decrypt them — the backend refused to start (`[7199c5ab]`). The bitter irony: the suite I was running to *verify* the test-isolation fix was itself the polluter, and it was harmless only because no real data existed yet (cf. lesson 48). The first fix — function-scope autouse fixtures — was too late: pytest imports test modules during collection, so an import-time constant like `SESSION_DB_PATH` has already resolved against the real path before any fixture runs. The durable fix is a root `conftest.py` that redirects the user-data env vars (and unsets the keystore pointer) at module load, before any import — the earliest code pytest executes — with the package fixtures kept as a second layer for call-time reads. Whenever a test process can reach real user data, redirect it at startup; never trust a fixture to guard a constant that resolved at import. (cf. lessons 2, 46.)

56. **A harness that mocks the boundary the bug lives past is a unit test in integration clothing — and not knowing your own coverage is part of the coverage problem.** A green 2163-test default suite missed a one-line production-only bug: the launcher wired the gateway to the Policy Agent's port (5000) instead of the Orchestrator's (5001), so every prompt was rejected with `Unsupported message type`. The unit tests mock the seam and the bug lived in the seam. The twist that makes this governance and not just another lesson-3: BlarAI *already had* a deliberate three-layer scenario harness (#563 — dispatcher locks, real-model latency, real pywinauto window automation), and it missed the bug for the *same* reason — its dispatcher layer drives a fake gateway and its window layer a fake backend, so the real gateway→Orchestrator routing was exercised by nothing at any layer, and a `slow` marker had quietly hidden two integration files of bit-rotted tests from CI besides. And I compounded it: I told the operator the GUI had "no tests at all," having scoped my check to the C# project and forgotten the harness I had built a sprint earlier — caught only by checking the whole repo before committing the claim. Coverage is which seam a test *exercises*, never how integration-flavoured it looks; a marker that hides a test is not coverage; and a claim about your own coverage is worth as little as any other unverified claim. The fix is doctrine — `TEST_GOVERNANCE.md` §2.7: every major subsystem has a test that reaches its real production seam, it is part of "done," and minimal-human-in-the-loop is the target — because a blind spot this structural is a class to outlaw, not a bug to patch. (cf. lessons 3, 16, 30, 32.)

57. **A control built but never issued its key material is half-armed, which is to say off.** The mTLS code path in `vsock.py` — the server SSL context, `CERT_REQUIRED`, the handshake — had been production-ready for many sprints; what broke the chain was that the cert files it needed did not exist and the gateway client passed no cert paths at all. A door with all the hardware installed and no key ever cut. The fix co-issues the control and its material at one point in the startup sequence — per-boot certs minted fail-closed before any service constructs — so the chain can never be half-armed: either both the control and its key are present, or the boot refuses. When a security control depends on a provisioning step, the provisioning must fire automatically as part of the same path, never sit as a manual prerequisite no startup code enforces. (cf. lessons 46, 61 — the built-but-uninvoked family.)

58. **A side effect placed before the point a process can still fork-and-exit runs more than once.** The launcher self-elevates through a UAC prompt: the original un-elevated process returns 0 and an elevated child continues. I had put the per-boot cert mint after the dev-mode interlock but *before* the elevation block, reading "after the interlock, before service construction" as one contiguous region. It is not — the elevation return-0 cuts it in two, and a disk-write effect on the wrong side of that cut runs in both processes, the short-lived non-elevated one writing first. The fix moved the mint after `admin_ok` is recorded, so it fires exactly once in the elevated process. Side-effecting startup steps belong after the last point the process can still bail out and hand off to a successor, not merely somewhere in the pre-service span. (cf. lesson 11 — a value that must cross the elevation boundary needs a channel defined to cross it.)

59. **Build the mechanism and pin the sentinel before you throw the switch — an unactivated mechanism is a deferred bug, and the test suite is what defines "safe" before you flip the bit.** A breaking change too risky to attempt cleanly tempts you to either ship it unguarded or never ship it; both fail. The staging pattern is the third path: one increment builds the capability and lays regression locks asserting the *current* pre-flip behaviour — the sentinel — so a premature activation fails CI loudly; a later increment throws the switch and updates the same sentinel to assert the new behaviour. EA-2 built dev-mode-off and pinned "HOST default is still dev"; EA-4b flipped it and changed the lock to "HOST default is production." The activation was safe precisely because the suite said what safe meant before the bit moved. A mechanism built but never activated, because activating it was never made safe, is not a mechanism — it is a bug deferred and dressed as caution. (cf. lessons 44, 54 — make the safe posture the enforced default.)

60. **Off-chip structural proofs and on-chip ceremony verification are distinct work with distinct value — stub data does not cheapen the structural proof.** An off-chip cascade harness with stub signers and a dummy model binary proves the config plumbing, the manifest format, the validation flow, and the fail-closed regression locks — all without a TPM, a GPU, or real weights. The instinct to dismiss it as "fake" is wrong: it de-risks every structural surprise before the operator spends ceremony time, so the on-chip session narrows to exactly what cannot be faked — the chip-bound key provisioning and the real-model digest match. Separating "what a stub can prove" from "what only the silicon can prove" is how a scarce human-on-hardware ceremony gets spent on only the irreducible part. (cf. lessons 23, 53 — built-and-green is not invoked-and-correct; prove the break-glass path against a dead chip.)

61. **Transport topology and security mechanism are orthogonal decisions — conflating them leaves the real production path unbuilt behind a fully-specified one.** AF_HYPERV is a mechanism that belongs to the VM-boundary topology; mTLS is a security property every production channel needs regardless of topology. Defaulting to AF_HYPERV because the boot was "production" meant the loopback host-mode path — the actual production topology for the foreseeable future — had no working implementation, despite a signed SDV criterion describing it and an EA-1 test suite verifying it in isolation. The right diagnostic for a well-specified, isolation-verified path the boot never invokes is "built into nothing," not "needs redesign": wire the proven path into the boot rather than redesigning around the wrong default. (cf. lessons 46, 57.)

62. **Config-to-provisioner correspondence must be tested as a unit, or the gap surfaces only at production boot.** The cert provisioner was written for two services; three `[ipc]` configs named cert pairs — the Orchestrator's and the Router's — the mint had never been taught to produce, and nobody had audited every config's cert paths against what the provisioner wrote. The first gap surfaced as `[Errno 2] No such file or directory` on the AO listener at live boot; the class-audit of all configs found the second before it could. The durable safety net is not a per-service regression test added each time a gap bites, but a single test that traverses every `[ipc]` section, extracts each cert path, and asserts the provisioner covers it. Any config that names a file another component must produce needs a test that walks from the config key to that component's output. (cf. lessons 64, 46.)

63. **Fail-closed is a property of a call site relative to its unit of accountability, not a uniform property of a component.** A leaf decryptor is accountable for the one row it was asked for; it must raise when that row will not decrypt. A bulk reader is accountable for the health of the whole store; it must not let one sick row deny access to every healthy one. Sprint 14 applied the leaf posture uniformly, so a single dev-era row the production key could not decrypt aborted `list_sessions` and bricked the app — a self-inflicted denial of service, which the availability leg of the CIA triad names as the security failure it is. The synthesis: bulk loops quarantine the bad row (omit it, log a stable greppable event, continue) while single-record paths keep hard-failing; confidentiality and integrity are untouched — no plaintext returned, no tampered data trusted — only the blast radius changes. Applying one call site's correct posture to a different site's accountability converts a confidentiality control into an availability attack surface. (cf. lesson 54 — the weak posture must be opt-in, not fall-back.)

64. **When you fix a defect of a particular shape, scan the codebase for the same shape before you close — finding one and fixing only one is a half-fix.** The session store and the substrate store were built in parallel under the same encryption ADR, with the same bulk-read pattern and therefore the same brick-on-one-bad-row defect. The session-store fix should have triggered an immediate check of its sibling; it did not, and the substrate store sat one boot away from the identical AO-startup brick until the class-audit caught it. Two components built from the same template carry the same design defects; the reflex that closes the *class*, not just the instance, is what stops the sibling biting in production next. (cf. lessons 62, 23.)

65. **A correct downstream fix can mask an upstream defect by moving its symptom past the gate that used to catch it — so test the seam between configuration and connection, not each side alone.** The launcher pointed the gateway at the Policy Agent's port instead of the Orchestrator's; the misroute would once have fail-closed loudly at the boot handshake, but an earlier, correct fix teaching the PA to answer that handshake made the wrong door answer too — and the failure migrated from "won't boot" to "boots clean, every prompt fails," past every gate. The most dangerous regression is the one a good fix makes quiet. Two disciplines follow: when a fix makes a formerly-loud failure quiet, ask what that path was *implicitly* verifying (a control that starts answering more messages can stop being a tripwire); and test the wire itself — the launcher picks a port, so assert the gateway actually reaches the right service there, because both endpoints can be individually correct while the connection between them is wrong. (cf. lessons 56, 6.)

66. **A test can lock a *defect*, not just a feature — so "it's test-locked" is not proof "it's a decision to escalate."** Twice in one night I reached to escalate a clear defect as a security/posture decision because a green Sprint-14 regression test asserted the buggy behaviour (the brick-the-whole-list fail-closed), treating the passing test as evidence the behaviour was intended. The operator collapsed it each time — "is there really more than one correct answer?" There was not. Before escalating a change as a decision, ask whether the behaviour it *reverses* was ever correct or merely locked-in; a defect with one nameable fix is yours to fix even when a passing test guards it — the test is then part of the defect, and correcting it is part of the fix. The boundary still holds the other way: a genuine capability or quality trade-off remains the operator's call. (cf. lessons 29, 38.)

67. **An integrity check on a multi-file artifact must cover the whole manifest — and reject whatever the manifest omits.** BlarAI's weight-integrity gate had verified only the primary model binary since Sprint 12; every other `.bin` (draft weights, embeddings, any future shard) loaded unverified, and nothing at all checked for files present on disk but absent from the manifest — so swapping the primary binary to a new filename and dropping a payload under the original name would have passed the gate clean. A manifest defines a set: the check must verify every member of that set *and* assert no extra members exist, or it is a false assurance with a silent injection window. (cf. lessons 3, 56.)

68. **A lock you cannot run green yet must still verify everything the missing resource does not gate — so its first real run is a confirmation, not a discovery.** The boot-cascade smoke and the GUI harness both need hardware a headless session lacks (the GPU, a display). Shipping them unrun and "verifying later" would make their first-ever run a debugging session — and a lock whose first run debugs it never locked anything. The discipline that makes them real: maximize what runs green stubbed now (the cascade invocation and teardown against a stubbed model; the XAML compiled with `dotnet build`; the tests proven collectable), leaving only the genuinely hardware-bound step for the deferred dev-machine run. "Built, not verified" is honest only when the un-verified part is precisely the part that needs the hardware. (cf. lessons 2, 56.)

69. **A gate checklist never re-verified against the code drifts in both directions.** The security-roadmap gate tracker listed two shipped controls (the TPM-signed audit stream, the armed runtime egress guard) as still-remaining, while implying a half-built one (weight integrity) was nearly done. An inherited tracker decays silently into both over- and under-statement; the only authoritative gate tracker is one walked against the code on disk, carrying file:line evidence, distinguishing DONE / PARTIAL / REMAINING, and naming the open decisions it finds (here: whether "attestation" means PCR hardware quotes or key-material validation) rather than smoothing them into uniform progress. (cf. lessons 1, 3.)

70. **A regression lock only locks if the gate that actually runs includes its scope.** The production-parity tests landed in `tests/integration/`, which the canonical Layer-A baseline command (`shared/ services/ launcher/`) excludes — so a green canonical run proves nothing about whether that lane passed. Where a lock lives and where the standing gate looks are two separate facts that must be reconciled, or the lock is decorative: the lock-before-modify guarantee for the next sprint holds only if the cascade gate is genuinely run in a `tests/`-inclusive scope before the first edit. (cf. lesson 56.)

71. **Version pins are containment, not supply-chain integrity.** Bounding the security-critical dependencies stops a silent major-version jump, but a PEP 508 pin cannot stop a malicious wheel served at the same version number — only a hashed lock file (`uv lock` / `pip-compile --generate-hashes`) closes that gap. The honest posture names the boundary it actually buys (version-containment now, hash-verification as the explicit next step) instead of letting a pin masquerade as a supply-chain guarantee. (cf. lesson 21.)

72. **Closing the debt a sprint exists to close is act-not-ask — escalate trade-offs, not the obvious completion of the stated goal.** Sprint 16's whole purpose was locks that *actually fire*; I found the production-parity lane (and, one layer worse, the security posture guards) sitting outside the gate that runs, named it lesson 70, and then *escalated the fix as a governance/baseline-scope decision* instead of doing it. The operator's correction was blunt — "why are you even considering not closing the debt fully? obviously that is the goal." He was right: a scope-or-baseline change with a genuine trade-off is the operator's call, but folding a just-built lock into the gate so it does its job has no trade-off — it *is* the work. Before escalating, ask whether the choice has a real downside the operator must weigh, or whether I am merely deferring the obvious completion of what the sprint set out to do. (cf. lesson 70; the Proactive-Defect-Fixing defect/decision boundary.)

73. **A long-"deferred" code path is where bugs hide from each other — activating it is a discovery exercise, not a one-liner.** The #615 guest boundary was carried for sprints as a single deferred item — "Windows AF_HYPERV addressing fix" — and the one word hid *two* opposite faults that had masked each other because nothing in the path had ever run: the shared transport had the right protocol but built a Linux `(cid, port)` address where Windows Hyper-V needs a `(VmId, ServiceId)` GUID pair, while the gateway had the right GUID address but omitted the protocol argument and raised `WSAEPROTOTYPE` before the address was ever used. Neither had thrown because the whole thing was dormant. When you finally exercise dormant code, budget for *plural* defects and find the empirical oracle — here the phase-2 `vsock_validation.json` from the real hardware was the spec — rather than trusting in-code comments that describe the intent, not the working form. Sibling to "committed ≠ done until verified": dormant ≠ correct until exercised. (cf. lessons 22, 46.)

74. **A `@hardware`/`@slow` tier must SKIP on every axis it needs, not just the headline one.** The production-boot model-loaded tier guarded only on `tpm_signer.is_available()` and then FAILED — not skipped — on a TPM-present dev box that simply had no models on disk, turning an incomplete-box run into a false red and defeating the marker's whole purpose. The point of the marker is that the tier is safe to run anywhere, which means the skip guard must cover the *full* prerequisite set — chip AND keys AND model material — and you only learn it does by actually running the tier under its own marker on an under-provisioned box rather than trusting the marker to spare you. A hardware tier you never tried to run is as untrustworthy as a stand-in tier you never ran green. (cf. lessons 30, 46.)

75. **Cross-stream seams that merge separately must connect by registration, not by import.** When two streams own opposite halves of one mechanism and merge at different times — here `egress_guard` (merges first) and the `exfil_screen` that plugs into it — the module that merges second imports the anchor and registers into it; the anchor never names the dependent. That keeps the dependency graph acyclic across the merge boundary and lets the first stream ship and test the whole seam end-to-end with a fake before the real implementation exists: the seam-lock test drives a fake screener through the real socket send path, so "the wiring works" is proven without the second stream's code. The tempting alternative — having the anchor reach into a known module path — creates a cycle the moment the dependent imports the anchor it lives inside. (cf. lessons 42, 46.)

76. **Put a carve-out where the rule actually fires, not where its name says it lives.** The brief and the SDV both located the egress deny in the PA `rule_engine` — and a `rule_engine.py` exists — but `DENY_EXTERNAL_NETWORK` is really RULE 3 of `DeterministicPolicyChecker.check()`, the single classmethod enforced at *both* the PA boundary and the AO tool loop. Amending the real chokepoint covered every call site with one change; putting the carve-out where the name pointed would have missed the AO path or forked the rule into two copies that drift, the more dangerous outcome in a security control. Verify the enforcement point on disk before trusting the module name in the brief — a filename is documentation, the call graph is the truth. (cf. lessons 43, 46, 47.)

77. **A red test is a claim about your test before it is a claim about the code.** The security-cascade rotation assertion went red and looked like a real fail-to-rotate defect — two byte-identical certs, identical serials — but the actual fault was reading the cert file *after* an in-place re-mint had overwritten it, diffing a file against itself. I nearly escalated a fail-to-rotate bug that did not exist. Scrutinise the harness's own timing and ordering before escalating a red as a code defect: the proactive-fix mandate is for *confirmed* defects, and an unconfirmed red can manufacture one that isn't there. (cf. lessons 30, 35.)

78. **Mechanism-exists is not disaster-tested — the missing catastrophe test is the actual deliverable.** A control can be fully built and even marked DONE in a tracker while the one test that proves it survives the specific disaster it exists for is missing. The offline DEK recovery wrap shipped in Sprint 14, yet every existing test faked "the chip is gone" by monkeypatching a *live* sealer to raise mid-process — the sealing key still one attribute-set away — which is not the hardware migration the recovery key exists for. What was missing, and what the criterion actually needed, was the fresh-machine lock: provision on machine A, carry only the keystore file and the recovery key to a machine B whose sealer cannot unseal at all, and decrypt real data byte-for-byte out the far side. When a control's whole reason to exist is a rare catastrophe (dead chip, hardware migration), the load-bearing test reconstructs that catastrophe end-to-end — fresh environment, only the surviving artifacts, real data out — not a monkeypatch of the failure symptom. (cf. lessons 46, 53.)

79. **A static config lock and a runtime posture guard are not redundant — they catch different regressions.** `test_secure_defaults.py` has asserted `dev_mode = false` in the shipped TOML since Sprint 12, but a static file-lock only proves the config on disk is secure; it cannot prove a boot in production posture actually *resolves* dev_mode to False at runtime, nor that the production-only fail-closed behaviours engage. An inverted override default, a stray env leak into the resolver, or a service quietly ignoring the override would sail past the static lock and surface only at the operator's terminal. When a security flag has a resolution path (env → override → mode default) and behaviours that branch on it, lock BOTH ends — the declared default AND the resolved runtime value — each paired with a dev-mode control so neither assertion can pass vacuously. (cf. lessons 44, 46.)

80. **A frozen-dataclass exception type masks an unexpected raise behind a confusing secondary error.** `ConfigResolutionError` is `@dataclass(frozen=True)` and also subclasses `ValueError`; when an unexpected one escapes a `pytest.raises`/`with` block, the machinery's attempt to attach a traceback hits the frozen wall and surfaces as `FrozenInstanceError: cannot assign to field '__traceback__'` — not the real message. The first time it bit, the true cause was a stricter-than-expected validator rejecting an incomplete dev-mode "control" config (which is a feature — the validator caught the thin fixture), but the frozen-type masking hid that behind the traceback-assignment error. When that signature appears, the real failure is an *unexpected* exception of a frozen type: read the wrapped value, don't chase the traceback assignment, and make control fixtures complete rather than minimal. (cf. lesson 30.)

81. **A posture test must own the outcome it asserts on, not borrow it from the environment.** Stream J's production-posture guard was twelve green tests in its isolated worktree, and the merge gate turned one red in nine seconds: a test that proved the AO security-material gate fail-closes "because the manifest isn't there" passed in a `git worktree` (no `models/` directory — the multi-gigabyte weights aren't in git) and failed on the dev machine's main checkout where the real manifest sits on disk. It was measuring whether the box happened to have models installed, not exercising the gate. The sharper half of the lesson came in the fix: even after forcing the precondition, pinning the *specific* failure code re-introduced the same host-dependence one layer down — a provisioned box trips on the JWT CA, a bare worktree on the manifest. Force the precondition explicitly AND assert the invariant (the gate is engaged / fail-closes), never the environment-decided detail. This is exactly why the gate is re-run on the merged main tree and a builder's in-worktree green is never trusted — the builder ran on a tree that structurally could not show the failure. (cf. lessons 44, 46, 79.)

82. **Verify an alarming conclusion against the evidence you already have before you escalate it — a symptom read correctly can still yield a wrong conclusion.** The #615 guest-boundary test failed `connect(): bad family`, and I read the symptom right — this venv's Python has no `socket.AF_HYPERV` — but escalated the wrong conclusion off it: that the approach was broken and wanted a ctypes rewrite, maybe a re-scope of the #598 gate. The LA's four-word push (*"didn't we test AF_HYPERV already?"*) was right: the Phase-2 evidence held a real host→Alpine AF_HYPERV round-trip, and the failure was a mundane interpreter-version gap (3.11.9 predates CPython's `AF_HYPERV`; the system 3.14 has it and the gateway that connects imports zero OpenVINO, so it runs ≥3.12 cleanly). The design had been sound the whole time. The same instinct, applied a beat later, paid off correctly: C2's `CERTIFICATE_VERIFY_FAILED` was called *test-tier* — not a real mTLS gap — precisely because the stubbed-GPU tier ran the identical handshake and passed, so the difference had to be environmental. An alarming red is a hypothesis about the system; price it against the evidence already on disk (a prior validation artifact, a passing sibling tier, a one-line probe) before you spend the LA's credibility on it. The proactive-fix mandate is for *confirmed* defects; an unconfirmed alarm can manufacture an architecture crisis that isn't there. (cf. lessons 66, 77, 39.)

83. **A WinUI element is in the UI-Automation tree only when THREE separate conditions hold — and when one is missing, instrument the tree before theorising.** A folder of `winui`+`slow` GUI critical-path tests had collected green and *run never* (they skip when `BlarAI.Desktop.exe` is absent, which is every headless gate run), so they were hypotheses, not regression locks — and the first time the screen actually drove them, six failed, all in the harness, none in the app. The fix took three foreground runs because the UIA tree is a *projection*, not the visual tree or the XAML, gated on three preconditions a hand-driven dev session hides from you: *realization* (the window must be foreground with a render pass run — a backgrounded WinUI 3 window has an unrealized tree); *peer existence* (only Controls get an AutomationPeer — `Grid`/`StackPanel`/`Border` get none and are never findable by AutomationId); and *non-empty/visible projection* (an empty or Collapsed `ListView` is not in the tree at all). Each run taught me one precondition and I mistook it for the whole; the screen corrected me three times, twice by failing a "fix" I had *just* shipped — a re-anchor onto a peer-less `RootGrid` (my own no-peer bug, fixing one no-peer assertion by introducing another) and a self-contradictory `MessagesList.exists()` on the empty-state greeting. Anchor on a Control or on child text (a `TextBlock` surfaces its text as the UIA Name), never on a container; for a templated element whose `x:Name` WinUI does not project, anchor on its rendered text; and when an element is missing, dump the AutomationId-bearing descendants and the item count rather than guessing virtualization. A GUI harness's real subject-under-test includes the projection preconditions its author never sees because their own session has the window in hand — and printing the tree is the cheapest way to stop guessing at them. (cf. lessons 2, 3, 32.)

84. **A sequencing or packing choice that changes nothing built is the orchestrator's to own, not a fork to hand the operator — and offering "or we defer half" reads as proposing they stop short.** Preparing Sprint 17 I put two "decisions" to the operator; one was genuinely his (the measured-boot attestation scope, ADR-028), but the other — build the egress machinery this wave or hold it to the next — changed nothing that gets built and no answer quality. It was sequencing, how disjoint work packs across waves, and dressing it as "…or we could defer half to Sprint 18" landed exactly as it should have: as suggesting he stop short of finishing. The honest tell: if the alternative you are offering is "do less of the agreed work later," you are not surfacing a decision, you are proposing to stop short — decide it yourself and keep moving. (cf. lesson 72.)

85. **Schedule the human's understanding pass BEFORE the irreversible decision, not after — with a remediation loop so what they learn can still change the outcome.** The #612 capstone security presentation was sequenced after the #598 air-gap-removal sign-off, the way its ticket read it; the operator moved it before, as his real deep-dive into the residual risks of going internet-facing, with any work it surfaces tasked and resolved first. A briefing after the gate can only justify a decision already taken; before it, with the authority to surface and resolve new work, it makes the sign-off an informed act rather than a ceremony. The governance counterpart to "committed ≠ done until verified": *signed ≠ safe until understood*. (cf. lesson 84.)

86. **Automate by default; a human step must be proven necessary, not inherited from a runbook's wording.** The Sprint-17 on-chip session was scripted "LA-driven, one command at a time" — correct only for a guiding session with no hardware access — and I handed that framing to the execution agent that was running on the dev box with full shell access, so the operator was made to run tests the agent could run itself ("my time is being wasted, again"). An agent on hardware it can reach runs every step itself; a step goes to the human only when proven to need human hands (an elevation the shell cannot obtain, a physical action, a presence assertion), never because a runbook is phrased for a human operator, and one-time-access blockers like shell elevation are solved once rather than routed through the human each command. The burden of proof is on routing a step out to the human, not on automating it. (Tracked as Vikunja #629.)

87. **A process leak that makes tests defensively skip looks like correct behaviour — verify the resource was released by a separate mechanism, because a stable skip count is not a safety net.** The standing gate drifted between 2342/0 and ~2333/9 with no code change: the WinUI harness's bare `proc.terminate()` killed `BlarAI.Desktop.exe` but orphaned the Python backend child it spawned, which kept holding AO loopback port 5001, so the *next* gate run's boot-cascade tests saw a held port and skipped — correctly, defensively, silently. The defect was not in the skip logic; it was upstream, in the teardown that should have released the port. A skip-guard on a shared resource needs its *release* verified by a separate fail-loud probe — here a session-scoped autouse detector that fires only on a free→held delta (the one transition that means "this session leaked"), staying a silent no-op on the clean free→free run so it never moves the baseline. Never trust the skip count to stay put. (cf. lessons 30, 35, 46.)

88. **A `dev_mode` round-trip proves the routing, not the posture — and the gap between them is the seam production burns on.** Two existing tests touched the gateway→AO path and both stepped around the hard part: one stubbed the GPU and ran `dev_mode=True` (no mTLS), the other drove the real 14B through an in-process fake transport (no socket, no handshake). Each was honest about its coverage; neither exercised the *composed* production walk — real socket, real `CERT_REQUIRED` mTLS, the signature-verified manifest boot, real generation, real PGOV. `dev_mode=True` is the single line that makes the easy tests easy and incomplete. The production-posture tier (`dev_mode=False`, real mTLS, `require_signed_manifest=true`) is a *separate lock*; build it explicitly and mark it `@hardware` so the gate is honest about what has actually run on the chip. When it first ran, the production boot fail-closed correctly on a test that pointed its model path under a tmp dir — the signed-manifest gate doing its job, proving the gate works before the round-trip did. (cf. lessons 44, 79, 81.)

89. **Reference paths are for reading; writes go to the worktree — verify the file landed where the branch lives, not just that the branch is right.** The worktree hazard is not only `git checkout` switching the main branch (lesson 52); it is also absolute paths from a brief — correct for *reading* the main checkout — silently becoming *write* targets. Stream M wrote both new test files into the main checkout it was told not to touch; collection-against-the-worktree surfaced it, and the non-destructive recovery (copy into the worktree, SHA-verify byte-identity, preserve a backup, then restore main) kept the never-discard rule. A `pwd`/`ls` of the file you just wrote, against the worktree root, catches it before commit — the filesystem-path counterpart of the `git branch --show-current == main` guard. (cf. lessons 35, 52.)

90. **A mocked seam plus a standalone test of the seamed component do not cover the path that connects them — only a test that drives both in sequence does.** Two Sprint-18 streams surfaced the same blind spot independently. C3: the AO unit tests mock `SemanticRouter.classify()` and the one real-router test exercises it standalone — so the cross-service router→AO path had zero coverage, and grepping the AO turn revealed it is not even wired (the router is called nowhere in `_handle_connection`; ticketed #632). C4: every TUI test drove `BlarAIApp` against a mock gateway — so `BlarAIApp`→`TransportGateway`→AO had never run as a system. "Component covered" and "seam covered" are different claims, and the gap between them is silent until one side changes. The lock is a single test that exercises both real components in sequence on the real path. (cf. lessons 3, 30, 46.)

91. **Reconcile a large audit with parallel adversarial verifiers, then resolve their disagreements on disk — never average the verdicts.** Fanning the #612 capstone status-mapping out to independent read-only verifiers turned their conflicts into signal: the tool-call allowlist (read FIXED by one, STILL-OPEN by another) and the exfil-screen (built vs unwired) each resolved, on disk, to a more precise truth than any single serial pass would have produced — the allowlist is the four built tools while ten unbuilt names linger only in a removal-comment, and the screen is *doubly* dormant (empty allowlist AND unregistered screener). The disagreement is where the real status hides; chase it to the file, do not split the difference. (cf. lessons 3, 21, 69.)

92. **The bug lives in the seam, not the parts.** Two components that are each individually correct can still leave a hole where they meet if the contract between them was never specified — in #638's capability-JWT containment a 5s nonce-seen window and a 30s token were both reasonable in isolation and together left a 25s replay window, while the `jti` spent-set the ticket prescribed was redundant because single-use was *already* enforced by the nonce. When a control depends on two independently-defaulted values agreeing, tie them together at one source of truth (`JWT_VALIDITY_SECONDS` in `shared/constants.py`) rather than trusting them to stay in step — and when a ticket names the *mechanism*, verify that mechanism against the code before building it, because the property may already hold by another means and the real defect may be hiding in the seam. (cf. lessons 6, 64.)

93. **Wiring a guard onto a hot path means asking what legitimate traffic it will see, not just what it is meant to catch.** A content screen pointed at a channel that carries the system's own secrets will fire on them: wiring #634's exfil-screen as the ticket literally said would have tripped the kill-switch on BlarAI's internal IPC (the loopback PA→AO gateway and the AF_HYPERV vsock carry the runtime's own capability JWTs and the user's PII), a self-inflicted denial of service dressed as a security control — and a verdict contract read by truthiness compounds it, because the real `exfil_screen.screen()` returns an always-truthy frozen `Detection` whose *clean* result the old normaliser would have read as a hit. Scope the guard to the traffic that actually crosses the boundary (tag a socket only after a successful connect to an external-allowlisted endpoint; never loopback, vsock, or accept/dup sockets), and read verdicts by their explicit flag — never by truthiness. (cf. lessons 22, 27, 75.)

94. **A merged capability is not an activated one — say which it is.** A control can be fully built, tested, and merged and still change nothing in production until a deliberate wiring step turns it on: #637's data-map DACL hardening tightens only for *other* principals (owner-preserving, so it can never brick the app), and #639's ESCALATE consumer is fail-closed on every path and registers nothing at import, so an ESCALATE is denied exactly as today until a verifier is wired at AO startup. The honest gate record distinguishes "the path exists and is proven" from "the path is live," because conflating the two is how a system gets credited for a protection it is not yet running. (cf. lessons 46, 60.)

95. **An activated control is not an exercised one — verify it against the real producers, not assumed ones.** A consumer can be built, wired, activated, and even live-demonstrated and still never fire in production because nothing yet produces its input: #639's ESCALATE consumer was activated on the TUI surface and live-demoed against a forced sample, but none of BlarAI's four shipped tools (time/date/day-of-week/calculate) can emit an ESCALATE at all — `_adjudicate_tool_dispatch` builds `resource="tool:<name>"` / verb EXECUTE, which trips none of the seven ESCALATE rules that key on file paths, large writes, cross-agent params, and unverified-code flags only a *future* file or network tool will carry. The honest record distinguishes "the path works" from "the path is exercised by real traffic," and a live-verify must use the actual toolset — an assumed trigger (a filesystem tool that does not exist) proves nothing. (cf. lessons 36, 46, 94.)

96. **Prove the property on the load-bearing artifact, not an equivalent of it.** A test that proves non-exportability on an ephemeral key built the same way is strong evidence — but a trust-root claim is about *those four specific keys* (the PA JWT signer, the audit-stream signer, the weight-manifest signer, and the RSA DEK-seal key), and "built the same way" is an inference a determined auditor will rightly poke. `verify_trust_root.py` closed it by pointing the exact vetted `NCryptExportKey` private-blob call at the *production* key handles and asserting the real STMicroelectronics TPM refuses every one — a refused export reads and mutates nothing — turning verified-by-equivalence into a per-key fact and grading all four `VERIFIED_LIVE` (resident, functional, non-exportable). When the direct proof is cheap and read-only, do the direct proof; the gap between "reads well" and "holds up" is exactly that last mile. (cf. lessons 18, 23, 51.)

97. **A security control must not claim more than it delivers.** When code documents a protection — zeroization, isolation, validation — the comment is a claim a future reader will trust, and if the implementation cannot honour it the honest move is to say so plainly and point at where the real mitigation is tracked. The #611 study found the DEK break-glass recovery ceremony "zeroizing" recovered key material with `recovery_key = bytes(len(recovery_key))`, which merely rebinds the name to a fresh zero buffer and leaves the original immutable `bytes` in the heap until GC; the governance-honest fix was to correct the comments to the true (weak) guarantee rather than change the code to *look* like it zeroizes, because in pure Python you cannot overwrite an immutable backing store and re-asserting the false claim with more confidence is the worse move. An overclaiming control is worse than an absent one, because it buys false confidence. (cf. lessons 47, 51.)

98. **When a read contradicts the premise of your task, the read wins — pause before you write.** A grep run to *do* a #612 follow-up task instead revealed the task was already done — by a concurrent session editing the same `make_outline.py` in the main tree, uncommitted, with a live preview server on another port; proceeding on the original premise would have re-applied a ticket sweep on top of another session's work and very possibly raced it into a lost edit, the exact failure this project has paid for before. Treat a result that contradicts your assumption as a stop signal — surface precisely what you found (the uncommitted diff, the foreign preview server, the worktrees) and confirm how to coordinate before mutating shared state. (cf. lessons 9, 35, 41.)

99. **A decision is only a decision if the options differ on security, capability, or quality — otherwise it is mechanics the builder owns.** Handing a novice operator an implementation choice with no governance delta inverts the responsibility the agent is paid to carry, and costs the operator exactly the involvement he is trying to minimize: scoping the #649 Windows-Hello biometric verifier, the route (a tiny C# helper subprocess versus an in-process WinRT binding) changed nothing the operator governs, and the only fork with a real posture delta — adding a new runtime dependency — had an obvious answer under this project's supply-chain posture (don't; the `net8.0-windows10.0.19041.0` `UserConsentVerifier` the WinUI already targets needs no new wheel). The check is cheap: name what each option changes for the operator; if the answer is "nothing he governs," decide, state the call, and keep moving. (cf. lessons 29, 38, 66.)

100. **An "unbounded by default" comment is often an unclosed gap wearing a decision's clothes.** The audit log's docstring framed never-rotating as a deliberate forensic stance, and it half was — but the load-bearing half (keep all the metadata-only records, which *are* the AIGP governance evidence and a rounding error on disk) was sound while the literal half (in one file, all in RAM, re-read whole every boot) was just deferred work that grew the working set without bound. #607's rotate-and-retain seals the active `audit.jsonl` into gzip segments pinned by a signed segment index so tamper-evidence spans files, and rejected time-based purge / FIFO size-cap / the status-quo single file on the record in ADR-029. When a docstring asserts a design choice that also happens to be the path of least resistance, check whether the *valuable* part of the claim and the *convenient* part are actually the same claim — here they were not. (cf. lessons 17, 39, 69.)

101. **Strip authority with an allowlist, not a denylist — and know which lever you're pulling.** Reducing a process's power should default to "remove everything not explicitly needed," so an unanticipated capability is dropped by default instead of surviving a hard-coded remove-list (fail-safe beats fail-complete): #652 strips every Windows token privilege off the elevated launcher — which hosts the PA, AO, and model as in-process threads, so every privilege it holds the LLM tool loop holds — down to a tiny keep-allowlist via `SE_PRIVILEGE_REMOVED` (gone for the process lifetime, unrecoverable from inside). And pick the right lever for the topology: integrity-lowering is per-process and useless for in-process worker threads (the launcher must stay High to drive Hyper-V), while privilege-stripping shrinks what the High-integrity process may do without changing the integrity it runs at. Keep-on-doubt for any privilege whose removal could *silently* disable a sibling fail-safe control — a skipped #637 DACL hardening reads identically to a successful one in the logs, so `SeRestore`/`SeTakeOwnership` were kept and the removal flagged as a post-live-verify tightening — that is a decision to weigh, not a defect to reflex-fix. (cf. lessons 20, 66, 94.)

102. **Reuse the project's canonical approval primitive instead of minting a second one.** When a new control needs "a human must approve this" — re-arming the egress kill-switch (#653) after it auto-trips — the right move is to consume the *same* `ApprovalVerifier` / `ApprovalResult` / bounded-wait machinery the ESCALATE consumer (#639) and the Windows-Hello verifier (#649) already established, not to invent an egress-specific auth path: re-arming is the same shape of single-user-local, human-in-the-loop approve/deny, and a second trust surface is a second thing to get wrong and to drift out of step with the one that already has 40 tests behind it. One audited approval primitive, reused, is the governance-grade choice — paid for by `egress_rearm` deliberately taking a hard dependency on `escalation_consent`'s shapes rather than rolling its own, fail-closed so only an explicit `approved is True` clears the latch and a missing verifier stays LOCKED. (cf. lessons 40, 75, 99.)

103. **Verify the vocabulary you teach against the current source before you teach it.** Domain knowledge in an agent's head has a shelf life, and a certification syllabus, a framework, or an API can restructure under you between training and use: the lessons-teaching deck (#650) "knew" the IAPP AIGP exam as seven domains, but live-source verification showed the Body of Knowledge had been restructured to *four* domains in v2.0.1 (Feb 2025), with v2.1 effective 2026-02-02 adding ISO/IEC 42005 and agentic architectures — so every framework claim would have taught a one-restructure-stale exam. The staleness was caught only because verification was made a *gate* (a pinned `aigp_vocabulary.md` enforced mechanically by `validate_acts.py`, alongside an anti-hallucination check that every cited entry and SHA exists verbatim in the journal), not a courtesy. An instance of the evidence-rot lesson applied to the teacher's own knowledge, and of verify-the-premise applied to oneself. (cf. lessons 39, 82.)

104. **An audit log is the one at-rest store where signed-plaintext beats encryption.** The project's default is encrypt-content-at-rest (ADR-025: sessions + substrate are DEK-sealed), and the reflex — reinforced by a standing lean toward strict posture — is to extend it to everything. A forensic audit log inverts that trade-off, and #607's sealed segments were ratified signed-plaintext + owner-only-DACL'd rather than DEK-encrypted. An audit trail's value is *independent verifiability*: it must be checkable by a reviewer (or future-you) and must survive the loss or compromise of the very key whose use it records — encrypting it couples the witness to the thing it witnesses, so the one record meant to outlive a key incident would die with the key. That holds *because* the records are metadata (decisions, resource identifiers, hashes), not content, so the confidentiality encryption buys is small; the principle has a named expiry — revisit when post-#556 tools start writing sensitive paths/URLs into the log. The governance-grade move was to *escalate* the deviation from encrypt-by-default rather than silently keep plaintext or reflexively encrypt, and to put the revisit trigger on the record (ADR-029). (cf. lessons 97, 100.)

105. **Demonstrating a capability is not deciding to deploy it — a proof-of-concept that ends in a deliberate "no" is a success, not a waste.** The 2026-06-09 PoC proved PCR-bound key-sealing works on this TPM (seal under `PolicyPCR`; the chip refuses the mismatched unseal); the right use of that proof was to let the LA *decline* it on 2026-06-10 with the cost fully weighed, not to read "it works" as "ship it." PCR-binding the production keys is feasible *and* operationally wrong here — it would brick the trust root on every firmware/OS update for a sliver of already-mitigated physical-threat reduction — so the keys stay hardware-resident + non-exportable + DACL-locked, not PCR-gated. Separate feasibility from desirability: a capability demo's highest governance value is often the informed *negative* it enables, recorded with the path-not-taken (ADR-028 Am.1). (cf. lessons 95, 104.)

106. **The only secret you can truly erase is a mutable one.** Python `bytes`/`str` are immutable — rebinding their name to zeros dereferences the secret but leaves its bytes in the heap, so "zeroization" of a `bytes` key is a comment, not a control. A numpy array is mutable, so `arr[:] = 0` genuinely overwrites it: #611's embedding cache (the largest, longest-lived, semantically-invertible plaintext-derived blob in RAM) was worth attacking precisely because it is the rare in-RAM secret we *can* scrub, where the DEK is not. When you claim a memory-residency control, the honest test is whether a *captured reference* to the buffer reads zero afterward — not whether the container merely dropped it. (cf. lessons 97, 104.)

107. **A "gate-blocking residual" must be reconciled against disk before it drives the plan.** Decision-support inventories — a residual register, a finish-list, a deep-dive's "one remaining proof" — drift from the code the moment work lands faster than the document is reconciled. The #612 deep-dive graded the controlled egress exercise (A.3) "BUILT-DORMANT, GATE-BLOCKING, the one remaining proof"; on disk it was already built (#643) AND passing in the standing gate, covering its exact criteria against real non-loopback traffic. Treating the residual as buildable work would have rebuilt a green proof; reconciling it to disk first turned "the gate's last blocker" into "already done, plus a genuine residual the register had folded into it" (the AO-tool-driven full chain, un-exercisable until an egress tool exists). Verify the gate state against the tree before you act on the inventory — and when they disagree, that drift is itself gate-relevant. (cf. lessons 98, 70.)

108. **A security artifact built on an active tree is stale the day after it ships** — so a pre-decision deep-dive's real value is not re-reading the deck but re-grounding every load-bearing claim against the code on disk. Here that reconciliation collapsed the gate-blocking list from a finish-list to one proof; had I presented the deck as written, the LA would have weighed a decision against three findings that were already false. Verify against disk, not against the narrative — especially your own.

109. **Sanitization order is a verdict-integrity property, not a style choice** — detect on what was actually submitted, then strip; strip-then-detect launders the quarantine verdict, and normalization must run before detection because invisible characters are a scanner-evasion vector. Lock the ordering with tests that attack it.

110. **A deterministic hash of secret content is content metadata, not a label** — anywhere ciphertext is the design (at rest, in a signed plaintext log), content-derived hashes must be keyed, and "it's only a hash" is the warning sign to check for an oracle.

111. **When you add the first instance of a capability a system was built to forbid, build the seam before the feature** — one guarded door every future caller must pass through, not a socket per feature — and encode the new posture as an ADR amendment, not just code, when it moves where a human gate sits. Two corollaries paid for themselves here: (1) a registration seam with **no** default wired is the honest fail-closed shape when no real in-process authority exists to call — ship the documented TODO, never a fabricated verdict; and (2) a fail-closed control will fight the legitimate use it's meant to allow the moment a real client behaves normally (an HTTP client connects to a resolved IP, not a name) — the fix is to teach the control about the legitimate path narrowly (pin the resolved IP, at the allowlisted port only, dropped on revoke) without ever loosening deny-by-default for everything else. Close an SSRF in both the layer that resolves and the layer that connects — a named host that resolves internal must be refused at the resolver (don't pin the internal IP) *and* re-checked at the door (resolve-and-recheck before any widen), because neither layer alone covers the other's blind spot. And when a fix narrows a class of attack without closing it — DNS rebinding here — name the residual and the robust fix (a peer-IP-validating transport) in code and journal rather than implying full coverage; a half-closed hole you've documented is honest, a half-closed hole you've implied is closed is a trap for the next reader.

112. **Audit the request side of every interception seam, not just the reply side** — a command surface that persists its own raw input as a user turn re-injects that input into future prompts with user-level trust, no matter how well the downstream pipeline guards the same content.

113. **Parallel sessions that each author "Amendment N" to the same ADR collide silently** — same number, disjoint content, both ratified. Like journal lesson numbers, amendment numbers want assignment at integration time, not at authorship time; until that discipline exists, the integrator must expect to unify competing same-numbered amendments into one record and document the unification in the register, never renumber a ratified limb after the fact.

114. **A test double must preserve the variance the assertion depends on.** My first stub embedding session was mathematically separable in (token, dimension), so L2 normalisation cancelled exactly the truncation effect the 128-vs-512 regression lock existed to detect — the test was green against a broken property until the stub itself was made non-separable. When a lock guards a numeric property, prove the lock can fail before trusting that it passes.

115. **When an ADR rejects an alternative that might come back, *hold* it explicitly** — name the option and the condition that would warrant it — rather than just rejecting it. ADR-027 held its hybrid per-action-consent option with the exact use case in a parenthetical; when that feature arrived three days later, the posture change cost one amendment with the reasoning pre-built, instead of a fresh debate over a policy that looked closed. The held option is a designed seam in the decision record, the same way a private search method is a designed seam in the code.

116. **An ownership-conditional cleanup ("only release what I acquired") is a ratchet whenever the ownership flag can latch off from a transient event** — once it does, the resource leaks forever and every subsequent clean run *looks* correct while skipping the release. Prefer deciding teardown from the resource's *current observable state* ("is it running now?") over a remembered "did I start it?" provenance bit; keep the provenance-based mode only as an explicit opt-in for the genuine shared-ownership case, and make the unrecognised/default path fall to *release*, never to *retain*. The signature to watch for is a teardown log that completes without the release line it should contain.

117. **A green fit-check can certify an unreadable artifact — gate on the property, not the mechanism.** The deck's render audit went through three definitions of "correct": first it measured ink against a scrollable fold (false alarms), then box-escape (which a readability-destroying shrink passes perfectly), and only third the actual property — every diagram renders at a scale a human can read, with overflow allowed where overflow is the better trade. A verification check is itself a design with a failure mode: when it flags what doesn't matter or blesses what does, iterate the check against the real consumer's experience (a screenshot, a read) until the thing it gates is the thing you care about. (Kin to lessons 21 and 30, applied to the test's own definition.)

118. **When an exception class subclasses a builtin, every `except` of the subclass is a claim that the base class cannot reach that spot** — and a `# pragma: no cover` comment asserting unreachability deserves adversarial review, not exemption from coverage. The catch must be at least as wide as the raise surface of the call it guards; here `except ParseChannelError` under a function whose framer raises plain `ValueError` turned one hostile frame into a dead listener, found only by adversarial review AFTER a green 131-test merge.

119. **A passing test suite proves the code does what the tests ask, never that the tests ask the right question.** The loopback tests were green because they exercised the loopback path; the boundary hop had no test because the API had no way to *name* it. When two independent properties (here: transport family and mTLS on/off) are welded to one switch, the suite can be exhaustive over that switch and still have a blind spot the size of the missing fourth combination. The live round-trip wasn't redundant with the unit tests — it was the only thing that could fail in the dimension the unit tests couldn't address. Build the explicit opt-in *before* you need it, so the thing you must test has a name.

120. **When a transport's bootstrap dependency lives on the far side of the transport itself, stop trying to automate the first hop** — design the smallest possible permanent manual step, then make everything after it automatic. The P5-era deploy treated "manual-in-guest" as a temporary embarrassment and left the channel unproven for three months; naming the one-time console bootstrap as a *designed* step (and proving the predicate and protocol around it with tests) is what makes the remaining automation trustworthy.

121. **At a security boundary, prefer a wire format with exactly one legal encoding per payload.** When the receiver can compute the single valid shape of every frame from the first frame's declaration, validation degenerates to equality checks, every attack or corruption class becomes a first-frame or per-frame rejection, and "tolerant reader" — a virtue in friendly protocols — stops being a euphemism for "unbounded trust." The 64 KB cap looked like an obstacle; it was the design pressure that produced a channel where nothing about a message is negotiable after its first frame.

122. **A runtime pinned below a capability its own new feature needs is a version-bridge problem, not a migration problem.** When the validated interpreter (here 3.11, for OpenVINO + the 14B weights) lacks a stdlib feature a low-frequency control path needs (`socket.AF_HYPERV`, 3.12+), isolate the jump in a short-lived out-of-process helper running the newer interpreter rather than migrating the whole stack or hand-marshalling the syscall in `ctypes`. Keep the in-process fast path so the bridge self-retires when the runtime catches up, and make the helper's import surface the forcing function for the air-gap (stdlib + pure shared IPC only, no network client). Paid for by the #655 Stage C `AF_HYPERV` 3.11/3.12 mismatch.

123. **A control proven in a harness that relaxes its own environment is not proven.** The most dangerous defect of this arc was invisible from the design and from every green test — it lived only in what the *armed* runtime does to the door's own DNS call, a path the proof harness had disarmed away. Verify-before-build is not just reading the code that will run; it is running it in the posture it ships into. A door that opens cleanly when the guard is off has told you nothing about the guard.

124. **Build the door, but make opening it a separate, single, reviewed act.** When a capability is dangerous to activate (here: the first real egress off an air-gapped box), wire it fully and unit-test the wiring, but leave the activation as one uncalled line with the trade-off named — never let "I built it" quietly become "I turned it on." The egress door denies until a consumer registers an adjudicator; building that adjudicator and declining to register it kept the air-gap provably intact while the corridor behind it was finished.

125. **Your confident technical answer is a hypothesis until the disk confirms it — and verifying often reveals the work was already done.** I told the LA "reuse the vsock connection" with conviction; the codebase had no such path and the real door-opening mechanism (the ADR-027 §2 carve-out) had been built and left dormant a sprint earlier. Reading the live adjudication path before wiring to it corrected a wrong build AND turned a "build the adjudicator" task into a "50-line adapter over an existing control" task. Don't build on remembered architecture; the cheapest sprint is the one whose mechanism already shipped.

126. **A host/guest runbook is unproven until the real console runs it.** The defects here weren't logic errors a reviewer could spot — they were environment truths (in-place rebuild strips the per-VM ACL; cold busybox won't auto-detect iso9660) that no mocked test encodes. Treat a freshly written boundary runbook as a hypothesis, not a procedure, until a real ceremony has executed every line of it; and when the ceremony finds a defect, the fix lands in the runbook in the same breath, with the failure left visible.

127. **Rewriting a file the hypervisor holds open creates a new object — re-grant its ACLs, because the permissions that made it work do not follow the path.** And disable automatic checkpoints on any VM whose state you update out-of-band: a silent Revert undoing a provision is a success-shaped failure.

128. **Dynamic Memory reclaim is set by the floor and the measured demand, not by the ceiling.** Size the minimum to the real steady-state demand (with headroom) and a generous maximum is free — a cap is never paid for unless demand actually rises to it, so "tighten the max" is a non-move when the floor already dominates. Paid-for corollary: a too-early snapshot of a just-booted guest can wear the exact face of a hardware-absent failure — design the measurement window to outlive the warm-up transient before trusting what it shows (the same verify-the-alarm-before-escalating instinct, in measurement form).

129. **When a feature carries the same content in two fields — an edit buffer and a render field — a visibility toggle that swaps which one is shown will silently display the stale one unless you sync on the transition.** A green headless build and a tamper-evident audit chain can both be correct while the screen the human approves against lies; the live-verify caveat on a `@winui` feature is load-bearing, not boilerplate.

130. **A regex escaper is only as safe as its parity with the consumer that renders its output.** The alt-escape bypass passed three layers of green tests because the escaper's URL grammar (`[^)\s]`) silently disagreed with the WinUI renderer's (`[^)]`) by one whitespace class, and a comment asserting they matched made the gap invisible. When code in one module sanitizes for a grammar parsed in another, the two grammars are a shared contract that must be tested against each other, not described in prose — and an adversarial reader told to *refute* the security claim is the screen that catches what same-module tests, written by the same author who set the grammar, structurally cannot. A sanitizer has two failure directions; a review that walks only one is half a review. My seven-dimension pass asked "can something dangerous survive?" and found the bypass, but never asked "does something benign get destroyed?" — so it missed an escape pass silently rewriting every legitimate relative / `#anchor` / `mailto:` link in real article bodies. Neutralization keyed to "not on the allowlist" over-eats; neutralization keyed to "carries a dangerous scheme" is the correct shape. When you adversarially review a filter, commission the false-positive lens (legit input mangled) as deliberately as the false-negative one (bad input survives).

131. **Before inventing a new transport for a feature, check whether an existing idempotent/dedup contract already carries it.** The editable-approve looked like it needed a new "approve-with-body" IPC verb; in fact the existing submit→dedup-replace→approve path already substitutes the stored body, so the edited-body transport added zero new AO wire surface — one additive metadata field on the existing INGEST_SUBMIT verb (the only new RPC method, `ingest_decide`, is a host-local UI-decision channel, not a body path). The cheapest correct change reused a contract that was already load-bearing — and dodged a second plaintext-on-IPC path the architecture forbids.

132. **When two individually-correct controls collide, add a tier, don't lower a bar.** A real false-positive (faithful recall flagged as a leak) tempts the cheap global fix — promote to trusted, disable the control, or raise the threshold — and every one of those buys the usability win by quietly dropping protection somewhere else (the action-lock, the whole detector, or the catch on real echoes). The durable fix names the *new combination the existing taxonomy couldn't express* — here "untrusted for the lock, but leak-exempt" — and gives it its own provenance tier, so the control narrows to its true target instead of being weakened. The 2026-06-04 trusted-content exemption and this 2026-06-14 knowledge exemption are the same move twice: the leakage axis is provenance, never a score.

133. **Verify the backend before scoping the backend.** Two of this session's "frontend + backend" items were actually frontend-only — the delete and rename paths were fully built and tested, just unwired. The reflex to scope a symmetric build (UI + matching backend) wastes the half that already exists; a five-minute read of the store + dispatcher + IPC client settled it. Pairs with the "committed ≠ done until wired" lesson from the other direction: here the code was done *and* tested but unreachable, so the only missing work was the door. And: an unload that drops a Python reference is not proof of RAM reclaim — the reclaim promise lives in the C++/driver finalizer, not the Python `del` + `gc.collect()`. Build the unload, prove the object is dropped headlessly, but make the RAM-delta a measured gate (a real-hardware run with a pass threshold) — and flag it as such up front, not after building, so a no-op unload can never masquerade as a working one.

134. **A regex's parity with its consumer includes the regex engine's own dialect, not just the pattern.** The forged-id gate `^[0-9a-f]{32}$` was a correct pattern and a wrong .NET regex — there `$` matches before a trailing `\n`, so a 33-character id carrying a control char passed a gate whose contract said "nothing trailing." This is the third instance of the grammar-mismatch class (after the alt-escape whitespace bypass and the control-char fail-open), and it sharpens the rule: when a sanitizer's output is parsed by a grammar in another module — or its own match is anchored by an engine in another language — the metacharacter *semantics* are part of the contract. Prefer the absolute anchors (`\A…\z`) that mean the same thing in every engine over the line anchors (`^…$`) whose newline behavior is per-engine, and have an adversarial reader who knows the *target* engine check it.

135. **Fail-closed is only *proven* when you watch the control refuse under the exact failure it was built for.** Reasoning about the deny path, and even a clean happy-path go-live, both leave it unexercised; it took a no-network accident mid-fetch to show the door actually slams shut on a dead wire. Design the rehearsal to include the failure, or wait for the accident to do it for you.

136. **One threshold can need two predicates, split by who owns the fail-closed.** When the same bound — here the UC-003 decompression-bomb ceiling — is enforced at a permissive boundary and a strict one, the "can't measure it" case resolves *oppositely*: the strict gate drops (cannot prove the image safe), the affirmative check returns "not flagged" (cannot prove it unsafe). The coordinator's fetch-time gate fails closed to DROP on an unreadable header; the at-rest store mirror returns "not provably too big" on the same input, because it already passed the magic-byte gate. Conflating them into one helper to look DRY forces a bad choice — break every existing `store_image` test or leave the exact hole TD-4 was meant to close. Name both predicates and write the boundary into both docstrings, or a later "simplification" re-introduces the hole. (cf. lessons 34, 63.)

137. **A policy decision rewrites the test surface, not just the code.** CD-1 — that pasted and file ingests fetch *no* remote images, only URL-ingested articles do — made the old paste-mode image-corridor tests encode an impossible scenario, because paste was simply the only mode wired when that corridor was first built. The faithful move was not to delete that coverage but to move it to the now-valid URL path and add the new-rule (paste/file no-fetch) tests beside it. Deleting them loses real coverage; leaving them green-but-impossible is worse, because a passing test for a scenario the system no longer permits is a false witness. When a rule change invalidates a path, migrate its tests to the path that is now real and add the tests the new rule earns. (cf. lessons 36, 90.)

138. **When a security primitive doesn't fit the new artifact, add a sibling — don't bend the locked one.** The weight-integrity sweep (`verify_all_manifest_entries`) is load-bearing for the 14B/PA/draft boot path and globs flat `*.bin` by bare filename; a diffusers-OpenVINO model puts its weights in subdirectories with the bare name `openvino_model.bin` repeating across five of them, so the flat function would find nothing and collide on the duplicates. The cheap moves — skip the verify, or widen the flat globber and risk the boot path — trade a real fail-closed guarantee for convenience. A clearly-named sibling (`verify_all_manifest_entries_nested`) that reuses the proven `load_manifest`/`compute_sha256` primitives and keeps the recursive extra-file rejection holds both layouts honest and leaves the locked path byte-identical. Flag the sibling as a deviation-of-necessity rather than pretend the plan covered it. (cf. lessons 64, 67.)

139. **Name the control you actually have, not the one that sounds reassuring.** The instinct on an "uncensored image generator" is to bolt on a content classifier so the project can say it has a guardrail; the honest answer is that a *local* classifier cannot hold the one boundary that matters (no distributable hash database for it, a prompt denylist bypassable three ways while it false-refuses legitimate prompts) and would be memory-costly against a tight ceiling and privacy-invasive against the operator's own output. ADR-033 says the true thing instead, and the go-live ceremony makes it mean something: the boundary is named explicitly, it is a recorded ACCEPTED-RISK the operator *signs sole responsibility* for, the control is governance plus a one-time attestation, and the consequences are bounded structurally (operator-initiated, audited, no-egress, no-distribution). That puts accountability where it actually lives rather than pretending a local model can adjudicate legality. Writing "robust local technical control here is not achievable" into a security ADR feels wrong until you realize the alternative is writing a comforting lie into one — a fiction in a security document is worse than a named, accepted, structurally-bounded, operator-signed risk. (cf. lessons 97, 105.)

140. **When a value cannot fit the transport frame, model the new chunked channel on the proven one — and pin the cap with a coupling-lock.** A decrypted PNG (up to 2 MiB) does not fit a 64 KB vsock frame, which dictated the whole resolve corridor: rather than invent framing, model it directly on the proven Stage-C parse channel — two verbs, total-bytes declared on the first frame, per-chunk exact-size validation, a hard cap that *equals* `image_staging.MAX_IMAGE_BYTES` with a coupling-lock test asserting the equality so the two siblings can never drift, and fail-closed assembly on every contract violation. The one shape the parse channel didn't need and this one does is the *placeholder*: a missing image (unknown id, decrypt-quarantine, dormant store) is a normal expected outcome a renderer draws as the inert alt box — so it must be a first-class small `found=false` frame, never an error and never a partial. Spend the assembler's strictness on proving that distinction (a placeholder after found-data raises, and vice-versa), so the corridor cannot leak half an image. (cf. lessons 121, 75.)

141. **A hardening that touches a shared primitive must be scoped to the path that needs it.** The brief said extend *both* the flat and nested manifest verifiers to require an `.xml` and `model_index.json`; that "both" would have refused the next real boot, because the flat verifier is the one the 14B/PA/draft go through at every boot and their signed manifests are `.bin`-only — re-covering them is not an edit but a TPM re-stage-and-re-sign ceremony. The green standing gate would never have caught it: it runs on synthetic fixtures and never boots the real 14B. Scope the `.xml`/`model_index.json` coverage and the `require_signed` gate to the nested verifier (one caller, image generation), leave the flat verifier and stager byte-identical, and add a flat `.bin`-only regression lock so a future agent who reads "we hash topology now" and tries to "finish the job" trips a test instead of the operator's boot. "Fix it everywhere" can refuse the very boot you were protecting, and the regression lock that proves the untouched path still works is not optional. (cf. lessons 64, 67, 138.)

142. **A docstring that names the wrong control as load-bearing is a defect, not a nicety.** Two fixes in the UC-010 post-review pass were comments confidently describing a door the code never built or wired — a "two-store by-id resolve" whose second store is keyed per-document and whose `get_display_image_by_id` fallback points at a method that does not exist, and a C# `ResolveImageAsync` doc that named the wrong control (an "unregistered dispatcher method" and a `PipeBackendException` catch) as the reason for its null return. A genuine fail-closed hole (an unguarded base64 decode that could escape the method) hid behind one of them. Read the call site, not the comment — and when the honest fix would require inventing a deferred decision (the by-id knowledge method) to make the comment true, fix the comment, flag the decision, and leave the seam inert rather than fabricate a door. (cf. lessons 47, 97, 146.)

143. **Least authority is a default, not a dogma.** Scoping a generated-image resolve per-session is the textbook least-authority reflex, but it was wrong here: the AES-GCM AAD already binds each ciphertext to `session_id|image_id`, so a relocated or edited row fails to decrypt regardless of how it was looked up — per-session is a *query-scope* choice, not a confidentiality one, and in a single-operator / one-DEK / DACL-locked-pipe / zero-egress system the boundary it would enforce is "the operator from themselves," which is not a real trust boundary. It would buy ~nil security while costing a concrete capability (yesterday's generation could no longer seed today's `/edit`). Match the grain to the data's *ownership* model (operator-owned → cross-session; document-owned → per-document), scope access only to a *real* trust boundary, and when a default is deliberate, record it — an unscoped default that isn't written down reads as an oversight, which is the only real defect. (cf. lessons 99, 101.)

144. **An operator-facing ceremony runbook must pin its interpreter and environment explicitly.** The UC-010 go-live ceremony tripped on its second step: staging the manifest ran clean on the dev box's default `python` (3.14, hashlib-only) and gave a false "environment is fine" signal, then signing died on `ModuleNotFoundError: No module named 'cryptography'` because the venv (3.11.9) has it and 3.14 does not. A bare `python` (or `pip`, or any tool) is a latent trap on every box with a second interpreter on PATH, and the danger is sharpest when an early, dependency-light step happens to succeed and masks it until a later step fails. Pin the exact venv in each command, not in the operator's memory, and add a top-of-runbook interpreter callout naming why. (cf. lessons 126, 86.)

145. **Measure the memory before you believe the design.** The UC-010 hires-fix pass (generate at 1024², upscale, low-strength img2img refine) was correct and its quality was real — *standalone*. Run live, co-resident with the always-resident shared 14B, the 1536² refine drove the box to 99.8% memory and paged the 14B out to disk (`proc_rss` collapsed to 142 MB), thrashing for two minutes and blowing the 90 s timeout. The standalone number (26 GB peak) plus the 14B (~14 GB) was always ~40 GB against a 31.3 GB ceiling; the design review nodded it through and only a measured `proc_rss` collapse made it legible. Pair it with two corollaries this go-live paid for: *a config knob is dead until you trace it to its call site* (the shipped `steps` default of `0` was honoured as "1 step" and floored every image to noise that still passed a byte-decode assertion — the screen caught what three layers of green tests did not), and *verify whose resource you are about to evict* (the "AO's 14B" was actually one `LLMPipeline` shared by the Policy Agent and the AO through a single wrapper, so eviction touched the classifier too — a premise that had to be confirmed on disk before it was safe). (cf. lessons 27, 37, 30.)

146. **A "zero-change" or "already handled" claim in a docstring is a load-bearing assertion — verify it against the code before trusting it.** Two docstrings in `imagine_coordinator.py` claimed the WinUI renders `blarai-img://` inline "with zero C# change"; it was never true — the renderer's old `Resolve()` returned null unconditionally, and the gateway's success reply was plain text so the image path never even fired. Two layers each silently defeated the feature the docs called done; the fix was as much killing the false claim (and the now-contradicted `ImageResolver` "dormant, not part of this build" comments) as writing the wire. Doc rot reads as confidence and hides exactly the hole you are there to close — treat your own predecessors' "done" the way you treat any technical premise: as a hypothesis to confirm against the call site. (cf. lessons 46, 142.)

147. **Backend-complete is not operator-reachable.** A feature with a green backend gate and a clean merge is still not done until the operator's actual surface reflects it — the definition of done runs to the surface the human touches, not to the last passing test. UC-010 image management shipped backend-complete and gate-green, and the operator typed `/images` and got "Unknown command," because a client-side surface had silently dropped it — the *identical* gap `/imagine` had hit at its own go-live, a capability merged and provably tested yet unreachable. Treat "the backend works and the suite is green" as a claim about the backend, never as a claim that the operator can reach the feature; verify the end-to-end surface, because the most expensive blind spot is the one that hides between a green gate and the human's screen. (cf. lessons 56, 149.)

148. **A "saved" flag is memory; a content hash is truth.** With `/images` finally working, the operator saw all 15 generated images flagged encrypted-only though he had saved most to disk — correct and useless, because the `saved` flag is forward-looking by construction and the whole library predated it. A boolean that means "was exported at least once, going forward" cannot describe a backlog older than itself, and drifts from "currently safe on disk" even going forward. The honest fix was to stop trusting the flag and check reality: with the app closed, replicate the AO's DEK unseal, decrypt each stored image, hash its plaintext, hash the PNGs in the operator's save folder, and mark only byte-identical matches (cryptographically safe by construction — a wrong key or non-match marks nothing). It corrected the record in both directions — disk truth was 8, not the remembered ~14. For a pre-existing backlog, only a decrypt-and-hash reconcile against the real save location is accurate, and it will correct the operator's memory, not just confirm it. (cf. lessons 48, 95.)

149. **A hand-copied cross-language allowlist drifts in silence — guard it with a gate test, not vigilance.** When a set must be mirrored across a language boundary that cannot share a literal — here a WinUI C# `BackendPassthroughCommands` array mirroring the Python-side command set — the mirror is a manual copy with no enforcement, and a missing entry fails *quietly* (the command is handled host-side or errors, no crash, no log). Vigilance is not a control: the array dropped a shipped command twice. The durable fix names ONE source of truth and writes a gate-time test that parses the mirror out of the *other* language's source as text (stripping comments so a mentioned-but-unlisted token isn't miscounted) and fails loudly, naming the specific missing element — and you verify the gate has teeth by simulating a drop and watching it go red before trusting it. A red test is the control; a human remembering to edit two files is not. (cf. lessons 147, 56.)

150. **A test double standing in for an async object must itself be async, or it proves a path production never runs.** The gallery's new dispatcher RPCs were modelled on the sync `_resolve_generated_image` (`asyncio.to_thread(fn, arg)`), but copied onto the gateway's `async def` list/manage legs that runs the async function in a worker thread and returns an *un-awaited coroutine*, which the JSON encoder refused and the fail-closed path turned into an empty list — so inline render (sync) worked while the gallery (async) silently showed nothing. Thirteen green dispatcher tests missed it because the `_FakeGateway` legs were synchronous `def`, so `to_thread` worked fine and the suite validated a path production never executes — the frozen-dataclass mock-shape divergence from this project's early days, back in a new costume. A fake's method signatures (sync vs async, return types, raised exceptions) are part of the contract under test, not incidental scaffolding; mirror the real shape including await-ability, and where the shape is the very thing that can go wrong, make the double the strictest possible version of the real contract so the bug becomes a red test, not a production surprise. (cf. lessons 114, 90.)

151. **Confirm the measurement and the ground-truth fact are about the same scenario before you let one override the other.** The milestone-1 memory measurement and the operator's daily experience were never actually in conflict: the instrument had measured the *swap's* tight pool (a 14B-release floor of ~20–22 GiB) while the operator was describing the *leaned standalone* box he builds on daily — two different scenarios. Treating his fact as a refutation of the number (or the number as a refutation of his fact) drove three successive wrong verdicts before the honest three-case landing (release sound / standalone marginal / bare swap sub-threshold). The numbers were real every time; the error was attaching a verdict to the wrong scenario. An earlier draft — "when the instrument disagrees with the operator, the instrument is suspect" — is itself too strong, exactly the framing that rationalises the next bad call. Pin each measurement to its precise scenario before letting it speak to a fact gathered under a different one.

152. **Dispatch to the capability; don't become it.** The headless-coding feature had a tempting wrong shape — give BlarAI its own coder — when the agentic-setup fleet already had a battle-tested one (30B, isolated worktrees, deterministic verify gate, auto-merge). The thinnest correct integration turned out to be a queue file and a summary read: BlarAI decomposes the request, enqueues through the fleet's documented entry points, and reads the result, hosting no coder in its own trust domain. That is only the thinnest *because* the hard parts — isolation, verify, merge, the model itself — already live on the other side of the fence; re-implementing any of them inside BlarAI would have traded a ~200-line dispatch engine for a second coder to contain. When a sound capability already exists, integrate to its seam rather than rebuild it inside your own boundary.

153. **Inject the danger so you can test the logic.** The model-swap driver's entire job is irreversible side-effects — evict the 14B, load the 30B, relaunch the app — the very subsystem you cannot exercise in an ordinary test. Building it with every side-effect as a passed-in callable (read-RAM, start-llm, run-fleet, stop-ovms, relaunch, health-probe, failure-signal) let the state machine be driven entirely in-process, turning the invariants that matter for a destructive subsystem — never-end-at-zero-models, disarm-before-stop, bounded-retry-then-out-of-band-signal, crash-recovery idempotency — into assertions instead of hopes (42 tests, no real model moved). The thin real subprocess wrappers stay the only thing left to verify on hardware. For a subsystem defined by its side-effects, make the side-effects injectable and the dangerous logic becomes ordinary, testable code. (cf. lesson 150.)

154. **Read the system you're dispatching to before you map onto it.** The headless-coding brief's tier table mapped a SMOKE criterion onto "the fleet gate (launch + console scan)" — a stage that does not exist. An hour reading the actual `new-agent-task.ps1`/`verify-project.ps1` showed the fleet builds, runs `pytest`/`npm test`, runs build/lint/typecheck, reviews, and auto-merges — with no launch-and-console-scan step — which turned a feature that would have silently under-delivered into one that's honest about what it can and can't check (SMOKE re-cast as a behavior-tier "it starts" test on the real TESTS stage). A diagram of a dependency is a hypothesis; the dependency's source is the fact — map onto the system that's there, not the one the spec describes. (cf. lessons 147, 152.)

155. **An unrun check is not a pass.** The most dangerous output of an assurance layer is a green light for something it never tested. The fleet has no `dotnet test`, so a C#/.NET behavior criterion's test never runs and comes back `none` — and an acceptance layer that rounded `none` up to "passed" would manufacture the exact false confidence it exists to remove, becoming the rubber stamp it was built to replace. `criterion_status` returns `verified` only on a real `pass`; `none`/`skip` is UNVERIFIED, surfaced as loudly as a failure, with the .NET build-only limit stated up front at the confirm step. When a gate cannot run a check, "didn't check" must never read as "looks good." (cf. lessons 149, 154.)

156. **When a value becomes config-driven, audit every consumer — a half-promoted config is worse than a hardcoded one.** Promoting the fleet's `~/agentic-setup` and `~/projects` roots to `[fleet_dispatch]` config keys looked done once the gateway honored them — but the boot-time swap *reconciler*, the crash-recovery that restores the 14B after a half-finished swap, still read the compiled-in fallback. On a default box it's invisible (fallback == config); on a custom root, once swaps go live, the EXECUTE path would *write* swap-state under the configured root while the reconciler *read* the old one, never finding the in-flight swap, so the never-end-at-zero recovery would silently never fire. A hardcoded value is at least consistent everywhere; a config only some readers honor desyncs the writer from the reader exactly where it's most dangerous. Grep every consumer of the old constant before calling the promotion done, and pin writer-path == reader-path with a test. (cf. lessons 147, 149.)

157. **Reply before you exit; a daemon thread can't take down its own process.** When a backend must terminate itself to free a resource — here the launcher exiting so the 30B can have the 14B's GPU memory — the last message to the operator has to be sent and flushed BEFORE the teardown, because the WinUI window dies with the process; the event order `["reply", "step_aside"]` is correctness, not polish. And the teardown can't come from the daemon thread the AO runs on — it would self-join its own serve-loop — so it signals the main thread (`_thread.interrupt_main()` → `KeyboardInterrupt` → `atexit`→`_cleanup`) to unwind the normal way. Wire that signal as an injectable seam, test the wiring and the ordering, and mark the actual process-death live-only: a green suite proves you *asked* to exit cleanly, never that the exit *was* clean. (cf. lesson 153.)

158. **A control tested in isolation can still rest on an unstated single-instance assumption.** The model-swap relaunch passed every unit test and the standing gate, then Fail-Closed-exited the first time it ran on the box — because four launcher instances were live and the per-boot cert provisioning writes a fresh Certificate Authority into one shared `certs/` dir, so the boots stomped each other's trust chain and the mTLS handshake failed. Nothing in the tests modelled "more than one of me," and nothing in the code guarded against it. The fix was a single-instance lock acquired before the shared write; the lesson is that "concurrency = 1" is an assumption that must be made explicit and enforced, not inherited silently — a shared per-boot artifact with no instance guard is a latent multi-instance bug waiting for the day two of you run. (And state the root cause precisely: it was host-side multi-instance, not the VM — the VM wasn't a cert consumer; a re-sync channel would have fixed a non-problem.) (cf. lessons 153, 156.)

159. **When you depend on a signal being delivered, guarantee termination — don't widen the window.** The swap's step-aside asked the launcher's main thread to exit by raising `KeyboardInterrupt` from a daemon thread; on the box the interrupt was never delivered (the WinUI native loop was parked), so the old process never exited and the relaunch deadlocked on it. The instinct is to wait longer — but a wider window cannot wake a thread that will never wake. The only fix is a watchdog that forces the process down (`os._exit`) when the graceful path doesn't complete: graceful first for a clean release, forceful as the guarantee, with a started-teardown flag to force the never-delivered case immediately while letting a real teardown finish. A green suite proves you *asked* to exit; only the watchdog proves you *will*. (cf. lessons 153, 157.)

160. **The log decides, not the hypothesis — verify the failure mechanism before you fix it.** The run-2 swap hang *looked* like a GPU out-of-memory: this box's 16 GB iGPU can't hold the 14B and 30B together, the symptom was a stuck load, and that mechanism was held confidently. But the OVMS log showed the 30B reaching AVAILABLE in thirty seconds with a zero-byte error log — a clean, successful load — and the real cause was elsewhere entirely: a captured-subprocess-pipe deadlock that hung the driver *after* the model was ready. Fixing the GPU would have shipped a real change against a non-existent bug and left the actual deadlock in place. A plausible failure mechanism is a hypothesis to test against the evidence, not a conclusion to build a fix on; read the log the system actually wrote before you touch the code. (cf. lessons 154, 158.)

161. **Capturing a subprocess's output deadlocks if it spawns a long-lived grandchild.** `subprocess.run(capture_output=True)` reads the child's stdout/stderr through a pipe and waits for EOF. On Windows the launched process's grandchildren inherit that pipe handle, so when the immediate child exits but a *long-lived* grandchild keeps running — here OVMS and the stateless coding-proxy daemon that start-llm spawns — the pipe never closes, EOF never comes, and the wait, including the timeout's own cleanup, blocks forever. The fix is to not capture an inheritable pipe for a launcher that backgrounds daemons: redirect to a file (or DEVNULL) with `close_fds`. Capture is for short subprocesses whose output you read; it is a trap for anything that starts a server. (cf. lesson 157.)

162. **Red-team the design before the diff, and let an independent gate certify what you certified — your own adversarial pass shares your blind spot.** Fixing the swap teardown, the brief called it a missing try/finally; the logs said the driver had wedged in the task loop and never reached teardown, so the real fix was elsewhere. But the costlier catch came from red-teaming the *design* before writing any code: three whole-class errors I'd otherwise have shipped — discharging a "the watchdog must not fire during the restore" guarantee with `thread.join(timeout)` (a bounded *wait*, not a kill), discharging a "no PID reuse" guarantee with a non-atomic `create_time` pre-read, and "resetting" a queue that was actually the operator's overnight data — were all invisible to test-after-build, because the tests would have been written against the same wrong mental model that produced the bug. I fixed them, built it, mutation-proofed every fix, ran my own seven-lens adversarial pass — and it came back MERGE-READY. The independent merge-gate disagreed: it found a real never-zero hole I'd left in plain sight — the two calls that *are* the 14B restore were `except Exception` while the sleep beside them was already `except BaseException`, so a signal escaped the retry loop and masked the original exception, flatly contradicting the docstring two lines above it. My own pass had read straight past it because it was reasoning about the watchdog and the holder, not the boring retry loop where the guarantee actually had to hold. The shape the cert track rewards: "I reviewed it adversarially" and "it is correct" are different claims. The cheapest place to catch a guarantee discharged by the wrong primitive is an adversarial pass over the design *before* any code; the cheapest place to catch the one your own adversarial pass missed is an independent gate that does not share your blind spot. (cf. lessons 3, 21, 30, 160.)

163. **Verify the brief's "where," not just its "what."** A builder brief written without source access points at a plausible file; the real cause can sit a layer downstream. The decomposition right-sizing brief sent me to the collapse ruler in `decompose.py` — but the collapse was already correct, and the spurious second fleet task was being re-created downstream in `acceptance.compile_prompts`. Confirming the root cause against the code (and a unit repro) before changing anything moved the fix to a different file *and* changed what a mutation-resistant test even had to lock: the brief's "revert the collapse" mutation would have guarded code the bug never touched, so the real lock had to sit at the layer that actually re-materialised the task. Pin the cause, not the brief's guess. (cf. lessons 3, 162.)

164. **A deterministic gate's dangerous direction is the one that drops work — verify *that* direction adversarially, in a separate pass, never with a green run.** The over-decomposition ruler shipped THREE false-collapses before it was right: a leaf-cap that dropped sibling features, a sibling-root test-drop that killed `verify-signature`, and — caught only by a *second*, post-build adversarial pass after I already believed it correct — a narrow integrity-noun allow-list that silently dropped an entire ordinary class (`verify-email`, `verify-otp`, `verify-age`). Each passed every happy-path test I had written. The fix was the same shape every time: stop *restricting when to drop* and instead *recognise the deliverable* — finally, any task that keeps a noun of its own after the tokens it shares with a sibling are removed. When an allow-list is the fix you usually have the wrong shape; a list of nouns is never complete. (cf. lessons 3, 162, 163.)

165. **Verification can shrink a change, not just confirm it.** The approved plan added a `_format_report` SKIPPED icon to surface the acceptance-task gating; tracing the real report path showed the swap driver writes through `write_cumulative_report` and never reaches `_format_report` at all, so the icon would have been dead code and `dispatch.py` could stay byte-unchanged. "Verify, don't assert" is usually framed as catching a wrong assumption, but it cuts the other way too — trace the actual consumer before adding code to serve an assumed one, and the honest build sometimes ends up *smaller* than the plan, deleting work rather than adding it. (cf. lessons 6, 146.)

166. **Ship the trigger, not a second mechanism.** A mid-flight `/dispatch stop` sounded like a hard cross-process problem — reach into a detached driver that has already torn the backend down and loaded the 30B — but the teardown was already there: the driver's CODE loop had polled a cancel sentinel at every task boundary since the #670 work, routing straight into the never-zero restore with partial work parked on its branch. The only thing missing was the operator-facing thing that writes the signal. The discipline that paid off was refusing to re-implement the teardown and instead reusing the existing `is_in_flight` gate (so a stop never poisons a future run) and the existing `prepare_and_launch_swap` cleanup (so no redundant unlink could race the driver). When a cross-process control already exists and is tested, the gap is usually just the trigger; build that and nothing else.

167. **An end-state assertion is not a guard on the step that produced it.** When a value is set in one place and re-set or defaulted in another on the same path, a test that checks only the final value stays green even when the first place is deleted — it cannot tell you the intermediate carry works. The build-signal's `_ensure_test_floor` carry was exactly this: the floor runs before the signal is attached, so the "survives the floor" end-to-end test passed even with the floor's carry mutated away, because the later attach satisfied the end state regardless. A mutation probe is the only thing that tells you which kind of test you wrote; the fix is a *direct* unit test of the intermediate step, not a more elaborate end-to-end one. (cf. lessons 3, 150.)

168. **Ground a "smart" heuristic in the real artifacts before you code the predicate.** The dispatch harness's doom-detector is trustworthy only because it was built from two real fleet runs read off disk first — a 59-minute-then-PARKED success and a frozen-at-`[1/5]` stall — and the stop condition is the *contrast* between them (stale fleet-log mtimes + idle coder CPU + no SUMMARY), never elapsed time, which a healthy long build can spend quiet. The same reading honored the false-positive warning concretely: the agent log echoes seed docs containing "parked"/"CS0246", so the predicate reads fleet-log mtimes and process state, never agent-log tokens. A monitor written from imagined signals would kill healthy long runs and wait out doomed ones; one written from the real files does neither. (cf. lessons 16, 24.)

169. **OOM is a commit-limit event, not a physical-RAM event — gate memory-risky loads on measured Available-RAM headroom, not on catching an allocation that won't fail.** A ~29 GB transient (the 30B coder) landing on a 31.32 GB box "should" have OOM'd and didn't, because Windows fires OOM when *commit charge* exceeds the *commit limit* (physical RAM + pagefile ≈ 42 GB here), not when the working set exceeds physical RAM. What actually happened was a measured page-storm: physical headroom cratered to ~67 MB and ~3.3 GB of modified pages spilled to disk. The correction to my own earlier model — "Committed < physical, so RAM never exhausted" — was wrong twice over: 31.3 GB is *physical*, the commit limit is higher, and physical headroom *did* exhaust. So the swap is gated on an Available-RAM headroom check, not an OOM catch: this box page-storms rather than raises, and only a headroom gate can abort before the storm. (cf. lesson 145.)

170. **Defence-in-depth only defends the paths that run it.** A fail-closed coercion — here, "an unresolved `ambiguous` surface becomes `unknown` before it reaches the fleet" — is worthless on a code path that bypasses the step that applies it. The clarifying-question fallback branch held the pre-coercion tasks and passed them straight through, so the safety net the rest of the flow relied on simply wasn't in the net's path; the test asserting the safe end-state is what exposed it. When you add a guard, audit *every* branch that produces the guarded value — the happy branch and the error/fallback branch both — and make a test exercise the fallback specifically, because the fallback is exactly the branch a tired implementer routes around the guard. (cf. lessons 22, 167.)

171. **A built-correct feature can be dormant in practice — the live model probe, not the green unit gate, is the acceptance test.** Increment 4's clarifying question shipped gated, merged, and 4300/0 green, yet when probed against the real 14B it never fired: the as-merged template told the model "a wrong guess beats over-asking," so it guessed `desktop-gui` for every platform-agnostic "app" and never reached `ambiguous`. The unit gate proved the wiring, not the capability. Prompt behaviour is empirical — a balanced four-clear/two-ambiguous few-shot prompt flipped a small model to 15/15 where a paragraph of instruction had failed — and for a small model a worked example beats a rule. The de-risk probe on the real model *is* the acceptance test; a green unit suite is a claim about the plumbing only. (cf. lessons 2, 14.)

172. **In a working tree holding other sessions' uncommitted changes, read the staged diff and commit only your own hunks — `git add -A` is the wrong reflex.** A reflexive `git add -A` to grab a diff swept in a change that was not mine — the operator's uncommitted `[fleet_dispatch].enabled = false → true` dispatch go-live flip — and a session that commits a foreign capability-enabling change inside an unrelated feature branch has quietly hijacked a governance decision doctrine reserves for the operator. The staged-diff review caught it; unstaging (non-destructive) and `git add -p` landed only the owned hunk. The lesson is not "be careful with git" — it is that on a shared tree the staged diff is evidence you *read*, never a net you cast. (cf. lessons 9, 162.)

173. **Decompose a model's output along the axis where it is strong vs weak, and discard the weak half rather than fighting it.** Playground v2.5 draws a decent flat subject and then fills every empty pixel with busy mosaic clutter — the SDXL negative-space horror you cannot beat out of it with prompts. The operator's call was to stop fighting the background and *remove* it: a one-second `u2net` cutout isolates the subject onto transparency, which is exactly the asset shape an embedded app graphic wants anyway, and reframes the only remaining quality lever as *generation* (steps, resolution, a singular prompt) rather than post-processing. The cheap separation along the model's competence boundary beats the expensive attempt to make one model good at everything.

174. **A stubbed test of a shell-out or a compiled-inline type proves the wiring, not the runtime — exercise the real compile/exec path at least once.** The VLM loop's foreground-capture fallback shipped broken and 63 green PowerShell tests sailed past it, because every test *stubbed* the capture: its inline `Add-Type` references `System.Drawing`, which on PowerShell 7 forwards through an endless assembly cascade and fails to compile at runtime — a failure no stub can see. Classic mock-passes-prod-crashes, the gap being environment (the target shell's available assemblies), not logic. The cheap guard is one un-stubbed test that actually compiles the type or runs the command on the target shell, so a stub can never again hide a runtime compile error. (cf. lessons 3, 16.)

175. **Unit-green is not seam-green.** 252 green tests (137 loop, 75 capture, 40 critique) declared the VLM auto-FIX loop done; an end-to-end run on the real box surfaced two bugs not one test could catch, both living on a seam the harness never reproduced. A render timeout tuned on an idle box was a different control under the resident-30B's memory pressure (a cold compositor init stalled past 30s); and a `.GetNewClosure()` rebound the rebuild closure's function lookup to global scope, which works one level deep — every `pwsh -File` unit test — but the real fleet runs *two* levels deep, where the dot-sourced fleet functions live in a nested scope the global-bound closure cannot see, so the callback threw "not recognized" and the whole design iteration silently skipped fail-soft. When a control's behaviour depends on its environment — invocation depth, what else is resident, a timeout tuned on an idle machine — only an end-to-end run at the real depth and real load crosses the seam; the green checkmarks from in-scope mocks are necessary, not sufficient. (cf. lessons 3, 9, 174.)

176. **The live test finds the bug the theory rationalized away.** The design loop's flakiness had a confident, plausible, written-down cause (GPU contention) that was wrong four times over — a silent screenshot arg bug, an async file write, a hung same-model reviewer, a false-doom heuristic. None was visible from the code; each took a real on-hardware run plus an empirical bisect to find. Build-dormant-then-live-verify is not a formality; it is where the actual defects live. (cf. lessons 171, 175.)

177. **Take the unreliable judge out of the critical path, don't just demote its verdict.** A slow/over-flagging/hanging reviewer that "is only a signal" still parks good work and stalls the pipeline while the path waits on it. The deterministic gate decides the merge; the cross-model critic reviews once, off the hot path. (cf. lessons 162, 176.)

178. **Before appending anything to a field that is later re-parsed, read the parser.** The design note looked like a free string to drop on `TaskOutcome.detail`, but `_classify_result` keyword-scans that exact line, so free-text feedback could silently re-classify a MERGED task — the #686 bug re-armed. A dedicated, never-re-parsed channel (the result field + the progress trail) carried the signal safely. A "harmless" append into a load-bearing parsed string is a defect until the parser proves otherwise. (cf. lessons 165, 167.)

179. **A "done" signal must be terminal, not per-step.** When a pipeline emits the same artifact at each unit of work (a per-task SUMMARY, a per-file checkpoint), don't treat its presence as whole-run completion — wait for the state machine's actual terminal phase. An early "done" is worse than a late one: whoever consumes it will act, and the action (kill the run, start the next thing, free the resource) can wreck work that was about to finish on its own. (cf. lessons 159, 166.)

180. **When a system underperforms, check which half the effort went to before you add more of it.** We had three reviewers and zero parallel sampling — polishing the judge while starving the generator. The fix wasn't a smarter critic; it was N independent candidates fed to the gate we'd already built. The bitter-lesson version: at the weak end, coverage beats criticism, and the lever that compounds is the one that rides the base model up. (cf. lessons 164, 177.)

181. **Default the PROVEN to live, not dormant.** "Ships dormant" is for the unproven and the security-sensitive; applied to a mature, gate-green, hardware-verified feature it only makes the operator carry a flip forever and the repository misrepresent what works. When something is proven, the mature move is to commit it on — not to preserve it as a perpetual uncommitted edit. (cf. lessons 171, 172.)

182. **Live-verify is a bug-finding instrument, not a rubber stamp — and the fixes are the verification, not a detour from it.** Four real defects (a scaffold config that lied, an unsynced agent, an empty post-merge diff, a cross-ecosystem prompt leak) all passed a fully green unit suite and surfaced only on hardware; landing the verify meant fixing them. Pairs with lesson 176 — the live test finds the bug the theory rationalized away. (cf. lessons 171, 176.)

183. **Prefer the wire format the model was trained on — a homemade dialect taxes both ends.** The model pays accuracy emitting a syntax it never saw in training, and your own parser accumulates silent defects nothing external exercises — BlarAI's system prompt taught an example (`calculate(2*(3+4))`) its own regex could not parse, for weeks. Adopt the trained format, and when the runtime offers decoder-level enforcement, answer the composition question (grammar × speculative decoding × streaming) empirically on the production pipeline shape with small models — not from documentation. (#718.)

184. **Match the failure posture to what failure threatens.** Fail-closed protects controls; applied reflexively to an *optimisation* it turns a driver update into a boot failure. The NPU embedding offload fails SOFT — deterministic `EMBED_OFFLOAD_FALLBACK` fingerprint, unchanged CPU path — because a slower embedder is a better posture than a system that refuses to start; the config-typo case stays fail-closed because a misconfiguration is a human error to surface. Ask what the failure actually threatens — a security boundary (closed) or a performance property (soft, loudly) — and match the posture to the threat, not the habit. (#720.)

185. **Tier the dispatch, deny the action — at separate layers.** A tool's risk tier prices what the model can steer (friction, consent-overridable); the irreversible action's deny belongs to the deterministic layer built for it (absolute, no override). Marking every egress-adjacent tool DANGEROUS would burn the /trust consent mechanism while adding zero real control — web_search dispatches GUARDED while the actual egress stays denied by RULE 3 + the empty allowlist + the unregistered runner, each independently. (#719; applies the #593 tier doctrine to the first tool where the two answers differ.)

186. **A quality yardstick pays for itself at the first seam it guards — and golden sets are commitments, not observations.** The eval harness caught a cross-branch semantic collision the night it landed: a golden case had encoded the old parser's nested-paren defect as *expected behavior*, and the format migration tripped it. When a contract changes on purpose, the baseline must be re-litigated in the same change — or yesterday's defect calcifies into today's specification. (#717 × #718.)

187. **Audit the deterministic plumbing before tuning the model when a quality miss is one-sided.** ISS-3's first measured 86.7% (all four misses false-DENIES of benign actions) looked like model over-caution; it was three individually-correct fail-closed defaults — a zeroed fallback confidence, a one-line output format that never asked for a confidence line, and a 10-token cap that could not fit one — composing into a classifier that could *never* emit its ALLOW verdict, invisible to any single component's tests. The eval harness measuring the full pipeline, not the model in isolation, is what made the composition visible. (#717 / ISS-3.)

188. **Re-verify an external contract against the live service before depending on it, not against the docs.** Kagi's v0 endpoint was documented, proven-shape, and coupling-tested — and deprecated; it 401'd the instant the governed egress chain first opened. Every axis of an external API (URL, method, auth scheme, body shape, response shape) is a fact that can rot between "built dormant" and "go live," and only a live probe with the real credential settles it. The controls around the call were all correct; the inherited fact about the call was the whole defect. (#719.)

189. **When a build must stop exactly one act short of live, enumerate the merge-time locks and make the release ONE reviewed governance act a tripwire forces into review.** The web_search go-live shipped every mechanism on main with the tree unable to make a single outbound request — three independent locks (flag off, no key, empty allowlist), released by one act. "Dormant" became a provable property (a loop test counts zero door calls), not a promise, and a golden tripwire forces the release into review. (#719.)

190. **The output validator has a direction, not just a similarity.** A verbatim-echo leakage control keyed on cosine alone cannot tell "leaked what you didn't ask for" from "answered with the public thing you did ask for" — it only sees two strings match. Twice (curated knowledge recall, web-search relay) the honest requested behaviour was ~verbatim to a grounded untrusted source and was held as a leak. The durable fix is never a threshold tweak (a faithful relay is cosine → 1.0); it is provenance — carve the intended-to-be-relayed sources out of the leak feed by an explicit auditable tier while keeping them untrusted for the action-lock, and keep the genuinely-external, not-requested source (`/external`) screened. (#719, ADR-023 Am.3.)

191. **Consent grain is set by judgeability, and judgeability lives on the action, not the content.** The blanket `/trust` failed not because it was too strict but because it asked the human to authorize a category on information he did not have — an unjudgeable request degrades to a rubber-stamp. The fix exempts a tool on the ground that its own danger is bounded (a non-exfiltratable local read), not on a property of whatever content happened to be in the session, because a content-keyed rule goes ambiguous under mixed input while an action-keyed one stays honest. Route danger to deterministic controls; ask the human only what he can actually evaluate, at the coarsest grain still meaningful. (#723 rung 1, ADR-023 Am.4.)

192. **When you replace a control, delete the old one in the same breath — and let the test that encodes the real goal tell you whether you did.** The rung-3 egress envelope was correct in isolation and useless in practice because the Layer-3 lock it was meant to supersede still stood in front of it; a behaviour-level integration test ("two searches share one fingerprint") caught what every unit test passed clean, because only the end-to-end test knew what the feature was *for*. Write the test that asserts the outcome the feature exists to deliver, not just the mechanism you built. (#723 rung 3, ADR-023 Am.4.)

193. **Verify the premise on disk before building the decision — a control that protects a threat the code cannot express is friction, not security.** The decided per-batch image-approval assumed the model-initiated `generate_image` tool generates images; the code showed it is a directive shim that generates nothing. Building the approval anyway would have shipped a gate that trains the operator to click through while claiming a protection the system does not have. When a decided premise meets contradicting code, stop and surface it — do not implement faithfully around a foundation that has moved. (#723 rung 2, ADR-023 Am.4.)

194. **A green gate proves the code does what you told it; only the live run proves you told it the right thing.** The whole trust-friction rework passed every test and still shipped an overstated claim — that removing the gate's refusal fixes the chat-poisoning — that only the operator's own machine could falsify, because the failure lived in model behaviour over real conversation history, not in any unit the harness exercises. Live-verify is not a formality after a green gate; it is the only place a "correct code, wrong belief" defect can be caught. And when the machine contradicts the claim, read the log before trusting the instinct — mine said "my exemption is broken," the log said "your exemption is fine, your model is imitating an old ghost." (#723 / #726 live-verify.)

195. **A memory measurement is valid only for the exact path and era it measured.** The Phase-0 gate proved the 1024² co-resident image case and stayed on the books as "image gen fits," but a later amendment bolted a 1536² hires pass on top and nobody re-measured the new peak against the ceiling — so a "circuit breaker" shipped set above the actual cliff, and on the live box the refine drove RAM to 100% and cascaded into a false timeout and a failed save. When you add a heavier path to a memory-bounded feature, the old headroom number is not evidence about the new one; re-measure the peak of the path you actually shipped. (UC-010 hires, #729.)

196. **An append-only lessons list dilutes its own signal — curation is part of the discipline.** At 195 entries the compounding surface had stopped compounding: recurring classes were re-minted as near-duplicate numbers (the mock-versus-live class alone owns fifteen-plus), environment-specific gotchas sat beside transferable judgment, and the one page a portfolio reviewer would actually read did not exist. Worse, the list learned faster than the practice — a class named in May recurred six more times in June, because writing a lesson down installs nothing. The fix is structural, not editorial willpower: a curated canonical tier naming the ~20 classes, a recurrence-tally rule instead of new numbers for repeat instances, a third-instance rule that converts a repeating class into an enforcement recorded on the lesson, and a separate `FIELD_NOTES.md` for mechanics. This is lesson 42 (a discipline's medium must scale with its load) and lesson 45 (find the doctrine producing the mess) applied to the journal itself. *(Control produced: this file's Rules section + `CLAUDE.md` §Build-Journal-Discipline SOP, 2026-07-03.)* (cf. lessons 42, 45, 103.)
