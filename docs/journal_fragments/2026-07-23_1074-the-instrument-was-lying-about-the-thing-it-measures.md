### 2026-07-23 — The instrument was lying about the thing it measures

*Plain summary: the coding fleet reported "the coder produced nothing" when what actually happened was that git failed — so about 8% of our capability data meant nothing, and we had no way to tell which 8%.*

Forensics on B4's parked `add-card` run turned up something worse than the park. All three candidates had genuinely written `app/card_entry.py` and run pytest green — it is there in the transcripts. The report said `CHANGES: none made / RESULT: Nothing to merge.` The work existed and vanished somewhere between `git status` seeing it and `rev-list` not.

The cause was three lines in `Invoke-CandidateBuild` that ran `git add -A` and `git commit` with both error channels piped to `Out-Null` and no exit-code check. A locked index, a rejecting hook, a sparse-checkout rule and an honest no-op all produced byte-identical output. The fail-open direction was "the coder produced nothing" — and because `hasChanges=false` skips the test and verify steps, the candidate could never win the gate, so best-of-N resampled a fault it had no way to see, three times, then parked.

What makes this the worst of the four defects that pass found is where it sits: inside the instrument that measures coder capability. Forty of 488 reports read `CHANGES: none made`, and every capability number we have rests on that string being a fact about the model rather than a possible fact about the box. It was not that some of those forty were wrong. It was that nothing in the system could say which.

The fix was easy to describe and the hard part was somewhere else entirely. Checking exit codes is trivial. The problem is that `git commit` exits non-zero on the honest "nothing to commit, working tree clean" — so the exit code is exactly as ambiguous as the silence it replaced. My first instinct was to match git's message, and I rejected it: that makes correctness depend on git's locale and version, which is a fine way to reopen this hole quietly in two years. What works is refusing to interpret at all. Read the *index*, and only attempt the commit when something is actually staged. Then the honest no-op never enters the failure channel, and a non-zero commit has exactly one meaning. The discrimination stopped being a judgement call and became a state. That is the piece the reviewer attacked hardest and could not break.

Then I nearly shipped the same defect backwards. I had merged git's stderr into the staged-set and status reads — and those outputs get counted. A warning line would have counted as a staged path, turning honest no-ops into false ERRORED tasks. My probe did not trip it, because git only warns on commands that write the index, and that is precisely the reasoning that produced this ticket in the first place: it works because of a property nobody wrote down. Stdout only now, with stderr re-read on the error path where it is a message rather than data.

And then the parts I got wrong stayed wrong through 124 of my own checks, a 49-suite regression sweep, and a written deliverables report.

Builder-1076 mentioned in passing that its scope had grown to the candidate worktree loop. I measured what my code does when a candidate's workspace was never created: `git add -A` returns 128, and I had made a capture fault terminal for the whole best-of-N run. One bad workspace would have stopped a run that today degrades to N-1 candidates and still merges a winner. My reasoning — "an infrastructure fault is not a model slip" — had treated a per-candidate fault as repo-wide, with the idle-stall carve-out making that exact counter-argument four lines above where I typed it.

So I made it non-terminal, and wrote a commit message defending that as safe because "the fault stays loud by four other routes". **That defence was false, and the lead measured it false on that exact commit.** Four of the five routes I listed — disqualification, rank, merge block, review-loop — only affect the candidate that faulted. The fifth, the ERRORED report, is the only one an operator ever sees, and it was precisely the one that stopped firing: with the fault no longer terminal, sampling continued, a sibling no-op outranked the faulted candidate, and the fault vanished into `fleet-report`'s "no change made" column. Measured in both candidate orderings:

```
[noop,FAULT,noop] ran=3 selectedIdx=0 selectedGitFailed=False anyFault=1  REPORT='none made'
[FAULT,noop,noop] ran=3 selectedIdx=1 selectedGitFailed=False anyFault=1  REPORT='none made'
```

Under the previous commit, terminality had accidentally been the thing keeping a lone fault visible — a fault in candidate 1 left it the only candidate, so it got selected and reported. Removing terminality was still the right call, but I removed the visibility along with it and then certified the result safe using a list I had not checked. For a stretch of that afternoon `CHANGES: none made` was *less* trustworthy than before I touched it, in the battery's default N=3 configuration — the ticket's own defect class, re-created by the fix for it, and shipped with a commit message asserting the opposite. `c8bf3c8` is the commit that makes that claim true, by aggregating the fault across all candidates instead of reading only the selected one. `7a9a87a`'s message is permanent and wrong; this paragraph is the correction, because a merged message reads as fact to whoever finds it next.

The lead's review then found two more, both of them this ticket's own defect reproduced inside the fix for it. One of my three `[kill]` assertions was written with double quotes, so PowerShell interpolated `$wt` to empty and the pattern could not match the commit swallow it existed to kill — a lock that passed against the bug, shipped inside the ticket about locks that pass against bugs. And a capture fault was silently dropped whenever any non-faulted candidate existed, because the report read the *selected* candidate and the ranking correctly sinks a faulted one below a plain no-op. The laundering I had just fixed at candidate level was still running at run level. The measured B4 instance only looked covered because all three candidates faulted together.

The trade-off I keep coming back to is one I got wrong in both directions on the same axis. First I bought loudness I did not need — terminality — at the cost of reliability I could not spare. Then, correcting that, I gave up the loudness that was actually load-bearing, because I had assumed the other routes covered it instead of checking which ones an operator can see. Detection and *surfacing* are different properties, and four of my five "routes" were detection. The design correctly detected the fault the whole time; the report layer simply never told anyone.

There is a sharper version of it. When I removed terminality I was reasoning about what the code would no longer *do*, and I checked that carefully. I did not check what the code would no longer *say*. In an instrument whose entire output is a sentence an operator reads, the saying is the function.

It happened a second time, one commit later, and that is what makes it a pattern rather than a slip. I wrote a disposition record marking F2 FIXED, and `verify_disposition.py` passed the file 11 of 11. The lead then reproduced F2's false alarm end to end against that exact tip: a completely healthy repo on branch `trunk`, handed the fallback base name `main`, was being classified as a capture fault — so every healthy build in such a repo would have become permanently, silently ERRORED. The narrowing was simply absent. The verifier passed anyway because it checks the *form* of a record — that a FIXED row cites a commit — and cannot see whether the claim is true. That limit is written into the verifier's own docstring; I had read it and still trusted the green.

So in one evening this change produced two claims that measurement contradicted: a commit message asserting the ERRORED route still fired when it did not, and a disposition row asserting FIXED when the fix was partial. Both were written by the person who had just done the work. Both sounded settled. Both were blessed by something structurally incapable of catching them — a test suite that validates what I built against what I meant, and a verifier that validates the shape of a sentence rather than its truth. In a ticket whose entire subject is a system reporting things it cannot support, that is not irony. It is the same defect operating one layer up, on the record a later session will trust instead of on the report an operator reads.

The fix for F2 was four lines: verify the base resolves before blaming the history, and give the two cases different reasons, because a fault that stays loud while naming the wrong cause is the original defect wearing a new label. What took the time was not the code. It was that I had already written down that it was done.

Six locks in this change were pinned to prose rather than to properties. Two ordering locks anchored on a *comment string*, so rewording a comment silently unpinned them. Two pinned a guard list exactly, so adding a guard failed them even though the loop had got stricter. One used a character-window proxy already widened 12000→18000 that same day. And the dead one was mine. Over-tight and under-tight are the same failure wearing opposite costumes. The generalisable control is the vacuity guard I added afterwards — assert that a `[kill]` pattern actually matches a literal specimen of the thing it kills — and it belongs on every `[kill]` assert in that repo, not just mine.

Proposed lesson (unnumbered): *A fix for a fail-loud defect can introduce a quieter version of the same defect, and the commit that does it will argue that it has not. Every checker in the loop that evening — the test suite, the disposition verifier — was structurally incapable of catching the specific thing that was wrong, because each validates the artifact against its author's intent rather than against reality. A test suite validates what you built against what you meant; a form-checker validates the shape of a claim, not its truth; and neither can see a lock that never bites. When a control's job is to detect a failure, the control needs its own negative control — and when you are about to record that something is done, that record needs one too.*
