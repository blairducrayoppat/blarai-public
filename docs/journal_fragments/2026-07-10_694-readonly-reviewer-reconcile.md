The interesting thing about "reconcile these two branches" was that there was almost nothing to reconcile — and seeing *why* was the whole job. `fix/694-review-truly-readonly` (`f219074`) was already in main's history: it had landed on 2026-07-07, then a config-sync (`3553b56`, author `harness-sync`) reverted just its `review.md` half three hours later, leaving its `new-agent-task.ps1` pre-gathered-diff half quietly in place. `feat/694-readonly-reviewer` (`e3ef0e3`) was forked *after* that, so it already sat on top of the surviving half, restored the reverted `bash: deny`, and added the harness digest guard the earlier fix never had. In other words e3ef0e3 was f219074-plus-restoration-plus-enforcement. The honest reconciliation was therefore not a hand-merge of two hunk sets — it was a single clean cherry-pick of the superset onto the current local main (`235b467`), which reproduced the entire intended end state: `review.md` with `edit/bash/todowrite: deny`, `new-agent-task.ps1` carrying both the pre-gathered diff *and* the digest fail-close, and both verify scripts. I took that path over re-applying both branches by hand precisely because the hand-merge was the more dangerous option: it risked double-applying the pre-gathered diff or re-opening the already-reverted `review.md` race. When the graph already encodes the answer, replaying it beats reasoning about it.

The trade-off worth recording is the ordering of the two defenses. The `review.md` `bash: deny` is a *request* — and this exact request has already been reverted once by a background sync, and the original incident's mutation came from *outside* the agent's own tool calls anyway. So the load-bearing enforcement is the harness digest guard (`Get-WorktreeDigest` snapshot around the review leg → `Test-ReviewLegMutated` → forced `FIX FIRST` + park), which no config-sync can touch. The contract deny stays as defense-in-depth, not as the primary control. That is the right shape for "the LLM reviewer is a signal, never an actor": you don't ask the actor not to act, you make the harness refuse to trust a verdict from a leg that moved the tree.

The runbook is where the #780 lesson actually gets operationalized. The previous fix failed for a mundane, portfolio-valuable reason: the change *needed two writes* (the repo copy and the live `~/.config/opencode/agents/review.md` the fleet reads) and made *one*, so the one-way LIVE→repo drift capture clobbered the repo edit with the stale live file at the next `start-llm` swap. So the runbook's merge motion is explicitly two-part — merge, then deploy repo→live in the same motion until the two hashes match — followed by a forced no-op `sync-harness.ps1` run as *proof* the trap is disarmed (HEAD unchanged, no "drift captured" commit, live still carries `bash: deny`). A fix for a silent-revert bug that can itself be silently reverted isn't a fix; the proof step is the part that makes it one.

**Recurrence of the shared-state / config-sync-clobber lesson** (#780 — a change that writes to shared state outside its declared scope; here the change *needed* two writes and made one). This is the second instance of that class on this exact surface, so it ships with a structural control, not just a re-description. The durable enforcement now exists in three layers: the harness digest guard (sync-proof by construction), the runbook's same-motion-deploy + no-op-sync proof, and — added on the team-lead's go-ahead — the #780 fix-(c) machine gate `verify-config-deploy-sync.ps1`. That gate is the interesting bit of judgment: divergence alone (two hashes differ) is not actionable, so it names the *direction* on LF-normalized content — `REPO_AHEAD` (live is a known past committed version → benign pending-deploy, warn) vs `LIVE_AHEAD` (live carries content the repo's history never held → the armed trap, fail). The trade-off I made deliberately: a live copy that is merely stale is a warn (the legitimate merge-window state, and failing it would red the suite on every undeployed box), while a live copy that has drifted *ahead* is fatal (a sync would capture unreviewed live content into the tracked tree). A `-Strict` mode collapses everything-but-`IN_SYNC` to a failure, which is what makes it a re-runnable post-deploy proof rather than an eyeballed hash: it is red before the deploy and goes green only once live matches repo. I left the live probe opt-in (`-CheckLive`) so the default gate stays hermetic like its siblings and reads nothing under `~/.config`. #780's fix (a) (deploy-in-motion) and fix (c) (this gate) are both in the branch; fix (b) — making sync-harness itself two-way-aware — I did not take, because the three existing layers cover the risk without touching the capture semantics the operator relies on to record real live drift.
