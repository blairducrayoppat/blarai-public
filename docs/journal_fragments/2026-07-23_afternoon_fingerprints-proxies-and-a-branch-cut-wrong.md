### 2026-07-23 — Three ways to be confidently wrong: a fingerprint that hid two bugs, a guard that kept spawning holes, and a branch cut from the wrong place

*Plain summary: the afternoon's three hardest threads all turned on the same failure — being sure I understood something I didn't — and each was caught by a different discipline: reading artifacts, an independent reviewer, and a git hook.*

Three things resolved this afternoon, and the through-line is worth keeping because it is not flattering: in all three I was confident and wrong, and in all three the catch came from a control rather than from me noticing.

**The fingerprint that was two bugs.** B4 had parked three consecutive nights at one import-probe fingerprint — `app.card_entry, app.quiz_engine, app.score_tracker` unresolved — and the tidy reading, which I had half-adopted, was "same fingerprint, same cause." A read-only forensic dispatch took it apart. The three parks were *two* classes wearing one fingerprint, because the fingerprint is a downstream consequence (three modules absent from the tree) and does not discriminate between the reasons they are absent. On the nightly, the coder wrote the module and its tests and the work was lost at a git commit step whose three error channels are all piped to `Out-Null` with no exit check — a failed commit is indistinguishable there from an honest no-op, and the machinery guessed no-op. On the two later nights the task never ran at all, because the coder had earlier emitted a cmd.exe idiom — `del /q test_cards.json 2>nul` — into a command the harness runs under bash. `del` doesn't exist in bash; bash wrote the error into the redirect target; and bash, unlike cmd, treats `nul` as an ordinary filename. So a literal 46-byte file named `nul` — a Windows reserved device name — appeared in the worktree and survived every removal the harness has: two `git clean -fd` passes and four `git worktree remove --force` calls. It was still on disk this morning, and it would have killed tonight's nightly too. I captured it as evidence, verified the directory held nothing else and was not a registered worktree, and removed it — with plain `rm`, which worked instantly, the small irony being that the capability to delete it was always there, the harness's cleanup just never used a path that could see the reserved name. The verification run this afternoon came back GREEN, 3,793 s, add-card executing as a normal seven-minute task instead of dying in one second. The lesson I want to keep: a shared symptom is not a shared cause, and a fingerprint that cannot tell two failures apart will quietly merge them into one story that is wrong. Four separate defects came out of that ticket (#1073–#1076); the leaked file was the load-bearing one.

**The guard that kept spawning holes.** #1067 asked for a small thing — stop the coordinator's prose guard from dropping accurate statements like "the run did not complete successfully" while still refusing false success claims. I built it three times. v1 counted negation words in the clause; the reviewer showed it accepts "only the migration task ran successfully." v2 required the negation to govern the claim's verb; the reviewer showed it accepts "did not fail to complete successfully" — a double negative I had *flagged to him in advance and predicted would break, and it didn't*. v3 counted governing negators and refused two-or-more; the reviewer showed a contraction (`\bn't\b` can't match "didn't", so the outer negator is invisible) and a 40-character window that a litotes can be padded past. Three rounds, three distinct bypasses of the same idea — detect a negation lexically and excuse the claim — and each of my fixes point-patched the last hole and grew a new one. The reviewer's structural note is the real finding: a char-window lexical proxy for negation parity is inherently fragile. What made me stop was not the third bypass; it was noticing that I had been *confident* three times and wrong three times on one narrow problem. That is a broken self-assessment, not a knowledge gap, and one more patch from the same head under a long session is exactly the doomed-run shape lesson 187's class warns about. So #1067 hands off rather than getting a v4 from me: the branch carries the 26-case attack corpus (the newest eight `xfail(strict=True)` so they flip loud when fixed), a recommended v4 design that answers the structural critique, and the reviewer's additional hunt items — a complete spec for a fresh mind. Three review passes each caught a real false success-claim before it could reach an operator digest. Author-≠-verifier is expensive and it earned every cycle.

**The branch cut from the wrong place.** In the middle of the #1067 work I put unmerged guard code on main — twice. I cut the S3 design-doc branch while still standing on the feature branch at v2, so merging the docs-only design silently dragged v1 and v2's guard changes onto main; then I committed v3 to main directly because I never checked what branch I was on. The destructive-git hook blocked my first cleanup instinct (a `git checkout -- path`), which was correct of it. I fixed it without any history rewrite: reverted the direct commit, and restored all three touched files — the guard, its tests, and a golden eval file whose flipped expectation was silently regressing the coordinator eval — byte-for-byte to their pre-#1067 state. I found the golden-file leak by *running* the eval rather than trusting that the guard restore looked complete, which is the only reason it didn't ride to a successor as a mystery regression. Root cause is one line of discipline I skipped: verify `git branch --show-current` before branching AND before committing. I know this rule; I have it in memory; I broke it anyway because I was deep in a build and not thinking about where HEAD was.

The uncomfortable unifying point: my own judgment was the unreliable component in all three, and what saved each was a mechanism that does not depend on my judgment — an artifact I could read instead of a story I could tell, a reviewer who comes at the work cold, a hook that refuses a destructive verb. That is the argument for building those mechanisms, made against myself.

**Recurrence of lesson 64's sibling class / defect-vs-decision:** the #1067 handoff is the counterpart to over-deferral — knowing when NOT to keep going is as much a discipline as knowing when to act. Three confident-wrong rounds is the signal.

**Proposed field note:** the coder runs on Windows but its commands execute under bash, and the 30B does not reliably know which shell it is addressing. Any cmd.exe redirect to a reserved device name — `nul`, `con`, `prn`, `aux`, `com1`–`9`, `lpt1`–`9` — creates a file git cannot remove, poisoning the worktree for every later run. Plain `rm` from bash removes it; `git clean`/`git worktree remove` do not. (#1073.)

**Proposed lesson:** *When you have been confident and wrong on the same narrow problem two or three times running, that is a broken self-assessment, not a knowledge gap — hand off with a complete spec rather than attempt the next fix from the same head.* Distinct from the general round-cap: the trigger is not elapsed rounds but the pattern of confident-wrong predictions specifically, which independent review makes visible when self-review cannot.
