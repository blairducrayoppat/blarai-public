### 2026-07-23 — The half that can only refuse more, and the measurement that only looked like a measurement

*Plain summary: a guard meant to catch false success claims had been letting "The build successfully completed all six tasks" through on failed runs the whole time; the fix ships alone, because the other half of the ticket grants excuses and has now been rejected twice — and separating them turned on a number I had been reading wrong.*

I came into #1067 to fix over-refusal: the coordinator's guard was dropping true sentences like "the overall run did not complete successfully", once in every thirty-four cycles. Four designs had already been rejected. Mine became the fifth, with thirty-four false acceptances found by an independent reviewer, and the sharpest of its findings was aimed at a paragraph I had written myself.

I had put into the file that "every gap in this grammar is a false refusal, never a false acceptance", and that splitting a sentence "only ever makes segments smaller, so an over-split fails safe." Both are false. Splitting does not only shrink the matched segment — it *detaches* it from the rest of the assertion, and the detached remainder is never screened, because it carries no success claim of its own. So "The run did not complete successfully. That statement is false." walked straight through. I had been warned, in the handoff brief, that the previous attempt's safety proof was exactly this shape: a one-direction argument about a two-direction operation, written in a register that reads as settled and therefore suppresses the hunt that would find the defect. I set out to avoid it and reproduced it inside a comment block whose own text names the three constructions that killed the earlier designs. The Lead Architect's read is that this is now the third instance across three tickets in one night, which makes it structural rather than three separate lapses, and I agree.

The more useful story is the one about splitting the ticket. There were two independent fixes in my change. One widened what the guard *catches* — closing a word-order blind spot where the pattern demanded the verb before the adverb, so "successfully completed" matched nothing at all. Fifty-six of fifty-six adverb-first inversions were accepted on a failed run against zero of twenty-four verb-first controls: not a partial gap, a total one, live in shipped code, in a guard whose entire job is that sentence. And it is the same inversion as the miss caught live on 22 July and golden-locked the same day — that fix closed one form and nobody swept the pattern for siblings, so the twin sat open in the neighbouring alternation for another day.

The other fix added the infinitive `run` beside `ran`. I argued on the ticket that the two had to ship together, because closing the second alone would start refusing accurate prose. I had a number for it: zero additional false suppression on the live window. The LA didn't accept it, and he was right not to. That figure measured both fixes *together*, on a window of thirty-four cycles carrying ten distinct statements — none of which contains the sentence in question. It was an N-limited result wearing the costume of a property, which is the same error the session before me made when it measured a design against a corpus written from the design's own theory of itself. I had criticised that in my own gate and then done a quieter version of it.

Measured properly — the two fixes separated, no carve-out, against constructed corpora chosen because the live window is silent on the question — they come apart cleanly. The adverb fix accepts all six constructed accurate cases and all five task-level ones while catching six of six false claims. The `run` fix refuses all six accurate cases. Worse than I had predicted, in fact: adding `run` also makes the pattern match the *noun* "run" before the adverb, so "the run successfully merged three tasks" gets refused too — a second suppression path I had not seen when I insisted the two were inseparable. So the adverb half ships and the `run` half waits for the carve-out it actually depends on, pinned by a characterisation test that asserts the gap is still open and must be flipped by whatever closes it.

One more thing fell out of the merge preparation. To set the gate baseline I measured collection on both sides and found main collecting 8883 against a recorded baseline of 8868 — stale by fifteen, arrived with a code merge that never updated either doctrine surface. The freshness gate did not catch it, and cannot: it only compares the two surfaces to *each other*, so a wrong-but-consistent pair passes silently. That limit is documented on the gate itself, and tonight is the second time it has been the thing that mattered. I recorded the new figure as explicitly derived rather than clean-measured, with the owed re-measurement written down, because the primary checkout was occupied by another session all night and a worktree cannot reach the model files.

The merge itself did not happen. Main is checked out in another session's merge worktree and two swap drivers are live, which is precisely the collision the shared-tree rules exist to prevent. The branch is finished and waiting. Stopping one command short at the end of a long night is the correct ending, and it is the same judgement I spent the night arguing for everywhere else.

Proposed lesson (unnumbered, for the integrator): *a figure measured on a sample that cannot contain the phenomenon is not evidence about the phenomenon. Before quoting a rate as a property, name the case that would have moved it and check the sample contains one.*

Second proposed lesson: *when a change has one half that can only tighten and one half that can only loosen, they are two changes. Ship the tightening half immediately; it is safe alone and holding it hostage leaves a live hole open for nothing.*
