### 2026-07-12 — The anchor list was never the whole tree

*Plain summary: closed F1, the non-anchor hardlink evasion of the ADR-039 governed-core boundary, in shared/coordinator/governed_core.py — added a link-count layer (an existing target with st_nlink > 1 is refused) so a hardlink aimed at a NON-anchor core file (pyproject.toml, launcher/config/default.toml, shared/coordinator/config.py, a PA policy module) is denied, not just one aimed at the four anchor files; 8 adversarial regression locks added; dormant module, no live wiring changed.*

The hole was subtle and it was in the defense I would have called finished. The boundary's hardlink guard, `_shares_inode_with_governed_core`, compares a proposed target's device+inode against a small anchor fileset — the `.blarai-governed-core` sentinel plus CLAUDE.md, DECISION_REGISTER.md, and dispatch.py. That reads like a hardlink defense, and against those four files it is one. But the governed core is not four files; it is the whole tree. A hardlink placed under an allowed workspace path and pointed at any *other* core file — `pyproject.toml`, the launcher config, or (the one that stings) `shared/coordinator/config.py`, the boundary's own policy data — aliases no anchor. Its inode matches nothing in the anchor set, and its own realpath is a genuinely innocent workspace path, so canonical containment waves it through too. Both of layer 1's sublayers miss it, and the write lands on the real core file. An independent re-verify proved it at execution phase: a hardlink to each of those four returned ALLOW and had to become DENY. This was a live violation of the "zero write path to the governed core" invariant, owned by neither the shipped fix nor the planned F2.

The trade-off worth recording is the one the LA already weighed and locked to option (a): deny any existing file whose link count is greater than one. A hardlink is the attack primitive, so instead of trying to enumerate which core files might be aliased — a list that is exactly as incomplete as the anchor set that created the hole — the check refuses the *primitive*. The cost is over-denial, and I want it named honestly rather than waved away: overwriting an already-multiply-linked existing file is now refused even when it is benign. The rejected alternative was to keep chasing completeness — extend the anchor comparison to walk the whole core tree per target — which trades a bounded, well-understood over-denial for an unbounded per-check stat storm and *still* leaves the "did I enumerate everything?" question open. For a constitutional fail-closed boundary the asymmetry is decisive: a false deny costs one refused overwrite of a file that was already hardlinked (a vanishingly rare shape for normal workspace work, where fresh files carry st_nlink == 1); a false allow costs BlarAI a write path into its own governing code. I took the over-denial.

I verified the Windows angle on the metal rather than trusting the platform note, because this is an NTFS host and `st_nlink` has a reputation for being unpopulated on Windows. It is not, for real files: a fresh file reported `st_nlink == 1`, and after `os.link` both directory entries reported `st_nlink == 2`. Python populates it from the file handle's `nNumberOfLinks` and it is reliable here. I still wrote the helper to treat a link count it cannot determine (0) as "not > 1" — so an unknown count never false-denies and never opens a path either; the other identity layers still apply. The docstring says exactly that and no more. The eight new locks prove the four evasion cases now DENY both directly and through `check_target` at EXECUTION, that the two prior layers genuinely miss them (the assertions pin `_canonical_containment` and `_shares_inode_with_governed_core` to False on the same target the new layer catches), that a normal new file still passes (the over-denial guard), and that an unexpected error in the new layer fails closed.

**Next:** author != verifier — an independent re-verify of this fix is pending (drive the four hardlink cases through `check_target` at EXECUTION on the merged branch, confirm 0 regressions against the prior 166, and confirm the empty behavioral diff on dispatch.py / live modules). After that clears, the coordinator folds the ADR-039 doctrine-correction — the boundary's layer-1 description and the §2.2 control-1 text must stop implying the anchor-inode comparison is the whole hardlink defense and name the link-count layer as the general one; that fold, and the DECISION_REGISTER touch if any, are the coordinator's, not this fixer's.

**Proposed lesson:** A fail-closed identity check that compares a target against an *enumerated* set of known-bad identities is only as complete as the enumeration, and the enumeration is almost never the whole protected surface — the anchor fileset here named 4 files to defend a whole tree. When the protected thing is "everything of a kind," prefer refusing the attack *primitive* (any hardlink / any symlink / any multiply-linked file) over enumerating the *targets*; the primitive is finite and the target list is not. Accept the bounded over-denial as the price of not having to be exhaustive.
