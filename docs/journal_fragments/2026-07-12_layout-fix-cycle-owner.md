### 2026-07-12 — The repair loop that fixed the survivor

*Plain summary: fixed the fleet's layout/import fix-cycle (`swap_driver._owning_task`) to re-run the task that OWNS an unresolved module — precisely, over every task regardless of status — instead of a merged sibling; the pre-fix version scanned only merged tasks and matched by a coarse `app`-substring, so it re-ran a survivor past the parked owner and resolved 0/3 on the 2026-07-12 battery (#790 defect #2).*

The dispatch fleet has a good idea buried inside it: when the whole job is about to score zero because the acceptance oracle can't import a module the coder was supposed to produce, don't wait for the opaque wave-final `ModuleNotFoundError` — probe the import contract on the integrated tree, name the exact unresolved entry, and spend ONE targeted fix cycle re-running the task responsible with that entry quoted. It was built for a specific failure shape, the one labelled B6n2: a coder who *merged* working code but placed the module one directory too deep (`app/cli_interface.py` where the oracle imports top-level `cli_interface`). For that shape the responsible task is *merged*, and the whole thing works.

The 2026-07-12 battery showed me the shape it was NOT built for, three times over, and it failed all three the same way. B7 is three independent Node helpers; `unit-converter-helper` parked ("Nothing to merge") while its two siblings merged clean. The oracle imports `convertUnit` from the missing `../src/unit-converter-helper.js`, the probe named it correctly — and then the fix cycle re-ran `password-generator-helper`. A *sibling*. The task that could never, by construction, produce `unit-converter-helper.js`. B2 re-ran `tokenize` for a missing `app.word_frequencies`; B4 re-ran `implement-data-storage` for a missing `flashcard_app` package. Zero of three resolved, and every one of them re-ran a task that had nothing to do with the gap.

Two faults compounded, and both are worth keeping. The first: `_owning_task` built its candidate list as `[t for t in tasks if t.status == MERGED]`. But the module is missing *because its owner failed* — the owner is the PARKED task, and the merged-only filter excluded it from ever being a candidate. A repair loop that structurally cannot point at the thing that failed will always point at something that didn't. The second fault is why it landed on that particular sibling: the module token `app.word_frequencies` was reduced to `app` (a `rsplit('.', 1)[0]` that, on a dotted Python module, keeps everything *before* the last dot), and `"app" in "app/__init__.py app/tokenize.py"` is true for every task in the `app` package — so the first `app/` task won. When no token substring-matched at all (the Node case), it fell through to `sinks[-1]`, the last merged task in plan order: `password-generator-helper`. The heuristic was ungrounded in the one artifact that already knew the answer — the plan graph, where every task's contract declares exactly what it `creates` and `exports`.

The fix maps a module to its owner *precisely* and over *every* task. A dotted `app.word_frequencies` normalizes to the file path `app/word_frequencies` and the leaf `word_frequencies`, matched by set-equality against each task contract's created-file forms — never a substring, so `app` can no longer match `tokenize`. A Node spec `../src/unit-converter-helper.js` strips its relative prefix and extension to `unit-converter-helper` and matches `unit-converter-helper`'s own contract, not `password-generator-helper`'s. Among the tasks that genuinely own an unresolved module, the parked/blocked one — the one that actually failed to deliver — is preferred over a merged one, which is preferred over a skipped one whose own dependency is still missing. When nothing precisely owns the module, it degrades to the exact pre-existing sink/last-merged heuristic, byte-identical, so no legacy path regresses.

The trade-off I deliberately did NOT fold in: re-running a *parked* owner that now merges leaves its plan node still stamped `parked`, so the job's verdict stays PARKED-HONEST even when the re-probe resolves and the oracle would pass. Promoting a recovered parked owner to `merged` (and re-running the dependents its park had skipped) would let the job go GREEN — but that changes verdict semantics, not just repair targeting, and "a job that parked can now be scored green" is a quality/capability call for the LA, not a defect I should silently decide. So this ships as owner-*selection* only, which is exactly what moves the stated A/B metric (layout-fix-cycle resolution rate: re-run the real owner, resolve the import), and I flagged the recovery-to-green as a separate decision rather than bundling it.

The failure that made this catchable-in-hindsight-only is the honest one: every existing test of this fix cycle had the owner *merged* (`_layout_tasks` builds core+cli, both of which merge). The parked-owner path — the *common* one in the wild — had zero coverage, so the merged-only filter was never exercised against a parked owner and the bug rode dormant behind a green suite. I added the parked-owner case and watched the new regression lock fail against the real fault first: stashing only the driver, the B7 end-to-end test reported `['password-generator-helper'] == ['unit-converter-helper']` — the live battery bug, reproduced in a unit test — before the fix turned it green. A lock you haven't watched fail is a decoration.

After it merges, the next battery A/Bs the resolution rate — B7 is the clean demonstrator (its three helpers are independent, so re-running `unit-converter-helper` resolves `convertUnit` with nothing else in the way). **Proposed lesson (recurrence of C3 / lesson 3):** *A repair loop must be able to target the thing that failed — check its candidate set, not just its matcher.* The layout fix cycle scanned only MERGED tasks, so the PARKED owner (the very thing that failed to produce the module) could never be its re-run target; it acted on a survivor instead. Sibling to "a test that shares the code's blind spot certifies the bug": every test of the mechanism had the owner merged, mirroring the mental model that a missing module's owner is always a merged-but-misplaced task, so the far more common parked-owner reality was never certified against. When you build a mechanism to *repair* a failure, the failed thing must be in scope for the repair, identified precisely (here: by the plan graph's `creates`/`exports` contract, not a coarse substring a survivor also satisfies) — and the regression lock earns trust only once you have watched it re-run the wrong task on the old code. *(cf. C3, C9 "ground heuristics in real artifacts", C15 "close the class" — the exec-smoke boot-fix cycle shares `_owning_task` and inherited the same fix in the same change.)*
