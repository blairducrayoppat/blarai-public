Should we build a proper front door?

Plain-language read of the Advanced Intake design sketch (ticket #1031) · written 2026-07-22 by the overnight session, from the sketch and the source it cites · you asked for this so you could decide on substance instead of reading 22,000 words of architecture

The decision in front of you Should the system interview you properly before it starts building — instead of asking a few quick questions and going?

That is the whole call. Build / don't build / later. Nothing here switches anything on: if you say build, it arrives switched off behind a flag, and turning it on is a separate occasion with you present.

Why this, and why now

Last night handed us the argument for it, by accident. We shipped a code fix meant to stop the builder from sprawling beyond its assigned task. It worked — it moved the number from 15.7% to 14.5% against an ideal of 20%. A real improvement, and small.

The much bigger improvement had already happened a week earlier, when we rewrote the exam. B4 — the flashcards test — had failed seven nights in a row. The cause was one module missing from the list of things its exam checked. Because the exam never mentioned that module, no task was ever told to build it, so the first task built everything instead. Repairing that one list fixed the behaviour before the code fix ever ran.

That pattern holds across all eight test cards, not just this one. Where every task owns exactly one testable piece, builds come out clean. Where tasks share pieces or a piece is unowned, they sprawl. It is not about the programming language, the size of the job, or how many tasks there are.

The blunt version: the quality of the specification predicts the quality of the build better than anything we have changed about the builder itself. We have been tuning the engine while the steering was the problem.

What it would actually do

Four stages, on the path you already use. Small jobs keep the current quick route — nothing here slows down "just build me X."

Stage 1 — a real interview. Instead of five fixed questions, it asks about purpose and audience, what you're actually delivering (a command-line tool, a web page, a service), whether you want real content or deliberate placeholders, the awkward cases in your own words, and — importantly — what is not in scope. Bounded to about one screenful and skippable; "just decide for me" already works and keeps working.
Stage 2 — writing the checklist together. The system already sorts every check into "the machine verifies this" versus "you look at this yourself." Two new rules get added. First, anything delivered as a web page must carry a machine-checked item saying the page actually loads and shows the thing — which is precisely the failure that parked the Bill Splitter test. Second, if a check claims to be machine-verifiable but nobody can name how a machine would verify it, it gets demoted to "you check this," rather than dressed up with a fake test.
Stage 3 — guaranteeing the coverage. The list of program pieces gets derived from your checklist, so that every build task owns exactly one testable piece by construction. Today the system notices coverage gaps and warns; in this mode it would try once to fix the gap, and if it still can't, it refuses to start and tells you where the hole is. You could still revise or reject — it is never a dead end.
Stage 4 — the approval card you already use. Unchanged. Same card, same six things you grade aloud, same approve / revise / reject.

An honest correction the sketch makes about itself

The first draft of this design overstated how much of it was new. It was rewritten against the actual source code, and the corrected version is considerably cheaper than it first appeared:

Sounds newActually already exists
Sorting checks into machine-verified vs judge-yourselfAlready built. Every check is sorted at creation, and the approval card already shows the two groups separately.
Storing the interview transcriptAlready built. A record is saved before every build, and it already carries your answers.
Measuring coverage gapsAlready built — last night's merge added it. Today it warns; this would make it block, in this mode only.
An interview stage at allAlready built and live. This extends its question set; it does not create it.

So the build is mostly extending three stages that already run, not adding a new front. No new model, no new background process, no new network access, and no change to the rule that the assistant cannot write to its own governed core.

A defect the design work found along the way

Your answers don't reach the exam author on multi-part jobs. When you answer the clarifying questions, those answers are supposed to inform whoever writes the acceptance exam. On single-task jobs they do. On multi-task jobs — the bigger, more valuable ones — the exam author receives the original short request instead, with your clarifications stripped. The code's own documentation explicitly claims both paths get them.

Verified directly, not taken on the sketch's word. The enriched version of your request is assembled and then handed to the single-task exam author; the multi-task exam author is handed the plain goal on the very next branch. One word apart, in the same function. The documentation above them describes the behaviour we want and not the behaviour that runs.

This is a bug, not a missing feature, and it is independent of the decision on this page — fixable on its own whether or not you green-light the intake. Two things make it low-risk to fix: the change is one identifier, and it is inert for the nightly test battery, whose cards supply their own plan and never go through the question stage. So it would only affect the jobs you start by hand — the ones where you actually answered questions. I have not fixed it; say the word and it's a small separate change with a regression test.

What it would buy — and what it cannot

Expected, based on what we've measured: fewer builds parked for reasons that were really specification gaps; web jobs crossing from "the logic exists" to "the page actually works"; no more invented placeholder content appearing without being asked for; less graphics-card time burned on vague requests; and every run's exam trustworthy because it was built to be, not because it happened to come out right.

What it cannot do: make the builder itself smarter — tasks stay bounded and machine-checkable either way. Make taste machine-checkable; "feels premium" stays your call by design. Or substitute for you on privacy, capability, and go-live decisions.

All four touch files that already exist. Each ships with tests proving it works and tests proving that with the flag off, the system behaves exactly as it does today — byte for byte.

The risks, stated plainly

  • Interrogation kills the tool. If the interview becomes tedious you will stop using the good path and go back to one-shotting. The design bounds it to one screenful and keeps "just decide for me" — but this is the real risk, and it is a judgement about your own patience that only you can make.
  • Blocking can frustrate. Stage 3 refuses to start when it can't close a coverage gap. That is the point, but a refusal you disagree with is annoying. It stays confined to the advanced mode; the quick path never blocks.
  • It is a capability addition to the part you touch. Hence dormant-by-default and a separate go-live with you present.
My recommendation: build it, in that order, dormant. The evidence for specification quality as the dominant lever is the strongest measured finding this system has produced, and it came from our own failure record rather than from theory. Piece 1 is worth having on its own merits even if you later decide the full interview isn't for you — the delivery check is exactly what the Bill Splitter failure asked for. I would also fix the exam-author bug regardless of what you decide here.

Three questions only you can answer

The sketch settled everything it could settle by reading the code. These three it deliberately left open, because they are product judgements rather than technical ones:

  • Should accepted specifications become reusable templates for similar future requests?
  • Should the interview be available by voice? The voice stack exists. Whether talking to it beats typing is your experience call.
  • Should the same machinery be used to author the test cards themselves? The seven-night B4 failure was a hand-written card with a flawed exam — this would prevent that class. It needs its own scoping because those cards live in the sibling installation.

None of them blocks the build decision. They shape what comes after it.
