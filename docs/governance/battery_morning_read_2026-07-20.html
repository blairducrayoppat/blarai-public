Battery Morning Read · Monday 2026-07-20 · revised

Cut off mid-sentence, nine times

Last night's coding experiment produced no usable result. Finding out why took three attempts, and my second answer — the one I sent you this morning — was wrong. An independent check caught it. This is the corrected read, and the real cause is both simpler and more fixable than what I told you.

The short answer

Two separate faults, one chronic and one acute.

Chronic: in our multi-part coding jobs, the first sub-task builds the entire application. Every downstream sub-task then has its nominal work already done. This happens on every night we have records for, across every job type — so our multi-part jobs have never really been testing multi-part building.

Acute: last night the coder was cut off mid-work nine consecutive times by a one-character formatting mismatch. It was not deciding anything. It was being severed.

The change we were testing never ran. The experiment has no result and must be re-run — on a differently-shaped job, or it will measure the same starved sub-task again.

First, what the battery is

Every night at 23:00 the machine gives your local coding assistant a small set of realistic programming jobs, from scratch, unattended, and grades them in the morning. It exists to answer one question over months: is the coding assistant actually getting better? It is the odometer, not the engine.

Last night ran two jobs. B2 is a text-statistics toolkit. B4 is a flashcard study program — the harder one, and the subject of this report.

The idea worth keeping In a column of elapsed times, “finished quickly” and “gave up early” are the same number. A duration means nothing without a measure of work completed alongside it.

Finding one: the job is finished before it starts

The flashcard job is deliberately split into six sub-tasks — save data, manage cards, run the quiz, track scores, build the command interface, write acceptance tests — so we can watch the assistant handle a build with real dependencies between parts.

The first sub-task builds the whole application. Asked only to handle saving data, it produced the card manager, the quiz engine, the score tracker and the main entry point. Verified as authored work, not merely as files lying around: the record is a comparison between two points in the project's own history, and the sandbox is rebuilt from empty every night. The first sub-task's commit added 170 lines across six files it was never asked to write.

And this is not last night's quirk. Across every one of the 18 recorded runs — all job types, not just flashcards — the first sub-task produces multiple source files. Not one run has a first task delivering the single piece it was assigned.

Why this matters more than one bad night We have been scoring these jobs as six deliverables. If the coder reasonably treats the request as one job and solves it in the first step, then every later sub-task is shadow-boxing, and the “N of 6 completed” figure has been measuring something other than capability — on every night in the archive. It changes how past results should be read, which is why it is the more important of today's two findings.

There is a genuine open question underneath, and it is a design question rather than a bug: a coder that solves the whole problem early is arguably doing well. The defect may be that we ask for six things and grade six deliverables when the work is really one. Forcing artificial fragmentation could make the output worse. That decision deserves to be made deliberately.

Finding two: nine severances, one character

Here is where I went wrong this morning, so I will show you the reasoning rather than just the conclusion.

I found that the sub-task which stopped had made 13 actions, every one a read, and no edits, and that the file it was meant to create already existed. I concluded it had looked, seen the work was done, and correctly declined to redo it — and that the harness had punished it for that restraint. It was a tidy story and I sent it to you.

It was wrong. The independent check read what the model actually said, which I had not done thoroughly enough. Every one of the nine attempts ends mid-sentence, announcing the next file it intends to open:

"Let me also check the score_tracker to see what's already implemented:" "Let me also check the main module to understand how the application flow works:" "I need to check the main.py to understand how the app is structured:"

Nine for nine. Not one produced a concluding statement. There is no already implemented, no nothing to do, no summary of any kind. The model was still gathering information when it stopped.

The cause is a single character. When the model asks to use a tool it writes a marker. Ours expects the form function=read. Last night the model wrote function/read — a slash instead of an equals sign. The component that repairs malformed markers matches only on the equals form, so it found nothing to repair, passed back an empty result, and the loop ended. The model's request to read a file simply evaporated, and from outside that is indistinguishable from the model choosing to stop.

Two details settle it beyond argument. The model had explicitly identified remaining work — it noted that the quiz function "doesn't seem to be fully implemented according to the requirements". And one attempt independently spotted the bug in our own grading script before being cut off. It was working, and working well, when it was severed.

The correction, stated plainly

I told you this morning that the coder was punished for being honest — that it correctly recognised completed work and our scoring penalised it. That is not true and I withdraw it. The harness did not punish restraint; it recorded a mechanical failure that looks like restraint from the outside.

I also called Saturday's work busywork. Also wrong: Saturday's sub-task wrote a genuine improvement — it made an interactive function testable by allowing input to be supplied programmatically, and added real tests for it. It spent a long time finding that adjacent work because its assigned work was already done, but the work itself was legitimate.

Why the experiment has no result

Last night was meant to test a change to how the coder's candidate work is filtered. The check confirmed directly: the part of that change which alters an outcome never executed. On the flashcard job no candidate produced the kind of file it acts on; on the text-statistics job it engaged but every file it examined was fine, so nothing was filtered. The change ran; its decision-making branch did not.

So there is nothing to compare. The A/B must be re-run — and on a differently-shaped job, or it will simply measure the same starved sub-task again.

One more thing worth knowing: neither night could have passed regardless of code quality. The script that grades the finished application is itself broken — it uses a component it never loads — and has been failing identically on every flashcard run since 2026-07-15. That is at least the sixth occurrence, not the third we believed. The fix is built and lands on its own night.

The safety check that has never once run

Separately, and found while investigating this morning's memory leak: the nightly run is supposed to refuse to start when free memory is too low, wait, and stand down at 04:00 rather than run badly.

That protection has never executed. The cause is one word: the logging helper was written in a form that, in this scripting language, quietly contributes to what the surrounding function returns. The memory check wrote a log line and then answered yes-or-no — and the log line joined the answer, producing a two-item list where a plain "no" was expected. A non-empty list counts as "yes". So the answer was always yes, and the waiting and the stand-down were unreachable from the day they were written.

The idea worth keeping This is “built but wired into nothing” in its purest form. The protection existed, was correct, was documented, and had never once been consulted. A control you have not watched refuse something is not yet known to work. It is now fixed, and ships with a test that deliberately puts the bug back and proves the test then fails.

The practical consequence: this morning's leaked assistant would not have skipped tonight — it would have let tonight run on a machine missing a third of its memory. A skipped night is a visible gap; a starved night produces plausible results that quietly mean less.

What changed today

The overnight leak is structurally fixed. A copy of the assistant had been left running since 00:13 holding 11.6 GB with nothing owning it. The night now claims that slot before it measures memory and releases it after the morning report, and the next night reclaims anything still holding it. All four ways an assistant can get started are covered — I found the fourth by accidentally reproducing the leak myself while testing. Your coordinator evidence was preserved: 252 records before the stop, 252 after.

The time-budget correction landed — the red banner atop every morning report is gone, and the calendar cutoff is now double-locked. Documentation asserting protections we do not have was corrected, including three claims found after the first pass, one of which was a message telling you at every startup that a ceremony had not run when it had.

Tonight's run is safe: memory is back to ~18.7 GB, and I tested the revived memory check against the real system to confirm it will let tonight proceed rather than wrongly stand down.

What happens next

The tool-call severance has a named fix site and is a small change — that goes first, because until it is fixed any night can lose its work the same way. The over-production question needs a decision from you rather than a patch: whether we re-grain these jobs so each sub-task has genuinely separate work, or accept that the coder solves them whole and grade them that way. And the A/B needs re-running once both are settled.

Where this comes from. Job records and task logs for runs 20260719-233631-bd and 20260719-002208-bd; the coder's own transcripts for all nine attempts; the sandbox project's git history; the tool-call repair component's source; a sweep of the first sub-task's output across all 18 recorded runs; and the memory-admission rules read from source rather than inferred from logs.

Whether the two remaining job types show the same over-production (they did not run in this window). What the right grain for these jobs actually is.
