# **Architectural Security Analysis of Local Multi-Agent AI Systems: Vulnerabilities, Controls, and Mitigations**

The rapid evolution of artificial intelligence from static, passive language models into autonomous, goal-directed agentic systems has fundamentally altered the cybersecurity landscape. Modern artificial intelligence agents are no longer confined to isolated text generation; they invoke computational tools, orchestrate multi-step workflows, query long-term memory structures, and interact with external networks and databases.1 This critical transition from probabilistic inference to real-world execution introduces profound security challenges, necessitating a radical departure from traditional perimeter-based network defenses. Traditional security infrastructure, which relies heavily on network packet inspection and static rule-sets, is ill-equipped to handle the semantic complexity of autonomous agents.2  
The comprehensive analysis presented in this report systematically evaluates the vulnerabilities inherent in multi-agent architectures. In accordance with the analytical parameters established, this report executes a phased methodology. Phase One extracts the foundational AI risks, vulnerabilities, and mitigations defined within the OWASP Top 10 for Large Language Model Applications (2025 version) and the provided architectural use cases, strictly without rendering recommendations. Phase Two synthesizes external research from highly trusted sources to explore the mechanics of these specific security risks—such as semantic transparency, intent-aware gatekeeping, hardware-backed memory isolation, and local inter-process communication—within the context of multi-agent architectures. Phase Three evaluates whether the proposed local multi-agent use cases properly address these contemporary threats. Finally, Phase Four concludes the analysis by providing strategic, architectural recommendations.

## **Phase 1: Extraction of Core AI Risks, Vulnerabilities, and Mitigations**

The foundation of this analysis requires a rigorous extraction of the known threat landscape. This extraction is derived from the OWASP Top 10 for LLM Applications 2025 and the specific architectural blueprints defining the local multi-agent system.

### **Vulnerabilities Identified in the OWASP Top 10 for LLM Applications 2025**

The OWASP Top 10 for Large Language Model Applications (2025 version) outlines the most critical vulnerabilities threatening generative AI deployments. A thorough review of this documentation reveals several risks that are uniquely amplified when applied to autonomous, agentic systems.4

#### **Prompt Injection (LLM01:2025)**

Prompt injection constitutes the most pervasive vulnerability in LLM integrations. It occurs when untrusted input alters the model's behavior or output in unintended ways.4 Crucially, these inputs do not need to be human-visible or readable; the exploit executes as long as the content is parsed by the underlying model.4 The OWASP framework delineates two primary vectors: direct prompt injections, where a user's input directly alters model behavior, and indirect prompt injections (IPI).4 Indirect injections occur when the LLM accepts input from external sources—such as websites, uploaded files, or external databases—that contain hidden data designed to alter the model's behavior when interpreted.4 The severity and nature of a successful prompt injection attack are largely dependent on the agency with which the model is architected.4 An agent equipped with execution capabilities can be manipulated to disclose sensitive information, manipulate critical decision-making processes, or execute arbitrary commands in connected systems.4 The OWASP documentation explicitly notes that techniques like Retrieval-Augmented Generation (RAG) and fine-tuning do not fully mitigate prompt injection vulnerabilities, and preventing complete "jailbreaks" requires ongoing updates to the model's fundamental safety mechanisms.4

#### **Sensitive Information Disclosure (LLM02:2025)**

Large language models, particularly those embedded within broader applications, risk exposing sensitive data, proprietary algorithms, or confidential details through their output.4 This vulnerability can lead to unauthorized data access, severe privacy violations, and intellectual property breaches. In multi-agent systems that constantly process user queries against private data stores, inadequate data sanitization can result in the model inadvertently regurgitating personal identifiable information (PII) or confidential business data.4 Mitigation strategies highlighted in the OWASP documentation include robust input validation, strict access controls based on the principle of least privilege, federated learning techniques, and data sanitization protocols.4

#### **Supply Chain Vulnerabilities (LLM03:2025)**

The LLM supply chain encompasses training data, third-party pre-trained models, and deployment platforms. The 2025 OWASP update emphasizes that open-access LLMs, fine-tuning adapters like Low-Rank Adaptation (LoRA), and on-device deployment frameworks introduce novel supply-chain risks.4 Vulnerable pre-trained models can harbor hidden biases or malicious backdoors that evade static inspection.4 Furthermore, compromised collaborative development processes or poisoned LoRA adapters can be merged with existing local models, providing attackers with covert entry points into the system.4 Mitigations require maintaining accurate Software Bill of Materials (SBOMs), utilizing third-party model integrity checks with cryptographic signing, and applying comprehensive AI red teaming during model selection.4

#### **Data and Model Poisoning (LLM04)**

Data poisoning represents an integrity attack where pre-training, fine-tuning, or embedding data is intentionally manipulated by malicious actors to introduce vulnerabilities, backdoors, or systemic biases.4 In local multi-agent architectures heavily reliant on RAG, poisoning the embedding data can compromise the model's ethical behavior and operational security, effectively turning the model into a "sleeper agent" waiting for a specific trigger.4 Mitigation requires strict verification of data origins, the use of data version control (DVC), robust sandboxing during training, and continuous adversarial robustness testing.4

#### **Improper Output Handling (LLM05:2025)**

Improper output handling refers to the insufficient validation, sanitization, and handling of the outputs generated by large language models before those outputs are passed downstream to other components and executing systems.4 Because LLM-generated content can be controlled by prompt input, failing to scrutinize the output is functionally equivalent to providing users indirect access to backend functionality.4 In agentic workflows, this can result in Server-Side Request Forgery (SSRF), privilege escalation, or remote code execution if the LLM output is entered directly into a system shell or database query.4 Mitigating this vulnerability requires treating the model as an untrusted user, applying zero-trust input validation on all model responses, and utilizing parameterized queries for all database operations.4

#### **Excessive Agency (LLM06:2025)**

The 2025 OWASP revision heavily expanded the concept of Excessive Agency to address the proliferation of agentic architectures.4 This vulnerability enables damaging actions to be performed in response to unexpected, ambiguous, or manipulated outputs from an LLM. The root cause of Excessive Agency is typically a combination of excessive functionality, excessive permissions, and excessive autonomy.4 When agents are granted open-ended extensions or privileged identities, a simple hallucination or indirect prompt injection can cascade into a catastrophic system compromise.4 The OWASP text emphasizes that mitigation requires limiting the extensions that LLM agents are allowed to call, minimizing the permissions granted to those extensions, avoiding open-ended execution environments, and enforcing human-in-the-loop approvals for all high-impact actions.4

#### **System Prompt Leakage (LLM07:2025)**

System prompt leakage refers to the risk that the foundational instructions used to steer the behavior of an LLM are extracted by an unauthorized user.4 While the prompt itself should not be considered a cryptographic secret, developers frequently embed sensitive information—such as internal rules, filtering criteria, and descriptions of backend infrastructure—within the system prompt.4 If an attacker maps the internal role structures and permission levels of the application via prompt extraction, they can formulate highly targeted privilege escalation attacks. Mitigation requires strictly separating sensitive data and credentials from system prompts and ensuring that security controls are enforced deterministically by independent systems outside the LLM's cognition.4

#### **Vector and Embedding Weaknesses (LLM08:2025)**

Retrieval-Augmented Generation (RAG) introduces profound vulnerabilities regarding how vectors and embeddings are generated, stored, and retrieved.4 Inadequate access controls can lead to the unauthorized retrieval of highly sensitive personal data. Furthermore, embedding inversion attacks demonstrate that mathematical vectors can be reversed to recover significant amounts of the original plaintext source information.4 Multi-tenant environments also face the risk of cross-context information leaks, where data from multiple sources contradict each other or bleed across user boundaries.4 Mitigations require implementing fine-grained, permission-aware vector stores, strict logical partitioning of datasets, and robust data validation pipelines.4

#### **Unbounded Consumption (LLM10:2025)**

The high computational demands of large language models make them inherently vulnerable to resource exploitation. Unbounded consumption occurs when users or autonomous agents conduct excessive inferences, leading to denial of service, rapid economic losses (Denial of Wallet), and systemic operational disruptions.4 Attackers can overload the system with variable-length input floods or continuously send queries that exceed the context window, draining CPU and memory resources.4 In local deployments, mitigating this risk demands strict input validation, rate limiting, dynamic resource allocation management, and the implementation of hard timeouts for resource-intensive operations.4

| :---- | :---- | :---- |
| LLM01: Prompt Injection | Malicious instructions embedded in retrieved data override agent goals. | Strict input handling; separation of instruction and context. |
| LLM06: Excessive Agency | Agents execute destructive tools due to unchecked autonomous permissions. | Human-in-the-loop controls; least-privilege tool access. |

### **Architectural Risks and Mitigations Extracted from Use Cases**

The architectural blueprint (use\_cases.txt and iteration\_directive v3\_4.md) outlines a complex, highly constrained local multi-agent system operating under a strict 32GB RAM ceiling. The design inherently recognizes several specialized risks and proposes built-in architectural mitigations.

#### **Semantic Transparency and Intent-Aware Gating (Use Case 001\)**

The primary risk identified in the system's core orchestration is "Semantic Transparency".4 Autonomous agents generate non-deterministic tool calls that can easily encapsulate highly sensitive data—such as proprietary source code—within legitimate-looking outbound payloads.4 A compromised agent could construct a syntactically valid HTTP POST request that actually serves as a covert exfiltration vehicle for private journal entries. Because standard network-layer devices (routers and firewalls) operate exclusively on IP, port, and protocol, they are entirely blind to the semantic intent of an AI-generated payload.4  
To mitigate this, the architecture proposes "The Policy Agent," a continuously resident, NPU-driven semantic gatekeeper.4 Operating on a 1.7B-class instruction-tuned LLM at INT4 quantization, this gatekeeper intercepts, classifies, and adjudicates every single tool-call and data-access request generated by any agent in the system.4 It enforces intent-level access control based on semantic sensitivity classification, rather than relying on network headers.4 Crucially, the Policy Agent operates under a strict "Fail-Closed" architecture: all unclassified, ambiguous, or potentially sensitive payloads are blocked by default and escalated to the user for manual approval.4

#### **Hardware Roots-of-Trust and Process Isolation**

To ensure the integrity of the Policy Agent and the broader system, the architecture mandates the use of hardware roots-of-trust. Microsoft Pluton serves as the candidate secure key vault for signing and verifying tool-call tokens, ensuring that only attested agents can issue requests.4 Intel SGX (Software Guard Extensions) enclaves are proposed for runtime attestation, guaranteeing that the Policy Agent's own code remains untampered.4 Furthermore, the Policy Agent sits architecturally inside the vsock (AF\_HYPERV) inter-process communication bus, ensuring all inter-VM messages are intercepted before reaching the traditional network stack.4

#### **Context Fragmentation and Zero-Trust Memory (Use Case 002\)**

The system identifies "Context Fragmentation" as a core operational risk. Without a unified memory structure, individual agents build siloed, incomplete views of the user's knowledge, duplicating storage and diverging in factual grounding.4 However, creating a centralized, universal "Personal Truth" index creates a massive exfiltration target—a semantic fingerprint of the user's entire intellectual life.4  
To mitigate this, the architecture introduces "The Substrate," a persistent vector index serving as the universal, passive, queryable long-term context provider.4 To prevent unauthorized access, the Substrate resides exclusively within a TDX-isolated Virtual Machine. Intel Trust Domain Extensions (TDX) ensure that the vector index, the embedding model, and the query interface are hardware-encrypted at rest and accessible only via vsock IPC calls explicitly authenticated by the Policy Agent.4

#### **Inference Pollution and Adversarial Sanitization (Use Case 003\)**

Feeding raw, noise-laden inputs into large language models leads to "Inference Pollution." Non-semantic noise—such as advertisements, tracking scripts, and HTML markup—wastes the highly constrained 32GB RAM capacity and critically degrades RAG retrieval quality.4 Furthermore, the ingestion of external data introduces the risk of Indirect Prompt Injection (OWASP LLM01), where web or email text contains hidden adversarial prompts.4  
The architectural mitigation is "The Cleaner," a modular data normalization utility operating inside a Hyper-V isolated container to sandbox potentially weaponized documents.4 The iteration directives mandate severe metric hardening for The Cleaner: noise-stripping fidelity must reach ≥ 99%, and semantic recall must hit ≥ 98.5% via a two-stage retrieval pipeline utilizing a Bi-encoder and a CPU-based Cross-Encoder re-ranker.4 Any low-confidence parses trigger a "Fail-Closed" quarantine.4 Furthermore, The Cleaner must include an explicit "adversarial sanitization layer" to neutralize stored prompt injections before they reach the Substrate.4 To bypass cloud surveillance during data ingestion, the system utilizes a Mobile LAN Ingress Proxy, transferring data securely over the local network via TLS-encrypted connections.4

#### **Unbounded Consumption and Model DoS (Use Cases 004 & 005\)**

The conversational "Assistant" and the interactive local "Coder" face immense risks regarding Unbounded Consumption (OWASP LLM04/LLM10). Recursive AI loops or infinite reasoning chains could rapidly exhaust the 25.6GB operational RAM ceiling, causing host operating system kernel panics.4  
To mitigate this, the architecture requires an explicit bifurcation of RAM footprints. The Assistant is split into a lightweight Semantic Router (\~128MB, BAAI/bge-small-en-v1.5 ONNX FP16 — measured 127.8 MB) and a heavier Assistant Orchestrator (\~1014 MB, Qwen3-1.7B OpenVINO INT4 — measured 1014.0 MB).4 Furthermore, the system mandates strict "circuit breakers" including maximum recursion depth limits and token output hard-caps.4 For the Coder agent specifically, the architecture limits risk by defining it as a "Headless Microservice" with no user interface, callable only via the Policy Agent.4 To prevent destructive whole-file overwrites, the Coder is restricted to deterministic Unified Diffs and is bound by mandatory Git atomic operations, ensuring strict rollbacks if the compile-test loop fails or exceeds the recursion limit.4

## **Phase 2: External Research Synthesis on Local Multi-Agent Architectures**

Having extracted the foundational risks and proposed mitigations from the internal documentation, it is necessary to contextualize these concepts against contemporary cybersecurity research. The transition to agentic AI fundamentally alters attack patterns, necessitating highly specialized hardware and software controls.

### **The Reality of Semantic Transparency and Data Exfiltration**

The risk of semantic transparency and the inadequacy of traditional network firewalls are extensively documented in recent security analyses. Traditional Data Loss Prevention (DLP) and Security Information and Event Management (SIEM) systems are architected to analyze network metadata, file hashes, and specific regular expressions; they are inherently incapable of interpreting the semantic context of an AI-based inference request.3 As a result, they fail to detect "semantic exfiltration," where an agent is manipulated into extracting sensitive information—such as API keys, database credentials, or private communications—and weaving that data into innocuous-looking text prompts or authorized API requests.3  
Real-world exploits demonstrate the severity of this vulnerability. At the Black Hat USA 2025 conference, security researchers demonstrated working exploits across major enterprise AI platforms, showing that a single crafted email could trigger an AI assistant to autonomously hand over access to connected cloud drives, while other agents leaked underlying CRM databases.5 The discovery of the "EchoLeak" vulnerability (CVE-2025-32711) further emphasized this threat; researchers proved that receiving a single crafted email was sufficient to trigger automatic data exfiltration without any user interaction.5 Furthermore, researchers utilizing "Pandora" Proof-of-Concept architectures have demonstrated how multi-modal AI agents can process hidden instructions embedded within seemingly benign images or MS Word documents, leading to zero-click exfiltration events.6 Because modern AI systems treat all ingested text as meaningful, a single hidden instruction in a web page or PDF can redirect reasoning, leak data, or trigger harmful tool actions—an attack vector fundamentally enabled by the blending of trusted system instructions and untrusted external data within the same context window.7

### **Intent-Aware Gatekeeping and the Action Authorization Boundary**

To combat the phenomenon of semantic exfiltration, the cybersecurity research community has developed the concept of the Action Authorization Boundary (AAB), a theoretical framework that aligns closely with the proposed "Policy Agent." The Faramesh protocol represents a leading implementation of this concept. Faramesh is defined as a protocol-agnostic execution control plane designed specifically for autonomous agent systems.9  
The core thesis of the AAB is that autonomous systems require a mandatory, non-bypassable architectural boundary placed explicitly between the agent's internal reasoning space and the external execution environment.9 Under the Faramesh protocol, all proposed side-effecting operations are reduced to a Canonical Action Representation (CAR).9 This canonicalization strips away incidental formatting, allowing for deterministic policy evaluation.9 The AAB then evaluates these actions against defined policies, issuing a cryptographic decision artifact (Permit, Defer, or Deny).9 The execution layer is hard-coded to reject any action lacking this valid cryptographic receipt. Crucially, the Faramesh protocol operates on a deterministic, "fail-closed" mechanism: any boundary failure, missing verification, or ambiguous intent immediately results in action denial.9  
This framework effectively separates cognition from control. Rather than attempting to force the primary agent to "behave safely"—a mathematically unproven proposition given the probabilistic nature of LLMs—the system enforces deterministic security constraints at the moment of execution.9 The adoption of Managed Control Planes (MCP) serves a similar architectural purpose, standardizing how AI agents interact with enterprise data while enforcing predetermined fail-safe behaviors and internal trust registries.12

### **Hardware-Backed Execution Environments: TDX, SGX, and Pluton**

The physical security of local AI memory and orchestration relies heavily on Trusted Execution Environments (TEEs). The "MemTrust" architecture exemplifies the cutting-edge application of TEEs in AI systems. MemTrust proposes a zero-trust AI memory framework where all critical functions—extraction, embedding generation, storage, and retrieval—are executed entirely within hardware-backed enclaves.13 This guarantees that even if an adversary achieves root access to the host operating system, they cannot observe the computation or decrypt the context data.14  
Intel Trust Domain Extensions (TDX) is a premier technology for enabling these environments. Intel TDX creates hardware-isolated virtual machines (Trust Domains) that isolate the guest OS and applications from the cloud host, hypervisor, and other VMs.15 It utilizes features like Secure Arbitration Mode (SEAM) and Multi-Key Total Memory Encryption (TME-MK) to protect the confidentiality and integrity of the CPU state.15 By removing the hypervisor from the trust boundary, TDX ensures that altering the underlying hypervisor does not grant access to the memory of the Trust Domain.16 However, hardware isolation is not infallible. A joint security review of Intel TDX 1.5 conducted by Microsoft and Intel scrutinized 29 attack vectors, yielding multiple vulnerabilities that compromised the core promise of integrity, including the ability to forge attestation data or execute a complete TDX module takeover under specific conditions.17 This highlights the necessity of continuous validation and defense-in-depth, even when utilizing advanced TEEs.  
While TDX secures entire virtual machines, Intel Software Guard Extensions (SGX) provides process-level isolation, allowing the creation of highly specific secure enclaves for individual applications.18 This is particularly useful for protecting localized algorithms and lightweight cryptographic processes without virtualizing a full operating system.  
Complementing these technologies is Microsoft Pluton, a secure crypto-processor built directly into the CPU die.19 Unlike discrete Trusted Platform Modules (TPMs) that are vulnerable to external communication channel attacks (such as intercepting the bus between the CPU and the TPM), Pluton integrates security at the core.19 Pluton generates its own cryptographic key-pairs directly in silicon during manufacturing, storing them persistently in e-fuses.21 Because these private keys are never exposed outside the protected hardware—not even to the Pluton firmware itself—it provides an unprecedented level of secure identity and remote attestation.22

### **Inter-Process Communication via Vsock**

In highly compartmentalized multi-agent systems, the method of communication between isolated components is as critical as the components themselves. The AF\_VSOCK address family (vsock) facilitates bi-directional, high-speed communication between a hypervisor and its guest virtual machines, or between isolated enclaves, utilizing a kernel-mediated transport layer.23  
Because vsock does not rely on the host's standard TCP/IP networking stack, it allows virtual machines to be configured entirely without network interfaces.24 This architectural choice drastically reduces the attack surface; malware or lateral movement tools scanning traditional IP subnets are completely blind to vsock communications. Connections are established using a 32-bit Context Identifier (CID) and a port number, routing traffic entirely within the kernel space.23 Frameworks like AWS Firecracker utilize virtio-vsock to provide secure, microVM isolation, achieving boot times under 125ms while maintaining robust boundaries.26 However, security researchers note that while vsock eliminates external network threats, it lacks inherent cryptographic security; if an attacker compromises a virtual machine that possesses a valid CID, they can utilize the vsock channel to attack internal host services unless additional protocol-level security (like mutual TLS) is enforced over the socket.28

| Security Concept | Mechanism of Action | Mitigation Focus |
| :---- | :---- | :---- |
| **Action Authorization Boundary** | Intercepts agent tool calls, reduces them to Canonical Action Representations, and applies deterministic policy rules. | Prevents semantic exfiltration and Excessive Agency. |
| **Intel TDX** | Hardware-level isolation of Virtual Machines using memory encryption, removing the hypervisor from the trust boundary. | Prevents host-level compromise of vector databases (MemTrust). |
| **Microsoft Pluton** | In-silicon secure crypto-processor utilizing e-fuses to generate and store immutable cryptographic keys. | Prevents bus-interception attacks; enables robust token signing. |
| **AF\_VSOCK** | Kernel-mediated IPC bridging VMs and hosts using CIDs without utilizing the traditional TCP/IP network stack. | Prevents network-based lateral movement and unauthorized port scanning. |

## **Phase 3: Evaluation of Use Cases Against Synthesized Risks**

The synthesis of external research provides a rigorous benchmark for evaluating the proposed local multi-agent architecture. By cross-examining the architectural blueprints (use\_cases.txt and iteration\_directive v3\_4.md) against the established threat landscape, it is possible to assess the efficacy of the proposed controls and identify residual vulnerabilities.

### **Evaluation of Use Case 001: The Policy Agent**

The implementation of the Policy Agent perfectly encapsulates the principles of the Action Authorization Boundary (AAB).4 By acting as an intent-aware semantic gatekeeper operating on a "fail-closed" architecture, it effectively addresses the vulnerabilities associated with LLM06 (Excessive Agency) and Semantic Transparency.3 The utilization of Microsoft Pluton for hardware-attested identity verification and tool-call token signing ensures that only authenticated agents can execute actions, effectively creating a zero-trust execution perimeter.4  
**Vulnerability Gap:** The architecture proposes running the Policy Agent on the local NPU utilizing a 1.7B-class INT4 quantized model.4 While this satisfies the strict latency and hardware efficiency requirements of the system, relying on a sub-3B parameter model for semantic security classification introduces severe risks. Contemporary evaluations of operational safety demonstrate that models within the 1.7B to 8B parameter range frequently fail to reliably identify Out-Of-Domain (OOD) or adversarially transformed queries.30 When malicious payloads are obfuscated or embedded within prompts that superficially resemble benign inputs (adaptive OODs), small-parameter models exhibit failure rates exceeding 70%.30 Consequently, an advanced attacker utilizing sophisticated prompt injection techniques could likely bypass the 1.7B Policy Agent, manipulating it into classifying a data exfiltration attempt as a legitimate operational tool-call.

### **Evaluation of Use Cases 002 & 003: The Substrate and Data Normalization**

The design of "The Substrate" represents a highly secure, privacy-preserving approach to long-term AI memory. By encapsulating the vector database, the embedding model, and the query interface within an Intel TDX-isolated virtual machine, the architecture closely mirrors the MemTrust zero-trust paradigm.4 This robustly mitigates the risks of Vector and Embedding Weaknesses (LLM08) and Context Fragmentation by ensuring that the user's comprehensive semantic fingerprint is cryptographically inaccessible to the host OS and external cloud providers.4  
The "Cleaner" utility addresses Inference Pollution and Unbounded Consumption by mandating a ≥ 99% noise-stripping fidelity and utilizing a sophisticated Bi-encoder/Cross-Encoder retrieval pipeline.4 Furthermore, deploying The Cleaner within a Hyper-V isolated worker jail neutralizes the threat of standard malformed file attacks, such as weaponized PDFs executing arbitrary code during the ingestion process.4  
**Vulnerability Gap:** To address Indirect Prompt Injection (LLM01), the iteration directives mandate the inclusion of an "adversarial sanitization layer" within The Cleaner.4 While necessary, pre-processing sanitization is an inherently probabilistic defense against an evolving semantic threat.8 Indirect prompt injection exploits the semantic processing capabilities of the LLM; because the system fundamentally treats all retrieved content as equally trustworthy within the context window, a sufficiently novel, obfuscated "jailbreak" hidden in a downloaded PDF may evade the sanitization layer.7 If the sanitized data retains semantic traces of the attack, the prompt injection will execute when the Assistant retrieves the data from the Substrate.

### **Evaluation of Use Cases 004 & 005: Assistant Orchestration and Coder Agent**

The architecture's approach to orchestration and code generation rigorously addresses Unbounded Consumption and Model Denial of Service (LLM10). The explicit bifurcation of the system into a lightweight Semantic Router (\~128MB, BAAI/bge-small-en-v1.5 — measured 127.8 MB) and a heavier Assistant Orchestrator (\~1014 MB, Qwen3-1.7B INT4 — measured 1014.0 MB) ensures that intent classification does not unnecessarily deplete the 32GB RAM ceiling.4 The implementation of strict circuit breakers—including maximum recursion depth limits and token output hard-caps—provides deterministic barriers against infinite reasoning loops.4  
The "Headless" configuration of the Coder microservice, constrained by mandatory Git atomic operations and forced to output deterministic Unified Diffs, excellently mitigates the risks of autonomous codebase destruction or runaway algorithmic editing.4  
**Vulnerability Gap:** The system correctly isolates skill execution and inter-agent communication using the vsock (AF\_HYPERV) bus, neutralizing external TCP/IP vulnerabilities.4 However, vsock inherently operates on a perimeter-defense logic *within* the hypervisor. It identifies connections via Context Identifiers (CIDs).23 If an attacker successfully executes an indirect prompt injection that compromises a Hyper-V worker jail (such as a specific Assistant Skill), that compromised container still retains its valid CID.29 Because the vsock protocol lacks internal cryptographic security, the compromised jail could potentially utilize its CID to attempt unauthorized queries against the Substrate or other internal microservices if the Policy Agent is bypassed.

## **Phase 4: Strategic Recommendations**

The architectural design outlined in the use cases demonstrates a sophisticated, defense-in-depth approach to securing local multi-agent systems, successfully neutralizing many of the vulnerabilities identified in the OWASP Top 10 for LLM Applications 2025\. However, the evaluation reveals specific limitations regarding probabilistic gatekeeping, side-channel memory risks, and internal communication protocols. Based on the synthesis of the preceding analysis, the following strategic recommendations are provided to harden the architecture.

### **1\. Augment the NPU Gatekeeper with Deterministic Execution Controls**

Relying exclusively on a 1.7B parameter LLM running on the NPU to adjudicate semantic intent is a profound security vulnerability, given the documented failure rates of small models against adversarially transformed queries.

* **Implementation:** The Policy Agent must transition from purely probabilistic classification to a hybrid deterministic model, adopting the precise mechanisms of the Faramesh protocol. The system must enforce Canonical Action Representation (CAR) for all inter-agent communications. Before the NPU model evaluates an action, the requesting agent must format its tool-call into a strict, deterministic schema.  
* **Cryptographic Artifacts:** The NPU gatekeeper must be supplemented by a secondary, deterministic algorithmic checker (e.g., regex constraints and semantic distance calculations). Only if both systems approve the action should the Policy Agent utilize the Microsoft Pluton secure crypto-processor to mint a cryptographically signed "Decision Artifact" or Agentic JWT. The destination microservices (such as the Coder or external API tools) must be hard-coded to reject any execution request that lacks this unforgeable receipt, guaranteeing a non-bypassable Action Authorization Boundary.

### **2\. Enforce Strict Context Spotlighting in RAG Pipelines**

The "adversarial sanitization layer" inside the Data Normalizer is a necessary defense but will eventually fail against novel indirect prompt injections. The architecture must assume that poisoned data will occasionally penetrate the Substrate.

* **Implementation:** To mitigate the impact of poisoned data during retrieval, the Assistant Orchestrator must implement strict context isolation, utilizing techniques such as "Spotlighting." The system prompt governing the orchestrator must utilize explicit, unforgeable structural markers (e.g., specialized data-fences or unique cryptographic delimiters) to strictly delineate untrusted retrieved data from foundational system instructions.  
* **Execution Isolation:** The architecture must enforce a least-privilege design regarding retrieved context. Data retrieved from the Substrate must be treated as inherently hostile and explicitly prohibited from influencing control-flow tokens, tool-invocation logic, or system-level write operations, ensuring that even if a prompt injection is ingested, it remains inert data rather than executable instruction.

### **3\. Fortify Vsock IPC with Mutual TLS (mTLS) Authentication**

While vsock successfully eliminates the risk of traditional network sniffing and external lateral movement, relying solely on Context Identifiers (CIDs) for internal routing is insufficient in a zero-trust architecture. If a worker jail is compromised, its CID can be weaponized.

* **Implementation:** The architecture must mandate mutual TLS (mTLS) or strict protocol-level cryptographic handshakes over all vsock connections. The receiving endpoint within the vsock architecture (such as the interface to the TDX-isolated Substrate) must validate the ephemeral Agentic JWT minted by the Policy Agent before allocating buffer space or processing the incoming Remote Procedure Call (RPC). This ensures that even a compromised jail cannot establish a functional connection without explicitly invoking the gatekeeper's authorization protocol.

### **4\. Implement Side-Channel Obfuscation within the TDX Substrate**

Intel TDX provides robust protection against direct memory reads by the host operating system or malicious hypervisors. However, TEEs remain theoretically vulnerable to advanced side-channel attacks, such as memory access pattern analysis or timing attacks, which could allow a highly privileged adversary to deduce the user's semantic profile.

* **Implementation:** The Substrate's retrieval logic must incorporate cryptographic obfuscation techniques. The system should integrate Oblivious RAM (ORAM) protocols or dummy/fake bucket sampling algorithms during vector retrieval. By systematically obfuscating the exact memory vectors being accessed during a query, the system neutralizes the threat of a compromised adjacent virtual machine deducing the nature of the data through precise timing analysis, ensuring the absolute privacy of the lifelong semantic memory backbone.

### Phase 3: Evaluation of Use Cases Against Synthesized Risks (Continued)

#### Evaluation of Use Case 009: Autonomous System Maintainer

The Autonomous Maintainer introduces the highest inherent risk in the architecture by explicitly requesting two highly dangerous capabilities: outbound network access and system-level write access. This directly invokes the risks of Supply Chain Vulnerabilities (OWASP LLM03) and Excessive Agency (OWASP LLM06). The architecture mitigates this by restricting network egress to an explicit URL allowlist and enforcing a "staging-only" write access model, ensuring no changes are pushed to production without human review.

**Vulnerability Gap:** The reliance on a "staging-only" area protected by explicit user approval is highly susceptible to "human-in-the-loop fatigue." Attackers continuously innovate in supply-chain poisoning by embedding malicious payloads deep within transitive dependencies, poisoned LoRA adapters, or obfuscated Python scripts. If the Maintainer agent autonomously fetches a poisoned dependency from a trusted source (e.g., a compromised PyPI package) and presents a massive, complex codebase diff to the user, the human operator lacks the capacity to manually verify every line. The human will eventually "rubber-stamp" the approval, effectively allowing the agent to bypass its own security constraints and introduce malicious code into the host.

### Phase 4: Strategic Recommendations (Continued)

To address the vulnerabilities identified in Use Case 009, the following architectural enhancements are recommended to finalize the local multi-agent security posture:

#### 5. Implement TEE-IO (TDX Connect) for GPU Workloads

To resolve the trust-boundary leakage when offloading code-generation inference to the Arc iGPU, the architecture must transition from basic CPU isolation to full device isolation.

* **Implementation:** The system must leverage **Intel TDX Connect**, which utilizes the TEE Device Interface Security Protocol (TDISP). TDX Connect extends the hardware-encrypted trust boundary across the PCIe/internal bus directly to the GPU. This ensures that the GPU memory utilized for sensitive inference context windows remains cryptographically inaccessible to the host OS, maintaining the integrity of the TDX enclave during inference and enabling secure post-inference cryptographic zeroing.



#### 6. Defend Against Human-in-the-Loop Fatigue via Cryptographic Supply Chain Verification

Human review of the Autonomous Maintainer's proposed updates must be supplemented by deterministic, automated supply-chain verification to prevent sophisticated poisoning attacks from slipping through the staging area.

* **Implementation:** Before the Maintainer agent is permitted to propose an update to the user, it must be required to generate or verify an AI Bill of Materials (AIBOM/SBOM) for the incoming dependency. Furthermore, the system must integrate cryptographic provenance checks (such as Sigstore or in-toto) to verify the digital signatures of the upstream packages. If the update lacks a valid signature or introduces unverified transitive dependencies, the Policy Agent must systematically deny the proposal before the user is even prompted, acting as a deterministic filter against human fatigue.
