## The false claim in `agentic-setup fix/1074-fail-loud-git-capture "capture fault must NOT be terminal"` — read this before trusting that commit message

`agentic-setup fix/1074-fail-loud-git-capture "capture fault must NOT be terminal"` is a real commit on `fix/1074-fail-loud-git-capture` and its message will read as fact to
anyone who finds it. **Its safety argument is false as written, and this record is the correction.**

The message defends making a capture fault non-terminal by asserting the fault "stays LOUD by every
other route", listing five: disqualification (`Test-IsCandidateGreen`), rank (`Get-CandidateRank`),
the merge block (`Test-ShouldMerge`), the review-FIX loop, and the ERRORED report. **Four of those
five only affect the candidate that faulted.** The fifth — the report — is the only one an operator
ever sees, and on that exact tip it did not fire. Measured, in both candidate orderings:

```
[noop,FAULT,noop] ran=3 selectedIdx=0 selectedGitFailed=False anyFault=1  REPORT='none made'
[FAULT,noop,noop] ran=3 selectedIdx=1 selectedGitFailed=False anyFault=1  REPORT='none made'
```

The mechanism: before `agentic-setup fix/1074-fail-loud-git-capture "capture fault must NOT be terminal"`, terminality meant a fault in candidate 1 left it the *only*
candidate, so it was selected and reported ERRORED. That was accidentally the one path making a lone
fault visible. Removing terminality — which was the correct decision — removed the visibility with
it: sampling continues, a sibling no-op outranks the faulted candidate (correctly), and the fault's
git stderr and uncaptured work vanish into `fleet-report`'s "no change made" column.

**Consequence, stated plainly: between `agentic-setup fix/1074-fail-loud-git-capture "capture fault must NOT be terminal"` and `agentic-setup fix/1074-fail-loud-git-capture "review fixes F3-F1-F2"`, `CHANGES: none made` was LESS
trustworthy than before this ticket began**, in the battery's default N=3 configuration. That is
#1074's own defect class, re-created by the fix for it, and certified safe by a commit message whose
author had not checked the list he was citing. `agentic-setup fix/1074-fail-loud-git-capture "review fixes F3-F1-F2"` makes the claim true by aggregating over
`$bon.Candidates` rather than reading only the selected candidate.

The decision in `agentic-setup fix/1074-fail-loud-git-capture "capture fault must NOT be terminal"` stands and should not be reverted; only its stated justification was
wrong. Anyone reading that message should treat "stays LOUD by every other route" as false until
`agentic-setup fix/1074-fail-loud-git-capture "review fixes F3-F1-F2"`, and should note the general form: **detection and surfacing are different properties, and
a route that only disqualifies a candidate is not a route that tells anyone.**

## A row that said FIXED while the fix was partial

The row `F2-revlist-exit-code-unchecked | FIXED | agentic-setup fix/1074-fail-loud-git-capture "review fixes F3-F1-F2"` went into this record after `agentic-setup fix/1074-fail-loud-git-capture "review fixes F3-F1-F2"`,
and `verify_disposition.py` passed the whole file 11/11. It passed because **the verifier checks the
FORM of a record that exists — it cannot see that a FIXED row is only partly fixed.** That is the
honest limit `deferral_discipline` states about itself, and this is a live instance of it: a green
gate sitting on top of an inaccurate claim.

What was actually missing: F2's narrowing. Faulting on a non-zero `rev-list` alone cannot tell "I
cannot resolve the baseline you were given" from "I cannot read this repo's history", so a healthy
repo on branch `trunk` handed the fallback base name `main` was classified as a capture fault and
would have become permanently, silently ERRORED. Reproduced end to end on the tip that carried the
FIXED row. Fixed at `agentic-setup fix/1074-fail-loud-git-capture "F2 narrowing"` by verifying the base resolves first
(`git rev-parse --verify <base>^{commit}`) and giving the two cases different reasons.

Measured on `agentic-setup fix/1074-fail-loud-git-capture "F2 narrowing"`, the same reproduction:

```
repo health: HEAD=agentic-setup fix/1074-fail-loud-git-capture "earlier build commit"  dirty=0  branch=trunk
rev-list with BaseBranch fallback 'main' : exit=128  raw=''
base 'main' resolvable = False
Resolve-CommitCapture -> Failed=False  Reason='base-unresolvable'
END-TO-END -> GitFailed=False  Reason='base-unresolvable'  HasChanges=True  SHA=agentic-setup fix/1074-fail-loud-git-capture "earlier build commit"...
```

The last line matters as much as the first: a healthy *build* with a mis-set base keeps its real
commit rather than being laundered into "none made".

**Two claims contradicted by measurement in one evening, one commit apart** — `agentic-setup fix/1074-fail-loud-git-capture "capture fault must NOT be terminal"`'s message
asserting the ERRORED route still fired, and this row asserting FIXED. Both were written by the
person who had just done the work, both sounded settled, and both were checked by something that
could not see the gap: a test suite that validated what I built against what I meant, and a verifier
that validated the shape of a sentence rather than its truth. In a ticket whose entire subject is a
system reporting things it cannot support, that is not irony, it is the same defect operating one
layer up — and the layer it operates on is the record a later session will trust.

## Notes a later session may want

**F4 and the gitignored-output row are the same class, not two bugs.** Both are "a thing that exists
in the worktree which `git add -A` will not stage". F4 is the case `status` *can* see (false alarm);
the gitignored case is the one `status` *cannot* see (false silence). Fixing the class means
deriving the invariant's dirty count from the same predicate `git add` uses, rather than from
`status` — at which point the two cannot disagree by construction. Fixing only F4 leaves the more
dangerous half open, so #1088 should be worked as the class.

**What the review validated and should not be re-litigated:** the honest-no-op discrimination.
Reading the index (`git diff --cached --name-only`) and only attempting a commit when the staged set
is non-empty means the honest "nothing to commit" never enters the failure channel, so a non-zero
commit exit code is unambiguous — no locale dependency, no git-version dependency, nothing keyed on
message text. The reviewer attempted to break it and could not.

**The recurring shape across F3, F5, and the four de-anchored locks is one lesson, not four
findings:** a lock pinned to prose rather than to the property it protects. Over-tight (breaks on
any nearby edit, trains the next session to widen it) and under-tight (accepts a neutered guard) are
the same failure wearing opposite costumes. The vacuity guard added as W2b — asserting that a
`[kill]` pattern actually matches a literal specimen of the thing it kills — is the generalisable
control, and it is worth spreading to the other `[kill]` asserts in the `agentic-setup` verify
suites.
