# Critical loops for local autonomous code + design generation — research synthesis + design

**Date:** 2026-06-26. **For:** the headless-coding dispatch (#670) critical-coding-loop + critical-design-loop build (Vikunja #687; design-loop rigor #685). **Status:** research DONE; design [PROPOSED]; build is the next focused effort (handoff). **Method:** 5 parallel web-search angles → 50+ primary sources → ~52 cited findings → cross-agent consistency check on load-bearing claims. Confidence HIGH except where marked.

> **Through-line the evidence forces:** deterministic execution is the anchor (swap-free → maximize it); the LLM/VLM critic is a noisy SIGNAL, valuable mainly because the 14B/8B is a DIFFERENT model than the 30B coder (free self-preference de-bias); feed the coder EXECUTION feedback, never prose-only; bound iterations (default 2, cap 3); SKIP debate/panels/large-N — their active ingredient (model diversity / large search) is exactly what one-model-at-a-time swaps make expensive.

## Applicability filter (judge every technique against THIS)
One Intel Arc iGPU, ~31 GB shared RAM, OpenVINO. **One model resident at a time** — switching = unload/load swap (~30–60s). Models: 30B coder (Qwen3-Coder INT4), 14B general, 8B vision (Qwen3-VL). No cloud/API, offline, bounded compute. Current pipeline: 30B builds a task in a git worktree → deterministic gate (build + pytest/npm test) → a SINGLE read-only LLM "code reviewer" reads the diff → MERGE/FIX-FIRST → bounded fix passes → merge/park. Plus the nascent visual loop (Phase 3). The coder writes its OWN tests (happy-path risk).

## Q1 — Iterative self-correction for code: what works, what's overstated
- **Intrinsic self-correction (no new external signal) does NOT reliably improve correctness and frequently degrades it.** Huang et al. *LLMs Cannot Self-Correct Reasoning Yet* (DeepMind, **arXiv 2310.01798**): GPT-3.5 self-correcting with no oracle drops GSM8K 75.9→75.1→74.7%, CommonSenseQA 75.8→**38.1%** (near-collapse). Decisive: prior "successes" (Reflexion/Kim) **used the gold label to decide when to stop the loop — an oracle/perfect verifier**; remove it and gains vanish or invert. Corroborated for planning by Stechly/Kambhampati (**2310.12397**, **2402.08115**): "performance collapse with self-critique and significant gains with sound external verification"; the *content* of the self-criticism is "largely irrelevant" — apparent gains are draw-more-samples + an external check.
- **What looked like self-correction was an external crutch:** an oracle stop-label, **execution feedback** (unit tests/interpreter), or **more sampling + a sound verifier.** CRITIC (**2305.11738**) ablation: remove the tool and GSM8K 72.2→68.3 (gain turns negative). Code-specific — Olausson et al. *Is Self-Repair a Silver Bullet?* (**2306.09896**): self-repair gains are "modest… sometimes not present at all," **bottlenecked by the model's ability to produce correct feedback on its own code**; human feedback raises success **1.58×**; GPT-4's own feedback was wrong in 32/80 cases vs 7/80 for humans.
- **What genuinely works for code = execution-guided repair.** Self-Debugging (**2304.05128**): "rubber-duck" explanation alone +2–3%; **execution/unit-test feedback +12%.** Reflexion HumanEval 80→**91%** (**2303.11366**) is unit-test-grounded, not verbal-reflection-alone. Survey — Kamoi et al. TACL (**2406.01297**): self-correction works only with "reliable external feedback (code executors/unit tests)" or large-scale fine-tuning. **Overstated-citation flag:** Self-Refine's ~20% (**2303.17651**) is on *quality* tasks (dialogue, code-optimization/readability), **not functional correctness**, and is **+0.2pp on its own math task** — don't cite it for correctness.
- **Constraint-fit:** strong, and it validates the pipeline's spine — but the lesson is sharp: **feed the coder failing tests / runtime errors, never prose-only self-critique.** A fix pass with no fresh execution signal is expected to regress.

## Q2 — LLM-as-judge for code: failure modes + rigor levers
- **Measured failure modes:** position bias — single-order verdicts flip 35–76% (GPT-4 65% consistency, GPT-3.5 46%, Claude 24%; Zheng et al. **2306.05685**); verbosity bias fools weak judges ~91%; self-enhancement +10–25% to own outputs. **Self-preference is causal** — Panickssery et al. (**2404.13076**): a judge favors text it recognizes as its own; inducing self-recognition *increases* the bias. Sycophancy (Anthropic, **2310.13548**): models abandon CORRECT answers under mild pushback (Claude conceding up to ~98%). Code-specific: judges swing **+26 to +35 pts** toward "correct" from a planted `// verified` comment (*Don't Judge Code by Its Cover*, **2505.16222**); badly miscalibrated (ECE ~39–74, clustered 90–100% confidence; **2508.06225**).
- **Rigor levers (ranked):** (1) **cross-family judge** — a different model removes the self-preference *mechanism* (2404.13076); (2) **diverse panel + vote** beats single GPT-4 on human agreement at **~7× lower cost**, lower variance (PoLL **2404.18796**); (3) **both-order pairwise > scalar pointwise**, multi-vote across orderings (CodeJudgeBench **2507.10535**; Zheng); (4) **execution/reference grounding** is the strongest correctness lever — reference-guided grading took a math-judge 70%→15% error (Zheng); CodeJudge fault-taxonomy rubric 0.61 Spearman vs real correctness (**2410.02184**); (5) structured rubric + decomposed scoring (G-Eval **2303.16634**); (6) **sanitize the diff** — strip author claims / "correct" comments / task-spin before judging.
- **Folklore flag (MODERATE conf):** "add CoT / 'explain and propose a fix' → rigor" is FALSE for code review — *Reliable Code Reviewers?* (**arXiv 2603.00539**, single-source 2026) finds "explain + propose a fix" prompting **raises** misjudgment (systematic over-correction of conforming code); the fix that worked was running the proposed fix as an executable counterfactual. Also (CodeJudgeBench): a *trained* judge and a *bigger* judge are both folklore — an 8B thinking model beat fine-tuned 70B judges.
- **Constraint-fit:** a single same-model critic is the weakest viable design. **Highest-leverage local move: use the 14B as the critic — already cross-model from the 30B coder, capturing the #1 de-bias (self-preference removal) for the one swap you were already paying.** A true diverse *panel* costs multiple swaps → not worth it (Q5).

## Q3 — Independent / adversarial test generation
- **Quantified risk:** tests generated against buggy code bake the bug in — bug-detection drops HumanEval 53.5→46.7%, **APPS 51.4→37.8%**, coverage ~−14pp (*Measuring the Influence of Incorrect Code on Test Generation*, **2409.09464**). This is the self-testing coder's failure mode. **Rule: generate tests from the SPEC, implementation-blind — never show the test-writer the code it must catch.**
- **Practical-at-small-model:** **Property-based testing** is the best calls-to-coverage ratio — one model call writes an invariant/metamorphic relation, the engine (Hypothesis / fast-check) fans out thousands of executions free; PBT + example tests 68.75%→**81.25%** bug detection (**2510.25297**, **2506.18315**); properties are implementation-blind by construction. **CodeT** (**2207.10397**): generate independent tests + rank solutions by **dual execution agreement** (RANSAC-style consensus) — HumanEval +18.8pp; degrades gracefully to N=3–5. **Mutation as a cheap test-quality gate** (**2406.09843**): use **traditional operators** (PIT/mutmut, ~free, 99.9% compilable) to score whether the coder's own tests kill bugs — LLM-generated mutants too costly for the hot loop. **CEGIS** counterexample-guided repair (AAAI 2025): one call/iteration, the failing test = the counterexample.
- **Constraint-fit:** EXCELLENT — mostly deterministic + swap-free. PBT + traditional-mutation-scoring + spec-derived tests written by the 14B (during its critic residency) solve the happy-path problem with ~zero extra model calls.

## Q4 — Verifier-guided generation / test-time compute
- A verifier buys a large effective-size boost (Cobbe **2110.14168**: 6B+verifier ≈ finetuned 175B), but **best-of-N saturates then *declines* past N≈400** as search finds adversarial solutions that fool the verifier; Gao et al. (**2210.10760**) overoptimization law — large N against a *learned* proxy is actively harmful. Process-reward best-of-N (Lightman **2305.20050**) needs N=1860 + 800K labels → not local. Snell et al. (DeepMind **2408.03314**): **compute-optimal allocation is >4× more efficient** than flat best-of-N; a small model + test-time compute can beat a 14× larger model where it already has non-trivial success — *match N to difficulty.* Self-consistency (**2203.11171**) gets most gain by **5–10 samples.**
- **Constraint-fit:** large-N + learned RM = wrong fit. **Right fit: small N (3–5) ranked by execution agreement on independent tests (CodeT), not text voting or a learned RM** — execution oracles resist the overoptimization that kills large-N learned-verifier search. Best-of-N is an optional upgrade (same model resident, no swap), not the baseline.

## Q5 — Multi-agent debate / ensemble critique
- **No, not in multi-round form.** Du et al. (**2305.14325**) +8–16pp but on a weak GPT-3.5 base; wins shrink with stronger bases. Smit et al. *Should we be going MAD?* (**2311.17371**, ICML 2024): debate does NOT reliably beat self-consistency/ensembling and costs ~2.4×. Zhang et al. (**2502.08788**): across 5 methods × 9 benchmarks × 4 models, MAD "often fails to outperform single-agent CoT/Self-Consistency" at far higher compute. **Decisive (equal call budget; Huang 2310.01798):** Self-Consistency beats debate (6 calls 85.3 vs 83.2; 9 calls 88.2 vs 83.0). Debate's one robust booster is **model heterogeneity** (2502.08788's "universal antidote") — which on a single-weights box = a 30–60s swap per turn = disqualifying. Multi-agent DOES pay as a **verifier-grounded role pipeline** (MetaGPT 85.9% HumanEval **2308.00352**; MapCoder +5.5–7.8pp **2405.11403**) — but MapCoder is ~36 calls/problem for single-digit gains.
- **Constraint-fit:** skip debate + multi-critic panels locally. Cross-family benefit captured by the single 14B-vs-30B asymmetry. If more signal is wanted, a **2-pass architect→editor** split (Aider) is the high-ROI shape, not an N-round panel.

## Q6 — VLM-as-judge for generated UI / visual design
- **The 8B VLM's bottleneck is perception, not reasoning** — exactly the UI-critique axes. VL-RewardBench (**2411.17451**): even GPT-4o 65.4%, open models near random, "fail predominantly at basic visual *perception*." MMVP / *Eyes Wide Shut* (**2401.06209**): GPT-4V 38.7% vs human 95.7% on orientation/count/position/presence/color. *VLMs Are Blind* (**2407.06581**): counting grid rows/nesting/intersections 35–58%. *What's Up* (**2310.19785**): left/right/above/below ≈56% vs 99% human. UICrit (**2407.08850**): zero-shot Gemini valid UI critiques only **13.1%**, "poor object localization."
- **What works:** **pairwise (render vs golden screenshot) is the VLM's most human-aligned mode** (0.696 vs 0.454 absolute scoring; MLLM-as-a-Judge **2402.04788**) — give it a reference image; **ground critiques to supplied bounding boxes** (don't trust its localization) — visual prompting closed ~50% of the human gap (**2412.16829**); deterministic **block-match (text/position/color) + CLIP similarity** from Design2Code (**2403.03163**) handle everything geometric/countable. Critique→regenerate loop validated (Idea2Img **2310.08541**, +43pts) **but with a frontier critic** — an 8B is near-random on perception, position bias up to 88%, length bias up to 76%; small-VLM judging is bimodal (0%→75% by checkpoint, **2505.08468**).
- **Constraint-fit:** the 8B VLM must be a **signal, never the gate.** Anchor visual QA on deterministic geometry/CLIP checks (free, swap-free, strongest exactly where the VLM is weakest); use the VLM only for pairwise-vs-reference subjective judgment with grounded boxes.

## Q7 — Loop control: stopping, oscillation, bounding
- **Diminishing returns are real + early:** Self-Refine caps at 4 iterations, "marginal improvement naturally decreases," most gain in iterations 1–2 (2303.17651). **Oscillation/regression measured:** intrinsic loops talk models out of correct answers (Huang: CSQA 75.8→38.1). **Refinement only works when the model can detect the error.** **Engineering norms:** Reflexion stops "until the Evaluator deems correct" (external check); SWE-agent (**2405.15793**) bounds ~25–30 steps typical, 128–250 hard cap (MODERATE: follow-on figures); OpenHands MAX_ITERATIONS ≈100; Aider re-runs lint/test-fix "until the linter is happy or it gives up after a sensible number of tries."
- **Stop criteria (priority):** (1) external check passes → stop; (2) no-improvement/oscillation guard → stop and **keep the best-verified candidate** (never accept the latest blindly); (3) hard cap as runaway ceiling.
- **Constraint-fit:** bound hard. Locally each fix iteration = up to **2 swaps (~1–2 min pure overhead)** → **default 2 iterations, hard-cap 3**, gate iterations 2+ on the **deterministic test delta**, not a re-review.

# Recommended architecture [PROPOSED]

**Design principles the evidence forces:** (i) deterministic execution is the anchor and is swap-free — maximize it; (ii) the LLM/VLM critic is a signal, valuable mainly because the 14B/8B is a *different model* than the 30B coder (free self-preference de-bias); (iii) feed the coder execution feedback, never prose-only; (iv) bound iterations (evidence + swap cost agree); (v) skip debate/panels/large-N (their active ingredient is what swaps make expensive).

## (a) CRITICAL CODING LOOP
1. **Deterministic gate = the real judge (zero model calls, zero swaps):** real build + the coder's own tests + lint/typecheck + **property-based tests (Hypothesis / fast-check)** + **mutation scoring with traditional operators (mutmut / Stryker)** that flags the coder's tests as happy-path when they fail to kill mutants. A diff cannot merge on a green LLM verdict alone — it merges on this gate.
2. **Independent tests (solves the self-testing risk, 2409.09464):** during the SINGLE 14B critic residency, have the 14B *also* emit spec-derived, **implementation-blind** tests + properties (do NOT show it the coder's code). Cost = zero extra swaps (amortized into the critic swap). PBT + mutation-scoring cover the rest free.
3. **Critic = one 14B pass (1 swap), signal not gate:** read the diff cross-model (30B-written → 14B-judged = self-preference removed). Rigor: structured rubric + forced issue enumeration + sanitize the diff (strip author claims / "correct" comments) + pairwise/reference framing; treat its MERGE/FIX-FIRST as one input alongside the deterministic gate. Do NOT ask it to "explain and propose a fix" as the rigor mechanism (2603.00539) — ask it to find concrete, test-checkable defects.
4. **Iterate (execution-grounded, bounded):** on FIX-FIRST or any failing independent/PBT test, swap to 30B and feed the **failing tests/errors** (not prose). **Default 2 fix passes, hard-cap 3.** Gate iterations 2+ on the **deterministic test delta only** (do not re-swap the critic each round). Keep the **best-verified candidate** (anti-oscillation); stop on green or no-improvement; **park** if cap hit without green.
5. **Swap/call budget:** ~**2 swaps** baseline (30B→14B→30B for one fix), **4 worst-case**; the critic runs **once**, not per iteration. Optional upgrade where compute allows: **best-of-N=3** from the 30B (same model, no swap) ranked by execution agreement on the independent tests (CodeT 2207.10397). NEVER multi-round debate or a multi-judge panel.

## (b) CRITICAL DESIGN LOOP (refines the Phase-3 build already on a branch)
1. **Deterministic anchor (swap-free, strongest exactly where the 8B VLM is weakest):** the layout-lint (Lever A, #685, already built) + block-match (text present / position / color) + CLIP similarity to a golden render (Design2Code 2403.03163) + DOM/bounding-box geometry checks (alignment, spacing, element counts, left/right/above/below). These catch the geometric/countable defects the VLM hallucinates on (2407.06581, 2310.19785, 2401.06209).
2. **8B VLM = signal only, never the sole gate:** run **pairwise (current render vs golden reference screenshot)** — its most human-aligned mode (2402.04788); supply the reference image; ground critique to supplied element boxes (2412.16829); randomize order (position bias→88%); cap verbosity (length bias→76%). Output = draft issues to verify, merged with the deterministic geometry failures. No golden reference → lean almost entirely on the deterministic checklist + discount the VLM further.
3. **Iterate:** feed {deterministic geometry failures + VLM pairwise verdict} to the 30B coder. A design fix cycle = 30B→8B→30B = 2 swaps; same bound — default 2, cap 3, keep best-scored screenshot, stop on metric convergence or no-improvement.

## Timing & liveness (a first-class axis, LA-flagged)
**Every long-running step runs under TWO clocks, never one fixed timeout:** (1) a PROGRESS clock — no new edit / test output / critic text for a short window → HUNG → stop fast; (2) a generous CEILING clock — the absolute max so slow-but-working finishes. A single fixed timeout is always wrong (short kills good slow work; long waits on hangs). GENERALIZE the #682 progress-aware coder timeout (idle-kill 240s + 60-min ceiling) to every level: coder build (exists), each shell command (command-timeout.js: 5-min cap, 10s for blocking servers — #688), the 14B critic (its own progress+ceiling — the OLD single 10-min wall-clock false-parked good work in run A), tests/PBT (Hypothesis deadline + max-examples), **mutation scoring (the slow one — SAMPLED + hard-time-boxed ~2-3 min, never exhaustive in the hot loop)**, each model swap (load-progress / GPU-clear poll), and the whole run (judged "hung" only when NOTHING is progressing; swap_run_budget_s watchdog = hard backstop). Budgets GROW to fit the loops' extra steps+swaps, but because stopping is PROGRESS-based, a bigger budget never lets a hang run longer. Loop iterations are count-bounded (2/3) AND get a total-time budget.

**Net:** rigorous because determinism anchors both loops and runs free; the 30B/14B/8B asymmetry gives cross-model critique for the one swap already paid; iterations are execution-gated + capped at 2 (cap 3); the locally-expensive techniques whose value depends on model diversity or large search (debate, panels, large-N, learned RMs) are correctly excluded.
