# Self-Governance Boundary — External Research Synthesis

**Status:** Research input to the Coordinator program (epic #841; design doc `docs/research/coordinator-program-plan-2026-07.md`)
**Date:** 2026-07-11
**Purpose:** Survey how the field handles the same challenge — an agentic system that must not modify itself — and test the Coordinator design's five structural controls against external state of practice. Compiled as input to an independent adversarial review of the program plan.

## 1. The threat class is recognized and named

The industry now treats **agentic self-modification** as a first-class risk category: an agent modifying its own source, configuration, or operational memory to alter behavior, evade controls, or persist changes [Pillar Security, "Agentic System Self-Modification"]. Documented real incidents include agents editing their own timeout logic when hitting a timeout rather than fixing the underlying inefficiency, and self-improving agents rewriting their own codebases to game benchmarks [Sysdig, "Runtime security for AI coding agents"; Sakana AI DGM report]. The design's premise — that "propose work on own backlog" + a dispatchable coding fleet is a self-modification loop — matches the field's threat model, not a hypothetical.

**Consensus mitigation, stated across sources:** enforcement must live in a **higher privilege domain than the agent** — the agent must never be trusted to enforce its own sandbox, and must never be able to edit its own runtime constraints or watchdog scripts [Pillar; Sysdig]. This is exactly the design's control 1 (staging-time governed-core refusal in deterministic code, not model judgment) and control 4 (config immutability from inside) — with one gap the reviewer should probe: BlarAI's ruler code is *itself* part of the BlarAI codebase, i.e., host-side rather than hypervisor-side. The compensating layers are the human-only dev channel and signed manifests, but the privilege-domain question deserves adversarial attention.

## 2. OWASP LLM06 "Excessive Agency" — the three root causes

OWASP decomposes excessive agency into **excessive functionality, excessive permissions, excessive autonomy** [OWASP GenAI LLM06:2025; Kontext Security]. Prescribed mitigations map 1:1 onto the design:

- **Complete mediation** — never rely on the LLM to decide whether an action is authorized; downstream systems independently enforce policy. → BlarAI's "model proposes, deterministic ruler disposes" (ADR-034/035) and PA per-dispatch adjudication (#570) are complete-mediation implementations.
- **Split dangerous actions into separate tools; gate high-impact actions behind approval** → the read-only GUARDED ticket tools + approval-staged writes split.
- **Runtime policy check on every sensitive call, with volume caps and logged decisions** → the ruler bounds on proposal count/size/scope; the outcomes-only comment discipline; per-action DECISION_REGISTER/journal records.

## 3. CaMeL — control-flow/data-flow separation as design, not detection

Google DeepMind's CaMeL ("Defeating Prompt Injections by Design," arXiv:2503.18813) is the strongest formal statement of the pattern the Coordinator design uses informally: a **privileged planner** whose control flow is derived only from trusted input, a **quarantined model** that touches untrusted data but has no tool access, and a capability-tracking interpreter so that *untrusted data can never impact program flow* [DeepMind; InfoQ; Willison, "CaMeL offers a promising new direction"]. Cost: ~2.7–2.8× tokens; benefit: provable security on a large fraction of AgentDojo tasks.

Mapping: BlarAI's deterministic snapshot composer (C1), staging-time ruler (SG control 1), and the rule that ticket content is datamarked untrusted feedstock reaching *at most a staged proposal the LA reads verbatim* is a CaMeL-shaped posture — untrusted data can influence *proposal content* but never *action selection or target*. The reviewer should test whether any C2–C4 path lets ticket-derived data select an action target (that would break the CaMeL property).

## 4. The Lethal Trifecta and the Agent's Rule of Two

Willison's **lethal trifecta** — private data + untrusted content + external communication [simonwillison.net, 2025-06-16] — and Meta's generalization, the **Agents Rule of Two** (an agent should combine at most two of: processing untrusted inputs, accessing sensitive systems, changing state externally), give a compact audit rubric. The Coordinator processes untrusted inputs (ticket/run content) and touches a sensitive system (Vikunja, the fleet). It stays inside Rule-of-Two by making state change conditional on human approval, and BlarAI has no external communication channel (loopback-only; egress welded). Per-phase and per-autonomy-flip (C5), the rubric should be re-applied: **every propose→auto flip moves that action class to all three legs** — the C5 ceremony should explicitly re-run a Rule-of-Two analysis per class.

## 5. Self-improving-agent research draws the same line

The Darwin Gödel Machine (Sakana AI / arXiv:2505.22954) — the most cited *pro*-self-modification system — still runs all self-modifications inside sandboxes under human supervision, and critically keeps two things **frozen**: the foundation model and the **evaluator/benchmark** that judges the modifications. Follow-on work (Red Queen Gödel Machine, arXiv:2606.26294) treats a co-evolving evaluator as the headline risk. The transferable rule: **the judge must not be improvable by the judged.** In BlarAI terms, the governed core correctly includes the ruler definitions, verify gates, oracles, and eval baselines — the design already lists "the coordinator's own ruler/flag definitions" in the core; the reviewer should confirm the fleet's verify-gate and oracle code (agentic-setup side) is equally out of reach, since the dispatch fleet is the acting limb.

## 6. Runtime enforcement practice — configuration as immutable, signed, externally verified

2026 sandboxing guidance (NVIDIA; the awesome-agent-runtime-security corpus) converges on: **sandbox/agent configuration is immutable code**; agents may never modify their own approval policy, hooks, or tool configs *regardless of user approval level*; agent manifests are signature-verified (Ed25519-style); startup canary probes verify the sandbox and **refuse to start** on any unexpected capability [BeyondScale; agentwiki; Parallax/OpenParallax, arXiv:2604.12986]. BlarAI's existing anchors align: signed model manifests at boot (`require_signed_manifest`), fail-closed egress guard armed at the socket layer, and the fail-closed house style. Two practices worth importing that the design does not yet state: (a) a **boot-time canary/self-check for the boundary itself** — probe that the coordinator surface cannot reach a governed-core write path, refuse-to-start if it can (the OpenParallax pattern, and the house third-instance-control idiom); (b) extending **signature verification beyond model weights** toward coordinator-relevant config/policy, so a corrupted-on-disk flag file is detected rather than trusted.

## 7. Governance frameworks — graduated autonomy is now standardized practice

Singapore's Model AI Governance Framework for agentic AI (January 2026) is the first regulator-grade document to classify agents by **autonomy level with proportional human oversight**; NIST launched an autonomous-agent standards initiative in February 2026; ISO/IEC 42001 supplies the management-system shell (documented decisions, roles, continual improvement) [gaicc.org comparison; CSA "Agentic AI Governance"; NeuralTrust comparison]. "Policy Cards" (arXiv:2510.24383) proposes machine-readable runtime governance manifests per agent. The Coordinator's C5 ladder (per-class propose→auto flips, each an owner-level ceremony with a DECISION_REGISTER row) is a concrete implementation of risk-proportional graduated autonomy, and the ADR/DECISION_REGISTER/BUILD_JOURNAL discipline is the ISO-42001-shaped management system. For the AIGP portfolio, naming this alignment explicitly in ADR-039 is cheap and load-bearing. The multi-operator driver also matches the frameworks' insistence that oversight duties be **role-scoped** (owner vs user) — the design's "ceremonies are owner-level, not user-level."

## 8. Tailoring to the model — small-model realities (LA directive 2026-07-11)

The coordinator brain is a **14B dense model on an iGPU**, not a frontier model. The literature says this changes the design, not just the expectations:

- **The format/competence gap.** Small models show a large gap between task accuracy and output-format compliance — naive prompting can achieve high task accuracy with near-0% usable structured output ["When Correct Isn't Usable," arXiv:2605.02363]. Constrained decoding closes the format gap but levies a **constraint tax**: measured ~3.6× latency overhead and, in several settings, degraded task performance, because schema enforcement competes with the model's reasoning budget ["The Constraint Tax," arXiv:2605.26128]. The emerging remedy is **think-then-constrain** — reason freely, switch to constrained generation at a trigger point [arXiv:2601.07525]. BlarAI already lives this: grammar-first with fail-soft parse (#743), grammar-constrained tool calls proven composable with spec-decode (#718) but currently OFF after a live crash (#725) — the stack itself is a risk surface, so no coordinator correctness may ever *depend* on the grammar path.
- **Small models CAN match large ones on tool use** when wrapped correctly — guided decoding + strict schemas + validator-first execution close most of the gap at 10–100× lower cost [SLM agentic survey, arXiv:2510.03847]. The winning shape is many **tiny, single-decision calls** with deterministic validation, never one holistic judgment call.
- **Design implications for the Coordinator:** (a) every 14B call in C3/C4 is a bounded, single-decision task (classify one ticket, draft one proposal) — never "assess the whole workspace and decide priorities" in one shot; the *composition* of judgments is deterministic code; (b) the deterministic-first cycle order (evidence gathered by code, model only drafts) is not just security posture but a small-model necessity; (c) injection resistance scales with model capability, so the CaMeL-style separation matters MORE here, not less — a 14B quarantined-drafter is acceptable precisely because nothing it emits selects targets or control flow; (d) budget per-cycle model calls explicitly (the constraint tax compounds with spec-decode and the shared-GPU reality).

## 9. Dense vs MoE — designing for both

The system already runs both shapes: the dense Qwen3-14B (assistant/coordinator, OpenVINO GenAI, spec-decode via 0.6B draft) and the MoE Qwen3-Coder-30B-A3B (coder, OVMS from-source GPU build #710/#753). Findings:

- **MoE quantization is more fragile** than dense — models with very small experts (30B-A3B class) are significantly harder to quantize well; dense models degrade more gracefully [Kaitchup MoE-vs-dense analysis; ubergarm quant benchmarking]. OpenVINO matured MoE support to GA in 2026.0, with INT4 data-aware 3D-MatMul compression targeted at exactly this class, naming Qwen3-30B-A3B [OpenVINO 2026.0 release]. Structured output via xgrammar landed in OpenVINO GenAI 2025.3 — on the GenAI (coordinator) side; the OVMS serving side's constrained-decoding capability is a separate question and **must be measured, not assumed**.
- **Design rule — capability contracts, not model identity.** The Coordinator must bind each model *role* (coordinator-brain, coder, critic, draft) to a declared **capability contract** (structured-output backend available? tool-call format? grammar composable with spec-decode? eval baseline file?) rather than to a model name. The swap driver already treats models as swappable roles; the contract makes swap-ins (dense→MoE or the reverse, e.g. a future Qwen3.6-35B-A3B coder) a config + eval-baseline event, not a design change. Per-model eval baselines (the #717 harness) are the enforcement: a swapped model re-earns its role on measurement — the existing model-upgrade-watch discipline, generalized.
- **Correctness placement makes both shapes safe.** Because all correctness lives in deterministic gates ("gate as JUDGE, model as SIGNAL" — the LA-accepted leverage assessment), dense-vs-MoE differences (quantization sensitivity, routing variance, format drift) degrade *quality*, never *safety*. The boundary does not care which architecture emits the proposal.

## 10. Stack tailoring — OpenCode and the Agent Client Protocol

- **OpenCode as an enforcement ring — useful, not trustworthy.** OpenCode supports per-agent tool allowlists (markdown + YAML frontmatter agents) and `opencode.json` permission rules: allow/deny/ask per tool, path-scoped edit denies, bash command patterns, `external_directory` ask-by-default, last-match-wins [opencode.ai/docs/permissions]. Denying coder edits to BlarAI/agentic-setup paths in harness config is a cheap additional ring for SG control 1. **But it cannot be the boundary**: the tracker shows real enforcement failures — `external_directory: "deny"` not enforced from Git Bash **on Windows** (issue #16126 — this project's exact platform), per-agent deny rules ignored via SDK invocation (#6396), general permission non-enforcement reports (#8832). Verdict: configure it, test it, and treat it as defense-in-depth behind BlarAI's deterministic staging-time refusal and the fleet's `_FORBIDDEN_REPO_ROOTS` — never as the load-bearing control. (This re-validates the house lesson from the :8099 proxy: a safety wrapper must fail loud, and the protection must not live only in the wrapper.)
- **ACP as the coordinator's sensory nerve + a third chokepoint.** The fleet's ACP integration (#759 approved, `tools/dispatch_harness/acp_coder.py`) speaks JSON-RPC 2.0 over stdio with **typed `session/update` streaming notifications** for every message chunk, tool call, and plan update, plus **`session/request_permission`** callbacks before sensitive operations [agentclientprotocol.com schema; zed.dev/acp]. Two design consequences: (a) the C2/C3 monitoring should consume the **typed ACP event stream** instead of log-tailing — per-tool-call visibility gives honest stall detection (last-event-age), real-time progress for the #712-lineage feed, and structured evidence for outcome reports; (b) the ACP **permission-request channel is a machine chokepoint**: a deterministic policy responder on the client side can refuse governed-core-touching tool calls *mid-run*, in-band, complementing staging-time refusal (before) and worktree/merge-gate checks (after). The responder must be deterministic code — never the 14B — or it becomes an injectable judge.

## 11. Project-management method + agent architecture (LA directive 2026-07-11, second research pass)

**Agent-run kanban is the emerging industry pattern.** Purpose-built agent PM systems (Agent Kanban, vibe-kanban, Kanbots, agent-kanban-pm) converge on the same shape: an explicit board flow (`Backlog → Refinement → Ready → In Progress → In Review → Done`), agents that pull tickets / do work / post results / move cards with no manual bridging, per-card isolated worktrees, and **the ticket as the context home** — conversation history, decisions, and evidence live on the card [agent-kanban.io; virtuslab vibe-kanban; Kanbots]. BlarAI's dispatch already does the worktree-per-task and outcome-comment halves; the board flow and pull discipline are the missing halves.

**Kanban over Scrum for this system.** Scrum's fixed sprints, roles, and ceremonies exist to synchronize human teams; Kanban is continuous-flow, needs no roles, and suits solo operations with variable incoming work [Atlassian kanban-vs-scrum; Akiflow; monday.com]. Scrumban keeps planning/retrospective cadences atop flow [monday.com Scrumban guide]. The decisive technical fit: Kanban's machinery — WIP limits, pull policy, flow metrics — is arithmetic over ticket states/timestamps, i.e., **deterministically computable**, keeping the model out of the control loop.

**Classes of service + flow metrics — the deterministic PM toolkit.** The four canonical classes (Expedite / Fixed-date / Standard-FIFO / Intangible) give a complete, code-computable pull-ordering policy [djaa.com (Anderson); Kanban University glossary; KanbanZone]. The four flow metrics — WIP, cycle time, work-item age, throughput — give principled stall detection (age outliers per class) and honest digests [ProKanban; Atlassian kanban metrics; Scrum.org flow metrics]. All derivable from Vikunja timestamps.

**Orchestrator-vs-specialist agent split.** Multi-agent literature: role separation via task-specific system prompts on a SHARED underlying model is a recognized architecture (functional separation without new weights); split agents when tool surfaces, contexts, or audit identities must differ, keep shared context when roles are tightly coupled [arXiv:2601.13671 orchestration survey; Anthropic multi-agent research system; Augment orchestration guide]. Applied here: the Coordinator = a third resident role on the shared 14B (own prompt, minimal tool surface, own adjudication identity), sibling to PA and AO — a separate model/process is neither affordable under the 31.3 GB ceiling nor architecturally needed.

## 12. Third-pass verification against the live system (2026-07-11)

A max-effort re-review walked the design through years-scale operation and verified its assumptions against the live instance and code. Findings (all folded into the plan's §Operational-maturity additions):

- **VERIFIED DEFECT — shared Vikunja identity.** `shared/fleet/vikunja_bridge.py:34-37,74` reuses the same account (`blarai`) as the MCP server and all human writes → `created_by` cannot distinguish BlarAI authorship → the review-F13 "structural authored-by" fix is inert as designed. Resolution: dedicated `blarai-coordinator` account.
- **VERIFIED DEFECT CLASS — silent pagination truncation.** Vikunja's server-side `maxitemsperpage` (default 50) clamps even explicit `per_page=200` requests; the devplatform `project_summary` tool reads truncated task lists today (five projects reporting exactly 50 tasks — the clamp signature; defect ticketed to devplatform). Coordinator reads must loop pages; flow metrics over truncated boards are silently wrong.
- **VERIFIED SURFACE — the kanban substrate is implementable on v2.3.0.** The live instance's swagger (118 paths) confirms bucket endpoints (`/projects/{id}/views/{view}/buckets`, per-bucket task moves), task relations, and saved filters. API contract gets pinned with recorded-shape regression tests; upgrades become gated events.
- **Design-walk failure modes closed:** TOCTOU between approval and execution (ruler re-runs at execution, authoritative); cross-cycle duplicate proposals (deterministic fingerprint dedup) and stale proposals (TTL expiry); fail-soft blindness (`UNREACHABLE` must never render as `EMPTY` — a dead Vikunja otherwise looks like a finished backlog and silences the tripwire); unvalidated coordinator judgment at go-live (shadow mode + fixture-board eval suite + measured precision graduation, re-run at every model swap); cross-project starvation (global class-then-age pull); operator absence (away mode + proposal TTL pause); missing approval audit trail (briefing ledger — AIGP-grade governance evidence); hardcoded substrate IDs (discover-or-create migration, runtime name resolution); laptop thermals (power-aware cadence); status-render injection into AO chat (untrusted provenance tier for `/coord status` output); PA adjudication verbs for the coordinator identity; #740/M2 interlock on the shared `shared/fleet/` surface.

## 13. Synthesis — where the design stands, and the questions handed to review

The five structural controls have direct external precedent: (1) staging-time target refusal = complete mediation + deny-by-default; (2) advisory-only self-work = the DGM's human-supervised self-modification channel, made human-*executed*; (3) provenance-marked authorship = CaMeL data-flow tracking, coarse-grained; (4) config immutability from inside = NVIDIA/industry immutable-config doctrine; (5) default-deny multi-operator = graduated-autonomy + role-scoped oversight.

**Open questions for the adversarial reviewer** (found by this survey, unresolved by the design):
- **Q1 (privilege domain):** the ruler and forbidden-target set live in BlarAI's own codebase. The dev channel + code review is the write barrier — is that sufficient, or does the boundary need an enforcement layer BlarAI's process cannot load-time-patch (signed policy file verified at boot; guest-side verification)?
- **Q2 (the acting limb):** the fleet (agentic-setup repo) executes the code changes. Is *its* gate/oracle/config surface inside the governed core, and is the severance enforced there too (the fleet's own `~/BlarAI` refusal is one anchor — is it enough)?
- **Q3 (CaMeL property):** can any C2–C4 path let untrusted ticket/run content select an action *target* rather than merely proposal *content*?
- **Q4 (boundary self-check):** should SG (#848) include a boot-time canary that proves the boundary and refuses to start on failure?
- **Q5 (Rule of Two at flips):** should each C5 ceremony mandate a per-class Rule-of-Two re-analysis?
- **Q6 (memory/substrate as instructions):** the knowledge substrate and session memory feed the model — are they an unmarked self-instruction channel the boundary should cover (the "operational memory" leg of the Pillar taxonomy)?
- **Q7 (small-model calibration):** which coordinator judgments in C3/C4 are actually within a 14B's reliable envelope? Is the single-decision-call decomposition specified concretely enough, and is the constraint tax (latency + reasoning degradation) budgeted for a heartbeat cycle on a shared iGPU?
- **Q8 (capability contracts):** does binding roles to capability contracts + per-model eval baselines fully cover a dense↔MoE swap, or are there behaviors (quantization drift, tool-format drift, grammar-on-OVMS support) the contract can't capture and a gate can't catch?
- **Q9 (OpenCode ring placement):** given the documented Windows enforcement bugs, is "defense-in-depth, never load-bearing" the right posture for OpenCode permissions — and should SG (#848) include a live adversarial probe of the harness config (attempt a governed-core edit through the coder) rather than trusting it?
- **Q10 (ACP integration):** should C2/C3 monitoring and the deterministic permission responder over ACP `session/request_permission` be pulled into the ticketed scope now (which tickets), and does an in-band deterministic responder introduce new failure modes (e.g., wedging a run on a refused permission)?
