## Context

The LA has defined the mature role the AI side of coding projects must fill: not just executing tasks, but **coordinating the entire effort** — monitoring live processes each cycle, keeping the work queue full (a quiet queue is a broken queue), taking direct actions in the gaps, running Vikunja as a living SSOT, making information durable and distributed, pushing work through completion, and proactively finding/assigning the next work — all behind reasonable periodic LA-approval gates. A parallel session is making the role durable for Claude sessions. **This program designs how the same role maps onto BlarAI itself, in BlarAI's idiom: fail-closed, deterministic-ruler, approval-gated, loopback-only.**

LA scope decision (2026-07-11): mandate = **dispatched coding projects + BlarAI's own backlog** — full coordinator, phased so the dispatched-projects tier lands first.

LA governance decision (2026-07-11, second): **BlarAI must not be able to modify its own code, configuration, instructions, policy, models, or governing documents.** The coordinator role made a latent blend visible — "BlarAI proposes work on its own backlog" sits one step from "BlarAI dispatches a coding fleet against itself." That loop is severed by doctrine and by structure (see §Self-governance boundary). Driver beyond safety: the LA intends to eventually put BlarAI in the hands of **non-technical operators** building their own custom software — so every control must be structural and default-deny, never dependent on operator vigilance or expertise.

## Foundation to reuse, not rebuild

- **Dispatch engine exists**: `shared/fleet/` (dispatch, decompose, acceptance, plan_graph/PlanStore, swap driver/state/ops, critic/design/oracle phases, battery). Lifecycle: `/dispatch <repo>|<goal>` → PLAN → confirm → `/dispatch approve` → swap → fleet → honest per-criterion report.
- **Vikunja from the runtime is sanctioned + solved**: `shared/fleet/vikunja_bridge.py` (#749) — loopback-pinned (`assert_loopback`), fail-soft (2 s cap, swallow errors), outcomes-only (anti-firehose), transport socket in `tools/dispatch_harness/vikunja_http.py` (outside the egress import-scan). Loopback is NOT egress (ADR-027); the one-network-client invariant (`httpx` only in `guarded_fetch`) is preserved by this split.
- **Model-write governance template exists**: `propose_preference` (#770) — GUARDED tool → staged proposal → confirm card → operator approval. Also the ADR-034/035 pattern: *model proposes, deterministic ruler disposes*; mandatory confirm; never rubber-stamp an unrun check.
- **The two genuine gaps (the whitespace this program fills)**:
  1. **No heartbeat** — BlarAI is entirely turn-driven/reactive; nothing wakes to check monitored processes or queue depth.
  2. **No work-origination loop** — nothing harvests outcomes, proposes next work, drives tickets to completion, or notices an idle fleet.
- Existing invariant to amend consciously: *"the coder/planner models get NO ticket tools"* (#749) — this design gives the model **read-only** ticket tools (GUARDED) and keeps all writes deterministic-or-approval-staged. The amendment is made explicitly in ADR-039, never silently.

## Self-governance boundary (LA decision 2026-07-11 — constitutional, applies to every phase)

The system is split into two domains with structurally different write rules:

**The governed core** — everything that defines what BlarAI *is* and how it behaves. BlarAI has **zero write path** to any of it, in any phase, at any autonomy level:
- Runtime code (`shared/`, `services/`, `launcher/`, `tools/`) and tests
- Configuration (`default.toml` and all `[coordinator]`/autonomy/policy flags)
- System prompts, Policy Agent rules, pgov policy, tool registrations, risk tiers
- **The operator-preference tier and any prompt-injected memory** (the `propose_preference` staging path writes into future system prompts — ADR-036 rates it the highest-risk auto-injected surface; a coordinator-role model instance must not carry that write-staging tool, or its proposals pass the same staging-time ruler)
- **The knowledge substrate and session memory insofar as they are injected as standing instructions** (the "operational memory" self-instruction leg)
- Model weights + signed manifests + signing keys
- Governing documents (ADRs, DECISION_REGISTER, CLAUDE.md-tier doctrine, this program plan)
- The coordinator's own ruler definitions, forbidden-target set, proposal-queue mechanics, flag ladder, **and the heartbeat's own trigger definition** (the scheduled task / timer that wakes it)
- **The acting limb's judge**: the agentic-setup fleet's verify-gate, oracle machinery, and harness config (`opencode.json`, agent definitions) — the judge must not be improvable by the judged (the Gödel-machine frozen-evaluator rule); severance is enforced from BOTH sides
- The proposal-staging store itself (append via sanctioned staging API only; no direct writes)

**The workspace** — what BlarAI coordinates and (with approval) acts on: target project repos in the configured projects dir, their fleet jobs, and Vikunja ticket data.

Changes to the governed core travel exactly one channel: the **human-governed development path** (LA + Claude dev sessions, feature branches, gates, ceremonies). BlarAI may *observe* its own backlog and *advise* on it; it may never *execute* against itself.

**Structural controls** (each becomes a regression-locked control at implementation; none rely on operator vigilance):
1. **Self-target severance at every action chokepoint — identity-based, not name-based.** The existing anchor — `_FORBIDDEN_REPO_ROOTS` in `shared/fleet/dispatch.py:40` refuses BlarAI as a fleet target — is generalized AND strengthened: every coordinator action class (redispatch proposals, work origination, any future class) passes a governed-core target check **at staging time**, not just at dispatch time. Membership is decided by **canonical realpath containment** (symlinks/junctions/worktrees resolved), modeled on the existing `relative_to(projects_dir)` containment check — never by path-component name matching, which a renamed clone or junction evades. Additionally, the execution target of any proposal is **re-derived by deterministic code from trusted structured fields** (e.g. a validated ticket repo id checked against `projects_dir`), never taken from model free-text — so untrusted content can shape proposal *content* but can never select an action *target* (the CaMeL property). A proposal that fails either check is refused before it reaches the proposal queue. The forbidden-target set is config-defined but **not modifiable via any BlarAI surface** (see controls 4 and 7).
2. **Self-work is advisory-only.** BlarAI-originated tickets about BlarAI itself are created (post-approval) with a **structural** provenance marking — the dedicated **`blarai-coordinator` Vikunja account's immutable `created_by`** plus a label set by deterministic code, never a text convention in the ticket body (text markers are forgeable by any injection that can author a comment; and a SHARED account makes `created_by` inert — verified live, see Operational-maturity item 1) — and are executable only by the human-governed dev channel. BlarAI's own dispatch path refuses them categorically — even if approved, approval routes them to humans, never to the fleet. Closure of these advisory tickets is a **human-dev-channel responsibility** (they are exempt from any BlarAI auto-close and from the shipping-closes-the-ticket automation, which would otherwise leave them silently drifting).
3. **Instruction-channel integrity.** Ticket/comment content BlarAI authors is provenance-marked, and all consumers (including BlarAI itself on later reads, and Claude dev sessions) treat it as untrusted proposal-grade input — closing the loop where the ticket store becomes a covert self-instruction channel.
4. **Configuration immutability from inside.** No tool, proposal class, or preference (`propose_preference` included) may read-write any `[coordinator]`, autonomy, policy, or security config. Flag changes are operator-only, made outside BlarAI, and ladder flips remain LA ceremonies with DECISION_REGISTER rows.
5. **Multi-operator readiness (the distribution driver).** The autonomy ladder ships fully OFF for any new installation; go-live ceremonies are owner-level actions, not user-level; the governed-core boundary is identical for a non-technical operator — nothing they can click hands BlarAI a write path to itself. The safety story must survive an operator who cannot audit code. Approval UX is sized for that operator: briefings render proposals as **outcomes in plain language** ("this will dispatch repo X to do Y"), batch size is bounded, and high-impact classes require per-item confirmation — batched approval must never become a rubber stamp (approval fatigue is a vigilance failure, and vigilance is disallowed as a control).
6. **Boot-time boundary canary + refuse-to-start.** At every backend boot, a deterministic self-check probes that the coordinator surface cannot reach a governed-core write path (the OpenParallax canary pattern); any probe succeeding means misconfiguration and the coordinator **refuses to start** — the same fail-closed idiom as the egress guard.
7. **Signed policy verification — the out-of-process anchor.** The forbidden-target set and the `[coordinator]`/autonomy/policy config are **signature-verified at boot** (extending the existing signed-manifest machinery beyond model weights), so a corrupted- or patched-on-disk policy is detected rather than trusted. This is the enforcement layer that does not depend on human code review — the control that makes the boundary hold for a **non-technical operator**, for whom the dev-channel write barrier does not meaningfully exist. **Claim discipline:** until control 7 ships, the boundary is honestly stated as "safe under expert operation," not "safe for distribution."

ADR-039 (C0) carries this boundary as a dedicated section — the two-domain model, the enumerated governed core, and the seven controls — so every later phase implements under it rather than re-deriving it.

## The project-management method — Kanban by design (LA directive 2026-07-11; evidence in `coordinator-self-governance-research-2026-07.md` §12)

Governance says what the Coordinator may not do; this section says what it actually DOES as a project manager. The method is **Kanban** (a flow-based agile method) with light Scrumban cadences — chosen over Scrum deliberately: sprints exist to synchronize human teams, which this system does not have; a solo operator + AI executors is a continuous-flow shop with variable-size incoming work, which is Kanban's home ground. Decisive fit: every core Kanban mechanism is **deterministically computable** from Vikunja timestamps and states — the method itself obeys "gate as JUDGE, model as SIGNAL."

**Vikunja is the PM substrate, mapped concretely:**
- **Workflow = Vikunja kanban buckets per project:** `Backlog → Ready → In Progress → In Review/Verify → Done` (the emerging agent-kanban industry standard flow). Card movement is deterministic, driven by real events (dispatch started → In Progress; oracle GREEN + merged → Done), never by model opinion.
- **Definition of Ready / Definition of Done as deterministic gates.** A ticket enters Ready only when a checklist passes (acceptance criteria present, target repo valid under the SG ruler, no unresolved blocker relation); Done only via the existing GREEN + oracle-passed close. The Coordinator *proposes* refinements to get Backlog items Ready (lane-a proposals); it never fakes readiness.
- **WIP limits, ruler-enforced.** Work-in-progress caps per stage are config policy. The hardware itself sets the deepest one — one dispatch at a time (the 14B⇄30B swap is exclusive) — and the ruler refuses pulls that would breach a limit. WIP limits are the mechanism that makes "keep the queue full" mean *flowing*, not *flooded*.
- **Classes of service via labels + due dates, with a deterministic pull policy:** `Expedite` (jump the queue, one at a time), `Fixed-date` (Vikunja due date; pulled early enough to make the date), `Standard` (FIFO — oldest Ready item pulls first), `Intangible` (maintenance/docs; pulled when nothing else is Ready). The pull ORDER is computed by code; the model never chooses who goes next — it only drafts *proposals to add or reclassify* work, which you approve.
- **Flow metrics, computed each heartbeat cycle** from ticket timestamps: cycle time, throughput, work-item age, and aging-WIP. Stall detection becomes principled (an item whose age is a statistical outlier for its class), the digest reports flow (not vibes), and the quiet-queue tripwire's real definition is "ELIGIBLE Ready items exist (resource-gated cards are visible in the gated-inventory line but do NOT arm the tripwire) + WIP below limit + nothing pulling."
- **The ticket is the context home** (already house rule, now doctrine): proposals, run outcomes, decisions, and evidence live as comments on the ticket they concern — an agent or human picking up a card gets its whole history with it.
- **Durability without distribution is half the job (the anti-drop tenet).** Recording work durably — a ticket comment, the outcome ledger, a doc — does NOT guarantee the next actor acts on it; the board must also DISTRIBUTE it into the pull view they read. **Cross-session continuity is a property of the durable board, never of session memory or a verbal handoff.** Two consequences the method enforces: (1) every defer/handoff writes BOTH halves — the durable record AND the queue placement with its unblock predicate; (2) **resource-gated deferral is a first-class, still-visible marking** (a marking + pull-eligibility rule, NOT a new bucket/lane) — a card that is Ready-in-principle but blocked on a RESOURCE window carries a registry-keyed `blocked_on` marking, **stays in Ready** (presence in the pull computation is the invariant; only pull ELIGIBILITY is suspended), and is re-surfaced automatically by the pull policy / replenishment when the resource frees. The predicate vocabulary is a **CLOSED registry** ADR-039 enumerates — each entry ships with a deterministic evaluator over host/board state (initial candidates: gpu-window/lean-box, disk-space, model-residency); marking a card with an unregistered predicate is REFUSED at staging (fail-closed). Boundary cases are OTHER classes, not resources: work-shaped blockers (a wheel rebuild) are ticket dependencies → the DoR blocker relation, which correctly gates *out* of Ready; external events (an upstream fix landing) have NO host-evaluable signal and the runtime may not poll for one (zero-egress) — they get a bounded human-check line in the briefing/digest, never a false automation promise. Gated time is EXCLUDED from stall/aging alarms (reported as a distinct "gated inventory" digest line — visibility without alert fatigue) while age still ACCRUES for pull ordering, so a released card catches up rather than re-queuing at the back. Origin: the 2026-07-11 #769 catch — the follow-up benches were durable on their ticket but absent from the pull queue, one forgotten verbal handoff from silent loss across the night→next-daytime-GPU session chain. Hardened by the same-day adversarial review (`docs/reviews/durability-distribution-tenet-adversarial-review-2026-07-11.md`: closed registry, marking-not-state, stays-in-Ready encoding, blocker taxonomy, alarm carve-out).
- **Scaffolding is retired by the thing that replaces it (the anti-sprawl counterweight — LA-raised 2026-07-11).** Distribution taken naively becomes sprawl, and sprawl is its own signal loss — so the anti-drop tenet carries a matched pair: *distribute AND curate*. Every interim mechanism, board structure, or one-off doctrine is created CARRYING its supersession predicate (the #859 queue-ticket pattern — "superseded once the board ships" — generalized), and the superseding change's Definition of Done includes EXECUTING the retirement: the structural sibling of shipping-closes-the-ticket, which is closure-curation for tickets. Board-structure hygiene is deterministic: the substrate-setup migration (Operational-maturity item 11) runs bidirectionally — discover-or-create AND flag-orphans (labels/buckets/filters no current doctrine references) as digest curation candidates, removals human-approved. The existing content-surface curation SOPs (LESSONS quarterly consolidation, fragments fold-then-delete, memory dedup, the #267 doc-sprawl work) stay authoritative for their surfaces; this tenet closes the STRUCTURE gap they do not cover.

**Cadences (the Scrumban borrowings):** the heartbeat digest = the standup analog (flow snapshot, blockers, aging outliers); the periodic briefing = the replenishment meeting (what enters Ready next, what gets reclassified — your approval gate); the existing monthly retrospective absorbs flow-metric trends. No sprints, no story points, no velocity theater.

## Agent architecture — the third resident role, not a new model

The Coordinator is **a distinct agent role on the shared 14B**, exactly as the Policy Agent and Assistant Orchestrator are distinct roles on one model today: its own system prompt, its own (minimal) tool surface, its own adjudication identity — same weights, same GPU residency, zero new RAM. Multi-agent research supports this split-by-prompt pattern for role separation on a shared model; a separate model or process is neither affordable (31.3 GB ceiling) nor useful (the roles never need to run concurrently with different weights).

Why a separate role rather than more duties for the AO: (1) **tool-surface minimization** — the Coordinator role carries ONLY read-only ticket/state tools + proposal staging; it structurally lacks the AO's `propose_preference`, image, and chat tools (review F3 demanded exactly this); (2) **prompt integrity** — a PM persona with flow-policy instructions must not share a prompt with a conversational assistant whose context is user-shaped; (3) **auditability** — coordinator actions carry a distinct identity end-to-end (PA adjudication, logs, ticket authorship). The AO remains the conversational front door (`/coord …` commands route through it); the Coordinator role is what the heartbeat and briefing machinery invoke.

## The design — five phases, each independently shippable, each ending dormant-until-flag

### Phase C0 — Doctrine
- This program plan (this document).
- **ADR-039 (Coordinator role + graduated-autonomy gates + self-governance boundary)** + DECISION_REGISTER row, authored at implementation kickoff (same-change rule). Carries the two-domain model (§Self-governance boundary) and its seven structural controls as a dedicated section.

### Phase C1 — Read surface (turn-driven, advisory)
- Extend `vikunja_bridge` with **read** ops (list open tasks per project, stalls, summaries) — same loopback/fail-soft/transport-split pattern.
- Deterministic **work-state snapshot** composer: fleet-swap `current.json` + battery campaign state + fleet queue + SUMMARY/scorecards + Vikunja open-ticket view → one structured "state of the work."
- Operator surface: `/coord status` (gateway command, mirrors `/dispatch status`) + a GUARDED `project_status`/`list_open_work` model tool so conversation can answer "where does the work stand." Model gets READ ONLY.

### Phase C2 — Lifecycle coordination (event-driven, deterministic)
- Ticket hygiene automation on existing dispatch events: stall detection (budget-exceeded → ticket comment + operator surface), PARKED-HONEST → staged redispatch **proposal**, close-on-GREEN discipline (already exists) extended with "CURRENT STATE" comments on partial lands.
- Information durability + distribution (BOTH halves, per the anti-drop tenet §Kanban): the per-project outcome ledger append + ticket comments are the DURABLE record; distribution is the separate second half — board/queue placement in the pull view the next actor reads — and the harvest/hygiene steps write both; feed the #712-lineage WinUI typed progress feed.
- Promote stall-kill patterns from `tools/dispatch_harness/monitor.py` into driver-integrated checks (stop-doomed-fast).

### Phase C3 — The heartbeat (the genuinely new surface)
- A bounded **coordinator wake cycle**: host-side scheduled trigger (launcher-managed timer or Windows scheduled task → a coordinator entrypoint), each cycle deterministic-first: read monitored-process state, harvest finished runs, compute queue depth, detect stalls; the 14B is invoked only to draft proposals/digests. **Swap-aware model dependency:** during an active dispatch swap the 14B is EVICTED (ADR-034 exclusive occupancy) — a mid-swap cycle runs deterministic-only and defers model drafting until the 14B is resident; the wake cycle never forces a swap.
- **Read + propose only**: every state-changing action lands in a **proposal queue** (`/coord approve <id>` — same confirm pattern as dispatch). At most one digest per cycle (anti-firehose), and the digest is an **operator-surface artifact only — never a Vikunja comment** (the bridge's outcomes-only-never-heartbeats invariant is untouched).
- **Quiet-queue tripwire**: approved, unblocked, resource-ELIGIBLE work exists + fleet idle → surfaced as a defect-grade alert, not silence (an all-gated Ready column is reported as gated inventory, never as a quiet-queue alarm) — **schedule- and swap-aware** (suppressed during active swaps, in the pre-23:00 overnight quiet period, and within a configured idle-grace window) so false alarms never retrain the operator to ignore it.

### Phase C4 — Work origination (model proposes, ruler disposes, LA approves)
- BlarAI reads the whole Vikunja workspace + vision docs (`Use Cases_FINAL.md`, program plans) and drafts **next-work proposals** (new tickets, priorities, redispatches) into staging; a deterministic ruler bounds count/size/scope; nothing is created/changed without approval.
- **Two proposal lanes, split by the self-governance boundary**: (a) **workspace proposals** — next work on dispatched coding projects; approvable and then executable via BlarAI's own dispatch path. (b) **self-advisory proposals** — observations/suggestions about BlarAI's own backlog; approvable only into `Advisory:BlarAI`-marked tickets consumed exclusively by the human-governed dev channel. The ruler enforces the lane split at staging time (control 1); BlarAI's dispatch path categorically refuses lane (b) items (control 2).
- **Periodic coordinator briefing**: batched approval gate — the "reasonable gates for live approval of everything periodically" from the LA's role definition. The briefing renders the lane of every proposal explicitly.

### Phase C5 — Graduated autonomy (per-class go-live ceremonies)
- Individual low-risk action classes flip propose→auto behind per-class flags, each an LA governance decision, default off (e.g., auto-redispatch-once on transient failure; auto-stall-comment). Ladder documented in ADR-039; every flip gets a DECISION_REGISTER row **and a per-class Agents-Rule-of-Two re-analysis** (each auto-flip moves that class to all three legs — untrusted input, sensitive system, autonomous state change — so the ceremony must show which leg is severed or bounded for that class).

## Model & stack tailoring (LA directive 2026-07-11; evidence in `coordinator-self-governance-research-2026-07.md` §8–10)

- **Small-model discipline (the 14B is the brain — design for it).** Every coordinator model call is a bounded, single-decision task (classify one ticket, draft one proposal, summarize one run) — never a holistic workspace judgment; the composition of judgments is deterministic code. Grammar-first with fail-soft parse (the #743 pattern); no coordinator correctness ever depends on the grammar path (the #725 crash is the standing warning). The constraint tax (latency + reasoning degradation under constrained decoding) is budgeted per heartbeat cycle; deterministic evidence-gathering first, model drafting last. This is both security posture and small-model necessity — injection resistance scales with capability, so the quarantined-drafter split matters more at 14B, not less.
- **Dense and MoE both, via capability contracts.** Each model role (coordinator-brain = dense 14B; coder = MoE 30B-A3B; future swaps either shape) binds to a declared capability contract — structured-output backend, tool-call format, grammar×spec-decode composability, eval-baseline file — not to a model name. A swapped-in model re-earns its role on measurement (#717 eval harness); MoE quantization fragility and format drift degrade quality, never safety, because all correctness lives in deterministic gates (gate as JUDGE, model as SIGNAL).
- **OpenCode permissions = defense-in-depth ring, never the boundary.** The coder harness gets path-scoped edit denies for governed-core paths in its config, but documented enforcement gaps on this exact platform (Windows/Git Bash, upstream issues #16126/#6396/#8832) mean the load-bearing severance stays in BlarAI's staging-time ruler + the fleet's `_FORBIDDEN_REPO_ROOTS`. SG (#848) includes a live adversarial probe of the harness ring rather than trusting its config.
- **ACP is the coordinator's sensory nerve + a third chokepoint.** C2/C3 monitoring consumes the typed ACP `session/update` event stream (per-tool-call visibility → honest stall detection, real-time #712-lineage progress) instead of log-tailing; the ACP `session/request_permission` channel gets a **deterministic** policy responder that refuses governed-core-touching tool calls mid-run (never the 14B — an LLM responder would be an injectable judge). **Wedge-safe refusal:** a bare refusal can hang an ACP agent on an unreturned tool call — refusal therefore translates into a clean cancel + run teardown + PARKED-HONEST outcome, with its own timeout and a wedge-recovery regression lock. Three rings total: staging-time refusal (before), ACP responder (during), worktree/merge gate (after).

## Operational-maturity additions (third design pass, 2026-07-11 — verified against the live instance)

The first pass built the fence, the second the method; this pass hardens the design for **years of unattended-adjacent operation**. Items 1–3 were verified live; the rest close failure modes found by walking the lifecycle.

1. **Dedicated Vikunja identity (verified defect).** The runtime bridge reuses the SAME account (`blarai`) as the MCP server and all human/dev writes (`vikunja_bridge.py:34-37`), so `created_by` cannot distinguish BlarAI-authored content — the structural-provenance control is inert under a shared account. Fix: a dedicated **`blarai-coordinator`** Vikunja user for all runtime-bridge writes. This makes `created_by` genuinely structural, enables least-privilege sharing per project, and gives audit queries a clean identity. (SG control 2 amended below.)
2. **Pagination-aware reads (verified defect class).** Vikunja paginates everything; the server's `maxitemsperpage` (default 50) silently clamps even explicit `per_page=200` requests — the dev-side `project_summary` is reading truncated task lists today (defect ticketed to devplatform). Coordinator reads MUST loop pages until a short page; flow metrics over a truncated board are silently wrong, which at a decades horizon is guaranteed.
3. **API-contract pinning (verified surface).** Buckets (`/projects/{id}/views/{view}/buckets`), relations, and saved filters confirmed present on the live v2.3.0 spec (118 paths). The bridge pins the API contract with regression tests against recorded response shapes; a Vikunja upgrade is a gated event (contract tests re-run), not a silent drift. The Vikunja SQLite DB joins the backup discipline — the board is load-bearing state.
4. **Execution-time re-validation (TOCTOU closure).** Approval is not freshness. Every approved proposal re-runs the FULL deterministic ruler at execution time (target containment, lane, WIP, DoR) — the world may have changed between staging and approval (ticket closed, repo moved, limit reached). The execution-time run is authoritative; a stale-invalid proposal is refused with a comment, never "best-effort executed."
5. **Cross-cycle proposal dedup + expiry.** Proposals carry a deterministic fingerprint (class + target + evidence hash); a condition detected every cycle (the same stall) stages ONE proposal, not one per cycle. Un-actioned proposals expire after a configured TTL (default 7 days) back to drafts with a note — so an operator absence never returns to a wall of stale asks.
6. **Tri-state substrate reads — fail-soft must never read as empty.** Every board/state read distinguishes `OK / EMPTY / UNREACHABLE`. The bridge's swallow-errors idiom is kept, but a `None` (unreachable) NEVER renders as "no open work": the snapshot shows "PM substrate unreachable," the quiet-queue tripwire suppresses (unknown ≠ quiet), and Vikunja-down is itself a surfaced condition. Without this, a dead Vikunja looks exactly like a finished backlog — the silent-stall the program exists to prevent.
7. **Shadow mode + measured graduation (validation harness, new ticket).** Before briefings ever go live, C3/C4 run in **shadow mode**: full cycles, proposals drafted and logged, nothing surfaced. The operator (or an eval pass) grades the shadow log offline; graduation to live briefings requires measured proposal precision ≥ a configured threshold over N cycles — the #717 eval-culture applied to the coordinator's judgment. A fixture-board **coordinator eval suite** (golden boards → expected classifications/pulls) joins `evals/` and re-runs at every model swap (the capability-contract enforcement).
8. **Global pull with starvation guard.** The pull queue is GLOBAL across projects (the dispatch WIP=1 is global — one GPU); ordering is class-then-age, so an Expedite-heavy project cannot starve others indefinitely (Standard items age into priority). Per-project WIP caps are optional policy on top.
9. **Operator-absence mode.** A config switch (or auto-detect via N unanswered briefings): only Expedite-class conditions surface; digests accumulate to a single catch-up brief; proposal TTLs pause. Absence must degrade to quiet safety, not alert debt.
10. **The briefing ledger (governance evidence).** Every briefing — proposals shown, lane, decision, timestamp, resulting action + outcome — appends to a durable per-month ledger file. This is the AIGP-grade audit trail: "what did I approve, when, and what happened" is queryable without archaeology. (Approval records are governance records.)
11. **Substrate setup is an operator-run migration, not a coordinator action.** The buckets/labels/filters the method needs are created by an idempotent discover-or-create setup script executed via the dev channel at C1 — the Coordinator never creates its own workflow structure. All Vikunja IDs (labels, buckets) are resolved by name at runtime, never hardcoded (the stale-label-ID lesson).
12. **Power/thermal-aware cadence.** The heartbeat honors the device: longer intervals on battery, alignment with the overnight-window schedule, and no model drafting while thermally throttled. A laptop coordinator that keeps the GPU warm all day fails the "decades of use" test.
13. **Status output is untrusted in chat.** `/coord status` renders ticket-derived text into the AO conversation — that content enters the AO's context under the UNTRUSTED provenance tier (datamarked, GUARDED-tool lock), or injection in a ticket title becomes injection in the assistant's chat. Same rule for digests.
14. **PA adjudication verbs for coordinator actions** are defined in ADR-039 (which Coordinator action classes map to which Policy Agent verbs/CARs), so coordinator-identity dispatches are adjudicated first-class, not as AO look-alikes.
15. **Program interlock with #740 (M2).** Both programs modify `shared/fleet/`; C2's driver-integrated checks and M2's plan-graph waves must sequence, not collide — the implementing sessions coordinate via the tickets, and merges follow the existing worktree discipline.

## Runtime topology & ownership (fourth pass, 2026-07-11 — LA Q&A clarifications)

1. **Heartbeat trigger = in-process launcher-managed timer**, primary. No new Windows Scheduled Task initially: a timer inside the running BlarAI backend needs no new privilege surface, sees swap state natively, and dies with the app (fail-safe). The existing elevated 23:00 overnight task remains the only app-not-running wake path. The trigger definition is governed core either way.
2. **At-rest encryption for coordinator stores.** The proposal-staging store, the briefing ledger, and the shadow journal are content-bearing (goals, ticket text) → born-encrypted, same posture as the session/knowledge stores (default-strict rule). The signed policy file is signed (integrity), not encrypted. The Vikunja DB stays third-party-native: loopback-only, encrypted disk, joins the backup discipline.
3. **Account topology (extends containment Decision 1, 2026-07-10):** the coder fleet runs in the **restricted** Windows account (untrusted model-written code executes in the contained domain); the **Coordinator runs in the operator's account** (trusted domain) — never in the restricted account, so the manager's state/policy is out of the worker's reach. The account boundary is an additional OS-level ring: restricted-account ACL-deny on coordinator state + governed-core policy paths (#848 records it as a supporting control).
4. **Merge boundary:** the Coordinator coordinates the lifecycle *around* merges (parked→proposal, merged→close-with-evidence, cross-job failure patterns in flow metrics) and NEVER the merge verdict — that stays with the deterministic fleet gate (author≠verifier applies to the PM too).
5. **Ownership migrations out of /dispatch:** ticket-lifecycle glue (the in-dispatch vikunja_bridge calls) → C2; "what runs next" (today: the operator types /dispatch) → C4 pull policy; ticket READINESS (DoR gate) → the PM, before a ticket may enter PLAN. Staying in dispatch: decomposition, acceptance criteria, clarify (#819)/revise (#820) — engineering, not management. **Single-approval UX:** a dispatch originating from an approved briefing item renders its plan-confirm inside that briefing — one approval moment per job, not two.
6. **Two-layer monitoring doctrine:** in-run tactical (swap-driver watchdogs, budgets, circuit breakers — the only layer alive mid-swap; seconds-to-hours) stays as-is; cross-run operational (aging, stalls across jobs, quiet-queue, flow trends; hours-to-weeks) is the Coordinator's. Same events, different timescales and response authority.
7. **The AO is front door + display only.** The Coordinator is a peer role: the heartbeat invokes it directly on the shared 14B with no conversation; C2 lifecycle hooks activate at the first dispatch/New-Project event with zero AO participation. The AO's roles: routing `/coord` commands and rendering digests/briefings in chat.

## Security/governance posture (constant across phases)

- **Self-governance boundary is constitutional** (see its section above): BlarAI has zero write path to its own governed core; self-work is advisory-only; config is immutable from inside; controls are structural, sized for a future non-technical operator.
- Loopback pin + bridge/transport split preserved; zero new egress; one-network-client invariant untouched.
- Model tool surface: READ-only GUARDED tools; ALL writes are deterministic reporting code or approval-staged proposals (`propose_preference` template). The #749 "no ticket tools" invariant is amended (read-only carve-out) explicitly in ADR-039.
- Ticket content read back into the model is **untrusted input** (datamark + GUARDED-tool lock per ADR-023 posture); prompt-injection reaches at most the proposal queue, never an action.
- Every phase ships dormant behind `[coordinator]` config flags; fail-soft toward the assistant (a coordinator failure never degrades chat); fail-closed on data.
