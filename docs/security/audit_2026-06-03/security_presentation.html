BlarAI Security Posture: What Protects You Today, and What Must Change Before You Go Online

A plain-language security review across 7 domains — verified against the actual code on disk — for the decision to remove the air-gap

Disk-rooted audit · 7 domains · 16-agent adversarial cross-check · 2026-06-03

What BlarAI Is — and Why It's Safe Today

  • BlarAI is your personal, locally-run AI: a Policy Agent (UC-001) that authorizes every action, and an Assistant Orchestrator (UC-004) that holds the conversation and your memory.
  • It runs entirely on one machine (Intel Lunar Lake + Arc GPU). Today it is air-gapped — there is no network connection at all.
  • The honest headline: the air-gap is doing almost all of the security work today. Most of the deeper controls are either built-but-not-switched-on, or built-but-not-wired-in.
  • This deck walks each of the 7 security areas in plain language, then shows the easiest ways someone could break in, then a prioritized to-do list before you ever go online.
  • Every claim here was checked against the actual code on disk, not the design documents. Where the docs over-promise, this deck says so plainly.

The Coming Shift: Going Internet-Facing Raises the Stakes on Everything

  • The plan is eventually to give BlarAI web access (navigate/retrieve). The day that happens, the single control protecting you today — the lack of a network — disappears.
  • Removing the air-gap inverts the risk profile: every 'soft' control (a convention, a docstring, a system-prompt instruction) suddenly becomes load-bearing, and every gap becomes remotely reachable.
  • Two compounding effects: (1) hostile web pages become a high-volume attacker-controlled input channel, and (2) any single foothold (a bug in a web tool) can reach your cleartext private data and the signing key.
  • The verified findings agree across all 7 domains: 'safe by environment, not yet safe by construction.' The design has the right seams; the enforcement is not yet built or not yet enabled.
  • Bottom line for the decision: a specific set of fixes must land BEFORE the air-gap is removed. Those fixes are the roadmap at the end of this deck.

Architecture Overview: What Actually Runs vs. What the Design Describes

  • The design describes isolated components talking over mutually-authenticated channels inside a locked-down Hyper-V VM. The code for that exists and is well-written.
  • What actually runs: one elevated host process. The Policy Agent, Assistant Orchestrator, the model, and your memory all run together, co-resident, on the host.
  • The Hyper-V VM is started and stopped on every boot — but it holds none of the security-critical code. The isolation boundary the design leans on is started-and-ignored.
  • The mutual-TLS channels exist in source but are inactive in the default (HOST) mode, which forces a development bypass (plain loopback, no certificate checks).
  • See the architecture diagram: the air-gap (bottom) is what actually separates this whole box from the world; the VM and mTLS (built-but-inactive) do not.
  • Evidence: launcher/__main__.py constructs PA/AO in-process; ADR-019 itself states they are 'in-process'; VM merely started at launcher/__main__.py:497-519.
  • The Hyper-V VM is a REAL, working isolation + file-transfer boundary — it is simply started EMPTY today (the security-critical code runs in the host process, not inside it). It is not vestigial: it could later hold the network-facing services so untrusted web content is contained, once its isolation is enforced.

Domain 1 — Hardware Trust Root & Measured Boot: Strong Parts, Unwired

  • What's good: there IS a genuine hardware trust root — a TPM key that the chip refuses to export (proven by a test that tries to export it and confirms the TPM says no). Real SHA-256 checks on the model weights do run at boot.
  • The catch: that TPM trust root is wired into nothing. A whole-repo search confirms the only thing importing it is its own test — zero runtime code uses it (shared/security/tpm_signer.py).
  • The 'measured boot' phase measures nothing: it just reloads a config file and checks files exist. It never touches the TPM, a hardware register, or any image hash (entrypoint.py:257-284). It can declare 'boot passed' on a compromised machine.
  • The 'known-good' list the integrity check trusts is a 342-byte plain JSON file with no signature (manifest.json). Both the boot check and the runtime check trust the same editable file — tamper it and both checks pass.
  • The default host launch silently runs in development mode (no mTLS, throwaway signing keys) because the launcher forces it — confirmed at launcher/__main__.py:570/636/715.
  • Severity: one Critical, several High. The air-gap, not the trust root, is what defends today.

Domain 1 — The Critical One: The Signing Key Is Sitting in Git

  • The Policy Agent's whole job is to authorize actions by issuing signed tokens. The private key it signs with — certs/pa_private.pem — is committed to the git repository in cleartext.
  • I confirmed it directly: it is a real EC private key (BEGIN PRIVATE KEY), tracked by git, and it is the configured production key path (default.toml:26). The matching public key (ca.pem) is what the Orchestrator trusts to verify tokens.
  • Anyone with the repository — or any future copy of its history (it was committed at 95c4f19) — can mint authorization tokens the system accepts as genuine. That defeats the entire authorization layer.
  • It cannot even be password-protected: the loader hardcodes 'no password,' so an encrypted key would fail to load (jwt_minter.py:146).
  • Why it's survivable today: the default live path runs in dev_mode with throwaway keys, and the air-gap means an attacker must already be on the box. Neither is true once you go online.
  • This is the #1 thing to fix: rotate the key, stop tracking it in git, and move signing into the TPM that already exists for exactly this.

Domain 2 — Policy Agent Authorization Gate: A Solid Skeleton With Inert Muscles

  • The good news is substantial: the deterministic core is genuinely fail-closed and verified. Missing config, malformed requests, integrity failures, identity mismatches — all deny. A deterministic DENY is final and skips everything downstream.
  • The gate has three layers: ordered rule engine, then a deny-list pre-filter, then an LLM classifier. Only an ALLOW yields a short-lived signed token. The deny direction is robust.
  • But the 'smart' layer is dead: the LLM is never asked to emit a confidence score, so a genuine LLM 'ALLOW' arrives at 0.0 confidence — below every threshold — and falls through to DENY. The LLM half can never actually approve anything (verified end-to-end).
  • The 'ask a human' safety valve (ESCALATE) has no consumer anywhere — it is silently treated as a DENY. There is no human-in-the-loop review queue built (verified by exhaustive search).
  • The revocation system is inert: the 'cut off a compromised agent' function is only ever called by tests, and the minter and validator track epochs as separate, never-reconciled objects.
  • Net: it fails safe (toward DENY) today, but the adaptive judgment, the escalation valve, and the revocation control the design relies on for online operation are all missing.

Domain 2 — Token Containment & A Denial-of-Service Lever

  • Authorization tokens are designed to be contained three ways: short life, revocable, single-use. In practice only one works: the short life (TTL).
  • Revocation is inert (above). Single-use is required but never enforced — the token ID is logged but never recorded as 'spent,' so a captured token can be replayed against a different instance or after a restart.
  • Worse, the short life isn't that short: both shipped configs set the token TTL to 30 seconds, not the 5 seconds the spec and code constant say — a silent 6x-wider replay window, and the config will accept up to 300 seconds.
  • Availability risk: before every single authorization, the system re-reads and re-hashes the entire multi-gigabyte model file. Correct, but it means an attacker who can drive traffic can stall the whole authorization choke-point.
  • Today none of this matters much — local trusted callers, no network. Online, the containment story degrades to 'TTL only,' and the per-request re-hash becomes a cheap way to grind the system to a halt.
  • Evidence: TTL 30s at both default.toml:25 and guest_runtime.toml:24; per-request full re-hash at adjudicator.py:318-325.

Domain 2 — The Worst Default: The Guest Profile Ships With Security Off

  • The architecture says services run inside the Hyper-V VM. The config selected for that VM (guest_runtime.toml) ships with dev_mode=true and measured_boot=false — verified on disk.
  • In that one profile, the entire perimeter drops at once: no mutual TLS, no caller-identity binding, no weight verification, no measured boot, and throwaway signing keys.
  • This is not a hypothetical stray flag — it is a committed, selectable runtime path that the code picks whenever deployment is set to 'guest' (runtime_config.py:106).
  • Air-gapped, a dev bypass is tolerable. Internet-facing, selecting this already-committed profile would authorize unauthenticated callers and mint forgeable-context tokens — a total perimeter collapse from one config choice.
  • Fix before going online: harden the guest profile to dev_mode=false (with real certs), or forbid dev_mode entirely outside an explicit local-test mode, and refuse to run dev_mode when network-facing.
  • This is the highest-leverage single-line risk in the authorization domain.
  • Why it is the worst: one config choice — deployment_mode=guest — collapses MULTIPLE controls at once (mTLS, identity binding, weight verification, measured boot). A single switch with whole-perimeter blast radius.

Domain 3 — IPC & Process Isolation: Built, Bypassed at Runtime

  • The mutual-TLS-over-vsock stack is fully written and sound (certificate checks, TLS 1.2+, bounded message sizes). The problem is it doesn't run: the default HOST mode forces a development bypass to plain loopback with no authentication.
  • Deeper gap found: the UI gateway that talks to the Orchestrator does no TLS at all on its production path — it never even imports the TLS library. So the 'authenticated channel' the design assumes does not exist in either mode (transport.py:416-436).
  • There are no production certificates on disk and no code to generate the per-boot certificates the design calls for — only a test fixture can mint them. A real production launch would fail to start the secure channel.
  • The VM doesn't contain what it's meant to: the security-critical code runs on the host, co-resident with the UI-facing process, so a host compromise is not contained by the VM.
  • Genuine live wins worth crediting: the WinUI window is de-elevated to Medium integrity (a real least-privilege improvement, ADR-019), and the local pipe rejects remote clients and is locked to your user account.
  • The local pipe authenticates only by OS file permissions, in neither direction — any process running as you can drive the backend or impersonate it to the UI.

Domain 4 — Prompt-Injection Defense (Input Side): The Deterministic Backstop Is OFF

  • What runs and is real: every loaded document and recalled memory gets its forged delimiters stripped and every line tagged as 'data, not instructions' with a fresh random marker per load, plus an always-on system prompt telling the model to never obey embedded instructions.
  • The strongest guarantee — 'even a fooled model still can't ACT' — is the tool-privilege gate (ADR-013). It is fully coded but DISABLED in both shipped configs (block_tools_when_documents_loaded = false), confirmed on disk.
  • Worse, the manual re-enable path is documented-broken: a 2026-05-26 user test found /trust didn't propagate and /unload didn't restore tool access (recorded in default.toml's own comment). So the action-side barrier is off by default AND can't currently be reliably turned on.
  • The spec's first sanitization layer — the 'Cleaner' service that would scrub and cryptographically sign every document — does not exist at all. Documents flow straight from disk to the model.
  • The spec calls the boundary markers 'cryptographic'; they are actually plain static text strings (the per-load tags ARE random, but the boundary tokens are code-visible). The governance doc honestly admits this.
  • Your own typed/pasted text is never scanned for injection — only loaded files are. Pasting hostile text from a web page into the chat box bypasses the warning.

Domain 4 — Why This Domain Is the Biggest Online Risk

  • Air-gapped, the only untrusted text is files you chose to load — and the controls that run (tagging + alignment prompt + warnings) are a credible defense against a passive injection.
  • Online, fetched web pages become a high-volume, adversarial injection channel — and the four-layer defense degrades to about two-and-a-half layers.
  • Missing for the online case: no Cleaner sanitization (Layer 1), the deterministic action-gate is off and its override broken, and the heuristic scanner never inspects fetched or pasted content.
  • What genuinely survives is the probabilistic stack: data-tagging + the alignment prompt + delimiter neutralization + an output-side check that catches the model echoing internal markers.
  • The three highest-value pre-internet fixes here: (1) re-enable the tool-gate AND fix the /trust path, (2) build the Cleaner with a signature gate, (3) extend the injection scan to fetched/pasted text.
  • Severity: two High (disabled gate, absent Cleaner), several Medium. This is the single largest spec-vs-build gap for the internet-facing future.

Domain 5 — Output Validation (PGOV): The Egress Guards Are Inert

  • PGOV is a 6-stage check on every response. The module is well-written and fail-closed on its own errors. Two of three data-leakage stages are inert in the shipped config.
  • The retrieval-leakage detector (catches the model parroting secrets from retrieved content) is fed an EMPTY list by the only production caller — I confirmed retrieved_chunks=[] hardcoded at entrypoint.py:1194. The detector is loaded and ready, but handed nothing to compare against.
  • The PII/secret detector ships OFF (pii_mode='off'). SSNs, API keys, credit cards, bearer tokens in model output are not detected or redacted on the live path.
  • What does run: the delimiter-echo check and the tool-call allowlist. Those are real and active.
  • Live streaming undercuts block-mode: tokens (even tool-call text) are shown on screen and spoken aloud BEFORE validation finishes — so suppression can't retract what you've already seen or heard.
  • The denial messages leak which stage fired (over the wire to the UI), contradicting the design's stated 'don't tell the adversary which control tripped.'

Domain 5 — The Allowlist Is Broader Than the Build

  • The tool-call allowlist names 14 tools — including high-impact ones like calendar_write, smart_home_control, note_create, code_agent — but only 4 read-only tools actually exist and execute.
  • Because the check only flags names OUTSIDE the allowlist, a model induced to emit an in-allowlist side-effecting tool name would NOT be flagged. Today it's latent (those tools don't run), but the allowlist is pre-approving a surface that isn't built or gated.
  • A credit-card detector claims 'Luhn-plausible' validation but performs no checksum — it flags any long number. The governance docs also cite line numbers that no longer match the code, which causes over-claiming of the security posture.
  • Net for PGOV: its deterministic 'verify the LLM behaved' promise is only partially delivered on the live path today.
  • Before going online, this domain needs: wire the retrieved chunks into the leakage check (a near-one-line fix), choose a non-off PII posture, buffer output until validated, prune the allowlist to built tools, and re-enable the tool-gate.
  • Severity: one High (leakage disconnected), one High (PII off), several Medium/Low.

Domain 6 — Privacy & Network Boundary: Real, But Enforced by Absence

  • The no-external-network mandate is real but enforced by ABSENCE and CONVENTION, not by an active guard. Verified: zero outbound network library calls in the runtime source code.
  • But there is no code-level kill-switch — no firewall rule, no DNS block, no egress allowlist, no socket guard. The mandate rests on (a) not importing network libraries, (b) ~20 docstrings saying 'no network,' and (c) a system-prompt instruction to the model.
  • The bigger picture (completeness gap): the runtime's Python environment SHIPS full network stacks (requests, httpx, aiohttp, openai, huggingface_hub, transformers) — about 270 packages. The guarantee rests entirely on source code not calling them; the capability is fully loaded in the process.
  • The Policy Agent's 'external network requests are always denied' rule (P-004) exists ONLY as a red-team test string — there is no code that classifies or denies a network-bearing action.
  • The 'search' tool is the seam internet access will arrive through — and it is BOTH pre-approved in the allowlist AND already advertised to the model in its system prompt as a live capability.
  • The same guest-profile dev_mode=true finding from Domain 2 reappears here independently.
  • Nuance (important): those ~270 network packages are PRESENT and loaded in the process, but NO runtime code CALLS them — egress is blocked by code discipline, not by a guard. There is no active network today; the boundary is one regression or one injection away, with nothing in code to stop it. That is why a code-enforced egress kill-switch is a Tier-0 fix.

Domain 7 — Data at Rest: Decades of Private Data, in Plaintext

  • Your entire history lives in two plaintext SQLite files: sessions.db (conversations) and substrate.db (a permanent searchable archive of every approved turn AND the full text of every document you load). No encryption at rest — confirmed: no SQLCipher, no DPAPI, no cipher anywhere.
  • substrate.db is the worst exposure: anyone who can read your user folder (malware, a stolen or backed-up disk, another local account, a sync tool) reads everything you've ever said or loaded.
  • There is no durable, tamper-evident audit log. The Policy Agent's 'complete audit record' lives only in memory and is discarded; the only on-disk trace is plaintext INFO lines in three deletable, unrotated logs (launcher.log, boot.log, crash.log).
  • The integrity 'known-good manifest' is unsigned JSON — an attacker with local write access can swap a model file AND edit the manifest, defeating the SHA-256 check entirely.
  • Diagnostic logs (including voice) are written unredacted into the same folder as your private data, one with a hardcoded developer home path.
  • The one strong piece — the non-exportable TPM key — is proven in hardware but connected to zero production code, so it protects nothing yet.

Cross-Cutting Completeness Gaps (Beyond the 7 Domains)

  • Supply chain: BlarAI's own dependencies are unpinned (floating '>=' with no upper bound) and there is NO lockfile and NO hash verification. A future 'pip install' pulls whatever the index serves that day — a compromise enters at build/update time, before the air-gap matters.
  • Model provenance: only ONE classifier weight file is ever hash-checked. The models that drive every visible feature (the 14B LLM, embeddings, Whisper, Kokoro, the vision model) are NEVER integrity-verified, and two of three tokenizer loads can fall back to fetching from the internet.
  • Dev/runtime co-residency: the Claude development session runs on the SAME machine with full internet and arbitrary-execution permissions, sharing the filesystem, the committed signing key, and your data DBs with the air-gapped runtime.
  • Error leakage: raw internal exception strings (paths, config, stack detail) are returned to the calling client — low risk on today's local pipe, but that dispatcher is exactly what a future network gateway would sit on.
  • Resource limits: the memory store grows unbounded (no retention), and the local pipe allows unlimited concurrent connections.
  • None of these are defended by the air-gap; several get worse the moment you go online.

Top Attack Paths — The Red-Team View

  • The easiest realistic compromises cluster around four enablers: the cleartext signing key, the disabled injection/output guards, the absent egress enforcement, and the plaintext data store.
  • Today the air-gap blocks the remote starting point for most of these. The diagram shows how a single online foothold chains into bulk data theft.
  • The marquee online path: hostile web page -> prompt injection -> tool call (gate is off) -> action or exfiltration, with the output leakage check inert to catch it.
  • The catastrophic path: any foothold reads substrate.db in cleartext (your whole history) and reads the signing key to forge unlimited authorization tokens.
  • The next slide's table ranks each path by severity and whether it opens or worsens once internet-facing.
  • Every path here traces to a specific verified finding — none are speculative.

Prioritized Hardening Roadmap — Before the Air-Gap Comes Off

  • TIER 0 (do first, blocking): Rotate the signing key + stop tracking it in git + move signing into the existing TPM. Build a code-level egress guard (default-deny + allowlist) — there is nothing to 'turn on' today.
  • TIER 1 (UC-001 / Policy Agent): Sign the weight manifest with the TPM; harden the guest profile to dev_mode=false and refuse dev_mode when network-facing; add a real network-egress adjudication rule (make P-004 enforced code, not a test string); stand up a tamper-evident audit log.
  • TIER 1 (UC-004 / Assistant Orchestrator): Re-enable the tool-privilege gate AND fix the broken /trust path (live-tested); wire retrieved chunks into the leakage check; switch PII detection to block/redact for online mode; prune the tool allowlist to the 4 built tools and stop advertising unbuilt 'Search' to the model.
  • TIER 2 (data + injection depth): Encrypt sessions.db and substrate.db (SQLCipher or TPM-wrapped key); build the Cleaner sanitization service with its signature gate; extend the injection scan to fetched/pasted content; add retention/pruning to the memory store.
  • TIER 2 (infrastructure): Actually run in GUEST/VM mode so mutual-TLS is exercised; build per-boot certificate generation; give the gateway a real mutual-TLS client path; put web-content-handling code inside the VM so it becomes a real containment boundary.
  • TIER 3 (supply chain + hygiene): Pin dependencies with a lockfile + hash verification; integrity-check ALL model weights; add a test that fails if a network library is imported; sanitize error strings before they leave the process.

The Decision in One Slide

  • Today: BlarAI is safe primarily because it's air-gapped. The deterministic deny-machinery, fail-closed posture, de-elevated UI, and local-only pipe are genuine and worth crediting.
  • The deeper controls the security story promises are either built-but-off (tool-gate, PII filter, leakage check) or built-but-unwired (the TPM trust root, mutual-TLS, the VM isolation).
  • Going internet-facing removes the one control doing the real work and makes every gap remotely reachable — most urgently the cleartext signing key, the absent egress enforcement, and the plaintext history store.
  • Recommendation: do NOT remove the air-gap until at least Tier 0 and Tier 1 of the roadmap are complete and live-verified.
  • The encouraging part: the design has the right seams everywhere. The work is mostly connecting and enabling what's already there, plus building three genuinely-missing pieces (egress guard, Cleaner, at-rest encryption).
  • This whole review was disk-verified — it's an honest baseline you can hand to anyone, including the AIGP portfolio.
