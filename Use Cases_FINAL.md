[USE-CASE-001]: Policy-Driven Security & Access Orchestration (The Policy Agent)
Description: A continuously resident, NPU-driven semantic gatekeeper that intercepts, classifies, and adjudicates every tool-call, data-access request, and outbound egress attempt generated by any agent in the system. It enforces intent-level access control using hardware-attested identity verification (Microsoft Pluton and Intel SGX as candidate solutions for key management and runtime attestation) and operates under a strict Fail-Closed architecture: all unclassified, ambiguous, or potentially sensitive payloads are blocked by default and escalated to the user for manual approval. The Policy Agent implements a hybrid deterministic-probabilistic adjudication model: all inter-agent tool calls are first reduced to a Canonical Action Representation (CAR) and evaluated by a deterministic algorithmic checker (candidate: regex constraint engine + semantic distance calculator) before the NPU-resident LLM performs probabilistic semantic classification. Only if both layers approve the action does the Policy Agent mint a cryptographically signed Decision Artifact (candidate: Agentic JWT signed via Microsoft Pluton) that the destination microservice must validate before executing the request. Destination microservices are hard-coded to reject any execution request lacking this unforgeable receipt, guaranteeing a non-bypassable Action Authorization Boundary (AAB). All vsock (AF_HYPERV) IPC channels under the Policy Agent's control are fortified with mutual TLS (mTLS) authentication, ensuring that even a compromised container with a valid Context Identifier (CID) cannot establish a functional connection without presenting a valid, Policy-Agent-issued ephemeral certificate. The Policy Agent is the first component to become operational during system startup via a deterministic, hardware-rooted Measured Boot Sequence — no other agent VM is permitted to launch, accept connections, or process data until the Policy Agent has completed attestation, derived its cryptographic identity, and established its internal Certificate Authority. This sequenced boot is the architectural root of trust for the entire multi-agent system. The Policy Agent holds exclusive NPU Priority 0 scheduling designation — its inference requests preempt all other NPU workloads, ensuring the security gate is never starved by lower-priority inference tasks. The shared mmap weight file is verified at boot via SHA-256 against the Pluton-sealed Known-Good Manifest and re-verified at runtime before every adjudication cycle via event-triggered integrity checking; the mmap region is established as a read-only section with no copy-on-write semantics, and any write fault terminates the offending process immediately (Fail-Closed). The Policy Agent VM holds a guaranteed hard memory reservation that is never reclaimable by Windows, never degraded under any execution tier (including Code Agent [005] activation), and is the highest-priority memory consumer in the system.

Agentic JWT Lifecycle Specification (ISSUE-006 Resolution — Option C):

The Agentic JWT is the sole authorization artifact within the Action Authorization Boundary (AAB). Every JWT minted by the Policy Agent conforms to the following lifecycle constraints:

1. Scope: Instance-Scoped, Single-Use.
	- Each JWT is bound to exactly one adjudicated CAR instance. The JWT payload includes a SHA-256 hash of the complete CAR (target CID, operation type, parameter set, payload hash). A JWT authorizes one specific action — not an action class, not a category of operations, and not a reusable capability. A JWT that does not match the exact CAR hash presented by the requesting agent at the destination is rejected.

2. Nonce: Cryptographic, Non-Replayable.
	- Each JWT contains a 128-bit cryptographically random nonce generated at mint time. Every destination microservice maintains a nonce-seen set — an in-memory data structure that records all nonces observed within the active TTL window. Any JWT presenting a nonce that exists in the destination's nonce-seen set is rejected immediately (replay prevention). The nonce-seen set is garbage-collected every TTL interval (5 seconds); expired entries are pruned deterministically.

3. TTL: 5 Seconds, Hard Expiry.
	- Each JWT carries an `exp` claim set to the current Policy Agent clock time + 5 seconds. Destination microservices reject any JWT whose `exp` claim is in the past relative to their own clock. Clock skew between VMs is bounded by Hyper-V's synthetic clock synchronization (sub-millisecond drift). A JWT that is not consumed within 5 seconds of minting is irrevocably dead — the requesting agent must re-submit the CAR to the Policy Agent for a fresh adjudication and fresh JWT. There is no renewal, extension, or refresh mechanism.

4. Signing: Pluton-Backed, Hardware-Rooted.
	- Each JWT is signed using the Policy Agent's Pluton-derived signing key. The signing key is generated during Phase 1 of the Measured Boot Sequence and never exists outside Pluton-protected memory. Signing latency is estimated at 5–20ms per JWT (Pluton TPM 2.0 HMAC/ECDSA operation). This cost is additive to the dual-gate adjudication latency (\~50–200ms for CAR check + NPU inference) and is budgeted within the per-tool-call authorization overhead at \~70–230ms total.

5. Epoch-Based Revocation.
	- The Policy Agent maintains a monotonically increasing epoch counter (uint64), initialized to 1 at boot. Every minted JWT includes the current epoch value in a custom `epoch` claim. When the Policy Agent revokes any agent's mTLS certificate (due to attestation failure, anomalous behavior, or administrative action), it atomically increments the epoch counter. The new epoch propagates to destination microservices lazily — piggybacked on the next JWT each destination receives. Each destination maintains a `last_seen_epoch` value. Upon receiving a JWT, the destination updates its `last_seen_epoch` to the JWT's epoch if the JWT's epoch is greater than the stored value. Any subsequent JWT presenting an epoch older than the destination's `last_seen_epoch` is rejected regardless of signature validity.
	- Residual Risk (Bounded): Between revocation (epoch increment) and the next JWT arrival at a given destination, the destination is unaware of the revocation. A compromised agent could replay a pre-revocation JWT against that destination during this window. This window is bounded to a maximum of 5 seconds by the JWT TTL — after 5 seconds, all pre-revocation JWTs are expired regardless of epoch propagation. Within this window, the nonce-seen set prevents re-replay of the same JWT. The bounded residual risk is therefore: at most one unauthorized action per destination, within a 5-second window, following a revocation event. This is accepted as a documented residual risk.

6. Destination Enforcement (Hard Constraint).
	- Every destination microservice implements a JWT validation gate as the first instruction in its request handler. The gate performs the following checks in strict order: (a) signature verification against the Policy Agent's public key, (b) `exp` claim validation against local clock, (c) `epoch` claim validation against `last_seen_epoch`, (d) nonce uniqueness check against nonce-seen set, (e) CAR hash match between JWT payload and presented request. If any check fails, the request is rejected and the failure is logged to the Policy Agent's audit stream. There is no fallback, degraded mode, or override. A destination microservice that cannot reach the Policy Agent to obtain the current public key (e.g., during a transient IPC failure) rejects all requests until connectivity is restored (Fail-Closed).

7. Memory Budget.
	- Nonce-seen set per destination: <1MB at peak throughput (5-second TTL window, 16–32 bytes per nonce entry, GC every 5 seconds). Epoch counter: 8 bytes per destination. Total JWT lifecycle state across all destination microservices: <5MB. This is negligible against the 31.5GB effective ceiling (32GB physical minus 512MB DVMT pre-allocation) and is included in each agent VM's existing memory reservation.

Core Problem Statement: The system suffers from Semantic Transparency — autonomous agents generate non-deterministic tool calls that can encapsulate high-sensitivity data (e.g., proprietary source code) within legitimate-looking outbound payloads. A hallucinating or compromised agent could construct a syntactically valid request that embeds fragments of private journal entries, sentiment scores, or health data encoded in query parameters, POST bodies, or header metadata. No network-layer device can inspect the semantic intent or payload sensitivity of an AI-generated tool call; it operates exclusively on IP, port, and protocol. However, relying solely on a sub-3B probabilistic LLM for semantic classification introduces a secondary, empirically documented vulnerability: contemporary evaluations demonstrate that models in the 1.7B-8B parameter range exhibit failure rates exceeding 70% against adversarially transformed or out-of-domain (OOD) queries that superficially resemble benign inputs (Phase 3, Gap 1). An advanced attacker utilizing sophisticated prompt injection obfuscation — including homoglyph substitution, instruction-data boundary blurring, or multi-turn semantic drift — could craft tool-call payloads that the probabilistic classifier misidentifies as operationally benign, allowing semantic exfiltration to proceed unchallenged. A purely probabilistic gatekeeper is therefore a mathematically insufficient security boundary for a system holding decades of private data. The system requires a hybrid gatekeeper combining deterministic schema enforcement (Canonical Action Representation) with probabilistic semantic classification, backed by cryptographic non-repudiation (Agentic JWT), to create a non-bypassable Action Authorization Boundary where no single-layer failure — whether from OOD evasion of the LLM or from schema manipulation of the deterministic checker — is sufficient to authorize a malicious action. Additionally, the vsock IPC bus — while eliminating traditional TCP/IP attack surfaces — relies on Context Identifiers (CIDs) that lack inherent cryptographic security (Phase 3, Gap 2); if an attacker compromises a Hyper-V worker jail, the jail retains its valid CID and could attempt unauthorized queries against internal microservices. The IPC layer must therefore enforce mTLS over vsock to ensure that a compromised CID alone is insufficient to establish a functional connection. Furthermore, the system suffers from a Bootstrap Trust Paradox (Red Team ISSUE-001): the Policy Agent is the root of trust for all inter-agent authentication (mTLS CA, Agentic JWT minting), but it must itself be launched, attested, and initialized before it can issue credentials to any other component. During this bootstrap window, the system is in an unauthenticated state — if the host OS or hypervisor is compromised before the Policy Agent completes attestation, a tampered binary could be loaded, and all downstream cryptographic guarantees would be structurally valid but issued by a compromised gate. The architecture must define a deterministic, hardware-rooted Measured Boot Sequence that chains trust from immutable silicon (Pluton) through the hypervisor to the Policy Agent's code and data, ensuring that no software-only attack can substitute or modify the Policy Agent before attestation completes. The system must also define explicit crash recovery behavior: if the Policy Agent fails during any boot phase, the AI system must hard-lock (Fail-Closed) while leaving the host OS fully operational, with a bounded automatic retry mechanism before requiring manual user intervention. Additionally, the Intel NPU on Lunar Lake is a single inference accelerator with a unified command queue whose concurrent multi-model scheduling behavior is undocumented (Red Team ISSUE-002). The Policy Agent's probabilistic gate is the single most latency-critical and security-critical NPU consumer — if it is queued behind lower-priority inference tasks, the security gate is starved and inter-agent authorization latency becomes unbounded. The architecture must designate the Policy Agent as NPU Priority 0 (preemptive) to guarantee that its inference is never delayed by other NPU consumers, and must define explicit KV-cache residency policies to prevent context-switch overhead from invalidating the Policy Agent's warm state. Furthermore, the shared mmap weight file creates an integrity coupling (Red Team ISSUE-003) between the Policy Agent and the Assistant Orchestrator: both agents execute inference against the same physical memory pages, meaning a single corruption event compromises both the security gate and the conversational agent simultaneously. The Policy Agent cannot detect its own corruption because its classification logic is derived from the same potentially corrupted weights. Additionally, a Copy-on-Write (CoW) attack could silently diverge the weights seen by one agent from the other if any process triggers a write fault against the mmap'd region. The architecture must enforce boot-time verification, event-triggered runtime re-verification before every security-critical adjudication, and strict read-only mmap semantics with write-fault termination to mitigate these risks. This shared failure mode is accepted as a documented residual risk for initial development, to be revisited after 30 days of production use of [001]+[004] concurrent operation. Additionally, the 31.5GB effective ceiling (32GB physical minus 512MB DVMT pre-allocation) is violated under concurrent peak load when all Priority 1+2+3 agents co-exist with realistic Hyper-V VM overhead, Windows root partition memory under TDX load, and iGPU reservation granularity (Red Team ISSUE-004). The Policy Agent — as the non-degradable security root — must hold a guaranteed hard memory reservation that is never reclaimed by Windows or Hyper-V dynamic memory, never reduced during any execution tier transition, and is the first reservation established during the Measured Boot Sequence. Additionally, the Agentic JWT lifecycle — including token expiration, replay prevention, scope granularity, and revocation propagation — must be precisely specified to prevent token replay, scope overreach, and stale-credential exploitation (Red Team ISSUE-006). The architecture mandates instance-scoped, single-use, nonce-bound JWTs with a 5-second hard TTL, Pluton-backed signing, and epoch-based lazy revocation propagation, ensuring that no captured JWT can be replayed, no JWT authorizes more than the exact action adjudicated, and revocation propagates deterministically with a bounded 5-second residual risk window.

Hardware & Security Feasibility Check:

Measured Boot Sequence (Red Team ISSUE-001 Closure):

The Policy Agent's operational integrity depends on a four-phase, deterministic, hardware-rooted boot sequence. No AI agent VM is permitted to launch, accept IPC connections, or process data until this sequence completes successfully. The host Windows 11 Pro OS and all non-AI applications are unaffected by the AI system's boot state — the Measured Boot Sequence governs exclusively the Hyper-V VM layer hosting the multi-agent system.

Phase 0 — Silicon Root of Trust (Pluton Measurement): Before any AI-related VM is launched, the Pluton secure crypto-processor (candidate solution) measures the integrity chain: UEFI firmware hash → Hyper-V hypervisor image hash → Policy Agent VM disk image hash → shared mmap weight file SHA-256 hash. These measurements are compared against a Known-Good Manifest (KGM) sealed in Pluton-protected storage. The KGM is provisioned via a one-time, user-initiated, physically-present provisioning ceremony performed on the existing device during initial system setup — this ceremony computes and seals the SHA-256 hashes of all trusted VM images and the shared model weight file into Pluton's hardware-protected storage using the Windows TPM/Pluton API. The KGM can only be updated by re-executing the provisioning ceremony with physical user presence; no remote attacker, compromised agent, or autonomous process can modify the sealed hashes. If any measurement in Phase 0 diverges from the KGM, the boot sequence terminates immediately — no VM is launched, and the system enters hard-lock state. The weight file hash verification (SHA-256 of the \~1GB quantized weight file on NVMe, completing in \~1 second) is performed during this phase, ensuring that the shared mmap weight file has not been corrupted or tampered with before any agent loads it into memory. This closes the shared-mmap boot-time integrity concern (Red Team ISSUE-003 partial — runtime integrity monitoring is addressed separately in ISSUE-003 resolution).
Phase 1 — Policy Agent Exclusive Launch: Only the Policy Agent VM is permitted to launch. Upon boot, it performs SGX local attestation (candidate solution) to verify that its own runtime code and deterministic rule set have not been tampered with since the Pluton measurement in Phase 0. It then derives its internal Certificate Authority (CA) private key from Pluton-sealed storage — the CA key material is generated during the initial provisioning ceremony and sealed to Pluton; it never exists in host OS RAM or in any non-TEE memory region. The Policy Agent generates ephemeral per-boot CA certificates (the CA key is persistent in sealed storage, but the operational certificates are freshly minted each boot, preventing stale certificate replay across boot cycles). The NPU-resident 1.7B INT4 model is loaded via OpenVINO (candidate runtime) with NPU Priority 0 scheduling designation — the Policy Agent's model is loaded first and its inference context is registered as the highest-priority workload on the NPU command queue. The deterministic CAR validator and regex constraint engine are initialized, and the mTLS listener on the vsock bus is activated. Phase 1 is complete when the Policy Agent's self-attestation succeeds, the CA is operational, and the hybrid gate (deterministic + probabilistic) is ready to adjudicate requests. Estimated Phase 1 duration: 5–10 seconds (NPU model load dominates).
Phase 2 — Sequenced Agent Credential Issuance: The Policy Agent issues mTLS ephemeral certificates to each agent VM in a deterministic, hardcoded launch order. For the Priority 1 Core Loop (concurrent development): Assistant Semantic Router VM, then Assistant Orchestrator VM. For Priority 2 (Memory & Ingress): Substrate VM, then Cleaner VM. For Priority 3 (Mid-Term): Code Agent VM. Each agent VM, upon launch, presents its measured disk image hash to the Policy Agent over the initial vsock handshake. The Policy Agent validates this hash against the corresponding entry in the Pluton-sealed Known-Good Manifest. If the hash matches, the Policy Agent issues an ephemeral mTLS certificate bound to that agent's CID and measured image hash. If the hash does not match, the agent VM is refused credentials, is forcibly terminated by the Policy Agent (via Hyper-V management API), and the event is logged. The agent VM cannot participate in any IPC without a valid certificate — it is architecturally inert. Phase 2 is complete when all agent VMs in the active priority tier hold valid mTLS certificates. Estimated Phase 2 duration: 3–8 seconds (dominated by VM launch and hash validation per agent).
Phase 3 — Operational Gate: Only after Phase 2 completes successfully does the Policy Agent open the user-facing interaction surface (the Assistant's vsock listener for the chatbot UI). The system is now in Operational state. Total estimated cold-start latency: 10–20 seconds from AI system launch trigger to user-ready state. This latency is accepted as the required cost of Fail-Closed bootstrap integrity (Lead Architect Decision 1).
Crash Recovery Posture (Red Team ISSUE-001 Closure):

If the Policy Agent fails during any boot phase (Phase 0 measurement divergence, Phase 1 SGX attestation failure, Phase 1 NPU driver fault, Phase 1 OOM, Phase 2 agent hash validation failure), the following deterministic recovery sequence executes:

Automatic Retry: The system attempts up to 3 automatic retries with exponential backoff (wait 2 seconds, then 4 seconds, then 8 seconds between attempts). Each retry re-executes the full Measured Boot Sequence from Phase 0. This handles transient failures (one-time NPU driver timeout, temporary resource contention during host OS startup).
Hard-Lock after 3 Failures: If all 3 retry attempts fail, the AI system enters a permanent hard-lock state: all AI agent VMs are terminated, the vsock bus is deactivated, and a persistent failure log is written to a user-accessible location (candidate: %LOCALAPPDATA%\BlarAI\boot_failure.log) containing the failure phase, failure type, and diagnostic fingerprint. The AI system remains inoperable until the user manually re-triggers the Measured Boot Sequence (candidate: a dedicated PowerShell launcher script or native application that re-initiates Phase 0 from the Pluton root).
Host OS Isolation Guarantee: At no point during the Measured Boot Sequence, automatic retry, or hard-lock does the AI system affect the host Windows 11 Pro OS. The laptop boots normally, all non-AI applications function normally, the desktop and file explorer are fully operational. The AI system is a service-layer tenant within Hyper-V — its failure is equivalent to a Windows Service failing to start. The user retains full use of their device and can diagnose the boot failure at their convenience.
Persistent Failure Resolution: If the boot failure is persistent (e.g., a Windows Update changed the UEFI measurement baseline, a firmware update altered the Hyper-V image hash, or the shared weight file was corrupted on disk), the user must resolve the root cause (e.g., repair or re-download the corrupted file) and then re-execute the provisioning ceremony to update the Known-Good Manifest with the new valid measurements before re-triggering the Measured Boot Sequence. This is the correct Fail-Closed behavior: the system refuses to operate with unverified components and provides the user with the diagnostic information and tools to restore integrity.
NPU Priority 0 Scheduling Designation (Red Team ISSUE-002 Closure):

The Policy Agent's 1.7B INT4 probabilistic classifier holds exclusive NPU Priority 0 — the highest scheduling priority on the Intel NPU's unified command queue. This designation has the following architectural consequences:

Preemptive scheduling: If the Assistant Orchestrator (the only other NPU-resident model in the Priority 1–3 stack) is mid-generation and a tool-call arrives requiring Policy Agent adjudication, the NPU pauses the Orchestrator's inference to service the Policy Agent's request. The Orchestrator resumes after the Policy Agent completes. This introduces micro-stalls (\~50–200ms) in conversational response generation during concurrent tool-call adjudication. This is accepted as the correct tradeoff: security gate latency must never be bounded by conversational generation throughput.
KV-cache residency guarantee: The Policy Agent's KV-cache (350–550MB) must remain resident in NPU-accessible memory at all times during Operational state (Phase 3 of the Measured Boot Sequence). The NPU runtime must not evict or invalidate the Policy Agent's KV-cache during context switches to service lower-priority workloads. If the NPU's context-slot architecture does not support persistent KV-cache across model switches (a hardware-dependent unknown to be determined via the Phase 2 Day-1 empirical gate), the Policy Agent's KV-cache must be backed by a CPU-side shadow copy that can restore warm state in under 500ms after a context switch.
Phase 2 Day-1 Empirical Validation Gate (Red Team ISSUE-002 Closure): Before any agent implementation code is written, the NPU's actual concurrent scheduling behavior must be empirically characterized on the physical Lunar Lake hardware. The validation gate consists of: (1) Load the Policy Agent 1.7B INT4 model and the Assistant Orchestrator 1.7B INT4 model simultaneously on the NPU via OpenVINO; (2) Measure actual concurrent throughput (parallel vs. time-sliced); (3) Measure KV-cache persistence across context switches; (4) Measure preemption latency (time from Policy Agent inference request to NPU service when the Orchestrator is mid-generation). If the NPU supports true concurrent dual-model inference with persistent KV-caches, the current architecture proceeds as designed. If the NPU implements time-sliced scheduling (the expected case), the architecture validates that the Policy Agent preemption + Orchestrator resume latency fits within the revised latency budgets defined in [USE-CASE-004]. If neither mode achieves acceptable latency, the architecture escalates to a fallback where the Policy Agent's probabilistic classifier migrates to CPU (accepting \~2–5x inference latency degradation for the security gate) while the Orchestrator retains NPU exclusivity — this is the last-resort fallback and requires explicit Lead Architect re-approval if triggered.

Shared mmap Weight File Integrity (Red Team ISSUE-003 Closure):

The Policy Agent and Assistant Orchestrator share the same 1.7B INT4 quantized weight file via zero-copy mmap to conserve RAM within the 31.5GB effective ceiling (32GB physical minus 512MB DVMT pre-allocation). This shared-weight design is a measured RAM optimization with a documented residual risk: both agents execute inference against identical physical memory pages, creating an integrity coupling where a single corruption event compromises both the security gate and the conversational agent simultaneously. This residual risk is accepted for initial development and is scheduled for revisit after 30 days of production use of [001]+[004] concurrent operation, at which point the Lead Architect will evaluate whether to upgrade to independent weight copies (\~1GB additional RAM cost) based on empirical data from the event-triggered hash checks (frequency of divergence detections, if any). The following three-layer integrity mitigation stack addresses this coupling:

Layer 1 — Boot-Time Verification (ISSUE-001 Phase 0): The shared weight file's SHA-256 hash is validated against the Pluton-sealed Known-Good Manifest during Phase 0 of the Measured Boot Sequence, before any agent loads it into memory. If the hash diverges, the boot sequence terminates immediately (Fail-Closed). This prevents a pre-corrupted or tampered weight file from ever being mmap'd. This layer is already integrated into the Measured Boot Sequence above.
Layer 2 — Event-Triggered Runtime Re-Verification: Before every Policy Agent adjudication cycle (i.e., every time the Policy Agent is about to invoke NPU inference to classify a tool-call payload as part of the hybrid gate), the Policy Agent computes the SHA-256 hash of the mmap'd weight region and compares it against the known-good measurement from Phase 0. If divergence is detected, the Policy Agent immediately halts all inference, blocks all pending tool calls (Fail-Closed), and escalates to the user via the boot failure log mechanism (%LOCALAPPDATA%\BlarAI\runtime_integrity_failure.log). This event-triggered verification ensures that the weights are intact for every security-critical decision — no JWT is ever minted against potentially corrupted weights. Performance cost: SHA-256 hashing \~1GB of memory takes \~200–500ms on Lunar Lake P-cores. This cost is added to every adjudication cycle, increasing the Policy Agent's per-adjudication latency from the current \~50–200ms (NPU inference only) to \~250–700ms (hash verification + NPU inference). This is within the 500ms user-approval prompt latency target when the hash verification is pipelined with the deterministic CAR validation (which runs concurrently on a separate CPU core while the hash is computed). Revisit milestone: This event-triggered cadence is the initial development posture. After [USE-CASE-009] (Autonomous Maintainer) achieves its first successful staging-and-approval cycle in production, the Lead Architect will revisit and evaluate upgrading to periodic background verification (candidate: 60-second intervals) to close the window between adjudication cycles during which a disk-level corruption could propagate to the mmap region undetected. During the [001]–[005] development window, the weight file is static on disk and mmap'd read-only — no autonomous process can modify it at runtime — making event-triggered verification sufficient.
Layer 3 — CoW Prevention (Hypervisor-Enforced): The mmap region is established as a read-only section with no copy-on-write semantics at the hypervisor level (Windows equivalent of MAP_SHARED | PROT_READ). The page table entries for the shared weight region are marked read-only in the Extended Page Tables (EPT) managed by Hyper-V. If any process — including the Policy Agent's own inference runtime or the Assistant Orchestrator's runtime — triggers a write fault against the mmap'd region (e.g., via an exploit in the inference runtime that attempts to modify weights in-place), the hypervisor raises an access violation and immediately terminates the offending process. If the terminated process is the Policy Agent, this triggers the Crash Recovery Posture (automatic retry with exponential backoff → hard-lock after 3 failures). If the terminated process is the Assistant Orchestrator, the Policy Agent logs the event, revokes the Orchestrator's mTLS certificate, and escalates to the user. This is a hard architectural constraint enforced at the hypervisor level — it does not depend on the inference runtime's own memory protection behavior.

Hard Memory Reservation and Execution Tier Immunity (Red Team ISSUE-004 Closure):

The Policy Agent VM is configured with a static (non-dynamic) Hyper-V memory allocation — a fixed amount of RAM assigned at VM creation that cannot be reclaimed by the Windows host under any memory pressure condition. The exact allocation is determined by the Phase 2 empirical memory map gate (mandatory prerequisite before implementation). The Policy Agent is never degraded under any execution tier: when the Code Agent [005] is activated and the architecture enforces a degradation posture on other agents (Substrate cross-encoder suspended, Orchestrator KV-cache flushed), the Policy Agent's memory reservation, KV-cache, NPU Priority 0 designation, and full hybrid gate capability remain completely unchanged. This is a non-negotiable architectural invariant: the security gate operates at full capacity regardless of system load. The Policy Agent's static memory reservation is the highest-priority reservation in the system — it is allocated before any other agent VM during Phase 1 of the Measured Boot Sequence, and if insufficient RAM remains for other agents after the Policy Agent's reservation, the lower-priority agents fail to launch (Fail-Closed) rather than the Policy Agent accepting a reduced allocation.

Phase 2 Empirical Memory Map Gate (Red Team ISSUE-004 Closure):

Before any agent implementation code is written, a precise, empirically measured memory map must be constructed on the physical Lunar Lake hardware. This gate is mandatory and runs alongside the NPU empirical gate (Red Team ISSUE-002). The memory map must measure: (1) Windows 11 Pro base memory consumption under active Hyper-V with TDX-enabled VMs (expected: 8–10GB, not the 6.4GB idle baseline); (2) Hyper-V root partition overhead including Extended Page Tables (EPT), IOMMU structures, and TDX metadata (TDMR) per VM; (3) each VM's guest OS footprint (guest kernel, virtio drivers, init system — expected: \~256–512MB per minimal Linux guest); (4) each agent's true measured resident set size (RSS) under load; (5) Arc 140V iGPU DVMT Pre-Allocation and Shared Memory Override Granularity (Red Team ISSUE-008 Closure — DVMT): The BIOS-level DVMT (Dynamic Video Memory Technology) pre-allocation is fixed at the minimum BIOS granularity of 512MB. This 512MB is unconditionally reserved from the 32GB physical LPDDR5X pool at boot, reducing the effective system memory ceiling to 31.5GB. All memory budget calculations, execution tier validations, and VM static memory limits are computed against this adjusted 31.5GB ceiling, not the raw 32GB hardware total. The 512MB DVMT floor is a deterministic, BIOS-enforced constant — it does not vary with GPU workload, driver state, or Intel Shared GPU Memory Override settings. The remaining iGPU shared memory (up to \~20GB via Intel Shared GPU Memory Override) is allocated dynamically from the 31.5GB pool as needed by GPU-accelerated workloads ([005]). The Phase 2 memory map must empirically confirm that 512MB is the actual minimum BIOS DVMT granularity on the physical Lunar Lake unit (VALIDATE_DVMT_BUDGET). The memory map deliverable determines the exact static memory limits for each VM and validates that the architecture fits within the adjusted 31.5GB ceiling under all defined execution tiers. If the empirical memory map reveals that the architecture exceeds the 31.5GB effective ceiling under any defined execution tier, the Lead Architect must resolve the overage before implementation proceeds — candidate resolutions include reducing the Code Agent's model size, permanently offloading the Substrate to the Lenovo Y700, or reducing the number of concurrent VMs.
Operational Runtime (Previously Established — Preserved):

The Policy Agent's probabilistic layer runs as a stateful model on the NPU via a candidate inference runtime (e.g., OpenVINO). Candidate reference model: 1.7B-class instruction-tuned LLM at INT4 quantization, consuming \~1-1.5GB resident RAM. The deterministic layer (CAR schema validator + regex constraint engine + semantic distance calculator) runs CPU-side at negligible cost (less than 50MB). The hybrid architecture ensures that the probabilistic model's empirically documented failure modes against adversarial OOD queries (>70% evasion rate for small models) are backstopped by deterministic schema enforcement: a tool call that passes the LLM but violates CAR structural constraints (e.g., unexpected destination CID, payload field outside declared schema, semantic distance from declared intent exceeding threshold) is blocked before reaching the cryptographic signing stage. Conversely, a tool call that satisfies CAR structural constraints but triggers probabilistic suspicion (e.g., the LLM detects anomalous semantic content within a structurally valid payload) is similarly blocked. This dual-gate architecture requires that both layers independently approve an action before the Agentic JWT is minted — neither layer alone is sufficient to authorize execution, closing the OOD evasion gap identified in Phase 3. Decision Artifact minting utilizes the Microsoft Pluton secure crypto-processor (candidate solution) for hardware-attested JWT signing; destination microservices validate the JWT signature before processing any request. Intel SGX enclaves are a candidate solution for runtime attestation that the Policy Agent's own code and deterministic rule set have not been tampered with. Both are present on the Lunar Lake SoC. The Policy Agent sits architecturally inside the vsock (AF_HYPERV) IPC bus — all inter-VM messages pass through it before reaching any network stack. Vsock channels are fortified with mTLS: each agent VM is provisioned with an ephemeral TLS certificate issued by the Policy Agent's internal certificate authority during Phase 2 of the Measured Boot Sequence; the receiving endpoint validates both the mTLS certificate and the Agentic JWT before allocating buffer space or processing the incoming RPC. This closes the CID-weaponization gap (Phase 3, Gap 2): even if an attacker compromises a worker jail and retains its valid CID, the compromised jail cannot present a valid mTLS certificate (which is rotated per-boot and bound to the Policy Agent's attestation chain) or a valid Agentic JWT (which requires passing both the deterministic and probabilistic gates). The Ubiquiti Dream Machine Router remains the independent network-layer boundary (blocking unauthorized ports/IPs/VLANs). Zero-copy mmap of the LLM weights — verified at boot via SHA-256 hash against the Pluton-sealed Known-Good Manifest, re-verified at runtime before every adjudication cycle, and protected against CoW divergence via hypervisor-enforced read-only page protections — allows the Policy Agent to share the same quantized weight file as the Assistant orchestrator (USE-CASE-004) without doubling RAM cost; only distinct KV-caches are instantiated per agent instance. Total incremental resident cost beyond shared weights: \~350-550MB for the Policy Agent's dedicated KV-cache, deterministic rule engine, CAR validator, and mTLS certificate store.

Validation Addressed: Creates a new validation dimension: VALIDATE_SEMANTIC_GATING — proves that network-layer firewalls are structurally insufficient for AI systems where agents autonomously generate tool calls containing non-deterministic payloads, that a sub-3B probabilistic classifier alone is empirically insufficient against adversarial OOD evasion, and that a hybrid deterministic-probabilistic semantic policy enforcement layer with cryptographic non-repudiation is a measured architectural requirement. **SGX TRUST-ROOT AMENDMENT (ADR-018, 2026-06-03):** Intel SGX is unavailable on the target Lunar Lake hardware (Intel removed SGX from client CPUs as of Alder Lake, 2021). All references in this document to SGX enclaves or SGX local attestation as the Policy Agent trust root, and to SGX/TDX *key sealing*, are **superseded by the TPM 2.0 trust root per ADR-018.** Concretely: Measured Boot Sequence Phase 1 (ISSUE-001) uses **TPM 2.0 local attestation** in place of "SGX local attestation"; CA-key and Cleaner-key sealing use **non-exportable TPM 2.0 keys** via the Windows CNG Platform Crypto Provider in place of SGX/TDX sealing. Hardware reality (ISS-4, `docs/TPM_CAPABILITY_FINDINGS.md`): the active TPM 2.0 is an STMicroelectronics TPM; **Microsoft Pluton is present but is not the active TPM**, so "Pluton Phase 0 / Pluton-sealed" references denote the platform TPM 2.0 / measured-boot root generically (targeted via the standard CNG surface). Formal PCR-quote attestation is staged; the non-exportable signing primitive is implemented and hardware-verified (`shared/security/tpm_signer.py`). TDX / TDX-Connect VM-isolation references are NOT affected by this amendment (separately reclassified as a conditional enhancement — see ISSUE-008).

**MODEL UPGRADE AMENDMENT (ADR-011 + ADR-012, 2026-02-28):**

The Policy Agent's probabilistic classification layer is migrating from a 1.7B-class INT4 model on NPU to **Qwen3-14B INT4 on GPU (Arc 140V)** with mandatory speculative decoding (draft model: Qwen3-0.6B INT4, \~367 MB). Key changes from the baseline architecture above:

- **Device:** NPU → GPU (Arc 140V). NPU retired from P1 Core Loop per ADR-011 (2026-02-27). NPU Priority 0 scheduling designation is superseded by GPU-resident inference.
- **Model:** 1.7B-class → Qwen3-14B (40 layers, 5120 hidden, 8 KV heads GQA, \~9.1 GB INT4 symmetric). Model path: `models/qwen3-14b/openvino-int4-gpu/`.
- **Shared weights:** PA and AO share the same Qwen3-14B model on GPU (single load, shared KV-cache pool). The mmap shared-weight architecture is preserved — both agents reference the same physical model artifact.
- **Thinking mode (ADR-012 §2.4 LOCKED):** `/no_think` system prompt + `stop_token_ids=[151645, 151668]` — defense-in-depth blocks `<|think|>` token at decode level, ensuring zero reasoning-trace leakage in classification output.
- **KV-cache:** Estimates will increase from 350–550 MB to levels commensurate with the 14B architecture. KV_CACHE_PRECISION locked at FP16 (ADR-012 §2.2).
- **Memory budget:** \~9.1 GB shared model + draft model (\~367 MB) + PA-specific KV-cache. The 31.323 GB effective ceiling (ADR-005) and static memory reservation guarantees remain in effect.
- **Latency:** Speculative decoding with `num_assistant_tokens=3` (PROVISIONAL) targets comparable per-adjudication latency despite the larger model.
- **Legacy model:** Qwen2.5-1.5B-Instruct demoted to legacy reference, retained on disk for rollback (`models/qwen2.5-1.5b-instruct/openvino-int4-npu/`). See ADR-012 §5 for rollback plan.
- **NPU references above** (NPU-driven, NPU Priority 0, NPU-resident, Phase 2 Day-1 NPU empirical gates, KV-cache residency in NPU-accessible memory) reflect the original operational baseline and are architecturally superseded by the GPU-based posture. The empirical NPU characterization data (ADR-008) remains valid as historical evidence.

**ACTION-AUTHORIZATION-BOUNDARY LIVE-POSTURE AMENDMENT (ADR-041, 2026-07-15):** Recorded at the Lead Architect's direction following the 2026-07 whole-system design review (`docs/design/july.15.2026/`, finding GP-1). This amendment corrects a divergence between the vision above and the shipped system. **The Agentic-JWT / dual-gate "Action Authorization Boundary" described throughout USE-CASE-001** — the "Destination microservices are hard-coded to reject any execution request lacking this unforgeable receipt, guaranteeing a non-bypassable Action Authorization Boundary" claim, the entire *Agentic JWT Lifecycle Specification*, and the *Destination Enforcement (Hard Constraint)* language — **is NOT the live action-authorization mechanism in the shipped default topology (host-mode).** It is built, tested, and instantiated as a Policy-Agent service, but in host-mode no live code sends a Canonical Action Representation to the Policy Agent over vsock, no Agentic JWT is minted for a real request, no destination validates a JWT receipt, and the probabilistic GPU classifier is not invoked for a live action decision. Presenting it as the *live* root of trust is superseded by this amendment.

**What actually enforces the action-authorization boundary today (host-mode, the shipped default):**
- **Per-boot ephemeral mTLS** on the loopback vsock transport (gateway ↔ Assistant Orchestrator ↔ Policy Agent; ADR-026) — real TLS, real handshake.
- **A fixed tool allowlist** — the Assistant Orchestrator is not a general remote-execution surface; the action set is a small, fixed list, which is what makes an in-process deterministic gate *proportionate*.
- **In-process deterministic adjudication** — every tool dispatch and the gateway egress door invoke the Policy Agent's own `DeterministicPolicyChecker` rule class **in-process** (single source of truth, no GPU, no vsock), **fail-closed** on any error (deny). The rule-layer verdict is authoritative and non-appealable.
- **Prompt/Output Governance (PGOV)** in the Assistant Orchestrator — leakage detection, delimiter/datamark handling, tool allowlist, and the untrusted-content action-lock (ADR-023).
- **The egress spine** — the raw-socket `egress_guard` (deny-by-default allowlist + latched kill-switch), the single `guarded_fetch` door (SSRF guard + exfil screen + injection scan), and the Windows-Hello turn envelope. This is a *structural* chokepoint, not a discipline-based one.
- **At-rest encryption** (DEK envelope, ADR-025) and **weight-integrity verification at model load** (ADR-018 posture).

**The Agentic-JWT / dual-gate AAB is VESTIGIAL.** It was designed as the cross-VM authorization boundary for the *original per-agent-VM topology* described throughout USE-CASE-001 — every LLM agent (Policy Agent, Assistant Orchestrator, etc.) in its own Hyper-V VM, with mTLS and JWT receipts between them. **That topology is foreclosed on this hardware:** LLM inference runs only on the host-resident Arc 140V iGPU (ADR-011), and a Hyper-V guest VM cannot reach that GPU, so the LLM agents can never run in separate VMs — meaning the cross-agent-VM receipt boundary can never become live enforcement. The Agentic-JWT / dual-gate machinery is therefore **slated for removal as dead code** (LA decision 2026-07-15; tracked #910), not reserved. (This is unrelated to the guest↔host boundary for CPU-only untrusted work — the parser/oracle — which uses vsock+mTLS directly, not the JWT AAB; #615 is that separate, legitimate handshake and is unaffected.) In the single-process host-mode default the deterministic plane plus the structural egress guard, sized to the fixed tool surface, are the live enforcement.

**Design principle this records (design-review synthesis):** the language model is treated as an *untrusted* component; the action-authorization trust lives in the deterministic, non-model plane (the rule checker + the structural egress guard), never in a probabilistic classifier or an un-driven cryptographic receipt. Where a probabilistic layer exists it may only *tighten* a deterministic ALLOW (to ESCALATE/DENY), never loosen a DENY.

**Scope of this amendment:** documentation and decision-record only — **no runtime behavior changes** (the controls above are exactly as shipped). The full decision, the trade-off, and the rejected alternative (wiring the receipt into host-mode now) are in **ADR-041**; the companion code-docstring correction (the code also over-describes the receipt as live) is a separate cleanup, tracked apart from this vision amendment.

[USE-CASE-002]: Personal Knowledge Substrate (The Substrate)

Description: A locally hosted, semantically indexed personal knowledge base that ingests, chunks, embeds, and retrieves the user's private data corpus — including journal entries, notes, documents, bookmarks, and structured records — using a hybrid retrieval pipeline (bi-encoder semantic search via HNSW + BM25 lexical fallback) running entirely on-device. The Substrate is the single authoritative data layer for all downstream agents: the Assistant Orchestrator [004], Health Analytics [007], and any future consumer. All Substrate operations are mediated by the Policy Agent [001] — every retrieval query, ingestion request, and index mutation requires a valid Agentic JWT (instance-scoped, nonce-bound, 5-second TTL, epoch-validated) before execution. The Substrate exposes no direct API to any agent; all access is brokered through mTLS-authenticated vsock IPC channels with Policy Agent adjudication as the mandatory first gate.

Retrieval Pipeline — Isochronous Timing Enforcement (ISSUE-007 Resolution — Option A):

The Substrate's HNSW vector index exhibits query-dependent traversal times structurally correlated with index region density and graph topology. To eliminate the timing side-channel through which a compromised agent could infer metadata about the user's private knowledge base by observing IPC round-trip latency, the Substrate enforces isochronous retrieval:

1. Fixed-Latency Deadline.
	- Every retrieval operation (HNSW traversal + optional BM25 fallback + result ranking) is governed by a fixed worst-case execution deadline. The deadline is set to 2× the empirically measured P99 HNSW traversal latency, calibrated during Phase 2 benchmarking against the target index scale. The 2× multiplier provides headroom against index growth, ensuring the deadline remains valid as the HNSW graph scales without immediate recalibration.

2. Constant-Time Release.
	- If the actual retrieval completes before the deadline, the Substrate holds the response in an internal buffer and releases it to the requesting agent only when the deadline elapses. The release mechanism is a deterministic `sleep_until(deadline)` barrier — no early return path exists. From the perspective of any external observer (including a compromised agent measuring vsock IPC round-trip time), all retrieval operations exhibit identical latency regardless of query complexity, index region density, or cache state.

3. Deadline Calibration.
	- Initial calibration: Performed during Phase 2 benchmarking using a representative index (target: 50K–100K entries) with adversarial query distribution (dense-region queries, sparse-region queries, boundary queries, out-of-distribution queries). The P99 latency is recorded; the deadline is set to 2× P99.
	- Runtime monitoring: The Substrate logs actual HNSW traversal time for every retrieval operation (internal metric, never exposed over IPC). If any traversal exceeds 80% of the current deadline, the Substrate emits a recalibration flag to the system's local maintenance log. Recalibration is a manual Phase 2+ maintenance task — the deadline is not auto-adjusted at runtime to prevent an attacker from inferring index growth rate from deadline shifts.
	- Growth thresholds: Recalibration is recommended when the index size crosses 100K-entry increments (100K, 200K, 300K, etc.) or when the recalibration flag has been emitted more than 3 times within a 24-hour window.

4. Latency Budget Impact.
	- Estimated isochronous deadline: 60–160ms (2× P99 of 30–80ms, scale-dependent). For a conversational turn requiring 3 retrieval operations, the isochronous padding adds up to \~180–480ms of total retrieval latency. This is charged against the [004] Assistant Orchestrator's 1-second first-token latency target. Combined with Semantic Router classification (\~50ms), Policy Agent adjudication + JWT minting (\~70–230ms per retrieval), and LLM first-token generation, the total pipeline latency is tight but within budget for single-retrieval turns. Multi-retrieval turns (3+) may approach or exceed the 1-second target under worst-case conditions; this is accepted as a documented latency tradeoff in exchange for complete timing side-channel elimination.

5. Threat Model Scope.
	- The isochronous enforcement is designed against the defined threat model: a compromised agent observing IPC round-trip times from within the Hyper-V VM isolation boundary. It does not defend against hypervisor-level memory access pattern observation. If the threat model is extended to include hypervisor-level adversaries in a future phase (e.g., post-TDX hardening), ORAM-backed retrieval can be evaluated as a supplementary layer at that time. This is a documented deferred decision, not an unaddressed gap.

Embedding Pipeline:

The Substrate's bi-encoder stage runs a 384-dimensional sentence transformer (INT8 quantized) on the Intel NPU at Priority 2 scheduling designation. The embedding model's mmap weight file is shared read-only with no copy-on-write semantics, verified at boot via SHA-256 against the Pluton-sealed Known-Good Manifest, and re-verified via event-triggered integrity checking before any embedding batch operation. The embedding model's NPU scheduling is preemptible by the Policy Agent (Priority 0) and the Semantic Router (Priority 1) — embedding operations are enqueued but not guaranteed real-time execution. Embedding latency variance caused by NPU preemption is absorbed by the isochronous retrieval deadline (the deadline governs the full retrieval operation including any NPU-dependent embedding of the query itself).

Memory Reservation:

The Substrate VM holds a memory reservation defined by the system's active execution tier. Under Tier 1 (Conversational), the Substrate holds its full allocation for HNSW index residency, BM25 index, embedding model weights, and operational buffers. Under Tier 2 (Code Agent [005] activation), the Substrate's HNSW index may be partially evicted to disk-backed mmap with accepted retrieval latency degradation — the isochronous deadline is not reduced during Tier 2 (the deadline remains at the Tier 1 calibrated value, meaning Tier 2 retrievals that exceed the deadline due to disk-backed access will trigger the recalibration flag rather than violating isochronous timing). The Substrate's memory reservation is never reduced below the minimum required to maintain the BM25 index, embedding model, and isochronous timing buffer in RAM. The Policy Agent's [001] memory reservation is never affected by Substrate memory pressure — the Policy Agent holds the highest-priority, non-degradable reservation in the system.

Cleaner Signature Validation Gate (ISSUE-009 Resolution — Option B):

The Substrate's ingestion endpoint enforces a mandatory three-field signature validation gate before accepting any document from the Cleaner [003]. This gate ensures that the Substrate's ingestion boundary attests not only source identity but processing integrity — that the sanitization pipeline version that executed against the document is cryptographically bound to the signature, preventing a compromised Cleaner from skipping sanitization and presenting a valid signature against arbitrary content.

1. Three-Field Signature Tuple.
	- Every document delivered by the Cleaner is accompanied by a signature covering the tuple: (document_content_hash, sanitization_pipeline_code_hash, timestamp). The document_content_hash is the SHA-256 of the cleaned document payload. The sanitization_pipeline_code_hash is computed once at Cleaner VM boot (during Phase 2 of the Measured Boot Sequence) from the Cleaner's sanitization pipeline binary/modules, cached for the session, and carried in every signature minted during that boot cycle. The timestamp is the Cleaner's monotonic clock at signing time.

2. Substrate Validation Sequence.
	- The Substrate's ingestion handler performs the following checks in strict order as the absolute first operation before any document enters the chunking or embedding pipeline: (a) signature verification against the Cleaner's signing public key, extracted from the Cleaner's Policy-Agent-issued mTLS certificate; (b) document_content_hash match against the received payload (SHA-256 recomputed on receipt); (c) sanitization_pipeline_code_hash match against the known-good pipeline hash propagated from the Pluton-sealed Known-Good Manifest to the Substrate during Phase 2 credential issuance; (d) timestamp recency check — signatures older than 60 seconds are rejected to prevent stale-signature replay. If any check fails, the document is rejected and the failure is logged to the Policy Agent's audit stream. There is no fallback ingestion path and no override. A document lacking a valid three-field signature, a mismatched payload hash, or a pipeline code hash that does not match the KGM entry cannot enter the Substrate under any operational condition (Fail-Closed).

3. Key Sealing and Tamper Resistance.
	- The Cleaner's signing key is derived from the Policy Agent's CA chain and sealed to the Cleaner VM's measured image hash via the SGX/TDX sealing mechanism instantiated during Phase 2 of the Measured Boot Sequence. If the Cleaner VM binary is modified — whether by an attacker attempting to bypass the sanitization pipeline or by any unauthorized software change — the measured image hash changes, rendering the sealed signing key cryptographically inaccessible. A tampered Cleaner cannot unseal its signing key and therefore cannot produce any signature the Substrate will accept. This tamper-resistance property is guaranteed by the hardware sealing mechanism and does not rely on any software-layer enforcement.

4. Residual Risk (Documented).
	- A sophisticated in-memory code injection attack that modifies the Cleaner's execution flow after boot attestation — without altering the on-disk binary and therefore without invalidating the sealed key or the cached pipeline code hash — could cause the sanitization pipeline to be skipped while the Cleaner retains its valid signing capability. This attack requires breaching SGX/TDX memory isolation within the Cleaner VM, an attack of equivalent difficulty to compromising any other TEE-attested agent in the system. This residual risk is accepted as consistent with the system-wide TEE trust boundary assumption documented and accepted across [001] through [008].

5. Offline Resilience.
	- If the Policy Agent is temporarily unavailable (e.g., during a scheduled restart or transient IPC failure), the Cleaner continues to process and sign documents using its session-cached pipeline code hash and its sealed signing key — neither requires live Policy Agent connectivity after the Phase 2 boot sequence completes. Signed documents are queued in the Cleaner's local staging area. The Substrate continues to accept signed documents with valid three-field signatures during the Policy Agent's temporary absence, because it holds the Cleaner's public key from the Phase 2 mTLS certificate exchange and the known-good pipeline code hash from the KGM propagated at boot. Staged documents are flushed from the Cleaner's queue to the Substrate upon Policy Agent recovery and re-attestation. This prevents ingestion pipeline stalls under transient Policy Agent unavailability while preserving all signature integrity guarantees.

[USE-CASE-003]: Local High-Fidelity Data Normalization with Mobile LAN Ingress (The Cleaner)
Description: A modular, UI-driven ingestion utility that accepts raw inputs (web articles, emails, PDF documents, exported message archives, health data exports) from both local sources and a Mobile LAN Ingress Proxy (receiving content from the user's Google Pixel 8 Pro via encrypted local-network transfer with strict certificate pinning and mutual TLS, completely bypassing cloud sync layers). The Cleaner performs two mandatory processing stages: (1) Noise Stripping — removes all non-semantic content (ads, CSS, HTML markup, navigation chrome, tracking scripts, comment sections, boilerplate headers/footers); and (2) Adversarial Sanitization — detects and neutralizes embedded prompt injection payloads (OWASP LLM01: Indirect Prompt Injection) hidden within ingested web or email text that could execute as stored instructions when later retrieved by the Assistant or other agents via RAG. Documents that fail either stage with low confidence are quarantined under a Fail-Closed policy for manual user review rather than silently ingested. The Cleaner operates as the mandatory preprocessing gate for the Substrate (USE-CASE-002): no raw data enters the vector index without passing through both normalization and adversarial sanitization. The Cleaner is explicitly recognized as Layer 1 of a mandatory three-layer defense-in-depth strategy against Indirect Prompt Injection; Layer 2 is Context Spotlighting at the Substrate retrieval boundary (USE-CASE-002); Layer 3 is context isolation enforcement in the consuming agent (USE-CASE-004). The architecture assumes that novel, sufficiently obfuscated injections will eventually evade Layer 1 (Phase 3, Gap 4), and designs Layers 2 and 3 accordingly.
Core Problem Statement: The system suffers from two compounding threats and one endpoint-level residual risk. First, Inference Pollution: feeding raw, noise-laden inputs into LLMs and embedding models wastes scarce 31.5GB effective RAM capacity (32GB physical minus 512MB DVMT pre-allocation) by forcing models to process irrelevant tokens (ad copy, tracking pixels, HTML attributes) and critically degrades RAG retrieval quality by injecting non-semantic vectors into the HNSW index. Over a multi-decade accumulation span, even a 1% noise ingestion rate compounds into thousands of polluted embeddings that degrade the Substrate's precision below operational thresholds. Second, Indirect Prompt Injection (OWASP LLM01 / Phase 3, Gap 4): adversarial instructions embedded in web articles, emails, or documents can persist in the Substrate as latent payloads; when a downstream agent retrieves this content via RAG, the injected instructions execute within the agent's context window, potentially overriding system prompts, exfiltrating data via tool calls, or manipulating agent behavior. Pre-processing sanitization is an inherently probabilistic defense against an evolving semantic threat — because the system fundamentally treats all retrieved content as meaningful within the context window, a sufficiently novel obfuscated injection hidden in a downloaded PDF may inevitably evade the sanitization layer. External research confirms that indirect prompt injection exploits the semantic processing capabilities of the LLM itself, making perfect pre-processing defense mathematically unachievable against a motivated adversary. The Cleaner must therefore operate as a high-confidence first line of defense while explicitly communicating to the architecture that residual risk persists and must be mitigated downstream via Context Spotlighting (USE-CASE-002, Layer 2) and consuming-agent context isolation (USE-CASE-004, Layer 3). Third, Mobile Endpoint Compromise (Phase 3, Gap 6): the user's primary content discovery and health telemetry surface is a mobile device (Pixel 8 Pro), but every commercial mechanism for transferring content from phone to PC routes through cloud infrastructure. Even with encrypted local-network transfer, the mobile endpoint itself is vulnerable: Android devices are susceptible to privilege escalation and malware, and if a malicious application on the mobile device compromises the operating system or intercepts the proxy's transmission process, the encrypted tunnel merely transports already-compromised data. Certificate pinning and mTLS guarantee the identity of the receiving server but cannot guarantee the integrity of the sending client's operating system. The architecture must acknowledge this residual mobile-side risk and enforce server-side validation of all ingested payloads regardless of transport-layer authentication. The core problem: how does one automate high-fidelity noise stripping and adversarial sanitization at lifelong scale, securely ingest content from a mobile device with cryptographic assurance of the receiver's identity while acknowledging residual sender-side risk, and ensure only semantically clean, injection-resistant content enters the Substrate?
Hardware & Security Feasibility Check: HTML-to-clean-text extraction (candidate algorithm class: Readability/Trafilatura-equivalent) is a lightweight CPU task consuming less than 50MB RAM per document. PDF text extraction (candidate library class: PyMuPDF/pdfplumber-equivalent) runs CPU-side at less than 100MB. Code-Hash-Bound Signing Mechanism (ISSUE-009 Resolution — Option B):

The Cleaner implements a code-hash-bound signing mechanism that attests processing integrity, not merely source identity, for every document it delivers to the Substrate. This closes the single-point-of-trust vulnerability identified in [ISSUE-009]: a compromised Cleaner that skips sanitization cannot produce a valid signature because the signature's pipeline code hash would mismatch the Known-Good Manifest entry held by the Substrate.

Boot-Time Pipeline Hash Computation: During Phase 2 of the Measured Boot Sequence — after the Policy Agent has validated the Cleaner VM's measured image hash against the Known-Good Manifest and issued its mTLS certificate — the Cleaner computes the SHA-256 hash of its sanitization pipeline binary (the compiled modules and rule sets constituting the noise-stripping and adversarial sanitization stages). This pipeline code hash is computed once per boot, cached in the Cleaner's process memory for the duration of the session, and never recomputed at runtime. The expected pipeline code hash is simultaneously propagated by the Policy Agent from the Pluton-sealed Known-Good Manifest to the Substrate's validation table during Phase 2 credential issuance.

Per-Document Signing: After both mandatory processing stages complete successfully for a document (Noise Stripping + Adversarial Sanitization), the Cleaner signs the three-field tuple (document_content_hash, sanitization_pipeline_code_hash, timestamp) using its sealed signing key. Signing latency is estimated at 5–25ms per document (SHA-256 of content + cached pipeline hash lookup + ECDSA signing operation), which is negligible against both the sanitization pipeline's own CPU execution time and the 100-documents-per-hour throughput target.

Signing Key Derivation and Sealing: The Cleaner's signing key is derived from the Policy Agent's CA chain (established during Phase 2) and sealed to the Cleaner VM's measured image hash via SGX/TDX sealing. The key material is accessible only within the attested Cleaner VM. If the Cleaner's on-disk binary is modified between boots, its measured image hash changes, the sealed key becomes inaccessible at the next boot, Phase 2 credential issuance fails, and the Cleaner receives no mTLS certificate and no signing capability — the ingestion pipeline halts cleanly under Fail-Closed rather than silently operating with a tampered Cleaner.

[USE-CASE-004]: Context-Aware Private Assistant with Modular Skills Framework
ISSUE-005 Integration Addendum (Deterministic Post-Generation Enforcement): The Orchestrator's system prompt Context Spotlighting directive is reclassified as Layer 2.5 alignment bias (probabilistic soft defense, not a security boundary). Layer 3 is implemented as a deterministic Post-Generation Output Validator (PGOV) that intercepts every generated response after token production and before dispatch/delivery. PGOV checks: (1) tool-call destination allowlist validation against Semantic Router expected skill set, (2) retrieved-content leakage detection using cosine similarity against Substrate-provided chunk embeddings (default threshold 0.85), and (3) delimiter echo detection using Substrate-provided session delimiter strings. Text tokens may stream; tool-call blocks are buffered until PGOV clearance. A failed check blocks execution Fail-Closed and logs reason codes.

Description: A locally hosted, conversational AI assistant composed of two distinct microservices: (1) a Semantic Router (fast-path intent classifier running on CPU) and (2) an Assistant Orchestrator (conversational response generator, tool-call dispatcher, and context manager running on NPU). The Assistant exposes a modular "Skills" architecture that dynamically discovers, loads, and executes task-specific capabilities (calendar analysis, document summarization, home automation commands, health queries, code requests via USE-CASE-005). It serves as the primary human-machine interface to the entire multi-agent system for decades. All user interaction flows through this surface; all inter-agent dispatch is mediated by the Policy Agent (USE-CASE-001) via Canonical Action Representation (CAR) and validated by Agentic JWT; all context retrieval draws from the Substrate (USE-CASE-002) with retrieved content structurally demarcated via Context Spotlighting delimiters. The Orchestrator enforces a four-layer defense-in-depth against Indirect Prompt Injection: Layer 1 (Cleaner probabilistic sanitization, USE-CASE-003), Layer 2 (Substrate cryptographic Context Spotlighting, USE-CASE-002), Layer 2.5 (the Orchestrator's system prompt alignment bias — a probabilistic directive treating all spotlighted content as inert data, reclassified from Layer 3 per Red Team ISSUE-005 as LLM compliance is not independently enforceable), and Layer 3 (a deterministic Post-Generation Output Validator (PGOV) that intercepts every generated response after token production and before dispatch/delivery, enforcing tool-call destination allowlisting, retrieved-content leakage detection at 0.85 cosine similarity threshold, and delimiter echo detection as Fail-Closed checks — see ISSUE-005 Integration Addendum). The Assistant also functions as the synthetic load generator for testing the Policy Agent's Fail-Closed interception logic during concurrent development. Both the Semantic Router and the Assistant Orchestrator enforce strict circuit breakers (OWASP LLM04: Unbounded Consumption) — hard-capped maximum token output limits and recursion depth limits per request — to prevent infinite loops or runaway generation from exhausting the 25.6GB operational RAM ceiling and causing host OS instability. All vsock IPC connections are mTLS-authenticated with Policy Agent Agentic JWT validation. The Orchestrator holds NPU Priority 1 scheduling designation — it yields to the Policy Agent's Priority 0 preemption but takes precedence over any future NPU consumers. The Orchestrator shares the mmap'd weight file with the Policy Agent under the same three-layer integrity mitigation (boot-time verification + event-triggered runtime re-verification by the Policy Agent + hypervisor-enforced CoW prevention); a write fault against the shared mmap region by the Orchestrator's inference runtime results in immediate Orchestrator process termination, mTLS certificate revocation, and user escalation. The Orchestrator VM operates under a static Hyper-V memory limit and supports a defined degradation posture: when the Code Agent [005] is activated, the Orchestrator's KV-cache is flushed, and subsequent user queries incur cold-start latency until the KV-cache is rebuilt. When the Code Agent is deactivated, the Orchestrator returns to warm-state operation.

Core Problem Statement: General-purpose cloud assistants (Alexa, Siri, Google Assistant) require continuous cloud connectivity and transmit all user queries — including intimate, context-rich conversational data — to corporate servers, fundamentally violating absolute privacy. No existing local assistant framework provides both (a) natural-language conversational quality sufficient for daily use and (b) a modular, extensible capability system where new skills can be added without retraining the base model. Additionally, on a memory-constrained 31.5GB effective device (32GB physical) running multiple concurrent agents, an unbounded generation loop or recursive tool-call chain (OWASP LLM04) in the assistant could exhaust system RAM and crash the host OS, taking down all other agents including the Policy Agent — a catastrophic single-point-of-failure. Furthermore, the Assistant is the primary consumer of RAG-retrieved content and therefore the primary target for Indirect Prompt Injection (OWASP LLM01 / Phase 3, Gap 4) — external research confirms that pre-processing sanitization is inherently probabilistic and that sufficiently novel obfuscated injections will eventually penetrate the Substrate. The Assistant must therefore implement deterministic post-generation enforcement as the final security layer of the defense-in-depth strategy: a Post-Generation Output Validator (PGOV) that deterministically verifies every generated response before dispatch, ensuring that all Substrate-retrieved content remains inert and is not leaked into tool-call payloads, that tool-call destinations match expected skill sets, and that spotlighting delimiters are not echoed into output — shifting Layer 3 from probabilistic LLM compliance to deterministic verification (Red Team ISSUE-005). Additionally, the Orchestrator's tool-call dispatch surface is the primary target for the OOD evasion vulnerability (Phase 3, Gap 1) — a compromised or hallucinating Orchestrator could generate tool calls that superficially resemble benign requests but embed sensitive data in payloads. All Orchestrator-emitted tool calls must be reduced to CAR format and adjudicated by the hybrid deterministic-probabilistic Policy Agent (USE-CASE-001) before reaching any destination, ensuring that the Orchestrator's own probabilistic reasoning cannot bypass the AAB. The user needs a unified interaction surface that routes intent to specialized agents while the Policy Agent enforces skill-level permission boundaries via CAR and Agentic JWT, the assistant enforces internal resource limits via circuit breakers, and the Orchestrator enforces strict context isolation of all Substrate-retrieved content — solving the problem of secure, resource-bounded, injection-resistant, modular capability discovery and execution on constrained hardware over a multi-decade timescale. The Intel NPU's undocumented concurrent scheduling behavior (Red Team ISSUE-002) introduces an additional constraint: the Semantic Router and Orchestrator cannot both assume guaranteed NPU access, and the architecture must define explicit compute allocation per microservice with empirically validated latency budgets. The shared mmap weight file (Red Team ISSUE-003) creates an integrity coupling between the Orchestrator and the Policy Agent — if the Orchestrator's inference runtime triggers a CoW write fault (e.g., via an exploit), the hypervisor must terminate the Orchestrator immediately to prevent silent weight divergence from the Policy Agent's view, and the Policy Agent must revoke the Orchestrator's mTLS certificate and escalate to the user. Additionally, the 31.5GB effective ceiling (32GB physical minus 512MB DVMT pre-allocation) is violated under concurrent peak load when the Code Agent [005] is active (Red Team ISSUE-004). The Orchestrator must accept KV-cache flush as a degradation posture during Code Agent operation, trading warm-state latency for RAM headroom.

Streaming Behavior: Text tokens stream in real-time from the Orchestrator via vsock to the Transport Gateway to the TUI Shell, with ≤5ms terminal write latency per token. Tool-call blocks are buffered in the Transport Gateway until PGOV clearance — the user never sees an unvalidated tool-call response. A 4,096 token hard cap per response (from `shared/constants.py`) acts as the circuit breaker. PGOV denials replace the streamed response with a styled panel displaying reason code labels (PII_DETECTED, DELIMITER_ECHO, TOOL_CALL_VIOLATION, LEAKAGE_DETECTED, TOKEN_BUDGET_EXCEEDED) and a user-actionable message.

Boot-Phase-3 Gating: The TUI Shell implements Fail-Closed startup — it displays a loading indicator and blocks all user input until the Transport Gateway confirms a successful vsock + mTLS handshake with the Policy Agent. If the handshake fails after 3 retries with exponential backoff (1s, 2s, 4s), the TUI displays "Policy Agent unavailable. System in Fail-Closed state." and input remains disabled. No prompt is accepted and no request is dispatched to the Orchestrator until the PA handshake succeeds.

Migration Path (Phase 4+): Per Lead Architect directive, the documented migration target is Native Desktop Shell (PyQt6 or Tkinter), not Web UI. The Transport Gateway's interface-agnostic API, shared SQLite session layer, and serializable PGOVResult dataclass ensure that only the UI Shell component is replaced during migration. A new ADR is required for Phase 4+ evaluating GPL license implications (PyQt6), Qt/asyncio bridging, high-DPI rendering, and memory delta (\~100–200MB).

Description: An agent that autonomously studies trusted external sources (package repositories, security advisories, model benchmark leaderboards, framework changelogs) and proposes system upgrades, dependency updates, and architectural refactors to the user for approval. Unlike the Interactive Local Software Engineer (USE-CASE-005), which executes only user-initiated requests with zero network access, the Autonomous Maintainer requires controlled, read-only outbound network access to monitored sources and system-level write-access to a staging area for proposed changes. Core Problem Statement: The system faces Architectural Entropy and Supply-Chain Risk — over a multi-decade timescale, dependencies rot, models become obsolete, security vulnerabilities accumulate, and the system's own codebase drifts from best practices. (b) Autonomous Write Escalation (OWASP LLM06) — a hallucinating or compromised agent with write-access could modify security-critical configurations (e.g., Policy Agent rules, TDX boundaries, firewall policies), creating a self-inflicted architectural compromise. Human-in-the-Loop Fatigue (Phase 3, Gap 7): The reliance on a "staging-only" area protected by explicit user approval is critically susceptible to approval fatigue. When the Maintainer autonomously fetches a poisoned dependency from a trusted source and presents a massive, complex codebase diff, the human operator lacks the capacity to manually verify every line and will eventually approve without sufficient review — effectively allowing the agent to bypass its own security constraints. External research confirms this is a known failure mode where the human becomes the weakest link in the authorization chain. The architecture must therefore supplement human review with deterministic, automated cryptographic supply-chain verification that filters proposals before the user is prompted, reducing the human's burden to reviewing only cryptographically verified, structurally valid proposals. All proposed changes are written to a staging area (candidate: a git branch or diff queue in Unified Diff format) requiring explicit user review and approval before merging. Circuit breakers (OWASP LLM04): same token output and recursion depth limits as USE-CASE-005 apply. Validation Addressed: Introduces a new validation: VALIDATE_AUTONOMOUS_WRITE — explicitly defines the extreme risk boundary of granting an agent network read + system write capabilities, and establishes that this capability must be the last to be implemented, with the most rigorous Fail-Closed controls, cryptographic supply-chain verification (closing Phase 3, Gap 7), and deterministic pre-filtering in the architecture. Critical Design Review — Red Team Assessment
[ISSUE-001]: Secure Bootstrap Paradox — Policy Agent Cannot Self-Attest at Cold Start
Severity: Critical
Affected Use Case(s): [001], [002], [003], [004], [005], [009]
Root Cause Analysis: The architecture mandates that every inter-agent IPC connection requires a valid mTLS certificate issued by the Policy Agent's internal CA, and every tool call requires an Agentic JWT minted by the Policy Agent after dual-gate (CAR + LLM) adjudication. However, the Policy Agent itself must boot inside a Hyper-V VM, load its NPU model weights via mmap, initialize its deterministic rule engine, and establish its Pluton-backed CA — all before any other VM can authenticate. During this bootstrap window:
The Policy Agent's own SGX attestation requires verifying its code hash and rule-set hash against a known-good measurement. Who verifies the verifier? The attestation root is Pluton/SGX silicon, but the chain from silicon to "Policy Agent VM is running unmodified code" requires a measured launch sequence that is not specified. If the host OS or Hyper-V hypervisor is compromised before the Policy Agent completes attestation, a tampered Policy Agent binary could be loaded, and all downstream JWTs would be valid but issued by a compromised gate.
During the bootstrap window (estimated 5–15 seconds for NPU model load + rule engine init + CA cert generation), no other agent can start because there is no CA to issue mTLS certificates and no JWT minter. This creates a hard serial dependency: the entire multi-agent system is blocked until a single microservice completes initialization. If the Policy Agent crashes during boot, the system is permanently inoperable until manual intervention — there is no fallback or recovery path defined.
The Policy Agent must issue its own mTLS certificate from its own CA. This is a self-signed trust root with no external anchor. If an attacker can substitute the CA key during the bootstrap window (before SGX attestation completes), they control the entire identity infrastructure.
Recommended Mitigation: Define an explicit Measured Boot Sequence as a numbered, deterministic phase-gate:
Phase 0 (Silicon Root): Pluton measures the UEFI → Hyper-V hypervisor → Policy Agent VM image hash chain before any VM launches. The measurement is sealed to Pluton's hardware key. This is the only trust anchor.
Phase 1 (Policy Agent Only): Only the Policy Agent VM is permitted to launch. It completes SGX local attestation, derives its CA key from Pluton-sealed storage (key material never exists in host RAM), and generates ephemeral per-boot CA certificates.
Phase 2 (Controlled Agent Launch): The Policy Agent issues mTLS certificates to each agent VM in a deterministic, sequenced order. Each agent VM presents its own measured image hash; the Policy Agent validates against a known-good manifest before issuing credentials.
Phase 3 (Operational): Only after all Priority 1/2 agents hold valid mTLS certificates does the system accept user input.
Crash Recovery: If the Policy Agent fails during any phase, the system remains in a hard-locked state (no agents operational). This is the correct Fail-Closed behavior — but it must be explicitly documented as an accepted availability tradeoff, and a user-initiated manual recovery procedure (re-trigger measured boot from Pluton root) must be specified.
[ISSUE-002]: NPU Multiplexing — Undocumented Concurrency Ceiling on Lunar Lake NPU
Severity: Critical

Affected Use Case(s): [001], [002], [004]

Root Cause Analysis: The architecture simultaneously assigns the following persistent workloads to the Intel NPU (48 TOPS):

[001] Policy Agent: 1.7B INT4 model — continuous inference on every IPC message.
[002] Substrate: 384-dim sentence transformer INT8 — inference on every retrieval query (bi-encoder stage).
[004] Semantic Router: BERT-mini INT8 — inference on every user query.
[004] Assistant Orchestrator: 1.7B INT4 model — continuous conversational inference.
[007] Health Analytics (Future): Lightweight time-series models INT8 — 24/7 background.
The Intel NPU on Lunar Lake is a single inference accelerator with a unified command queue. Current Intel OpenVINO NPU plugin documentation does not guarantee true concurrent multi-model execution on the NPU — it implements time-sliced scheduling with context-switch overhead. The architecture treats the NPU as if it were a multi-core GPU capable of running 4–6 models simultaneously with independent latency guarantees.

The physics failure: When the user sends a chat query, the following must execute near-simultaneously on the NPU: (1) Semantic Router classifies intent (\~50ms assumed), (2) Policy Agent adjudicates the resulting tool call, (3) Substrate bi-encoder embeds the retrieval query, (4) Assistant Orchestrator generates the response. If these are time-sliced on a single NPU queue, the actual latency is additive, not parallel: 50ms + Policy Agent inference + embedding + generation. The 1-second first-token latency target for [004] becomes physically unachievable under concurrent load. If [007] and [008] are also active (future phase), the NPU becomes the single bottleneck for the entire system.

Additionally, the KV-cache for the Policy Agent (350–550MB) and Assistant Orchestrator (300–500MB) must remain resident in NPU-accessible memory to maintain "warm" state. If the NPU has a limited context-slot count (Intel has not publicly documented this for Lunar Lake), swapping between model contexts could invalidate KV-caches, converting every context switch into a cold-start.

Recommended Mitigation:

Empirical NPU concurrency test must be a Phase 2 Day-1 validation gate: load the Policy Agent model and Assistant Orchestrator model simultaneously on the NPU via OpenVINO, measure actual concurrent throughput, and determine whether the NPU supports parallel inference or only time-sliced scheduling. This is a hardware-dependent unknown that cannot be resolved architecturally.
Define NPU scheduling priority hierarchy: Policy Agent inference is Priority 0 (preemptive — never queued behind other workloads). All other NPU consumers yield to it. This ensures the security gate is never starved.
Fallback compute allocation: If empirical testing proves the NPU cannot sustain concurrent multi-model inference within latency budgets, the architecture must define which workloads migrate to CPU (candidate: Semantic Router and Substrate bi-encoder are small enough for CPU inference with acceptable latency degradation). The 1-second latency target for [004] must be re-evaluated under the measured NPU scheduling model.
KV-cache residency policy: Explicitly define whether the NPU supports persistent KV-cache across context switches. If it does not, the "warm state" assumption collapses, and the architecture must budget for KV-cache reconstruction latency on every model switch.
[ISSUE-003]: Shared mmap Weight File — Single Corruption Point Across Security Boundary
Severity: High

Affected Use Case(s): [001], [004]

Root Cause Analysis: The architecture specifies that the Policy Agent [001] and Assistant Orchestrator [004] share the same 1.7B quantized weight file via zero-copy mmap to conserve RAM. This means both the security enforcement agent and the user-facing conversational agent execute inference against identical physical memory pages.

This creates two failure modes:

Integrity coupling: If the shared weight file is corrupted (bit-flip, disk error, malicious modification before mmap), both the security gate AND the conversational agent fail simultaneously with the same corruption. The Policy Agent cannot detect its own corruption because it has no independent reference — its classification logic is derived from the same corrupted weights. This violates the principle that the security gate must be independently verifiable and must not share failure modes with the agents it monitors.
Copy-on-Write attack surface: While mmap is read-only in the standard case, if any process triggers a copy-on-write fault (e.g., via an exploit in the inference runtime that attempts to write to the mmaped region), the OS creates a private copy for that process. An attacker who can trigger CoW in the Assistant Orchestrator's address space could then modify the weights used by the Orchestrator without affecting the Policy Agent's view — or vice versa. This is subtle because the system assumes both agents always see identical weights, but CoW semantics break this assumption silently.
Recommended Mitigation:

Boot-time integrity verification: The Measured Boot Sequence (ISSUE-001 mitigation) must include a SHA-256 hash verification of the weight file against a Pluton-sealed known-good measurement before mmap. If the hash fails, the system refuses to boot (Fail-Closed).
Runtime integrity monitoring: Periodically (candidate: every 60 seconds) verify the mmap region's hash against the known-good measurement. If divergence is detected, halt all inference and escalate to user (Fail-Closed).
Explicit CoW prevention: The mmap must be established with MAP_SHARED | PROT_READ (no write permission) at the OS level. Any process that triggers a write fault against the region must be immediately terminated by the hypervisor. This should be documented as a hard architectural constraint, not an assumed inference-runtime behavior.
Architectural acknowledgment: The shared-weight design is a measured RAM optimization with a documented residual risk: the Policy Agent and the Orchestrator share a failure mode. If this coupling is unacceptable after further analysis, the fallback is loading two independent copies (\~2GB additional RAM), which fits within the 25.6GB operational ceiling but increases the Priority 1 Core Loop footprint.
[ISSUE-004]: 25.6GB Operational Ceiling Violated Under Concurrent Peak Load
Severity: Critical

Affected Use Case(s): [001], [002], [004], [005]

Root Cause Analysis: The architecture defines a 32GB hard ceiling with \~6.4GB reserved for Windows 11 + Hyper-V hypervisor, yielding 25.6GB operational. The Use Cases enumerate the following concurrent resident costs:

Component	Stated Resident RAM
[001] Policy Agent (incremental)	350–550MB
[002] Substrate (HNSW + embedder + re-ranker + ORAM + spotlighting)	1.9–3.0GB
[004] Semantic Router	66–80MB
[004] Orchestrator (KV-cache, incremental)	1.0–1.2GB
Shared 1.7B mmap weights	\~1.0GB
Priority 1+2 Subtotal	\~4.3–5.8GB
[005] Code Agent during active synthesis	12–14GB
Total with [005] Active	\~16.3–19.8GB
This appears to fit. However, the analysis omits several critical memory consumers:

Hyper-V VM overhead: Each Hyper-V VM has a minimum memory footprint for its guest kernel, virtio drivers, and runtime. A minimal Linux guest consumes \~256–512MB. The architecture implies at minimum 4 concurrently resident VMs during Priority 1+2+3 operation (Policy Agent VM, Substrate VM, Assistant VM, Code Agent VM) plus the Cleaner VM (on-demand). That is 1–2.5GB of guest OS overhead not accounted for.
mTLS state per connection: Each mTLS handshake maintains TLS session state (\~50–100KB per connection). With N active vsock connections (conservatively 6–10 during active operation), this is trivial individually but contributes to unaccounted overhead.
Windows 11 memory pressure: The "6.4GB" Windows reservation is a baseline idle figure. During active operation with Hyper-V managing multiple TDX-enabled VMs, the Hyper-V root partition memory consumption can spike to 8–10GB due to extended page tables (EPT), IOMMU remapping structures, and TDX metadata (TDMR structures consume \~1/256th of assigned memory for integrity metadata).
iGPU Shared Memory Override: The 20GB iGPU allocation for [005] comes from the unified 32GB LPDDR5X pool. When the Code Agent is active with \~9GB model + KV-cache in GPU-accessible memory, this memory is simultaneously subtracted from system RAM. The Use Case states 12–14GB resident for [005], but this may not account for the GPU memory controller's reservation granularity (Intel iGPU shared memory is allocated in large contiguous blocks, potentially reserving more than the model actively uses).
Worst-case concurrent scenario (user is chatting while Code Agent is active):

Windows + Hyper-V root: 8–10GB (realistic under TDX load)
VM overhead (4 VMs): 1–2GB
Priority 1+2 agents: 4.3–5.8GB
[005] Code Agent: 12–14GB
Total: 25.3–31.8GB
The upper bound exceeds the 32GB hard ceiling. The system will trigger Windows memory pressure responses (paging to SSD, process termination by the OOM killer), causing cascading agent failures including potential Policy Agent termination — a catastrophic Fail-Open condition.

Recommended Mitigation:

Construct a precise memory map as a Phase 2 deliverable before any agent implementation begins. Every consumer must be measured empirically, not estimated. The map must include: Windows base, Hyper-V root partition (with TDX metadata overhead), each VM guest OS, each agent's true measured resident set size, iGPU reservation granularity, and KV-cache actual sizes.
Define exclusive execution tiers: The Code Agent [005] and the full Priority 1+2 stack cannot safely co-exist at peak utilization within 32GB. The architecture must enforce a memory-aware scheduler at the Hyper-V level: when [005] is activated, the Substrate's cross-encoder re-ranker and ORAM state machine are paged down (accepting degraded retrieval latency), and the Assistant Orchestrator's KV-cache is flushed (accepting cold-start latency on the next user query). When [005] completes, Priority 1+2 agents are restored. This must be specified as an explicit architectural tradeoff.
Hard memory reservation per VM: Each Hyper-V VM must have a hard maximum memory limit (not dynamic memory) to prevent any single agent from consuming more than its allocated share. The Policy Agent VM gets a guaranteed minimum reservation that Windows cannot reclaim (highest priority).
Lenovo Y700 offload trigger: The Substrate offload strategy (noted in [002]) must be formalized with a specific RAM pressure threshold (e.g., if system committed memory exceeds 28GB, the Substrate automatically migrates to the secondary node). This converts it from a "future scaling path" to a defined degradation mode.
[ISSUE-005]: Context Spotlighting Layer 3 Relies on LLM Compliance — Not Enforceable
Severity: High

Affected Use Case(s): [002], [004], [005]

Root Cause Analysis: The three-layer defense-in-depth against Indirect Prompt Injection is:

Layer 1: Cleaner probabilistic sanitization [003].
Layer 2: Substrate cryptographic spotlighting delimiters [002].
Layer 3: Consuming agent's system prompt "hardcoded directive" that treats spotlighted content as inert [004], [005].
Layer 3 is architecturally unenforceable. The "hardcoded Context Spotlighting enforcement directive" in the Orchestrator's system prompt is a natural-language instruction to a probabilistic model. The model's compliance with this instruction is a function of its training, RLHF alignment, and the specific adversarial payload — it is not a deterministic constraint. Current research demonstrates that system-prompt override attacks are a primary vector for prompt injection, and that no system prompt instruction is guaranteed to be followed by an LLM under adversarial conditions.

The Use Case states: "even if an injection attempts to close or spoof the delimiter structure, the Orchestrator's instruction hierarchy treats any retrieval-sourced content block as inert regardless of its internal structure." This claim is not physically enforceable. The LLM processes all tokens in its context window through the same attention mechanism; it has no hardware-enforced separation between "system prompt tokens" and "retrieved content tokens." A sufficiently sophisticated injection that successfully spoofs or escapes the delimiter structure can influence the model's next-token prediction, including tool-call token generation, regardless of what the system prompt instructs.

The architecture implicitly acknowledges this by requiring three layers, but claims Layer 3 provides independent reduction in attack probability. The actual probability is unknown and unmeasurable — it depends on the specific model, injection, and context window state.

Recommended Mitigation:

Do not claim Layer 3 is independently enforceable. Reclassify it as a "probabilistic alignment bias" rather than a "security control." This is a documentation accuracy issue with design implications.
Add a deterministic Layer 3 alternative: After the Orchestrator generates a response, but before it is executed or returned to the user, a deterministic post-generation validator (running CPU-side, outside the LLM) inspects the output for:
Tool-call tokens that reference destinations not present in the original user query's expected skill set.
Tool-call payloads containing content fragments that match bi-encoder similarity (above a threshold) to the retrieved spotlighted content — indicating the model "leaked" retrieved content into a tool call.
Any output that references or repeats the spotlighting delimiter structure, indicating the model is processing delimiters as content rather than boundaries.
This post-generation validator is architecturally equivalent to the deterministic layer in the Policy Agent's hybrid gate (ISSUE: consistency across the architecture). It transforms Layer 3 from "ask the LLM to behave" to "verify the LLM behaved" — shifting from trust to verification.
[ISSUE-006]: Agentic JWT Replay and Lifetime Management Undefined
Severity: High

Affected Use Case(s): [001], [002], [004], [005], [009]

Root Cause Analysis: The architecture mandates that every approved action is accompanied by an Agentic JWT minted by the Policy Agent via Pluton. Destination microservices reject requests lacking a valid JWT. However, the JWT lifecycle is unspecified:

Token lifetime: If JWTs have no expiration (or a long TTL), a compromised agent that captures a valid JWT can replay it indefinitely against the destination microservice. The destination validates the signature (which is valid) and the CAR schema (which matches the original approved action), and executes it. This permits arbitrary replay of any previously approved action.
Token scope: A JWT minted for "query Substrate for document X" — is it valid for querying document Y? If the JWT authorizes a class of action (e.g., "Substrate read") rather than a specific instance (e.g., "read embedding ID 47291"), then a compromised agent can use a single captured JWT to perform unlimited operations within that class.
Token revocation: If the Policy Agent detects anomalous behavior from an agent mid-session and revokes its mTLS certificate, do previously issued JWTs remain valid at the destination? If destinations only check the JWT signature (using the CA's public key, which hasn't changed), revoked agents can continue operating with cached JWTs.
Recommended Mitigation:

Single-use, nonce-bound JWTs: Each JWT must contain a unique nonce and a cryptographic binding to the specific CAR instance (hash of the full CAR payload). The destination microservice must maintain a nonce-seen set and reject any JWT with a previously observed nonce. This makes replay impossible.
Short TTL: JWTs must expire within a tight window (candidate: 5–30 seconds). The destination rejects expired JWTs. Combined with the nonce, this limits the replay window to seconds even if the nonce-seen set is compromised.
Instance-scoped claims: The JWT claims must encode the specific action instance (e.g., target CID, operation type, payload hash), not just the action class. The destination validates that the JWT claims match the actual request parameters exactly.
Revocation signal propagation: When the Policy Agent revokes an agent's mTLS certificate, it must simultaneously broadcast a revocation event to all destination microservices via a signed revocation message. Destinations must check both JWT validity and certificate revocation status before processing.
[ISSUE-007]: ORAM and Isochronous Execution — Unquantified Performance and Feasibility on Lunar Lake
Severity: High

Affected Use Case(s): [002]

Root Cause Analysis: The architecture prescribes ORAM for the Substrate [002] and isochronous (constant-time) execution for future sensitive temporal databases to close timing side-channel gap (Phase 3, Gap 5). These are cited as "candidate mitigations" without quantifying their actual cost on the target hardware:

ORAM overhead: Oblivious RAM protocols (Path ORAM, Ring ORAM) incur O(log N) bandwidth blowup per access, where N is the number of stored elements. For a Substrate with 5 million embeddings, each retrieval operation requires \~23 additional dummy accesses (log₂ 5M ≈ 22.3). Each dummy access is a full vector read from the HNSW index. The stated 10–15% latency overhead is drastically underestimated — published ORAM benchmarks on conventional hardware show 10–100x latency overhead for database operations, depending on implementation. For a 2-second latency budget, this could push actual retrieval to 20–200 seconds.
Isochronous execution on variable-latency hardware: Constant-time execution requires that every code path takes exactly the same number of cycles regardless of input. This is achievable for simple cryptographic operations (e.g., constant-time comparison) but is architecturally incompatible with (a) HNSW graph traversal (which is inherently data-dependent — the number of hops depends on the query's position in the embedding space), (b) SQLite or equivalent temporal database queries (which have data-dependent I/O patterns), and (c) YOLO inference (which has data-dependent NMS post-processing). The architecture states these operations will be constant-time but does not acknowledge that the underlying algorithms are fundamentally variable-time.
The fallback ("If strict constant-time is not feasible, randomized noise is injected into processing delays") is acknowledged but not quantified. The amount of noise required to achieve statistical indistinguishability (p > 0.05) depends on the actual timing variance of the operations, which is unknown on Lunar Lake hardware.
Recommended Mitigation:

Replace full ORAM with a pragmatic threat-model-scoped alternative. The ORAM threat model assumes a compromised adjacent VM performing timing analysis on the Substrate's memory access patterns. In the architecture, all VMs are launched by the same user on the same machine — if an adjacent VM is compromised, the attacker already has host-level access (since they compromised a Hyper-V guest, likely via an exploit in an agent's inference runtime). At that point, ORAM is insufficient because the attacker has access to the hypervisor's EPT mappings. The actual threat boundary should be scoped: if the threat is a compromised skill container (Hyper-V worker jail, not a TDX VM), then vsock mTLS + JWT + the Policy Agent's AAB already prevent data access. ORAM may be unnecessary given the existing isolation stack. If retained, re-scope it as a defense-in-depth measure with realistic overhead estimates and a degraded-latency fallback, not a primary control.
Replace "isochronous execution" with "bucketed-delay execution": Rather than attempting true constant-time (which is infeasible for the underlying algorithms), pad all operations to the nearest time bucket (e.g., all Substrate queries take exactly 2.0 seconds, all health DB queries take exactly 1.0 seconds, regardless of actual completion time). This is achievable, quantifiable, and provides timing indistinguishability within the bucket granularity. Document the residual risk: an attacker who can measure sub-bucket timing (unlikely over vsock) could still distinguish operations.
Phase 2 empirical gate: Measure actual ORAM overhead and bucketed-delay feasibility on Lunar Lake before committing to either mitigation. If ORAM overhead exceeds 5x latency, formally downgrade it to "accepted residual risk" with documented compensating controls.
[ISSUE-008]: TDX Connect (TDISP) for iGPU — Unverified Hardware Dependency Bearing Critical Security Claims
Severity: High

Affected Use Case(s): [005]

Root Cause Analysis: The architecture designates TDX Connect with TDISP as the mitigation for Phase 3, Gap 3 (iGPU trust-boundary leakage). [005] states: "The architecture mandates Intel TDX Connect (candidate solution) utilizing TDISP to extend the hardware-encrypted trust boundary to the iGPU."

The problem: TDX Connect is a specification, not a product. As of the document's date, TDX Connect availability on the Lunar Lake SoC is listed as "a pending hardware validation item." Intel's published TDX specifications (through 1.5) do not confirm TDISP support for integrated GPUs — the specification targets discrete PCIe devices. The Arc 140V is an integrated GPU sharing the same die and memory controller, which presents a fundamentally different trust boundary topology than a discrete PCIe device with its own memory.

The architecture builds Use Case [005] with critical security claims ("GPU memory cryptographically inaccessible to host OS during inference") on an unverified hardware capability. The fallback posture (if TDX Connect unavailable) relies on "compensating controls" including the Policy Agent's AAB and zero-outbound-network — but these compensating controls protect against exfiltration, not against local host-level memory scraping. If the host OS or hypervisor is compromised, the attacker can read raw iGPU memory without needing any network access or tool call. The AAB is irrelevant in this threat model because the attacker is not using an agent to extract data — they are reading GPU memory directly.

This means the actual fallback security posture for [005] under a compromised-host threat model is: standard TDX CPU isolation only, with GPU memory unprotected. The architecture should not describe this as "accepted residual risk with compensating controls" — it should describe it as "GPU memory is unprotected against a compromised host" with specificity about what that means for each data category.

Recommended Mitigation:

Reclassify TDX Connect as a conditional enhancement, not an architectural mandate. Remove the phrasing "The architecture mandates" and replace with "The architecture requires TDX Connect as a prerequisite for operational approval of [005] under the compromised-host threat model. If TDX Connect is unavailable on Lunar Lake, [005] operates under a reduced threat model that excludes host-level memory scraping, with this exclusion explicitly documented in the security attestation."
Quantify the fallback explicitly for each data category:
[005] Source Code: If host is compromised and GPU memory is readable, source code fragments in the context window are exposed. Compensating control: source code is regenerable and rotatable (lower severity).
Phase 2 hardware validation gate: Before any code is written for [005], empirically verify TDX Connect TDISP support on the physical Lunar Lake unit. This is a binary go/no-go gate.

**Resolution (ISSUE-008 — Option A: Conditional Enhancement with Data-Category Fallbacks + Option A: Fixed DVMT Pre-Allocation):**
- **TDX Connect/TDISP:** Reclassified from architectural mandate to conditional enhancement. Phase 2 binary go/no-go gate determines operational posture:
  - **[005] Source Code (Conditional Enhancement):** If TDX Connect available, [005] operates on iGPU with full trust boundary. If unavailable, [005] operates on iGPU under a reduced threat model excluding host-level memory scraping — source code is regenerable/rotatable (lower severity residual risk). Compensating controls (AAB + zero-outbound-network) remain active.
- **DVMT Pre-Allocation:** Fixed at 512MB BIOS minimum granularity. Effective system memory ceiling adjusted from 32GB to 31.5GB. All memory budget calculations, execution tier validations, and Phase 2 memory map gate outcomes computed against the adjusted 31.5GB ceiling. VALIDATE_DVMT_BUDGET introduced as Phase 2 empirical confirmation gate.

[ISSUE-009]: Cleaner [003] Signs Output, but Signing Key Provenance Is Unspecified
Severity: High

Affected Use Case(s): [002], [003]

Root Cause Analysis: The success metrics for [003] state: "Substrate rejects non-Cleaner-signed inputs." The Substrate's architectural gate enforces that only documents bearing a valid Cleaner signature are ingested. However, the signing mechanism is not specified:

Who issues the Cleaner's signing key? If it is the Policy Agent's CA, then the Cleaner depends on [001] for signing capability. If the Policy Agent is rebooting or unavailable, the Cleaner cannot sign cleaned documents, and the ingestion pipeline halts.
Where is the signing key stored? If it is in the Cleaner's VM memory, a compromised Cleaner can sign arbitrary (unsanitized) documents and the Substrate will accept them — the signature proves provenance from the Cleaner VM but not that the sanitization pipeline actually executed.
This creates a single-point-of-trust problem: the Substrate trusts any document signed by the Cleaner key, but the signature only attests to source identity, not to processing integrity. A compromised Cleaner could skip sanitization entirely and still produce valid signatures.
Recommended Mitigation:

Processing attestation, not just source attestation: The Cleaner's signature must attest not only that the document originated from the Cleaner VM, but that the specific sanitization pipeline version (code hash) executed against the document. Candidate: the signature covers a tuple of (document hash, sanitization pipeline code hash, timestamp, Cleaner VM attestation report). The Substrate validates all four fields.
Key derivation from Pluton via Policy Agent CA: The Cleaner's signing key is derived from the Policy Agent's CA chain (bound to the Measured Boot Sequence from ISSUE-001). The key is sealed to the Cleaner VM's measured image hash via SGX/TDX sealing — if the Cleaner VM's code is modified, the sealed key becomes inaccessible, and the compromised Cleaner cannot produce valid signatures.
Offline ingestion resilience: If the Policy Agent is temporarily unavailable, the Cleaner queues cleaned documents in a local staging area (signed with its sealed key). The Substrate accepts staged documents only after the Policy Agent validates the Cleaner's attestation report on recovery. This prevents the ingestion pipeline from permanently halting on Policy Agent unavailability.

**Resolution (Approved — Option B: Code-Hash-Bound Signing Key):**

The Lead Architect selected Option B for [002] Substrate / [003] Cleaner. Per-document SGX attestation reports (Option A) are rejected as redundant: the Cleaner VM's binary integrity is already attested during Phase 2 credential issuance (the Policy Agent validates the Cleaner's measured image hash against the KGM before issuing mTLS credentials), making per-document re-attestation a re-proof of an already-established guarantee at 10–50ms additional cost per document. Option C is rejected as architecturally regressive: routing every ingested document through the Policy Agent's hybrid gate (70–230ms per document) converts the Policy Agent into an ingestion bottleneck, consumes NPU Priority 0 inference cycles on content validation rather than tool-call adjudication, and reintroduces probabilistic dependency at the ingestion boundary.

**Option B Selected — Mechanism:**
- **Cleaner Signing Key:** Derived from the Policy Agent's CA chain during Phase 2; sealed to the Cleaner VM's measured image hash via SGX/TDX sealing. If the Cleaner binary is modified, the sealed key becomes inaccessible at the next boot, Phase 2 credential issuance fails, and the ingestion pipeline halts (Fail-Closed). Key material never exists outside the TEE-attested Cleaner VM.
- **Signature Tuple (Three-Field):** Each processed document is signed over `(document_content_hash, sanitization_pipeline_code_hash, timestamp)`. The `sanitization_pipeline_code_hash` is computed once at Cleaner VM boot from the sanitization pipeline binary, cached for the session, and carried in every per-document signature for that boot cycle.
- **Substrate Validation:** The Substrate's ingestion handler validates all three fields in strict order — (a) signature verification against the Cleaner's CA-issued public key, (b) document_content_hash match, (c) `sanitization_pipeline_code_hash` match against the KGM-propagated known-good value, (d) timestamp recency check (>60 seconds rejected). Fail-Closed: any check failure rejects the document with no fallback ingestion path.
- **Offline Resilience:** Cleaner queues signed documents locally if Policy Agent is temporarily unavailable. Substrate accepts documents using the boot-time-propagated KGM pipeline hash during Policy Agent absence. Queue flushed on Policy Agent recovery.
- **Signing Latency:** 5–25ms per document (negligible against sanitization pipeline and 100 docs/hour throughput target).
- **Residual Risk (Bounded and Documented):** A runtime in-memory code injection into the Cleaner VM that modifies execution flow post-boot without altering the on-disk binary could bypass sanitization while retaining the valid signing key and cached pipeline code hash. This attack requires breaching SGX/TDX memory isolation, equivalent in difficulty to compromising any TEE-attested agent in the system. Accepted as consistent with the system-wide TEE trust boundary assumption.
